## 4.1 指令级并行（ILP）基本概念

### 4.1.1 定义

**指令级并行（ILP）**：指令之间存在的潜在并行性。

$$IPC = \frac{1}{CPI}$$

$$CPI_{实际} = CPI_{理想} + Stall_{结构} + Stall_{数据} + Stall_{控制}$$

本章的核心目标：**如何减少各类Stall，提高IPC**。

### 4.1.2 基本块

**基本块（Basic Block）**：一段连续的代码，除了入口以外没有其他转入，除了出口以外没有其他转出。

- 基本块内的指令数很少（平均4~7条）
- 要挖掘更多的ILP，必须**跨越基本块**——即跨越分支边界

### 4.1.3 循环级并行

**循环级并行（Loop-Level Parallelism）**：不同循环迭代之间存在的并行性。

实现方式：
1. **循环展开**（Loop Unrolling）
2. **向量指令/数据**（Vectorization）

### 4.1.4 必须保持的两个关键属性

为了正确处理异常和维护程序语义：

1. **保持数据流**（Maintain Data Flow）：指令之间的真正数据传输关系不能改变
2. **保持异常行为**（Maintain Exception Behavior）：
   - **精确异常**：异常发生时保存的机器状态与顺序执行完全一致
   - **不精确异常**：由于乱序执行，异常时的状态可能不同于顺序执行
   - **弱化要求**：指令的重排序不能导致新的异常出现

3. **控制相关不是必须保持的**：如果可以确保不影响正确性，可以打破控制相关来并行执行

## 4.2 静态调度 vs 动态调度

| | 静态调度 (Static) | 动态调度 (Dynamic) |
|---|---|---|
| **调度者** | 编译器（编译时） | 硬件（运行时） |
| **优点** | 不增加硬件复杂度 | 可处理编译时未知的依赖；代码可移植性好 |
| **缺点** | 编译器不了解实际运行情况 | 硬件复杂度大幅增加 |
| **现实** | 两者往往结合使用 | |

**动态调度的核心思想**：将ID段拆分为：
1. **流出段（Issue）**：译码 + 检查结构冲突
2. **读操作数段（Read Operands）**：等待数据冲突清除 → 读操作数

这样可以在发生数据冲突时让后面的指令先执行（**乱序执行**）。

## 4.3 Tomasulo算法（考纲大题#8 — 最核心内容！）

### 4.3.1 核心思想

Tomasulo算法是IBM 360/91的浮点部件中使用的动态调度算法，有两大核心创新：

1. **操作数一旦就绪就立即执行**——最小化RAW导致的停顿
2. **通过保留站实现寄存器换名**——消除WAR和WAW冲突

### 4.3.2 与记分牌算法的区别

| 记分牌 | Tomasulo |
|--------|----------|
| 集中式冲突检测 | **分布式**冲突检测和执行控制 |
| 结果经过寄存器传递 | 结果通过**CDB**（公共数据总线）直接广播 |
| 乱序执行完成后写寄存器 | 通过保留站换名 |
| 有WAR和WAW停顿 | **消除**了WAR和WAW停顿 |

### 4.3.3 硬件结构

```
        指令队列
           ↓
    ┌──────────────┐
    │   流出段(Issue)  │ ← 取指、译码、分配保留站
    └──────────────┘
           ↓
    ┌──────────────────────────────────┐
    │         保留站 (Reservation Stations) │
    │  ┌──────┐ ┌──────┐ ┌──────┐       │
    │  │ ADD1 │ │ ADD2 │ │ ADD3 │       │ ← 浮点加法保留站(3个)
    │  └──────┘ └──────┘ └──────┘       │
    │  ┌──────┐ ┌──────┐                │
    │  │MULT1 │ │MULT2 │                │ ← 浮点乘法保留站(2个)
    │  └──────┘ └──────┘                │
    │  ┌──────┐ ┌──────┐                │
    │  │Load1 │ │Store1│                │ ← 存取缓冲器
    │  └──────┘ └──────┘                │
    └──────────────────────────────────┘
           ↓                    ↑
    ┌──────────┐         ┌──────────┐
    │  浮点加法器  │         │    CDB    │ ← 公共数据总线
    └──────────┘         └──────────┘
           ↓                    ↑
    ┌──────────┐               │
    │  浮点乘法器  │───────────────┘
    └──────────┘
```

#### 保留站（Reservation Station）字段：

| 字段 | 含义 | 说明 |
|------|------|------|
| **Op** | 操作类型 | ADD/SUB/MULT/DIV/L.D/S.D |
| **Qj** | 产生第一个源操作数的保留站号 | 0表示操作数已就绪 |
| **Qk** | 产生第二个源操作数的保留站号 | 0表示操作数已就绪 |
| **Vj** | 第一个源操作数的值 | 当Qj=0时有效 |
| **Vk** | 第二个源操作数的值 | 当Qk=0时有效 |
| **Busy** | 该保留站是否被占用 | yes / no |
| **A** | （仅Load/Store）有效地址 | 先存立即数，后存有效地址 |

#### 寄存器状态表（Register Status / Qi）：

为每个浮点寄存器记录：**哪个保留站将把结果写入该寄存器**。

- Qi = 0：表示该寄存器的内容就是当前值，数据已就绪
- Qi = 保留站名：表示该寄存器正在等待某个保留站的结果

#### 公共数据总线 CDB（Common Data Bus）：

- 连接所有功能部件和保留站
- 当某个功能部件完成计算后，将结果**广播**到CDB上
- 所有保留站**同时监听**CDB
- 如果某个保留站的Qj或Qk等于正在广播的保留站号，就抓取数据

### 4.3.4 三段执行流程

#### 阶段1：流出（Issue）

从指令队列头部取指令：
- **条件**：有空闲的保留站 → 否则结构冲突，暂停
- **操作**：
  1. 将指令发送到空保留站
  2. 若操作数在寄存器中就绪（Qi=0）→ 放入Vj/Vk
  3. 若操作数不在寄存器中（Qi≠0）→ 将Qi值记入Qj/Qk（记录产生该操作数的保留站号）
  4. 将本保留站号写入目标寄存器的Qi字段（**换名！消除了WAW**）

#### 阶段2：执行（Execute）

- **条件**：Qj=0 且 Qk=0（两个操作数都就绪）
- **操作**：执行运算
- Load/Store：先计算有效地址，然后读/写存储器

#### 阶段3：写结果（Write Result）

- **条件**：执行完成
- **操作**：
  1. 将结果放到CDB上广播
  2. 所有保留站检查：若Qj或Qk等于本保留站号 → 抓取数据，清除对应的Q
  3. 所有寄存器检查：若Qi等于本保留站号 → 写入数据，Qi清零
  4. 释放本保留站（Busy ← No）

### 4.3.5 完整例题走通（重中之重！）

**这是考纲大题#8最可能出现的题型，务必反复练习！**

#### 题目

给定以下MIPS浮点指令序列，各功能部件的执行延迟（时钟周期数）为：
- Load/Store：1个周期
- 浮点加法/减法：2个周期
- 浮点乘法：10个周期
- 浮点除法：40个周期

```asm
L.D   F6, 34(R2)       ; 1.
L.D   F2, 45(R3)       ; 2.
MUL.D F0, F2, F4       ; 3.
SUB.D F8, F6, F2       ; 4.
DIV.D F10, F0, F6      ; 5.
ADD.D F6, F8, F2       ; 6.
```

#### 硬件配置

- 3个浮点加法保留站：ADD1、ADD2、ADD3
- 2个浮点乘法保留站：MULT1、MULT2
- 3个Load Buffer：Load1、Load2、Load3
- 3个Store Buffer

#### 逐周期跟踪

**初始状态**：所有保留站空闲，所有寄存器Qi=0

---

**周期1**：流出指令1 `L.D F6, 34(R2)`

- Load1被分配
- Load1.A = 34 + (R2的值)
- Qi[F6] = Load1（表示F6等待Load1的结果）

**状态**：

| 保留站 | Busy | Op | Qj | Qk | Vj | Vk | A |
|--------|------|-----|----|-----|----|----|---|
| Load1 | Yes | L.D | | | | | 34 + (R2) |
| Load2 | No | | | | | | |

| 寄存器 | F0 | F2 | F4 | F6 | F8 | F10 |
|--------|-----|-----|-----|------|------|------|
| Qi | 0 | 0 | 0 | **Load1** | 0 | 0 |

---

**周期2**：流出指令2 `L.D F2, 45(R3)`

- Load2被分配
- Load2.A = 45 + (R3的值)
- Qi[F2] = Load2

**状态**：

| 保留站 | Busy | Op | Vj/Vk | A |
|--------|------|-----|----|----|
| Load1 | Yes | L.D | | 34+(R2) |
| Load2 | Yes | L.D | | 45+(R3) |

| 寄存器 | F0 | F2 | F4 | F6 | F8 | F10 |
|--------|-----|-----|-----|------|------|------|
| Qi | 0 | **Load2** | 0 | Load1 | 0 | 0 |

---

**周期3**：Load1执行完成，指令3 `MUL.D F0, F2, F4` 流出

- Load1写结果：F6得到值M1，Qi[F6]=0
- 流出MUL.D到MULT1：Qj=Qi[F2]=Load2（F2未就绪），Qk=Qi[F4]=0（F4就绪），Vk=F4的值
- Qi[F0] = MULT1

**状态**：

| 保留站 | Busy | Op | Qj | Qk | Vj | Vk | A |
|--------|------|-----|----|-----|----|----|---|
| Load1 | **No** | | | | | | |
| Load2 | Yes | L.D | | | | | 45+(R3) |
| MULT1 | Yes | MUL.D | **Load2** | 0 | | F4值 | |
| MULT2 | No | | | | | | |

| 寄存器 | F0 | F2 | F4 | F6 | F8 | F10 |
|--------|-----|-----|-----|------|------|------|
| Qi | **MULT1** | Load2 | 0 | **0** | 0 | 0 |

---

**周期4**：Load2执行完成，指令4 `SUB.D F8, F6, F2` 流出

- Load2写结果：F2得到值M2，Qi[F2]=0（Load2广播到CDB，MULT1监听并抓取Vj=F2值，清除Qj）
- 流出SUB.D到ADD1：Qj=Qi[F6]=0（F6就绪，Vj=F6值=M1），Qk=Qi[F2]=0（F2刚被Load2更新，就绪），Vk=M2
- Qi[F8] = ADD1
- MULT1的Qj被清除（Load2写回了F2），Vj获取到M2

**状态**：

| 保留站 | Busy | Op | Qj | Qk | Vj | Vk |
|--------|------|-----|----|-----|----|----|
| Load2 | **No** | | | | | |
| MULT1 | Yes | MUL.D | **0** | 0 | **M2值** | F4值 |
| ADD1 | Yes | SUB.D | 0 | 0 | M1值 | M2值 |

| 寄存器 | F0 | F2 | F4 | F6 | F8 | F10 |
|--------|-----|-----|-----|------|------|------|
| Qi | MULT1 | 0 | 0 | 0 | **ADD1** | 0 |

---

**周期5**：ADD1开始执行（两个操作数都就绪），指令5 `DIV.D F10, F0, F6` 流出

- ADD1的SUB操作需要2个周期，从周期5开始执行
- 流出DIV.D到MULT2（除法使用乘法部件）：Qj=Qi[F0]=MULT1，Qk=Qi[F6]=0，Vk=M1
- Qi[F10] = MULT2

---

**周期6**：ADD1完成SUB.D，写结果；指令6 `ADD.D F6, F8, F2` 流出

- ADD1写结果：CDB广播F8值 = M1 - M2，Qi[F8]=0
- 流出ADD.D到ADD2：Qj=Qi[F8]=0（刚被更新，Vj=M1-M2），Qk=Qi[F2]=0，Vk=M2
- Qi[F6] = ADD2（**这实现了换名！F6原来与Load1相关，现在用ADD2换名——消除了WAW**）

---

**周期7**：ADD2开始执行

- Qj=Qk=0，可直接执行

---

**周期8**：ADD2完成ADD.D，写结果

- CDB广播F6值
- 注意：指令6的ADD.D往F6写了一个新值，这是通过换名实现的——指令1写的F6和指令6写的F6用不同的保留站来区分

---

**周期13**（假设）：MULT1完成MUL.D（乘法需要10周期）

- CDB广播F0 = M2 × F4
- MULT2监听：Qj被清除，Vj获得乘法结果
- Qi[F0] = 0

---

**周期53**（假设）：MULT2完成DIV.D（除法需要40周期）

- CDB广播F10值
- Qi[F10] = 0

### 4.3.6 Tomasulo如何处理各类冲突

| 冲突类型 | Tomasulo的处理方式 |
|---------|-------------------|
| **RAW** | 通过Qj/Qk等待产生操作数的保留站结果（CDB广播） |
| **WAR** | 流出时就读取操作数放入Vj/Vk（或记录Q），**寄存器换名消除了WAR** |
| **WAW** | 最后一次写操作的保留站号覆盖之前的值在Qi中 → **寄存器换名消除了WAW** |
| **结构冲突** | 保留站全忙时暂停流出 |

### 4.3.7 Tomasulo的优缺点

**优点**：
1. 分布式冲突检测逻辑，各保留站独立判断
2. CDB广播结果给所有等待的指令，并行度高
3. 消除了WAR和WAW导致的停顿

**缺点**：
1. 硬件复杂度高（大量关联比较器）
2. 不精确异常（乱序完成，异常时状态与程序顺序不一致）
3. CDB是瓶颈（多发射处理器需要多条CDB）

## 4.4 动态分支预测

### 4.4.1 与静态分支预测的区别

| 静态方法（第3章） | 动态方法（第4章） |
|-------------------|-------------------|
| 编译器决定预测策略 | 硬件运行时根据历史记录预测 |
| 预测策略固定 | 可根据实际执行情况调整 |

### 4.4.2 分支历史表 BHT（Branch History Table）

**基本思想**：记录分支指令最近是否成功的信息。

**两位分支预测（最常见）**：

```
        预测成功          预测成功
    ┌──────────┐    ┌──────────┐
    │  强预测  │◄───│  弱预测  │
    │  成功    │    │  成功    │
    └──────────┘    └──────────┘
         │               │
    实际失败         实际失败
         │               │
         ▼               ▼
    ┌──────────┐    ┌──────────┐
    │  强预测  │───▶│  弱预测  │
    │  失败    │    │  失败    │
    └──────────┘    └──────────┘
         ▲               ▲
    实际成功         实际成功
```

- 一个分支连续两次预测错误才会改变预测方向
- 2位以上的预测器收益递减

### 4.4.3 分支目标缓冲器 BTB（Branch Target Buffer）

**BTB的结构**：一个小型Cache，存储：
- 分支指令的地址（作为Tag）
- 预测的分支目标地址

**工作过程**：
- 在IF段用PC查找BTB
- 若命中且该指令为分支指令 → 从预测的目标地址取指
- 若未命中 → 按正常顺序执行

**延迟情况**：

| 在BTB中？ | 预测 | 实际 | 延迟周期 |
|-----------|------|------|---------|
| 是 | 成功 | 成功 | 0 |
| 是 | 成功 | 失败 | 2 |
| 否 | — | 成功 | 2 |
| 否 | — | 失败 | 0 |

### 4.4.4 四种动态预测技术

1. **1位BHT**：只记录上次结果
2. **2位BHT**：记录两次结果（最常用）
3. **相关分支预测器**：利用不同分支之间的相关性
4. **BTB**：直接提供目标地址

### 4.4.5 相关分支预测器（特别是(2,2)预测器）

**(m,n)相关预测器**：
- $m$：最近m个分支的行为模式
- $n$：使用n位预测器
- **(2,2)预测器**：根据最近2个分支的历史（共4种模式：00,01,10,11），每种模式使用2位预测器

**(2,2)预测器容量计算**：

$$容量 = 2^m \times 2^n \text{位} = 2^2 \times 2^2 \text{位} = 4 \times 4 \text{位} = \text{16位}$$

对于有k个表项的BHT：

$$容量 = k \times 2^m \times 2^n \text{位}$$

## 4.5 前瞻执行（Speculative Execution）与ROB

### 4.5.1 核心思想

在前瞻执行中，不仅指令的**执行**可以乱序，**确认**（写寄存器或存储器）也需要按程序顺序进行。

- Tomasulo算法：乱序执行 + 乱序写回 → **不精确异常**
- Tomasulo + ROB：乱序执行 + **顺序确认** → **精确异常**

### 4.5.2 ROB（Reorder Buffer，重排序缓冲器）

**ROB每条记录的字段**：

| 字段 | 含义 |
|------|------|
| **指令类型** | 分支 / Store / 寄存器操作 |
| **目的地址** | 寄存器号（操作指令）或存储器地址（Store指令） |
| **数据值** | 指令执行的结果 |
| **就绪标志** | 是否已执行完成 |

### 4.5.3 四段执行流程

#### 阶段1：流出（Issue）

- **条件**：有空闲保留站**且**有空闲ROB表项
- 在ROB中分配一项，记录指令信息

#### 阶段2：执行（Execute）

- 与Tomasulo相同：等待操作数就绪后执行

#### 阶段3：写结果（Write Result）

- 将结果写入ROB对应表项和CDB广播（给其他等待的保留站）
- **但此时不写寄存器！**

#### 阶段4：确认/提交（Commit）

- **条件**：指令到达ROB头部
- 若不是分支/Store：将结果写入寄存器，从ROB中移除
- 若是Store：将数据写入存储器
- 若是分支，且预测错误：**清空ROB中所有该分支后的指令**，从正确方向重新取指

### 4.5.4 前瞻执行如何解决Tomasulo的不精确异常

在Tomasulo中，ADD可以在DIV之前完成并写寄存器 → 如果DIV发生除零异常，F0已被ADD修改

在ROB中：
- ADD即使先执行完，结果也只写入ROB
- DIV如果发生异常，ADD还没有确认（COMMIT），可以安全丢弃
- **实现了精确异常**

### 4.5.5 ROB如何做到"乱序执行、顺序确认"

```
指令1 (DIV, 40周期)   ← 先流出，后完成
指令2 (ADD, 2周期)    ← 后流出，先完成
指令3 (SUB, 2周期)    ← 后流出，先完成
```

- ADD和SUB执行完成后数据写入ROB
- 但ADD和SUB必须等待DIV确认后才能确认 → 顺序确认
- 虽然ADD/SUB先执行完，但结果暂存ROB，不写寄存器

## 4.6 循环优化技术（考纲大题#9）

### 4.6.1 本节的流水线延迟约定

| 产生结果的指令 | 使用结果的指令 | 延迟（周期数） |
|-------------|-------------|------------|
| 浮点计算 | 另一个浮点计算 | 3 |
| 浮点计算 | 浮点Store(S.D) | 2 |
| 浮点Load(L.D) | 浮点计算 | 1 |
| 浮点Load(L.D) | 浮点Store(S.D) | 0 |
| 整数运算 | 整数运算 | 0（通过转发） |

### 4.6.2 指令调度

**目标**：通过重排指令顺序来消除流水线停顿。

**例**：对 `x[i] = x[i] + s` 循环体进行调度

**原始代码**（单次迭代）：

```asm
L.D    F0, 0(R1)       ; 取x[i]
ADD.D  F4, F0, F2      ; x[i] + s
S.D    F4, 0(R1)       ; 存回x[i]
DADDIU R1, R1, #-8     ; R1指向下一个元素
BNE    R1, R2, Loop    ; 循环结束判断
```

**调度前用时**（考虑延迟）：
- L.D → ADD.D：延迟1周期 → 需暂停1周期
- ADD.D → S.D：延迟2周期 → 需暂停2周期
- 总共10周期/次迭代

**调度后**（将DADDIU提前）：

```asm
L.D    F0, 0(R1)       ; 取x[i]
DADDIU R1, R1, #-8     ; 提前做，不影响任何依赖
ADD.D  F4, F0, F2      ; x[i] + s
BNE    R1, R2, Loop    ; 提前判断
S.D    F4, 8(R1)       ; 存回，注意偏移量需调整
```

**调度后用时**：6周期/次迭代（减少了4个暂停周期）

### 4.6.3 循环展开

**基本思想**：将循环体复制多份，减少循环开销（分支、递减等），同时暴露更多指令级并行。

**展开规则**：
1. 确保循环边界正确（处理不能整除的情况）
2. 不同迭代使用**不同的寄存器**（寄存器重命名，消除名相关）
3. 消除冗余的测试和分支
4. 重新调度展开后的代码以消除停顿

**例**：将上例循环展开3次（即每次处理4个元素）

展开后的寄存器使用（消除依赖）：
- 原迭代0：F0, F4
- 迭代1：F6, F8
- 迭代2：F10, F12
- 迭代3：F14, F16

```asm
L.D    F0, 0(R1)
ADD.D  F4, F0, F2       ; 迭代0
S.D    F4, 0(R1)

L.D    F6, -8(R1)
ADD.D  F8, F6, F2       ; 迭代1
S.D    F8, -8(R1)

L.D    F10, -16(R1)
ADD.D  F12, F10, F2     ; 迭代2
S.D    F12, -16(R1)

L.D    F14, -24(R1)
ADD.D  F14, F14, F2     ; 迭代3
S.D    F14, -24(R1)

DADDIU R1, R1, #-32
BNE    R1, R2, Loop
```

进一步调度后可在14周期内完成4个元素 = **3.5周期/元素**（原始10周期/元素）

### 4.6.4 循环展开的限制

1. **代码膨胀**：展开N倍，代码也变为N倍
2. **寄存器压力**：需要更多寄存器存放不同迭代的数据
3. **分支预测精度**：展开过多可能降低预测准确性

### 4.6.5 软件流水（Software Pipelining）

**基本思想**：重叠不同迭代的操作，使流水线持续充满。

```
展开前（时间→）：
Iter1: L.D ADD S.D
Iter2:           L.D ADD S.D
Iter3:                     L.D ADD S.D

软件流水后（时间→）：
Prologue:  L.D1
           L.D2 ADD1
Kernel:    L.D3 ADD2 S.D1     ← 主体：每个周期同时有L.D/ADD/S.D
           L.D4 ADD3 S.D2
Epilogue:       ADD4 S.D3
                     S.D4
```

## 4.7 向量处理机与链接技术（考纲大题#10）

### 4.7.1 向量处理方式

| 方式 | 描述 | 特点 |
|------|------|------|
| **水平处理**（横向） | 逐个元素完成所有操作 | 不适合向量化 |
| **垂直处理**（纵向） | 对所有元素逐一完成操作 | 存储器-存储器结构 |
| **分组处理**（纵横） | 向量分段后混合处理 | 寄存器-寄存器结构（CRAY-1） |

### 4.7.2 CRAY-1向量功能部件

| 部件 | 流水深度（启动时间） |
|------|---------------------|
| 整数加法 | 3 |
| 逻辑运算 | 2 |
| 移位 | 4 |
| 浮点加法 | 6 |
| 浮点乘法 | 7 |
| 浮点求倒数 | 14 |

### 4.7.3 向量链接（Chaining）

**链接**：当两个向量指令存在RAW相关时，将两个功能部件链接起来形成一条流水线。

**效果**：第二个指令无需等待第一个指令的全部元素完成，当第一个结果出来就可以直接送入第二个流水线。

### 4.7.4 向量执行时间计算

$$T = T_{startup} + N$$

其中 $T_{startup}$ 为启动时间（第一条结果出来所需的时间）。

**链接时的启动时间** = 各链接部件的启动时间之和 + 1（链接操作）

**例**：D = A × (B + C)，向量长度 N=64

不使用链接（串行）：
- B+C（浮点加法）：$T_1 = 6 + 64 = 70$
- A×(B+C)（浮点乘法）：$T_2 = 7 + 64 = 71$
- 总时间 = $70 + 71 = 141$ 周期

使用链接（并行第一条，与第二条链接）：
- 加法启动后立即开始乘法链接
- 链接启动时间 = (6+7) + 1 = 14
- 总时间 = $14 + 64 = 78$ 周期

### 4.7.5 向量处理中的相关与冲突

| 类型 | 说明 | 处理 |
|------|------|------|
| **Vi冲突** | 两条并行向量指令使用相同Vi | 串行执行或换名 |
| **功能部件冲突** | 两条并行向量指令使用相同功能部件 | 串行执行 |

### 4.7.6 判断能否使用链接的条件

两条向量指令之间如果满足以下条件就可以链接：
1. 存在RAW数据相关（前一条写某个V寄存器，后一条读同一个V寄存器）
2. 没有Vi冲突
3. 没有功能部件冲突

### 4.7.7 向量处理机性能评价参数

**第一个参数（最重要的）**：向量指令执行时间 $T = T_{startup} + N$

- $T_{startup}$：向量流水线的填充时间（即第一条结果出现的周期数）
- $N$：向量长度

另外的参数包括：
- $R_{\infty}$（峰值性能）：当 $N \to \infty$ 时，每个时钟周期产生的结果数
- $N_{1/2}$（半性能长度）：达到一半峰值性能所需的向量长度

## 4.8 多流出处理器（仅需了解概念）

> 考纲说明：4.4.1-4.4.5仅需了解基本思路。

| 类型 | 英文 | 指令流出 | 调度 | 实例 |
|------|------|---------|------|------|
| **超标量** | Superscalar | 每周期多条（可变） | 硬件动态 | Pentium, MIPS R10K |
| **超长指令字** | VLIW | 每周期多条（固定） | 编译器静态 | Trimedia, i860 |
| **超流水** | Superpipeline | 流水段细分更深 | 硬件 | MIPS R4000 |
| **超标量+超流水** | Combined | 两者结合 | | Pentium 4 |

**超标量**：通过设置多个功能部件同时流出多条指令（IPC > 1）

**VLIW**：编译器将多条可并行指令打包成一个长指令字，硬件简单
