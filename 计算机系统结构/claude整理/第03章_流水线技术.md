## 3.1 流水线的基本概念

### 3.1.1 定义

**流水线技术**：将一个重复的时序过程分解为若干个子过程，每个子过程由专用的功能部件完成，多个过程在时间上**错开**，流水般通过各功能段。

**核心本质**：时间重叠（Time Overlapping）——各子过程在时间上并行执行。

### 3.1.2 流水线的特点

1. **协同性**：各段由专用功能部件实现，分工协作
2. **瓶颈性**：各段时间应尽量相等，否则最慢的段成为瓶颈
3. **流水寄存器**：段间必须设置流水寄存器（锁存器），隔离各段
4. **适用性**：适合大量重复的时序过程，需要连续输入任务
5. **通过时间 + 排空时间**：第一条指令从进入到出来需要一定时间(drain)，最后一条指令从进入到出来也需要时间(flush)

### 3.1.3 与多周期的关系

> 流水一定是多周期的，但多周期不一定是流水的。

- 流水线将一个周期的工作分解到多个段中实现，因此**单个指令的执行时间反而增加**（流水寄存器开销）
- 但流水线的**吞吐率大幅提高**——它可以同时处理多条指令

## 3.2 流水线的分类

| 分类维度 | 类型 | 说明 |
|---------|------|------|
| **按功能** | 单功能流水线 | 只能实现一种固定功能 |
|  | 多功能流水线 | 可实现多种功能（通过不同连接方式） |
| **按连接方式** | 静态流水线 | 同一时间各段只能执行同一种功能 |
|  | 动态流水线 | 同一时间各段可以执行不同功能 |
| **按级别** | 部件级（运算级） | 浮点加法器等 |
|  | 处理机级（指令级） | 指令流水线 |
|  | 处理机间级（宏流水线） | 多个处理机流水处理 |
| **按反馈** | 线性流水线 | 无反馈回路，各段只用一次 |
|  | 非线性流水线 | 有反馈回路，某段可能被多次使用 |
| **按顺序** | 顺序流水线 | 任务流出顺序与流入顺序相同（FIFO） |
|  | 乱序流水线 | 后续任务可能先于前序任务流出 |

## 3.3 流水线性能指标（大题计算核心）

### 3.3.1 基本参数

- $k$：流水线的段数（深度）
- $n$：任务数
- $\Delta t$：时钟周期（各段时间相等时）
- $T_k$：流水线处理n个任务的总时间

### 3.3.2 各段时间相等的情况

#### 时钟周期与总时间

$$\Delta t = \text{各段时间（相等）}$$

$$T_k = k\Delta t + (n-1)\Delta t = (k+n-1)\Delta t$$

**理解方法（时空图）**：
- 第一条任务需要 $k\Delta t$ 时间通过流水线
- 之后每隔 $\Delta t$，就有一条任务完成
- 所以 $n$ 个任务 = 第一个任务的通过时间 + 剩余 $n-1$ 个任务各 $\Delta t$

#### 吞吐率 (Throughput)

$$TP = \frac{n}{T_k} = \frac{n}{(k+n-1)\Delta t}$$

$$TP_{max} = \frac{1}{\Delta t}$$

当 $n \gg k$ 时，$TP \approx TP_{max}$。

#### 加速比 (Speedup)

$$S = \frac{T_{顺序}}{T_{流水}} = \frac{n \cdot k \cdot \Delta t}{(k+n-1)\Delta t} = \frac{nk}{k+n-1}$$

$$S_{max} = k \quad (\text{当 } n \to \infty)$$

#### 效率 (Efficiency)

$$E = \frac{n\text{个任务实际占用的时空区}}{k\text{个段的总时空区}} = \frac{n \cdot k \cdot \Delta t}{k \cdot (k+n-1)\Delta t} = \frac{n}{k+n-1}$$

$$E = \frac{S}{k} = TP \cdot \Delta t$$

### 3.3.3 各段时间不相等的情况

设各段时间为 $\Delta t_1, \Delta t_2, ..., \Delta t_k$。

**时钟周期** = $\max\{\Delta t_i\}$（瓶颈段时间决定了时钟周期）

**瓶颈问题解决方案**：
1. **细分瓶颈段**：将瓶颈段再分解为多个更小的子段
2. **重复设置瓶颈段**：设置多套瓶颈功能段，交替工作

### 3.3.4 时空图绘制方法

时空图是流水线分析的核心工具。以横轴为时间，纵轴为空间（各流水段）。

**例**：$k=4$ 段流水线，各段时间均为 $\Delta t$，$n=5$ 个任务。

```
空间
S4 │     T1  T2  T3  T4  T5
S3 │   T1  T2  T3  T4  T5
S2 │ T1  T2  T3  T4  T5
S1 │T1  T2  T3  T4  T5
   └─────────────────────────▶ 时间
   0   Δt  2Δt 3Δt 4Δt 5Δt 6Δt 7Δt 8Δt
```

- 每一个 $\Delta t$ 为一个时钟周期
- T1在 $4\Delta t$ 时完成，T5在 $8\Delta t$ 时完成
- $T_k = (4+5-1)\Delta t = 8\Delta t$

### 3.3.5 计算例题

**例1**：4段流水线，各段时间均为2ns。问：
1. 连续执行1000条指令的吞吐率
2. 加速比
3. 效率

**解**：

$k=4, \Delta t=2\text{ns}, n=1000$

$T_k = (4+1000-1) \times 2 = 2006\text{ns}$

$TP = \frac{1000}{2006} \approx 0.499 \text{条/ns}$

$S = \frac{1000 \times 4}{4+1000-1} = \frac{4000}{1003} \approx 3.99$

$E = \frac{1000}{4+1000-1} = \frac{1000}{1003} \approx 0.997$

$TP_{max} = \frac{1}{2\text{ns}} = 0.5 \text{条/ns}$

---

**例2**：有瓶颈段的情况

4段流水线，各段时间：S1=50ns, S2=50ns, S3=100ns, S4=50ns。求最大吞吐率。

**解**：瓶颈段S3=100ns

$TP_{max} = \frac{1}{100\text{ns}} = 10\text{M条/秒}$

若将S3细分为两个50ns的子段，则 $k=5$，各段均匀50ns：

$TP_{max} = \frac{1}{50\text{ns}} = 20\text{M条/秒}$（翻倍）

## 3.4 经典5段RISC流水线（MIPS）

### 3.4.1 五段功能

| 段 | 名称 | 操作 |
|----|------|------|
| **IF** | Instruction Fetch（取指） | IR ← Mem[PC]; PC ← PC + 4 |
| **ID** | Instruction Decode（译码/读寄存器） | A ← Regs[rs]; B ← Regs[rt]; Imm ← sign_extend(IR[16:31]) |
| **EX** | Execute（执行/地址计算） | 访存：ALUo ← A + Imm；RR型：ALUo ← A func B；分支：ALUo ← NPC + (Imm<<2); cond ← (A==0) |
| **MEM** | Memory Access（存储器访问/分支完成） | load：LMD ← Mem[ALUo]；store：Mem[ALUo] ← B；分支成功：PC ← ALUo |
| **WB** | Write Back（写回） | ALU结果：Regs[rd] ← ALUo；RI型：Regs[rt] ← ALUo；load：Regs[rt] ← LMD |

**各指令所需周期**：
- ALU指令（RR/RI型）：5个周期（经过所有5段）
- Load指令：5个周期
- Store指令：4个周期（不需要WB段）
- 分支指令：4个周期（不需要WB段）

### 3.4.2 流水寄存器

4个流水寄存器将5段流水线隔开：

| 寄存器 | 位于 |
|--------|------|
| **IF/ID** | IF段和ID段之间 |
| **ID/EX** | ID段和EX段之间 |
| **EX/MEM** | EX段和MEM段之间 |
| **MEM/WB** | MEM段和WB段之间 |

**流水寄存器的作用**：
1. 隔离各流水段
2. 保存本段的执行结果
3. 向后传递数据和控制信息

命名规则：`段名.寄存器名[字段]`，如 `ID/EX.IR[op]` 表示ID/EX寄存器中IR的操作码字段。

### 3.4.3 流水线中的结构冲突预防

| 冲突 | 原因 | 解决方法 |
|------|------|---------|
| IF和MEM同时访存 | 冯诺依曼结构共享存储器 | **分离指令Cache和数据Cache** |
| ID读寄存器和WB写寄存器同时 | 同一寄存器端口 | **前半周期写、后半周期读**，或使用双端口寄存器 |

## 3.5 流水线的相关与冲突（大题核心#7）

### 3.5.1 相关 vs 冲突

| | 相关 (Dependence) | 冲突 (Hazard) |
|---|---|---|
| **定义** | 指令间存在的一种**依赖关系** | 流水线执行中可能发生的**现象** |
| **性质** | 程序固有的属性 | 流水线实现中的属性 |
| **关系** | 相关是冲突的**必要但不充分条件** | 有相关不一定产生冲突，还要看流水线实现 |

> **核心理解**：存在相关 ≠ 一定产生冲突。有相关只是存在冲突的可能性，是否真正发生冲突取决于具体的流水线实现方式以及指令间的距离。

### 3.5.2 三种相关性

#### (1) 数据相关（True Data Dependence）→ RAW

指令j用到指令i产生的结果，或指令j与指令k数据相关而k与i数据相关（**传递性**）。

```
i: ADD R1, R2, R3    ; R1 ← R2 + R3
j: SUB R4, R1, R5    ; R4 ← R1 - R5    ← j与i数据相关(RAW)
```

**数据相关是真正的数据流动关系，不能消除。**

#### (2) 名相关（Name Dependence）

**反相关（Anti-dependence）→ WAR**：
指令j**写**一个名，指令i**读**同一个名，且j在i之后执行时可能先写。

```
i: ADD R3, R1, R2    ; R3 ← R1 + R2（读R1, 读R2, 写R3）
j: ADD R1, R4, R5    ; R1 ← R4 + R5（读R4, 读R5, 写R1）
                      ← j写R1, i读R1，反相关(WAR)
```

**输出相关（Output Dependence）→ WAW**：
指令j和指令i都**写**同一个名。

```
i: ADD R1, R2, R3    ; 写R1
j: ADD R1, R4, R5    ; 写R1  ← 输出相关(WAW)
```

**名相关的消除**：通过**换名技术**（Register Renaming）——将指令中的寄存器名换成另一个不冲突的名字。

#### (3) 控制相关（Control Dependence）

由**分支指令**引起的相关。分支指令之后的指令是否执行取决于分支指令的结果。

### 3.5.3 三种流水线冲突

#### 结构冲突（Structural Hazard）

**原因**：硬件资源不能满足指令重叠执行的要求。

**解决**：增加硬件资源（如分离I-Cache和D-Cache）；或插入**暂停周期**（stall bubbles）。

> 实际中常常**允许某些结构冲突存在**，以降低硬件成本。

#### 数据冲突（Data Hazard）

| 冲突类型 | 对应相关 | 说明 | 在基本5段流水线中出现？ |
|---------|---------|------|---------------------|
| **RAW**（写后读） | 数据相关 | j读之前i没写完 | **是** |
| **WAR**（读后写） | 反相关 | j写之前i没读 | 否（顺序发射、顺序写回） |
| **WAW**（写后写） | 输出相关 | j写之前i没写 | 否（顺序发射、顺序写回） |

> **经典5段流水线中**：只有RAW冲突可能发生。WAR和WAW只在**乱序执行**的流水线中才可能出现。

**RAW冲突解决方法**：

1. **定向技术（Forwarding/Bypassing）**：将计算结果直接从产生处送到需要处，不等写回寄存器。
   - 可以在EX段开始前就将ALU结果转发给EX段
   - 可以在MEM段后将结果转发给EX段

2. **插入暂停（Stall / Pipeline Interlock）**：当定向也无法解决时（如Load-use相关），硬件插入一个时钟周期的"气泡"。

> Load指令的结果在MEM段结束时才得到，但下一条指令在EX段开始时就需要操作数，所以即使使用定向，也需要1个暂停周期。

3. **编译器指令调度**：编译器重排指令顺序，将有依赖关系的指令拉开距离。

```
调度前（有stall）：
  LD  R1, 0(R2)
  DADD R4, R1, R5   ← RAW，需暂停1周期
  SD  R4, 0(R6)

调度后（无stall）：
  LD  R1, 0(R2)
  LD  R3, 0(R7)     ← 插入独立指令
  DADD R4, R1, R5   ← RAW仍在但解决
  SD  R4, 0(R6)
```

#### 控制冲突（Control Hazard）

**原因**：分支指令或改变PC的指令会导致流水线不确定接下来取哪条指令。

**在5段流水线中**：分支指令在MEM段结束时才能确定是否跳转，此时已有3条后续指令进入流水线（IF、ID、EX段各一条）。

**解决方法**：

**（一）静态方法（软件/编译器）：**

1. **预测分支不成功**（Predict Not Taken）：
   - 沿顺序方向继续取指
   - 若预测正确：0延迟
   - 若预测错误：将已进入的指令转为NOP

2. **预测分支成功**（Predict Taken）：
   - 一旦算出目标地址就从中取指
   - 需要能在ID段就算出目标地址

3. **延迟分支（Delayed Branch）**：
   - 在分支指令后设置**延迟槽**（1个或多个周期）
   - 延迟槽内的指令不管分支是否成功都执行
   - **三种调度策略**：

| 策略 | 方法 | 适用条件 |
|------|------|---------|
| **从前调度**（From Before） | 将分支前的一条独立指令移入延迟槽 | **首选**，总是有效 |
| **从目标处调度**（From Target） | 将分支目标处指令复制到延迟槽 | 分支成功时有效，需要调整目标地址 |
| **从失败处调度**（From Fall-through） | 将不成功路径上的指令移入延迟槽 | 分支不成功时有效 |

4. **分支取消**（Branch Cancellation/Nullifying）：
   - 若预测错误，延迟槽指令变成NOP

**（二）硬件方法（动态，在第4章详述）：**
- 分支历史表 BHT
- 分支目标缓冲器 BTB

## 3.6 经典5段流水线相关性分析例题

### 例1：分析以下MIPS指令序列中的相关性

```asm
1.  DADD  R1, R2, R3       ; R1 ← R2 + R3
2.  DSUB  R4, R1, R5       ; R4 ← R1 - R5
3.  AND   R6, R1, R7       ; R6 ← R1 AND R7
4.  OR    R8, R1, R9       ; R8 ← R1 OR R9
5.  XOR   R10, R1, R11     ; R10 ← R1 XOR R11
```

**解**：
- 指令2与指令1：**RAW**（指令2读R1，指令1写R1）——**数据相关**
- 指令3与指令1：RAW（数据相关）
- 指令4与指令1：RAW（数据相关）
- 指令5与指令1：RAW（数据相关）
- 指令2-5之间：读后读(RAR)，**不相关**

在5段流水线中：指令2在EX段需要R1，但指令1在WB段才写R1 → **需要转发或暂停**

### 例2：分析包含名相关的序列

```asm
1.  DADD  R1, R2, R3       ; R1 ← R2 + R3
2.  DSUB  R2, R4, R5       ; R2 ← R4 - R5（写R2）
3.  OR    R6, R2, R1       ; R6 ← R2 OR R1（读R2, 读R1）
```

**解**：
- 指令2与指令1：名相关（**反相关WAR**——指令2写R2，虽然指令1没有读R2，但如果指令1和指令2之间有其他指令读R2则可能相关）；这里检查：指令1读写R1,R2,R3，指令2写R2 —— 等等，指令1是**读**R2，指令2是**写**R2。但这在5段顺序流水线中：指令1在ID段读R2，指令2要到WB段才写R2，所以不会发生WAR冲突。

在**乱序执行**中才可能发生WAR/WAW冲突。

### 例3：包含控制相关的序列

```asm
      DADD  R1, R2, R3
      BEQZ  R1, Label       ; 若R1==0则跳转
      DSUB  R4, R5, R6      ; ← 控制相关：是否执行取决于BEQZ
Label: OR   R7, R8, R9
```

BEQZ在MEM段才确定是否跳转，DSUB已经进入ID段 → 若不跳转OK，若跳转则DSUB需要被冲刷。

## 3.7 流水线实现（MIPS数据通路）

### 3.7.1 必须掌握的多路选择器（MUX）

5段MIPS流水线数据通路中有5个关键MUX：

| MUX | 位置 | 选择对象 |
|-----|------|---------|
| MUX1（PC） | IF段 | 分支成功→目标地址(ALUo)；否则→PC+4(NPC) |
| MUX2（EX段A输入） | EX段 | 来自ID/EX.A → 一般情况；来自转发 → 定向 |
| MUX3（EX段B输入） | EX段 | 来自ID/EX.B → RR型；来自ID/EX.Imm → 访存/立即数 |
| MUX4（WB写回数据） | WB段 | 来自MEM/WB.ALUo → ALU结果；来自MEM/WB.LMD → Load结果 |
| MUX5（写回寄存器号） | WB段 | rd → RR型ALU；rt → RI型/load |

### 3.7.2 各段寄存器传输操作

详细内容建议对照教材图3.37理解，以下是核心点：

- **IF段**：取指地址来自MUX1（NPC或分支目标）
- **ID段**：读rs、rt，符号扩展立即数，计算分支目标（若为分支指令）
- **EX段**：ALU计算（A func B 或 A+Imm），确定分支条件
- **MEM段**：访存或分支完成
- **WB段**：选择写回数据和目的寄存器号，写入寄存器堆

## 3.8 课后思考题参考答案要点

1. **引入流水线的原因**：提高指令吞吐率，不是减少单条指令时间
2. **流水线深度不是越深越好**：流水寄存器开销、功耗、时钟偏移都增加，单条指令延迟也增加
3. **瓶颈**：最慢的段决定时钟周期；解决：细分或重复
4. **流水寄存器作用**：隔离各段、保存结果、传递数据/控制信息
5. **流水线特点**：协同性、瓶颈性、流水寄存器必要性、适合大量重复任务
6. **三种数据相关**：RAW（真数据相关，有传递性）；WAR（反相关，名相关）；WAW（输出相关，名相关）
7. **名相关消除方法**：寄存器换名（编译时或硬件——如Tomasulo算法）
8. **控制冲突解决**：预测、延迟分支、冲刷流水线
9. **延迟槽调度策略**：从前调度、从目标处调度、从失败处调度
10. **定向技术**：在EX段开始前将上一条指令的ALU结果或MEM结果转发过来
