1. primary index（clustering index）：in a sequentially ordered file，the index whose search key specifies the sequential order of the file.

	The search key of a primary index is usually but not necessarily the primary key.

2. secondary index（nonclustering index）：an index whose search key specifies an order different from the sequential order of the file.

3. dense index: index record appears for every search-key value in the index file.

4. sparse index:contains index records for only some search-key values.

5. 注意:the reason why secondary indices have to be dense:

	If we use sparse index, we can't find all the data by the order of sparse index becausethe orders in two files are different.

6. index的缺点：They impose serious overhead on database modification: whenever a file is updated,every index must be updated.【它们对数据库修改造成了严重的开销：每当更新文件时，都必须更新每个索引。】

	every record access may fetch a new block from disk.

	If index does not fit in memory, access becomes expensive.

	Solution:treat index kept on disk as a sequential file and construct a sparse index on it

# Ch14 索引 Index 总脉络

## 0. 索引到底是干嘛的？

**一句话：索引就是给数据文件建目录，用空间和维护代价换查询速度。**

数据库中的数据文件可能很大，如果没有索引，查一条记录可能要从头扫描到尾。索引保存的是：

> 搜索码值 + 指向记录的指针

其中：

**搜索码 search key**：用于查找记录的属性或属性集。注意，搜索码不一定是主键，一个文件可以有多个搜索码。比如 instructor 表可以按 `ID` 查，也可以按 `dept_name` 查，还可以按 `(dept_name, salary)` 查。学长笔记里也明确说，文件索引的作用是加速查询，搜索码是用于在文件中查找记录的属性或属性集。

**索引项 index entry / index record**：一般由“搜索码值 + 指针”组成。指针通常包含磁盘块号和块内偏移量，用来定位真实记录。

所以你可以把它想成：

```
真实数据文件：[很多很多记录]索引文件：
张三 -> 第 5 块第 2 条
李四 -> 第 9 块第 1 条
王五 -> 第 12 块第 6 条
```

# 1. 索引的两大基本类型：顺序索引 vs 散列索引

学长笔记里把基本索引分成两类：

|类型|核心思想|适合查询|
|---|---|---|
|顺序索引 ordered index / sequential index|按搜索码值的顺序排列|等值查询、范围查询、排序、顺序扫描|
|散列索引 hash index|用散列函数把搜索码值分到桶里|等值查询|

顺序索引是“有序目录”，比如按 ID 从小到大排好。你要查 `ID=10086`，可以二分或树查找；你要查 `ID between 10000 and 20000`，也可以找到开头后顺着往后扫。

散列索引是“按 hash 值进桶”。比如 `h(ID)=3`，就去 3 号桶找。它查 `ID=10086` 很快，但查 `ID between 10000 and 20000` 不方便，因为 hash 之后原来的大小顺序没了。学长笔记里也写到顺序索引基于值的顺序排序，散列索引基于把值平均分到若干散列桶中。

考试写法：

> 顺序索引维护搜索码的有序性，适合范围查询和顺序访问；散列索引通过散列函数将搜索码映射到桶中，适合等值查询，但不适合范围查询。

# 2. 评价一个索引好不好，看五个方面

学长笔记里列了索引评价指标，这块很适合出简答题。

## 2.1 访问类型 access type

看它支持什么查询：

```
where ID = '10086'
```

这是等值查询。

```
where salary between 50000 and 80000
```

这是范围查询。

顺序索引支持等值和范围；散列索引主要支持等值。

## 2.2 访问时间 access time

找到目标记录要多久。比如 B+树高度低，通常几次 I/O 就能找到。

## 2.3 插入时间 insertion time

插入一条新记录时，不仅要插入数据文件，还要更新对应索引。如果有 5 个索引，就可能要维护 5 个索引。

## 2.4 删除时间 deletion time

删除一条记录时，也要从索引中删除对应索引项。

## 2.5 空间开销 space overhead

索引本身也占空间。索引越多，占用空间越多。

这就引出一个高频问题：

> **索引能加快查询，但不是越多越好。**

因为索引会增加存储空间和更新维护成本。

# 3. 主索引 / 聚集索引 vs 辅助索引 / 非聚集索引

这是 2006A 第七题、2020 第六题的重点。

## 3.1 主索引 primary index / 聚集索引 clustering index

定义：

> 如果数据文件本身就是按照某个搜索码的顺序存储的，那么建立在这个搜索码上的索引就是主索引 / 聚集索引。

比如 instructor 文件本身按照 `ID` 从小到大存储：

```
数据文件：ID=101ID=105ID=110ID=120
```

那么在 `ID` 上的索引就是主索引 / 聚集索引。

注意一个容易错的点：

> primary index 的 search key 通常是 primary key，但不一定必须是 primary key。

也就是说，主索引的“主”不是因为它一定建在主键上，而是因为它的搜索码顺序决定了数据文件的物理顺序。

学长笔记里说，如果包含记录的文件是按某个搜索码指定顺序排序的，那么该搜索码对应的索引称为聚集索引 / 主索引。

## 3.2 辅助索引 secondary index / 非聚集索引 nonclustering index

定义：

> 搜索码定义的顺序和数据文件的物理存储顺序不同，这种索引叫辅助索引 / 非聚集索引。

比如数据文件按 `ID` 排序，但你又在 `dept_name` 上建了索引：

```
数据文件物理顺序：按 ID 排辅助索引顺序：按 dept_name 排
```

这时 `dept_name` 索引就是辅助索引。

学长笔记里也明确说，搜索码定义的顺序与文件中记录的物理顺序不同的索引称为非聚集索引 / 辅助索引。

## 3.3 主索引和辅助索引的核心区别

|比较点|主索引 / 聚集索引|辅助索引 / 非聚集索引|
|---|---|---|
|数据文件物理顺序|和搜索码顺序一致|和搜索码顺序不一致|
|一个文件能有几个|通常只能有一个|可以有多个|
|范围扫描效率|高|可能很低|
|能否用稀疏索引|可以|一般不可以，必须稠密|
|查到索引后访问数据|数据通常连续|数据可能到处散落|

为什么一个文件通常只能有一个聚集索引？

因为真实记录在磁盘上只能按一种主要顺序摆放。你不可能让同一份物理文件同时既按 ID 排，又按 name 排，又按 salary 排。

# 4. 辅助索引做顺序扫描为什么效率不高？

这个是 2006A 第七题、2020 第六题经常问的点。

假设数据文件按 `ID` 排，但是有一个辅助索引按 `salary` 排。

你想按 salary 从小到大扫描：

```
salary 索引顺序：
3000 -> 指向数据块 100
4000 -> 指向数据块 7
5000 -> 指向数据块 235
6000 -> 指向数据块 11
```

虽然索引本身是顺序的，但它指向的数据记录在真实文件中是分散的。结果就是：

```
读索引项 1 -> 跳到数据块 100
读索引项 2 -> 跳到数据块 7
读索引项 3 -> 跳到数据块 235
读索引项 4 -> 跳到数据块 11
```

每访问一条记录都可能读一个新的磁盘块，磁盘随机 I/O 很贵。所以：

> 使用聚集索引做顺序扫描效率高，因为数据物理上连续；使用辅助索引做顺序扫描效率低，因为数据记录物理上分散，可能每条记录都要访问新的磁盘块。

你的笔记里那句 “sequential scan using clustering index is efficient, but a sequential scan using a secondary index is expensive on magnetic disk” 就是这个意思。

考试写法：

> 辅助索引的搜索码顺序与数据文件物理顺序不同，按辅助索引顺序扫描时，索引项相邻并不代表数据记录相邻，因此可能产生大量随机磁盘 I/O，所以顺序查找效率不高。

# 5. 稠密索引 dense index vs 稀疏索引 sparse index

这是 2006B 第七题重点。



**聚集索引因为真实数据已经按这个字段排好了，所以同一个值的记录会挨在一起；非聚集索引因为真实数据没按这个字段排，所以同一个值的记录可能散在各处。**


## 聚集索引

**真实数据文件本身就按这个搜索码排好。**

比如真实数据就是按 `dept_name` 排的：

```
位置1：Biology 张三
位置2：Biology 李四
位置3：CS 王五
位置4：CS 赵六
位置5：Finance 孙七
```

你看，同一个部门的记录是挨着的。

---

## 非聚集索引

**真实数据文件不是按这个搜索码排的。**

比如真实数据是按学号排的，不是按部门排的：

```
位置1：1001 Biology 张三
位置2：1002 CS 王五
位置3：1003 Finance 孙七
位置4：1004 Biology 李四
位置5：1005 CS 赵六
```

这时候同一个 `CS` 的记录不一定挨着。

---

## 2. 稠密聚集索引：为什么只指向第一条就行？

假设真实数据文件按 `dept_name` 排好了：

```
真实数据文件：
位置1：Biology 张三
位置2：Biology 李四
位置3：CS 王五
位置4：CS 赵六
位置5：CS 钱八
位置6：Finance 孙七
```

因为数据是按 `dept_name` 排的，所以：

```
Biology 的记录一定连续CS 的记录一定连续Finance 的记录一定连续
```

所以索引可以写成：

```
Biology -> 位置1
CS      -> 位置3
Finance -> 位置6
```

查 `CS` 的时候：

```
1. 看索引：CS -> 位置32. 跳到位置33. 从位置3开始往后读：   位置3：CS 王五   位置4：CS 赵六   位置5：CS 钱八   位置6：Finance 孙七，停！
```

因为一旦读到 `Finance`，说明 `CS` 结束了。

所以聚集索引只需要指向：

```
CS 这一组的第一条记录
```

后面的 CS 记录自然连续跟着。

这就是：

```
CS -> 第一条 CS
```

的意思。

---

## 3. 稠密非聚集索引：为什么必须指向所有记录？

现在换一种情况。

真实数据不是按 `dept_name` 排的，而是按学号排的：

```
真实数据文件：
位置1：1001 Biology 张三
位置2：1002 CS 王五
位置3：1003 Finance 孙七
位置4：1004 Biology 李四
位置5：1005 CS 赵六
位置6：1006 Math 周九
位置7：1007 CS 钱八
```

你看，`CS` 的记录在：

```
位置2位置5位置7
```

它们不是连续的。

如果索引只写：

```
CS -> 位置2
```

那查 `CS` 的时候：

```
1. 跳到位置2，找到 CS 王五2. 往后读位置3，发现是 Finance3. 你会以为 CS 结束了
```

但其实后面还有：

```
位置5：CS 赵六位置7：CS 钱八
```

所以非聚集索引不能只指向第一条。

它必须写清楚所有 CS 在哪里：

```
CS -> 位置2，位置5，位置7
```

也就是：

```
CS -> pointer1, pointer2, pointer3
```

其中：

```
pointer1 = 指向位置2
pointer2 = 指向位置5
pointer3 = 指向位置7
```

## 4. 什么叫“指针桶 bucket”？

如果同一个值对应的记录很多，比如 `CS` 有 1000 条，那索引里直接写：

```
CS -> pointer1, pointer2, pointer3, ..., pointer1000
```

会很长。

所以可以弄一个“桶”专门存这些指针：

```
CS -> 指针桶
```

桶里面是：

```
pointer1 -> 位置2
pointer2 -> 位置5
pointer3 -> 位置7...
```

所以：

```
CS -> bucket
```

意思就是：

```
CS 这个索引项不直接写所有地址，而是指向一个小盒子，小盒子里装着所有 CS 记录的地址。
```

---

## 5. 为什么这两个都叫“稠密索引”？

这点最容易绕。

你们这本书里的 **稠密索引** 是：

> 每一个搜索码值都有索引项。

不是说每一条记录都必须单独有一行索引。

比如搜索码是 `dept_name`，出现过三个值：

```
BiologyCSFinance
```

那么稠密索引至少要有：

```
Biology -> ...
CS -> ...
Finance -> ...
```

这就叫稠密。

---

## 6. 聚集 vs 非聚集的根本区别

你直接这样记：

```
聚集索引：真实数据按搜索码排好了。所以同一个值的记录挨在一起。索引只要指向第一条，后面顺着读就行。非聚集索引：真实数据没有按搜索码排。所以同一个值的记录可能散在各个地方。索引必须能找到所有相关记录。
```
## 7 稠密索引 dense index

定义：

> 文件中每一个搜索码值都有一个索引项。

比如数据文件里有：

```
ID: 1, 2, 3, 4, 5, 6
```

稠密索引就是：

```
1 -> pointer
2 -> pointer
3 -> pointer
4 -> pointer
5 -> pointer
6 -> pointer
```

学长笔记里写得很直接：稠密索引中，文件中的每个搜索码值都有一个索引项。

### 稠密聚集索引

如果数据文件按搜索码排序，并且有重复值：

```
dept_name = Biology
dept_name = Biology
dept_name = CS
dept_name = CS
dept_name = Finance
```

稠密聚集索引可以只指向每个搜索码值的第一条记录，因为相同值的记录在后面连续存放。

```
Biology -> 第一条 Biology
CS -> 第一条 CS
Finance -> 第一条 Finance
```

### 稠密非聚集索引

辅助索引必须能找到所有记录。因为相同搜索码值的记录在数据文件中不一定连续，所以索引项必须能指向所有相关记录，可能是：

```
CS -> pointer1, pointer2, pointer3
```

或者：

```
CS -> 指针桶 bucket
```

学长笔记里也写了，稠密非聚集索引必须存储指向所有具有相同搜索码值的记录。

## 5.2 稀疏索引 sparse index

定义：

> 只为搜索码的某些值建立索引项。

比如数据文件按 ID 排好，每个磁盘块第一条记录建一个索引项：

```
块1：ID 1, 2, 3
块2：ID 4, 5, 6
块3：ID 7, 8, 9
```

稀疏索引：

```
1 -> 块1
4 -> 块2
7 -> 块3
```

查 `ID=5` 时：

先在索引中找到不超过 5 的最大索引项：`4 -> 块2`，然后进入块2顺序扫描，找到 5。

学长笔记里说，稀疏索引只为搜索码的某些值建立索引项；定位时先找到最大搜索码值小于所查找记录搜索码值的索引指针，再从该指针指向的记录开始顺序查找。

## 5.3 稀疏索引为什么只能用于聚集索引？

这是一个很重要的“理解题”。

稀疏索引成立的前提是：

> 数据文件本身已经按搜索码顺序排好了。

因为你查到某个索引项后，要从那个位置开始顺序往后找。如果数据文件没有按这个搜索码排序，你根本不知道目标记录是不是在后面。

比如文件物理顺序按 ID 排：

```
ID顺序：1 CS2 Math3 CS4 Physics5 CS
```

你想在 `dept_name` 上建稀疏索引。假设只存：

```
CS -> 某个位置Math -> 某个位置
```

那查 `CS` 时，只指向一个 CS 记录，但其他 CS 记录可能散落在文件各处，不能靠“从这里往后顺序扫”找到全部 CS。

所以：

> 辅助索引必须是稠密索引，因为辅助索引的搜索码顺序和数据文件物理顺序不同，不能用少量索引项推断未列出的搜索码值或记录位置。

学长笔记里也明确强调：只有当关系按照搜索码顺序存储时才能使用稀疏索引，换句话说，只有聚集索引才能使用稀疏索引。

## 5.4 稠密索引和稀疏索引比较

|比较点|稠密索引|稀疏索引|
|---|---|---|
|索引项数量|多，每个搜索码值都有|少，只为部分搜索码值建|
|查找速度|更快|稍慢，需要定位后顺序扫|
|空间开销|大|小|
|插入删除维护|开销较大|开销较小|
|是否可用于辅助索引|可以|一般不可以|
|是否适合聚集索引|可以|很适合|

学长笔记里也写到，稠密索引定位记录更快，但稀疏索引空间更小，插入删除维护开销也更小，需要在时间和空间上权衡。

## 5.5 为什么“每个块一个稀疏索引项”是好折中？

数据库 I/O 的主要代价是读磁盘块，而不是在内存里扫描一个块。

所以如果每个块建立一个稀疏索引项：

```
索引项 -> 定位到目标块目标块读入内存后 -> 在块内顺序找
```

这样索引很小，而且访问次数也少。学长笔记里提到，为每个块建立稀疏索引是较好折中，因为查询开销主要由把块读入主存的时间决定，一旦块进入主存，扫描整个块的时间可以忽略。

# 6. 多级索引：为什么要引入二级索引？

这是“稀疏的二级索引意义”的重点。

## 6.1 问题：索引文件本身也可能很大

假设数据文件很大，稠密索引也很大，大到不能全部放入内存。那查索引本身也要读很多磁盘块。

所以出现一个想法：

> 既然数据文件可以建索引，那么索引文件本身也可以再建索引。

## 6.2 二级索引的思想

结构：

```
外层索引：稀疏索引，索引“内层索引块”
内层索引：通常是稠密索引，索引“数据记录”
数据文件：真实记录
```

查找流程：

```
先查外层索引 
    ↓
找到对应的内层索引块
    ↓
在内层索引块中找到数据记录指针 
    ↓
访问真实数据
```

学长笔记里也说，二级索引是在稠密索引作为内层索引之外，在外层构建稀疏索引；查找时先查外层索引，定位到内层索引块，再继续找到目标文件块。

## 6.3 为什么外层索引常用稀疏索引？

因为内层索引本身是按搜索码有序存储的，满足稀疏索引使用条件。外层只需要记录每个内层索引块的第一个搜索码值即可。

简单理解：

```
内层索引块1：1, 2, 3, 4
内层索引块2：5, 6, 7, 8
内层索引块3：9, 10, 11, 12
外层稀疏索引：
	1 -> 内层块1
	5 -> 内层块2
	9 -> 内层块3
```

这样外层索引很小，甚至可以放入内存。

## 6.4 多级索引继续推广

如果外层索引还是太大，就继续给外层索引再建索引：

```
三级索引  
  ↓
二级索引
  ↓
一级索引
  ↓
数据文件
```

B+树本质上就是一种动态维护得很好的多级索引。

# 6.5 插入和删除索引

> **真实数据发生插入/删除以后，索引这本目录要怎么同步修改。**

你先抓住一句话就行：

```
数据文件 = 正文索引文件 = 目录正文变了，目录必须跟着变。
```

---

## 0. 先分清楚：为什么更新索引会麻烦？

比如原来有数据：

```
真实数据文件：第1块：  Biology 张三  Biology 李四  CS 王五  CS 赵六第2块：  Finance 孙七  Math 周九
```

索引可能是：

```
Biology -> 第一条 BiologyCS      -> 第一条 CSFinance -> 第一条 FinanceMath    -> 第一条 Math
```

如果现在插入一条：

```
CS 钱八
```

那索引要不要变？

答案不一定。

因为要看：

1. 这是 **稠密索引** 还是 **稀疏索引**；
2. 这个搜索码值原来有没有；
3. 这条记录是不是某个块里的第一条；
4. 删除时是不是删掉了索引正指向的那条记录。

---

## 1. 插入时：先问一个问题

插入一条新记录，比如：

```
CS 钱八
```

数据库先看搜索码值：

```
dept_name = CS
```

然后问：

> **索引里原来有没有 CS？**

---

## 2. 稠密索引的插入

稠密索引的特点：

> **每一个搜索码值都要有索引项。**

比如出现过：

```
BiologyCSFinance
```

那么索引里就必须有：

```
Biology -> ...CS      -> ...Finance -> ...
```

---

### 情况 1：新搜索码值原来没有

比如原来没有 `Physics`：

```
索引原来：Biology -> ...CS      -> ...Finance -> ...
```

现在插入：

```
Physics 刘十
```

因为 `Physics` 是一个新的搜索码值，所以必须新增索引项：

```
Biology -> ...CS      -> ...Finance -> ...Physics -> 新记录的位置
```

这就是图里说的：

> 如果该搜索码值不在索引中，系统就在索引中合适的位置插入具有该搜索码值的索引项。

傻瓜理解：

```
目录里原来没有 Physics 这一项，现在正文里有了 Physics，所以目录必须补上 Physics。
```

---

### 情况 2：新搜索码值原来已经有

比如原来已经有 `CS`：

```
CS -> ...
```

现在又插入一条：

```
CS 钱八
```

这时候不需要新增一个 `CS` 搜索码值，因为 `CS` 这一项已经存在。

但是要看索引项里面存的是什么。

---

### 2.1 如果索引项存“所有 CS 的指针”

比如非聚集索引经常这样：

```
CS -> pointer1, pointer2
```

其中：

```
pointer1 -> CS 王五pointer2 -> CS 赵六
```

现在插入：

```
CS 钱八
```

那就变成：

```
CS -> pointer1, pointer2, pointer3
```

也就是：

```
pointer3 -> CS 钱八
```

所以图里说：

> 如果索引项存储的是指向具有相同搜索码值的所有记录的指针，系统就在索引项中增加一个指向新记录的指针。

傻瓜理解：

```
目录里 CS 这一项下面原来列了 2 个地址，现在多了 1 个 CS，就把新地址也加进去。
```

---

### 2.2 如果索引项只指向“第一条 CS”

聚集索引常常这样，因为相同值是连续放的：

```
真实数据文件按 dept_name 排：Biology 张三Biology 李四CS 王五       ← 第一条 CSCS 赵六Finance 孙七
```

索引：

```
Biology -> 第一条 BiologyCS      -> 第一条 CSFinance -> 第一条 Finance
```

现在插入一条：

```
CS 钱八
```

因为真实数据文件是按 `dept_name` 排的，所以它会被放在其他 CS 附近：

```
Biology 张三Biology 李四CS 王五       ← 第一条 CSCS 赵六CS 钱八Finance 孙七
```

索引仍然可以是：

```
CS -> 第一条 CS
```

不用新增指针。

所以图里说：

> 否则，索引项存储一个仅指向具有相同搜索码值的第一条记录的指针，系统把待插入的记录放到具有相同搜索码值的其他记录之后。

傻瓜理解：

```
索引只负责告诉你 CS 从哪里开始。因为所有 CS 都挨在一起，所以新 CS 插到 CS 队伍后面就行。
```

---

## 3. 稀疏索引的插入

稀疏索引的特点：

> **不是每个搜索码值都有索引项，只给某些值建索引。**

最常见的是：

> **每个数据块建一个索引项，索引项指向这个块的第一条记录。**

比如真实数据文件：

```
第1块：  10 张三  20 李四  30 王五第2块：  40 赵六  50 孙七  60 周九
```

稀疏索引：

```
10 -> 第1块第一条40 -> 第2块第一条
```

你看，它没有给 20、30、50、60 建索引。

---

### 稀疏索引插入时只关心两件事

#### 第一：有没有产生新块？

比如原来第2块满了，现在插入后产生第3块：

```
第3块：  70 钱八  80 吴九
```

那么索引必须新增：

```
70 -> 第3块第一条
```

因为稀疏索引是一块一个目录项。

所以图里说：

> 如果系统创建一个新的块，它会将新块中出现的第一个搜索码值插入到索引中。

傻瓜理解：

```
多了一页正文，就要在目录里加上这一页的开头。
```

---

#### 第二：新插入的记录是不是这个块里最小的搜索码值？

比如原来：

```
第2块：  40 赵六  50 孙七  60 周九
```

索引：

```
40 -> 第2块
```

现在插入：

```
35 钱八
```

如果它被放进第2块，而且它变成了第2块第一条：

```
第2块：  35 钱八  40 赵六  50 孙七  60 周九
```

那索引不能还写：

```
40 -> 第2块
```

因为第2块的第一条已经变成 35 了。

所以要改成：

```
35 -> 第2块
```

这就是图里说的：

> 如果新插入的记录含有块中的最小搜索码值，那么系统就更新指向该块的索引项。

傻瓜理解：

```
这一页正文的开头变了，目录里这页的开头也要改。
```

---

#### 如果既没有新块，也没有改变块的第一条？

比如：

```
第2块：  40 赵六  50 孙七  60 周九
```

索引：

```
40 -> 第2块
```

现在插入：

```
55 钱八
```

插入后：

```
第2块：  40 赵六  50 孙七  55 钱八  60 周九
```

第2块第一条还是 40。

所以索引不用变。

这就是图里说的：

> 否则，系统对索引不做任何修改。

---

## 4. 删除时：先问一个问题

删除一条记录时，比如删除：

```
CS 王五
```

数据库先找到这条真实记录，然后问：

> **索引里面有没有东西正指向这条记录？**

如果没有，可能不用改索引。

如果有，就要改。

---

## 5. 稠密索引的删除

还是记住：

> 稠密索引每个搜索码值都有索引项。

---

### 情况 1：被删除的是这个搜索码值的唯一记录

比如索引里：

```
Physics -> 刘十
```

真实数据中只有一条 `Physics`。

现在删除：

```
Physics 刘十
```

那么数据里已经没有 `Physics` 了。

所以索引项也要删：

```
删除 Physics -> ...
```

图里说：

> 如果被删除的记录是具有这个特定搜索码值的唯一一条记录，系统就从索引中删除相应的索引项。

傻瓜理解：

```
正文里已经没有 Physics 了，目录里也不能再有 Physics。
```

---

### 情况 2：还有其他相同搜索码值的记录

比如原来有：

```
CS 王五CS 赵六CS 钱八
```

删除：

```
CS 王五
```

但还有：

```
CS 赵六CS 钱八
```

所以 `CS` 这个索引项不能删。

但是要看索引项的形式。

---

### 5.1 如果索引项存所有指针

原来：

```
CS -> pointer1, pointer2, pointer3
```

其中：

```
pointer1 -> CS 王五pointer2 -> CS 赵六pointer3 -> CS 钱八
```

删除王五后：

```
CS -> pointer2, pointer3
```

就是把 `pointer1` 删除。

这就是图里说的：

> 如果索引项存储的是指向所有具有相同搜索码值的记录的指针，系统就从索引项中删除指向被删除记录的指针。

傻瓜理解：

```
CS 这个栏目还在，只是把王五的地址划掉。
```

---

### 5.2 如果索引项只指向第一条记录

原来：

```
真实数据：CS 王五     ← 第一条 CSCS 赵六CS 钱八
```

索引：

```
CS -> CS 王五
```

现在删除的如果不是第一条，比如删除 `CS 赵六`：

```
CS 王五     ← 第一条 CS 仍然是它CS 钱八
```

索引不需要改。

但是如果删除的是第一条：

```
删除 CS 王五
```

删除后：

```
CS 赵六     ← 新的第一条 CSCS 钱八
```

那索引必须从：

```
CS -> CS 王五
```

改成：

```
CS -> CS 赵六
```

图里说：

> 如果被删除的记录是具有该搜索码值的第一条记录，系统就更新索引项，使其指向下一条记录。

傻瓜理解：

```
目录原来指向 CS 队伍的队头。队头被删了，就要改指向新的队头。
```

---

## 6. 稀疏索引的删除

稀疏索引最容易绕，但你只要抓住一句话：

> **稀疏索引只关心每个块的第一条记录。**

比如：

```
第1块：  10 张三  20 李四  30 王五第2块：  40 赵六  50 孙七  60 周九
```

索引：

```
10 -> 第1块第一条40 -> 第2块第一条
```

---

### 情况 1：删除的记录不在索引里

比如删除：

```
50 孙七
```

它不是第2块第一条。

索引还是：

```
40 -> 第2块第一条
```

不用改。

所以图里说：

> 如果索引不包含具有被删除记录搜索码值的索引项，则索引不必做任何修改。

傻瓜理解：

```
删的不是目录指向的那一行，目录不用变。
```

---

### 情况 2：删除的记录正好是索引指向的记录

比如删除：

```
40 赵六
```

它是第2块第一条，也是索引指向的记录：

```
40 -> 第2块第一条
```

删除后第2块变成：

```
第2块：  50 孙七  60 周九
```

那么第2块新的第一条是 50。

索引应该改成：

```
50 -> 第2块第一条
```

这就是最常见的情况。

傻瓜理解：

```
这一页正文的第一行被删了，目录要改成指向新的第一行。
```

---

### 情况 3：如果下一个搜索码值已经有索引项，为什么删除而不是替换？

这个是书上最绕的一句。

你可以这样理解：

稀疏索引不能乱加重复目录项，它本来就是“只保留必要的目录项”。

比如原来索引：

```
10 -> 第1块
40 -> 第2块
70 -> 第3块
```

现在删除 `40` 以后，本来想用下一个搜索码值 `70` 替换它。

但是 `70` 已经在索引里有一项了：

```
70 -> 第3块
```

那就没必要再搞一个重复的 `70`。

所以直接删掉 `40` 这一项就行。

傻瓜理解：

```
目录里本来已经有“70”这个入口了。40 被删后，不需要再把 40 改成 70，否则目录会重复。所以直接删 40。
```

这个点考试一般不深挖，你记住原则就行：

> 稀疏索引删除时，如果被删的是索引项指向的记录，就让索引项指向本块下一条合适记录；如果替换后的搜索码值已经在索引中存在，就直接删除旧索引项，避免重复。

---

## 7. 把插入和删除总流程串起来

### 插入

```
插入一条记录  ↓看索引类型  ↓如果是稠密索引：    该搜索码值原来没有 → 新增索引项    该搜索码值原来有：        索引存所有指针 → 加一个新指针        索引只存第一条 → 把记录放到同值记录后面，索引一般不变如果是稀疏索引：    产生新块 → 为新块第一条记录新增索引项    没产生新块，但新记录变成块内第一条 → 更新该块索引项    其他情况 → 索引不变
```

---

### 删除

```
删除一条记录  ↓看索引类型  ↓如果是稠密索引：    这是该搜索码值唯一记录 → 删除整个索引项    不是唯一记录：        索引存所有指针 → 删除该记录的指针        索引只存第一条：            删除的不是第一条 → 索引不变            删除的是第一条 → 改指向下一条同值记录如果是稀疏索引：    删除的记录不在索引中 → 索引不变    删除的记录在索引中：        改为指向块内下一条记录        如果替换值已经有索引项 → 删除旧索引项
```

---

## 8. 用一句话概括这一页

**稠密索引更新时，要保证每个搜索码值仍然能被索引找到；稀疏索引更新时，要保证每个被索引的块仍然能通过它的第一条记录被定位到。**

或者更傻瓜：

```
稠密索引关心：每个值有没有目录项。稀疏索引关心：每个块的开头有没有目录项。
```

## 9. 最好背的考试版

```
索引更新分为插入和删除。插入时，若为稠密索引：如果新记录的搜索码值原来不存在，则插入新的索引项；如果已存在，则根据索引项存储方式，增加指向新记录的指针，或把新记录放在相同搜索码值记录之后。若为稀疏索引：通常假设每块有一个索引项。如果插入导致产生新块，则把新块中的第一个搜索码值插入索引；如果新记录成为所在块的最小搜索码值，则更新指向该块的索引项；否则索引不变。删除时，若为稠密索引：如果被删除记录是该搜索码值的唯一记录，则删除相应索引项；否则删除对应指针，或者当索引项指向第一条记录且第一条被删时，将索引项改为指向下一条同搜索码值记录。若为稀疏索引：如果被删记录没有对应索引项，索引不变；如果有对应索引项，则根据情况用下一条记录或下一个搜索码值替换，若替换值已有索引项，则删除原索引项。
```

小兔你就抓这个底层逻辑：**稠密索引维护“值”；稀疏索引维护“块开头”**
# 7. B+树索引：这章最核心的大题

B+树插入和删除分别对应 2006A 第八题、2006B 第八题、2020 第九题。

## 7.1 为什么需要 B+树？

传统索引顺序文件的问题是：随着文件增长，新增记录可能放到溢出块里，文件顺序性变差，查询和顺序扫描性能下降。如果频繁重组文件，代价又很高。

B+树的目标就是：

> 在动态插入、删除的情况下，仍然保持树平衡，让查找、插入、删除都比较稳定。

学长笔记里说，索引顺序文件的缺点是随着文件增大，索引查询性能和数据顺序扫描性能都会下降；B+树采用平衡树结构，根到叶的每条路径长度相同。

---

# 8. B+树的结构

## 8.1 B+树最重要的性质

一棵 B+树有这些核心特征：

1. **平衡**：从根结点到任意叶结点的路径长度相同。
2. **非叶结点只负责导航**：内部结点存搜索码值和子指针。
3. **叶结点存真正的索引项**：叶结点保存搜索码值和记录指针。
4. **叶结点按搜索码顺序链接**：方便范围查询和顺序扫描。
5. **每个结点至少半满**：除根结点特殊外，插入删除后要通过分裂、借用、合并维持利用率。

可以画成：

```
              [20 | 40]             
              /    |     \  
      <20      20~40     >=40
叶子层：[5, 10, 15] -> [20, 25, 30] -> [40, 50, 60]
```

注意：真实记录指针主要在叶子层，所以查找最终一定要到叶子。

## 8.2 B+树查询流程

查 `K=25`：

```
从根开始看 25 落在哪个区间进入对应子结点重复直到叶结点在叶结点中找到 25通过指针访问真实记录
```

范围查询 `20 <= K <= 50`：

```
先找到 20 所在叶结点然后沿叶结点链表往后扫直到超过 50
```

这就是 B+树特别适合数据库的原因：既能点查，又能范围查。

# 9. B+树插入：考试步骤

## 9.1 插入总流程

插入一个搜索码值 `K`：

```
第一步：从根开始查找，找到 K 应该进入的叶结点。
第二步：把 K 插入叶结点，并保持叶结点内部有序。
第三步：如果叶结点没满，结束。
第四步：如果叶结点满了，分裂叶结点。
第五步：把新叶结点的第一个搜索码值复制到父结点中，作为分隔值。
第六步：如果父结点也满了，继续分裂父结点。
第七步：如果根结点分裂，创建新根，树高加 1。
```

学长笔记里也写到，插入时先用和查找一样的方法找到搜索码值将出现的叶结点，然后在叶结点中插入新记录并保持有序。

## 9.2 叶结点没满：直接插入

比如叶结点最多放 4 个值：

```
插入前：[10, 20, 40]插入 30插入后：[10, 20, 30, 40]
```

结束。

## 9.3 叶结点满了：分裂叶结点

比如叶结点最多放 4 个值：

```
插入前：[10, 20, 30, 40]插入 25 后临时：[10, 20, 25, 30, 40]
```

溢出，要分裂：

```
左叶子：[10, 20]右叶子：[25, 30, 40]
```

然后把右叶子的第一个搜索码值 `25` **复制到父结点**：

```
父结点新增分隔值 25
```

注意这里是 **copy up 复制上去**，不是从叶子里删掉。叶子中仍然保留 25。

考试易错点：

> B+树叶结点分裂时，提升到父结点的键值仍然保留在叶结点中。

## 9.4 父结点也满了：继续分裂内部结点

如果父结点插入分隔值后也溢出，就分裂父结点。

内部结点分裂时，中间的分隔值要 **push up 推上去**，它不再保留在原内部结点中。

简化记法：

```
叶子分裂：copy up，复制到父结点，叶子保留内部结点分裂：push up，推到父结点，内部不保留
```

这是 B+树插入题最容易丢分的地方。

## 9.5 根结点分裂

如果一路分裂到根，根也满了，就新建一个根：

```
       新根      /    \   左子树  右子树
```

树高加 1。

学长笔记里举了插入 Lamport 的情况：叶结点满了就分裂并向父结点插入新项，父结点满了继续分裂，还需要向根结点插入新项。

# 10. B+树删除：考试步骤

## 10.1 删除总流程

删除搜索码值 `K`：

```
第一步：从根开始查找，找到 K 所在叶结点。
第二步：在叶结点中删除 K 对应索引项。
第三步：如果叶结点仍至少半满，结束。
第四步：如果叶结点太空，先尝试向兄弟结点借一个。
第五步：如果兄弟结点不能借，就和兄弟结点合并。
第六步：合并后父结点中对应分隔值也要删除。
第七步：如果父结点也太空，继续向上处理。
第八步：如果根结点只剩一个孩子，则删除旧根，树高减 1。
```

学长笔记里也写到，删除时先用查询一样的方法找到待删除项所在叶结点，然后从叶结点移除该项；删除可能使叶结点太空，所以需要合并或重新分配。

## 10.2 删除后仍然够半满：直接结束

比如叶结点最多放 4 个，最少放 2 个：

```
删除前：[10, 20, 30]删除 20删除后：[10, 30]
```

还够 2 个，结束。

## 10.3 删除后太空：先借

比如：

```
左叶子：[10]右兄弟：[20, 30, 40]
```

左叶子太空，右兄弟有多余项，可以借一个：

```
左叶子：[10, 20]右兄弟：[30, 40]
```

然后更新父结点中的分隔值。

这叫 **redistribution / borrow 重新分配 / 借用**。

## 10.4 兄弟不够借：合并

比如：

```
左叶子：[10]右兄弟：[20]
```

两个都很少，没法借，就合并：

```
合并后：[10, 20]
```

合并后，父结点中原来区分这两个叶子的分隔值要删除。父结点删除后如果也太空，就继续向上处理。

## 10.5 根结点特殊处理

如果根结点因为合并只剩一个孩子：

```
旧根 |唯一孩子
```

那么删除旧根，让唯一孩子成为新根，树高减 1。

学长笔记里提到删除 Gold 后，合并使树高度减少一层，这就是根结点特殊处理。

# 11. B+树插入删除题怎么做笔记？

你可以直接记这个模板：

```
B+树插入：
1. 查找到目标叶结点。
2. 插入后若不溢出，结束。
3. 叶结点溢出：分裂叶结点，右叶第一个键 copy up 到父结点。
4. 父结点溢出：分裂内部结点，中间键 push up 到父结点。
5. 若根溢出，新建根，树高 +1。
   
B+树删除：
1. 查找到目标叶结点并删除。
2. 若仍满足最小占用，结束。
3. 若下溢，先向左右兄弟借，并更新父结点分隔键。
4. 若借不了，与兄弟合并，并删除父结点中对应分隔键。
5. 父结点若下溢，递归向上处理。
6. 根若只剩一个孩子，则孩子成为新根，树高 -1。
```

# 12. B+树文件组织

B+树不仅可以当索引，还可以直接组织文件。

普通 B+树索引：

```
叶结点：搜索码 + 指向记录的指针真实记录：在另外的数据文件里
```

B+树文件组织：

```
叶结点：直接存储记录
```

学长笔记里写到，B+树结构不仅可以作为索引，也可以作为文件中记录的组织者；B+树文件组织的叶结点存储的是记录而不是指向记录的指针。

优点：文件增长时仍然能保持较好查询性能。

缺点：记录比指针大，所以叶结点能存的项更少；插入删除后，连续叶结点在磁盘上的物理连续性会逐渐丢失，顺序扫描可能需要更多磁盘寻道，必要时需要重建索引。

# 13. B树索引 vs B+树索引

这是 2006C 第八题重点。

## 13.1 B树和 B+树共同点

二者都是平衡树，都能用于索引，都可以降低查找路径长度。

## 13.2 最大区别

|比较点|B树|B+树|
|---|---|---|
|搜索码出现位置|每个搜索码值只出现一次|搜索码值可能同时出现在内部结点和叶结点|
|记录指针位置|内部结点和叶结点都可能有记录指针|主要在叶结点|
|范围查询|不如 B+树方便|很方便，叶结点链表顺序扫描|
|非叶结点扇出|较小，因为还要存记录指针|较大，因为主要存键和子指针|
|树高|可能更高|通常更低|
|删除|更复杂|相对简单|
|数据库常用|较少|更常用|

学长笔记里说，B树和 B+树类似，区别在于 B树去除了搜索码值存储中的冗余；B+树中搜索码值可能重复出现在非叶结点和叶结点中，而 B树中只允许出现一次。笔记还总结说，B树空间优势不足以弥补缺点，所以数据库通常采用 B+树。

## 13.3 为什么数据库更喜欢 B+树？

重点写三点：

第一，**B+树非叶结点只存搜索码和指针，扇出大，树更矮，I/O 次数少。**

第二，**所有实际索引项在叶结点，叶结点有序链接，范围查询和顺序扫描很方便。**

第三，**插入删除主要发生在叶层，结构更统一。**

考试答案：

> B树虽然减少了搜索码重复，可能节省部分空间，并且当搜索码恰好位于非叶结点时可以更快返回，但它需要在非叶结点保存记录指针，降低扇出，可能增加树高；同时范围查询和删除处理不如 B+树方便。因此数据库系统通常采用 B+树索引。

# 14. 不唯一搜索码

有些搜索码不是唯一的，比如很多老师都属于 `dept_name='CS'`。

问题：

```
CS -> 可能对应很多记录
```

如果每条都单独放，会造成大量重复键值；删除某一条记录时也可能要遍历很多项。

常见处理方法：

## 方法一：复合搜索码 + uniquifier

给搜索码补一个唯一属性：

```
(dept_name, ID)
```

原来 `dept_name` 不唯一，但 `(dept_name, ID)` 唯一。

## 方法二：桶 bucket

每个搜索码值只存一次：

```
CS -> bucket -> pointer1, pointer2, pointer3
```

优点是节省搜索码重复空间；缺点是桶可能很大，删除和访问需要额外 I/O。学长笔记也提到，不唯一搜索码可以用复合搜索码确保唯一，或者每个码值只存一次并维护记录指针桶。

# 15. 多码索引 / 复合索引

复合搜索码 composite search key：

> 一个包含多个属性的搜索码。

比如：

```
CREATE INDEX idx_dept_salaryON instructor(dept_name, salary);
```

它的搜索码是：

```
(dept_name, salary)
```

学长笔记里也说，复合搜索码是包含多个属性的搜索码，索引结构和其他索引类似，只是搜索码变成属性列表。

## 15.1 复合索引适合什么查询？

对 `(dept_name, salary)` 建 B+树索引，适合：

```
where dept_name = 'Finance' and salary < 80000
```

因为先按 `dept_name` 分组，再在组内按 `salary` 排序。

也适合：

```
where dept_name = 'Finance'
```

因为它满足最左前缀。

不太适合：

```
where salary < 80000
```

因为索引第一排序关键字是 `dept_name`，不同部门里的 salary 分散在不同位置。

## 15.2 多个单码索引 vs 一个复合索引

假设有两个单码索引：

```
Index(dept_name)Index(salary)
```

查询：

```
where dept_name = 'Finance' and salary < 80000
```

可以分别通过两个索引得到记录指针集合：

```
S1 = 满足 dept_name='Finance' 的记录指针集合S2 = 满足 salary<80000 的记录指针集合结果 = S1 ∩ S2
```

但如果有复合索引 `(dept_name, salary)`，通常更直接更高效。

# 16. 合取选择和析取选择中的索引操作

这是 2006A 第十题：**对连词选择的索引文件操作对析取选择是否适用？**

先明确：

```
合取选择：AND析取选择：OR
```

## 16.1 合取选择 AND

例如：

```
where dept_name = 'Finance' and salary < 80000
```

如果两个条件都有索引：

```
S1 = dept_name='Finance' 的记录指针集合S2 = salary<80000 的记录指针集合AND 结果 = S1 ∩ S2
```

所以合取选择常用 **交集 intersection**。

## 16.2 析取选择 OR

例如：

```
where dept_name = 'Finance' or salary < 80000
```

如果两个条件都有索引：

```
S1 = dept_name='Finance' 的记录指针集合S2 = salary<80000 的记录指针集合OR 结果 = S1 ∪ S2
```

所以析取选择应该用 **并集 union**，不是交集。

## 16.3 那“合取选择的索引文件操作”能不能直接用于析取？

不能直接照搬。

考试答案：

> 对合取选择，多个索引结果应取交集；对析取选择，应取并集。因此合取选择中的交操作不能直接用于析取选择。若析取条件中每个子条件都有可用索引，可以分别通过索引得到指针集合并取并集；若某个子条件没有索引，则仅使用其他条件的索引会漏掉满足无索引条件的记录，此时通常需要全表扫描或其他访问策略。

举例：

```
where A=1 or B=2
```

如果只有 `A` 上有索引，没有 `B` 上的索引，你不能只找 `A=1` 的记录，因为还有很多 `B=2` 的记录可能不满足 `A=1`，会被漏掉。

## 16.4 位图索引和多条件选择

位图索引特别适合多个码上的选择操作。比如每个位代表一条记录，满足条件为 1，不满足为 0。AND 就做按位与，OR 就做按位或。学长笔记里也写到，位图索引主要用于多个码上的选择操作，往往对几个位图进行交运算即可得到想要的记录。

# 17. 散列索引 Hash Index

## 17.1 散列索引基本思想

散列函数：

```
h(search_key) -> bucket number
```

例如：

```
h("CS") = 3h("Math") = 7h("Finance") = 2
```

桶里存放记录或记录指针。

学长笔记里说，数值或字符串通过散列函数得到有限散列值，每个散列值对应一个散列桶，散列值相同的项放在同一个桶内。

## 17.2 散列索引适合等值查询

```
where ID = '10086'
```

直接算：

```
h(10086) = 5
```

去 5 号桶找。

但散列索引不适合范围查询：

```
where ID between 10000 and 20000
```

因为 hash 后大小顺序没了。

考试写法：

> 散列索引适合等值查询，不适合范围查询；顺序索引和 B+树索引既适合等值查询，也适合范围查询。

## 17.3 静态散列

静态散列就是桶数固定。

问题：随着记录增加，桶可能装不下，发生桶溢出。

溢出原因：

1. 记录太多，桶空间不足。
2. 桶倾斜：某些桶分到太多项。
3. 散列函数不好，分布不均匀。
4. 很多记录本身有相同搜索码值。

处理方法：加溢出桶。

```
桶3 -> 原桶 -> 溢出桶1 -> 溢出桶2
```

但溢出桶多了之后，查找性能会下降。学长笔记也写到，静态散列桶数固定，桶空间用完会发生桶溢出，原因包括记录太多和桶倾斜，通常通过链接溢出桶处理。

# 18. “理想的散列函数既是均匀的又是随机的”怎么理解？

这是 2006A 第九题。

## 18.1 均匀 uniform

均匀的意思：

> 每个桶分到的记录数量差不多。

如果有 1000 条记录，10 个桶，理想情况是每个桶大约 100 条，而不是：

```
桶1：800条桶2：10条桶3：20条...
```

均匀性好，查找平均时间就稳定。

## 18.2 随机 random

随机的意思：

> 搜索码值之间的规律不应该导致散列结果也有规律地集中。

比如 ID 是：

```
1000, 2000, 3000, 4000
```

如果散列函数是：

```
h(x) = x mod 10
```

那么这些 ID 全部进 0 号桶：

```
1000 mod 10 = 02000 mod 10 = 03000 mod 10 = 04000 mod 10 = 0
```

这就很糟糕。虽然函数简单，但不随机，导致严重倾斜。

一个更好的散列函数应该让相近或有规律的键值也被打散：

```
1000 -> 桶32000 -> 桶73000 -> 桶14000 -> 桶9
```

## 18.3 考试例子

可以这样写：

> 例如用学生学号作为搜索码，如果散列函数取学号最后一位，而某一批学号最后一位分布很集中，就会导致大量记录进入同一桶，产生桶溢出。这说明该散列函数不均匀、不随机。理想散列函数应使搜索码均匀分布到各桶中，并使搜索码中的规律不反映为桶号中的规律，从而减少桶倾斜和溢出。

学长笔记里也写到，对于理想散列函数，得到的散列值是均匀分布的，所有项均匀分布在各个桶内，可以得到最佳平均查找时间。

# 19. 动态散列：为什么需要？

静态散列的问题是桶数固定。数据库文件会增长，所以桶数固定会造成：

```
记录越来越多桶越来越满溢出桶越来越多查询越来越慢
```

解决思路：

> 动态散列允许散列函数和桶空间动态改变，以适应记录增多或减少。

学长笔记里介绍了可扩充散列：使用具有均匀性和随机性的散列函数，产生很大范围的 b 位二进制值，但开始不为每个散列值都建桶，而是按需建桶。

## 19.1 可扩充散列的大概逻辑

先算一个很长的 hash 值，比如 32 位：

```
h(key) = 101101001010...
```

但一开始只看前 i 位或者后 i 位：

```
i = 2只看 2 位：00, 01, 10, 11
```

目录 directory：

```
00 -> 桶A01 -> 桶B10 -> 桶C11 -> 桶D
```

如果某个桶满了：

1. 如果局部深度还小，就分裂这个桶。
2. 如果目录不够用，就目录翻倍。
3. 重新分配桶内记录。

你记到这个程度一般够考试简答。

# 20. 辅助索引和记录重定位

这块属于“额外但很有理解价值”的内容。

如果数据文件用 B+树文件组织，插入时叶结点可能分裂，记录可能被移动到新结点。问题是：

> 辅助索引里如果直接保存真实记录的物理指针，那么记录位置一变，所有指向这些记录的辅助索引都要更新。

这会非常贵。

解决办法：

> 辅助索引不直接存真实记录物理地址，而是存主索引搜索码值。通过辅助索引找到主索引键，再通过主索引定位真实记录。

这样记录物理位置变化时，辅助索引不需要全部改。学长笔记里也提到，B+树文件组织结构中叶结点分裂会导致记录移动，此时所有指向重定位记录的辅助索引都必须更新；解决方案是在辅助索引中存储主索引搜索码属性的值，而不是直接存记录指针。

代价是查询多一步：

```
辅助索引 -> 主索引键值 -> 主索引 -> 真实记录
```

# 21. SQL 中建立和删除索引

这部分不一定是 Ch14 主体，但笔记里有，考试小题可能问。

```
CREATE [UNIQUE] [CLUSTER] INDEX 索引名
ON 表名
 (列名 [ASC|DESC], ...);
```

含义：

- `UNIQUE`：索引值唯一。
- `CLUSTER`：聚簇索引。
- `ASC`：升序。
- `DESC`：降序。

删除索引：

```
DROP INDEX 索引名;
```

学长笔记里也写到，索引可以建在一列或多列上，`UNIQUE` 表示每个索引值只对应唯一记录，`CLUSTER` 表示建立聚簇索引。

注意：

> 索引由 DBA 或表拥有者建立，由 DBMS 自动维护，优化器自动选择合适索引作为存取路径，用户一般不显式指定索引。

# 22. 为什么引入索引？索引是否越多越好？

这是 2006C 第七题。

## 22.1 为什么引入索引？

因为没有索引时，查询可能要全表扫描：

```
找一条记录：可能读很多块
```

有索引后：

```
先查较小的索引结构再通过指针定位数据
```

可以显著减少磁盘 I/O。

考试答案：

> 引入索引是为了加快查询速度，尤其是等值查询、范围查询、连接和排序等操作。索引通过保存搜索码值和记录指针，使系统能够避免对整个数据文件进行顺序扫描，从而减少磁盘 I/O。

## 22.2 索引是不是越多越好？

不是。

原因：

1. **占用额外存储空间。**
2. **插入时要维护每个相关索引。**
3. **删除时要从每个相关索引删除索引项。**
4. **更新搜索码属性时，要更新索引。**
5. **索引过多会增加优化器选择成本和系统维护成本。**
6. **对写多读少的表，过多索引可能明显降低性能。**

考试答案：

> 索引不是越多越好。索引可以加快查询，但会带来存储开销和维护开销。每次插入、删除或更新记录时，相关索引也必须同步更新。因此应根据查询频率、选择率、更新频率和空间成本合理建立索引。

# 23. 高频考点逐题对应

## 2006A 第七题、2020 第六题：解释主索引和辅助索引，辅助索引顺序查找效率高吗？

答题模板：

> 主索引 / 聚集索引是建立在决定数据文件物理顺序的搜索码上的索引；
> 辅助索引 / 非聚集索引是建立在与数据文件物理顺序不同的搜索码上的索引。
> 主索引的搜索码通常是主键，但不一定必须是主键。
> 辅助索引可以有多个，但由于其搜索码顺序与记录物理顺序不同，按辅助
> 索引进行顺序扫描时，相邻索引项指向的记录可能分布在不同磁盘块中，会产生大量随机 I/O，因此效率不高。

## 2006A 第八题：B+树插入

答题模板：

> 先按查找过程定位到应插入的叶结点；
> 若叶结点有空间，则按序插入；
> 若叶结点溢出，则分裂叶结点，并将新叶结点的第一个搜索码复制到父结点；
> 若父结点也溢出，则继续分裂内部结点并将中间键上推；
> 若根结点溢出，则新建根，树高加一。

## 2006B 第八题、2020 第九题：B+树删除

答题模板：

> 先定位待删除项所在叶结点并删除；
> 若删除后结点仍满足最小占用要求，则结束；
> 若下溢，则优先向相邻兄弟结点借项并更新父结点分隔键；
> 若兄弟结点不能借，则与兄弟结点合并，并删除父结点中相应分隔键；
> 若父结点下溢，递归向上处理；
> 若根结点只剩一个孩子，则删除旧根，树高减一。

## 2006A 第九题：“理想散列函数既是均匀的又是随机的”

答题模板：

> 均匀是指散列函数应把记录尽量平均分布到各个桶中，避免某些桶过满；
> 随机是指搜索码中的规律不应导致散列值也呈现规律性集中。
> 比如若学号都以 0 结尾，而散列函数取最后一位，则所有记录进入同一桶，导致桶倾斜和溢出，这不是理想散列函数。
> 理想散列函数应减少桶倾斜，降低平均查找时间。

## 2006A 第十题：对连词选择的索引文件操作对析取选择是否适用？

答题模板：

> 不可直接适用。
> 合取选择 AND 中，多个索引得到的记录指针集合应取交集；
> 析取选择 OR 中，应取并集。
> 如果 OR 的每个子条件都有索引，可以分别取出指针集合后求并集；
> 如果某个子条件没有索引，仅使用其他条件索引会漏掉满足无索引条件的记录，通常需要全表扫描。

## 2006B 第七题：比较稠密索引和稀疏索引，理解稀疏二级索引意义

答题模板：

> 稠密索引为每个搜索码值建立索引项，查找快但空间和维护开销大；
> 稀疏索引只为部分搜索码值建立索引项，空间小、维护开销低，但查找时需定位后顺序扫描。
> 
> 稀疏索引只能用于文件按搜索码有序存储的情况，即通常只能用于聚集索引。
> 二级索引的意义是：当内层索引太大不能放入内存时，可以把内层索引看作有序文件，再为其建立外层稀疏索引，从而减少磁盘块访问次数。

## 2006C 第七题：为什么引入索引，索引是否越多越好？

答题模板：

> 引入索引是为了加快查询，减少全表扫描和磁盘 I/O。但索引不是越多越好，因为索引需要额外空间，并且插入、删除、更新时都要维护相关索引，可能显著降低更新性能。因此应根据查询模式和更新代价合理建立索引。

## 2006C 第八题：B树索引与 B+树索引比较

答题模板：

> B树中搜索码值只出现一次，记录指针可出现在内部结点和叶结点；
> B+树中搜索码值可能在内部结点和叶结点重复出现，实际记录指针主要保存在叶结点，叶结点按序链接。
> B树可能节省部分空间，且命中内部结点时可较快返回；但 B树非叶结点需要保存记录指针，扇出较小，树高可能增加，范围查询和删除更复杂。B+树扇出大、树矮、范围查询方便、结构统一，因此数据库系统通常采用 B+树。

# 24. 最后一版压缩笔记

你复习时可以最终背这个：

```
Ch14 索引
1. 索引作用：加速查询。
   索引项 = 搜索码值 + 记录指针。
2. 搜索码：用于查找记录的属性或属性集，不一定是主键。
3. 索引类型：   
   - 顺序索引：按搜索码有序，适合等值、范围、排序。   
   - 散列索引：按 hash 分桶，适合等值，不适合范围。

4. 主索引 / 聚集索引：   
   数据文件物理顺序与搜索码顺序一致。   
   一个文件通常只能有一个。
5. 辅助索引 / 非聚集索引：   
   搜索码顺序与数据文件物理顺序不同。   
   可以有多个。   
   顺序扫描效率低，因为会产生随机 I/O。   
   一般必须是稠密索引。
6. 稠密索引：   
   每个搜索码值都有索引项。   
   查找快，空间大，维护开销大。
7. 稀疏索引：   
   只为部分搜索码值建索引项。   
   空间小，维护开销小。   
   只能用于聚集索引。   
   查找：先找小于等于目标的最大索引项，再顺序扫描。
8. 多级索引：   
   索引太大不能放内存时，把索引文件当成有序文件，再建外层稀疏索引。   B+树可以看作动态多级索引。
9. B+树：   
   平衡树，根到叶路径等长。   
   非叶结点导航，叶结点存索引项，叶结点有序链接。   
   点查和范围查询都好。
10. B+树插入：    
    找叶子 
    -> 插入 
    -> 溢出则叶子分裂 
    -> 右叶第一个键 copy up 到父结点    
    -> 父结点溢出则内部结点分裂，中间键 push up    
    -> 根溢出则新建根
    
11. B+树删除：    
    找叶子 
    -> 删除 
    -> 若下溢，先借；借不了就合并    
    -> 删除父结点分隔键    
    -> 父结点下溢递归处理    
    -> 根只剩一个孩子则树高减一。
12. B树 vs B+树：    
    B树键不重复，记录指针可在内部结点；
	B+树键可重复，记录指针在叶结点，叶子链表支持范围查询。    
	数据库更常用 B+树。
13. 散列索引：    
    h(key) -> bucket。    
    理想 hash 函数要均匀且随机。    
    均匀：各桶记录数差不多。    
    随机：键值规律不会造成桶集中。    
    静态散列可能桶溢出；
    动态散列可扩展桶空间。

14. 多条件查询：    
    AND：多个索引结果取交集。    
	OR：多个索引结果取并集。    
	OR 若某个条件无索引，不能只用其他索引，否则漏记录。
15. 索引不是越多越好：    
    优点：加快查询。    
    缺点：占空间；插入、删除、更新都要维护索引。
```
