# 第9章：Application Development（应用开发）

## 9.1 应用架构的演变 [了解]

| 时代 | 架构 | 特点 |
|------|------|------|
| 1960s-70s | 大型机（Mainframe） | 所有处理在服务器，终端只显示 |
| 1980s | 个人电脑（PC） | 客户端有处理能力 |
| 1990s中-2010s | Web | 浏览器+Web服务器+数据库，三层架构 |
| 2010s至今 | Web + 智能手机 | 移动端+云端 |

## 9.2 Web基础概念 [了解]

| 术语 | 全称 | 含义 |
|------|------|------|
| **HTML** | HyperText Markup Language | 网页内容标记语言 |
| **HTTP** | HyperText Transfer Protocol | 无连接协议，服务器回复后关闭连接 |
| **URL** | Uniform Resource Locator | 统一资源定位符，如 `http://www.acm.org/sigmod` |
| **Cookie** | — | 服务器发给浏览器的小段识别文本，用于维持会话状态 |
| **CGI** | Common Gateway Interface | Web服务器与应用服务器之间的标准接口 |

> **关键理解**：HTTP是无连接的——服务器不知道两次请求来自同一用户。Cookie就是为了解决这个问题：服务器在第一次请求时给浏览器一个Cookie，浏览器后续请求都带上这个Cookie，服务器就能认出是同一个会话。

## 9.3 三层Web架构 vs 两层架构 [重点]

```
三层架构（Three-Layer）：
┌──────────────┐
│  浏览器/客户端    ← 表示层 (Presentation)
├──────────────┤
│  Web/应用服务器 │  ← 业务逻辑层 (Business Logic)
├──────────────┤
│   数据库服务器  │  ← 数据层 (Data)
└──────────────┘

两层架构（Two-Layer / Client-Server）：
┌──────────────┐
│  客户端(胖客户端) │  ← 表示层 + 业务逻辑
├──────────────┤
│   数据库服务器  │  ← 数据层
└──────────────┘
```

| | 三层架构 | 两层架构 |
|---|---|---|
| 优点 | 瘦客户端，易部署，安全 | 减少间接开销 |
| 缺点 | 多一层间接开销 | 胖客户端难维护 |

## 9.4 Servlet 和 JSP [了解]

- **Servlet（Java）**: 继承 `HttpServlet`，重写 `doGet()` / `doPost()` 方法，在Java代码中用JDBC查询数据库，输出HTML
- **JSP（Java Server Pages）**: HTML中嵌入Java代码 `<% ... %>`，最终编译成Servlet
- **PHP**: 服务端脚本，用 `<?php ... ?>` 嵌入
- **会话管理**: `request.getSession()` — 用Cookie跟踪用户会话

**关键理解**：Servlet = Java代码里写HTML，JSP = HTML里写Java代码。都是为了动态生成网页内容。

## 9.5 MVC架构 [重点]

| 字母 | 全称 | 职责 | 类比 |
|------|------|------|------|
| **M** | Model | 业务逻辑 + 数据访问 | 数据库和业务规则 |
| **V** | View | 展示层，用户看到的东西 | HTML页面 |
| **C** | Controller | 接收用户事件，调用Model，选择View | 分发器/路由器 |

**为什么重要？** 分离关注点（Separation of Concerns）：改界面不改业务逻辑，改业务逻辑不改界面。

## 9.6 ORM（对象关系映射）[了解]

- **工具**：Hibernate (Java), Entity Framework (.NET)
- **关键操作**：`session.save(object)` — 自动生成INSERT语句

## 9.9 🔴 应用安全（重点！简答题高频） [重点]

### SQL注入（SQL Injection） [必考]

**攻击原理**：用户输入被直接拼接到SQL字符串中。

```sql
-- 正常查询字符串拼接：
"SELECT * FROM instructor WHERE name = '" + name + "'"

-- 攻击者输入：X' OR 'Y' = 'Y
-- 实际执行的SQL变成：
SELECT * FROM instructor WHERE name = 'X' OR 'Y' = 'Y'
-- 结果：永远为真，返回所有行！
```

**防御方法**：永远使用**预编译语句（PreparedStatement）**，用参数占位符而非字符串拼接。

```java
// 安全做法：
PreparedStatement pstmt = conn.prepareStatement(
    "SELECT * FROM instructor WHERE name = ?"
);
pstmt.setString(1, name);  // 参数绑定，不会被解释为SQL
```

### XSS（跨站脚本攻击 / CSRF）

- **XSS**: 攻击者在网页中注入恶意脚本，窃取用户Cookie
- **CSRF**: 利用用户已登录状态，伪造用户请求
- **防御**：过滤HTML标签、验证HTTP Referer、不用GET做更新操作、使用CSRF Token

### 密码安全

- 绝不在可被Web服务器访问的脚本文件中明文存储密码
- 密码应加盐（Salt）后散列存储

### 认证方式

| 方式                        | 说明                       |
| ------------------------- | ------------------------ |
| 单因素认证                     | 只有密码                     |
| **双因素认证**                 | 密码 + 一次性密码/短信/硬件令牌       |
| **单点登录(SSO)**             | 一次登录访问多个系统（SAML, OpenID） |
| **LDAP/Active Directory** | 集中式认证服务                  |

### 加密基础 [了解]

| 类型 | 密钥 | 算法 | 特点 |
|------|------|------|------|
| **对称加密** | 加密解密同一密钥 | DES, AES | 快，但密钥分发困难 |
| **非对称加密（公钥）** | 公钥加密，私钥解密 | RSA | 安全，但慢 |
| **混合方案** | 两者结合 | — | 用公钥加密对称密钥，用对称密钥加密数据 |

**其他安全概念**：
- **数字签名**：保证消息真实性和不可否认性（发送方用私钥签名，接收方用公钥验证）
- **数字证书**：由CA（认证机构）签发，验证公钥的真实性
- **质询-应答**：避免直接传输密码
- **审计追踪（Audit Trail）**：记录操作日志，用于事后安全审计

# 第10章：Big Data（大数据）

## 10.1 大数据的3V特征 [重点]

| V | 英文 | 含义 |
|---|------|------|
| **Volume** | 体量 | 数据量大到传统数据库无法处理 |
| **Velocity** | 速度 | 数据产生和插入的速度极快 |
| **Variety** | 多样性 | 数据类型多（结构化+半结构化+非结构化） |

## 10.2 分布式文件系统（HDFS）[重点]

**为什么需要？** 单台机器的磁盘装不下大数据，需要横跨多台机器存储。

**HDFS架构**：
```
NameNode（名称节点）
  ├── 管理文件名 → Block ID映射
  ├── 管理Block ID → DataNode位置映射
  └── 单点（单一命名空间）

DataNode（数据节点）× N台
  ├── 每个节点存储实际数据块
  └── Block ID → 物理磁盘位置映射

典型块大小：64 MB（远大于传统磁盘的4KB）
每个块在多个DataNode上复制（默认3份）
```

**特点**：
- 一次写入，多次读取（Write-once-read-many）
- 适合百万级大文件，不适合数十亿小元组
- 针对海量数据吞吐量优化，不追求低延迟

## 10.3 分片（Sharding）[了解]

将数据按分片键（如user_id）水平拆分到多个数据库中。

| 优点 | 缺点 |
|------|------|
| 扩展性好，每个分片独立 | 应用层需自己路由 |
| — | 跨分片查询困难 |
| — | 数据重均衡困难 |

## 10.4 NoSQL / Key-Value 存储 [重点]

| 类型 | 代表系统 | 数据模型 |
|------|---------|---------|
| 不可解释字节 | Amazon S3, Dynamo | `put(key, value)`, `get(key)`, `delete(key)` |
| 宽表 | BigTable, Cassandra, HBase, DynamoDB | 列族，稀疏矩阵 |
| 文档存储(JSON) | MongoDB, CouchDB | JSON文档 |

**特征**：
- 有限或不支持事务更新
- 没有完整的查询处理
- 牺牲一致性换取可用性和分区容忍性

## 10.5 CAP定理（Brewer's Theorem） [重点]

> CAP三者只能同时满足两者，必须牺牲其一：

| 字母 | 含义 | 说明 |
|------|------|------|
| **C** | Consistency（一致性） | 所有节点在同一时刻看到相同数据 |
| **A** | Availability（可用性） | 每个请求都能获得（非错误的）响应 |
| **P** | Partition-tolerance（分区容忍） | 系统在网络分区（节点间通信中断）时仍能工作 |

| 选择 | 典型系统 |
|------|---------|
| CP（牺牲A） | 传统关系数据库 |
| AP（牺牲C） | 大多数Web应用、NoSQL系统 |

> **关键理解**：网络分区是不可避免的（机器宕机、网络故障等），所以实际上是在C和A之间二选一。传统数据库选C，互联网应用选A。

## 10.6 🔴 MapReduce 编程模型 [重点]

**核心思想**：程序员只需提供 `map()` 和 `reduce()` 函数，系统自动处理并行化、协调和容错。

```
 输入数据
         ↓ 分割
┌──┴──┬──┴──┬──┴──┐
│ Map │ Map │ Map │  ← 并行执行
└──┬──┴──┬──┴──┬──┘
         ↓  中间结果(key, value列表)
└─── Shuffle（按key分组）───┘
  ↓
┌──┴──┬──┴──┐
│Reduce│Reduce│  ← 并行执行
└──┬──┴──┬──┘
  ↓ 最终结果
```

### 词频统计（Word Count）—— 经典示例

```
输入: "One a penny, two a penny, hot cross buns"

Map阶段:
  → ("One", 1), ("a", 1), ("penny", 1), ("two", 1), 
    ("a", 1), ("penny", 1), ("hot", 1), ("cross", 1), ("buns", 1)

Shuffle阶段（按key分组）:
  ("One", [1]), ("a", [1,1]), ("penny", [1,1]), ("two", [1]), ...

Reduce阶段（求和）:
  → ("One", 1), ("a", 2), ("penny", 2), ("two", 1), 
    ("hot", 1), ("cross", 1), ("buns", 1)
```

# 第11章：Data Analytics（数据分析）

## 11.1 数据分析概述 [了解]

**别名**：Business Intelligence (BI)，Decision Support（决策支持）

**通用流程**：
```
数据收集 → ETL(抽取-转换-加载) → 
生成聚合/报表 → OLAP分析 → 
统计建模 → 预测模型
```

## 11.2 数据仓库（Data Warehouse）[重点]

| 概念 | 说明 |
|------|------|
| **定义** | 从多个数据源收集的、统一模式下的历史数据存储库 |
| **与操作数据库的区别** | 操作库存当前数据，仓库存历史数据；操作库用于事务处理，仓库用于分析查询 |

### 事实表 vs 维度表

| | 事实表（Fact Table） | 维度表（Dimension Table） |
|---|---|---|
| 大小 | **大**，记录最多的行 | **小** |
| 内容 | 数值型度量值 | 描述性属性 |
| 示例 | sales(item_id, store_id, date, number, price) | store(store_id, city, state), item(item_id, name, color) |

### 星型模式 vs 雪花模式

```
星型模式（Star Schema）:        雪花模式（Snowflake Schema）:
    维度表                          维度表(标准化)
      ↓                                ↓
事实表 ← 维度表              维度表 ← 事实表 → 维度表
      ↓                                ↓
    维度表                          维度表(标准化)
```

- **星型**：维度表直接连接事实表，非规范化，查询快但冗余
- **雪花**：维度表进一步规范化，减少冗余但需要更多连接

### 列式存储 [了解]

- 传统行存储：一行的所有列连续存放
- **列式存储**：同一列的所有值连续存放
- 优点：分析查询通常只访问少数列，列存储减少I/O；压缩效果更好
- 缺点：元组重建代价高

## 11.3 🔴 OLAP 操作（重点中的重点！）[必考]

### 基本OLAP操作

| 操作 | 英文 | 方向 | 说明 |
|------|------|------|------|
| **上卷** | Rollup | 细→粗 | 日→月→年，聚合掉一个维度 |
| **下钻** | Drill down | 粗→细 | 年→月→日，展开更细粒度 |
| **切片** | Slice | 固定一维 | 只看"2024年的数据" |
| **切块** | Dice | 固定多维 | 只看"2024年+上海的数据" |
| **旋转** | Pivot | — | 交换行/列维度 |

### 数据立方体（Data Cube）

多维交叉表的泛化。一个有3个维度（如商品、颜色、尺码）的数据立方体可以汇总不同维度组合下的销量。

### SQL中的OLAP操作（高频考点！）

**CUBE**：对指定属性的**所有子集**做GROUP BY
```sql
-- 3个属性 → 2^3 = 8种分组
SELECT item_name, color, size, SUM(number)
FROM sales
GROUP BY CUBE(item_name, color, size);
-- 生成的分组：
-- (item_name, color, size)
-- (item_name, color)
-- (item_name, size)
-- (color, size)
-- (item_name)
-- (color)
-- (size)
-- ()  -- 全表总计
```

**ROLLUP**：只对**前缀子集**做GROUP BY
```sql
-- 3个属性 → 3+1 = 4种分组（只保留前缀层级结构）
SELECT item_name, color, size, SUM(number)
FROM sales
GROUP BY ROLLUP(item_name, color, size);
-- 生成的分组：
-- (item_name, color, size)  -- 最细粒度
-- (item_name, color)        -- 按商品和颜色汇总
-- (item_name)               -- 按商品汇总
-- ()                        -- 全表总计
```

**对比记忆**：
- CUBE: 2^n 种分组（所有可能的维度组合）
- ROLLUP: n+1 种分组（只有前缀层级）

**GROUPING SETS**：精确指定要哪些分组
```sql
GROUP BY GROUPING SETS((color, size), (size, item_name));
-- 只生成这两种分组，不多不少
```

**grouping() 函数**：判断某个NULL是否是ROLLUP/CUBE生成的"汇总行"
```sql
-- 返回1表示这一行的NULL是汇总产生的（代表"all"），不是真实数据
SELECT item_name, color, SUM(number), 
       GROUPING(item_name), GROUPING(color)
FROM sales
GROUP BY CUBE(item_name, color);
```

**PIVOT 操作**：
```sql
-- 把颜色值变成列
SELECT * FROM sales
PIVOT (SUM(quantity) FOR color IN ('dark', 'pastel', 'white'));
```

### OLAP实现方式

| 类型 | 全称 | 方式 |
|------|------|------|
| **MOLAP** | Multidimensional OLAP | 内存中的多维数组 |
| **ROLAP** | Relational OLAP | 纯关系数据库 |
| **HOLAP** | Hybrid OLAP | 摘要存内存，基表存关系库 |

## 11.4 数据挖掘（Data Mining）[重点]

**定义**：半自动地从大型数据库中找出有用的模式。属于**KDD（知识发现）**的一部分。

### 两大类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **预测型** | 分类、回归 | 信用评级、房价预测 |
| **描述型** | 关联规则、聚类 | 买了A的人也买B、用户分群 |

### 🔴 分类（Classification）[重点]

**决策树（Decision Tree）**：
- 每个内部节点按某属性的条件划分数据
- 叶节点是分类标签
- 选择最佳划分属性的标准：

| 指标 | 公式 | 说明 |
|------|------|------|
| **Gini** | `Gini(S) = 1 - Σ p_i²` | p_i是第i类在S中的比例 |
| **熵（Entropy）** | `Entropy(S) = -Σ p_i · log₂(p_i)` | 值越大越混乱 |
| **信息增益** | `Gain = Entropy(S) - Σ(|S_v|/|S|)·Entropy(S_v)` | 父节点熵减子节点加权熵 |
| **信息增益率** | `Gain Ratio = Gain / SplitInfo` | 克服信息增益偏好多值属性 |

**朴素贝叶斯分类器**：基于贝叶斯定理 `P(c_j|d) = P(d|c_j)·P(c_j) / P(d)`，假设各属性独立分布。

**SVM（支持向量机）**：找最大间隔的分隔线/面（Maximum margin），用核函数处理非线性分类。

**神经网络/深度学习**：多层网络，用反向传播（Backpropagation）算法训练权重。

### 回归（Regression）

预测**连续值**而非分类标签。如 `Y = a₀ + a₁X₁ + ... + aₙXₙ`（线性回归）。

### 🔴 关联规则（Association Rules）[必考]

**经典案例**：买了面包的人通常也会买牛奶。

| 指标 | 英文 | 公式 | 含义 |
|------|------|------|------|
| **支持度** | Support | P(A∪B) = 包含A和B的交易数 / 总交易数 | A和B一起出现的概率 |
| **置信度** | Confidence | P(B|A) = 包含A和B的交易数 / 包含A的交易数 | 买了A的人中同时买B的比例 |

**计算示例**：
```
交易记录：
T1: bread, milk
T2: bread, diaper, beer, eggs
T3: milk, diaper, beer, cola
T4: bread, milk, diaper, beer
T5: bread, milk, diaper, cola

规则 bread → milk:
  Support = 包含{bread, milk}的交易数 / 5 = 3/5 = 0.6
  Confidence = 包含{bread, milk}的交易数 / 包含{bread}的交易数 = 3/4 = 0.75
```

**Apriori算法核心思想**：如果一个项集不是频繁的（支持度不够），那么它的所有超集也不可能是频繁的。用它来逐层剪枝。
