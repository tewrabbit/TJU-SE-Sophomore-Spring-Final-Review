## 总体概述

索引是数据库**性能**的核心。没有索引，每次查询都要全表扫描（Linear Scan）；有了索引，可以快速定位目标数据。但索引不是免费的——每次数据修改都必须更新索引。

1. **顺序索引**（概念多，简答题常考）
2. **B+树**（操作题必考：插入和删除）
3. **散列**（概念+对比）
4. **特殊索引**（位图、LSM树、空间索引）

# 14.1 索引基本概念 [必考]

## 什么是索引？

索引是一个辅助结构，包含**搜索码(Search Key)** 和**指针(Pointer)** 的配对记录。

| 术语 | 含义 |
|------|------|
| **搜索码 (Search Key)** | 用于查找记录的属性或属性集 |
| **索引文件 (Index File)** | 包含 `<search-key, pointer>` 记录的文件 |
| **索引项 (Index Entry)** | 索引文件中的一条记录 |

## 索引的评价指标

| 指标 | 含义 |
|------|------|
| 支持的访问类型 | 等值查询? 范围查询? |
| 访问时间 | 找到目标数据多快? |
| 插入时间 | 插入新数据时更新索引多快? |
| 删除时间 | 删除数据时更新索引多快? |
| 空间开销 | 索引占用多少额外空间? |

## 两大索引分类

| 分类方式 | 类型 | 说明 |
|------|------|------|
| 按**顺序** | **顺序索引 (Ordered Index)** | 搜索码按顺序排列 |
| 按**顺序** | **散列索引 (Hash Index)** | 搜索码用散列函数均匀分布到桶中 |
| 按**与文件的关系** | **聚集索引 (Clustering/Primary)** | 文件按搜索码**物理排序** |
| 按**与文件的关系** | **辅助索引 (Nonclustering/Secondary)** | 索引顺序与文件物理顺序**不同** |

# 14.2 🔴🔴 顺序索引的核心概念 [必考]

## 14.2.1 聚集索引 vs 辅助索引

| | 聚集索引 (Clustering / Primary Index) | 辅助索引 (Secondary / Nonclustering Index) |
|---|---|---|
| 搜索码顺序 | **与文件物理顺序一致** | **与文件物理顺序不同** |
| 一个表能有几个？ | 最多**1个**（文件只能按一种顺序排列） | 可以有**多个** |
| 顺序扫描效率 | **高**（顺序读磁盘即可） | **低**（每次读一个索引项，然后跳到完全不相关的磁盘位置读数据） |
| 也叫 | 主索引 (Primary Index) | 非聚集索引 |

> **经典考题**（2006A-Q7）：为什么辅助索引的顺序扫描效率不高？
> 
> **答案**：因为辅助索引的搜索码顺序与文件物理存储顺序不一致。按索引顺序扫描时，每找到一个索引项就要跳到磁盘上另一个位置读实际数据，导致大量随机I/O，每次都可能需要一次新的磁盘寻道。而聚集索引因为文件顺排好了，可以连续读。

## 14.2.2 🔴 稠密索引 vs 稀疏索引 [必考]

| | 稠密索引 (Dense Index) | 稀疏索引 (Sparse Index) |
|---|---|---|
| 定义 | **每个**搜索码值都有一个索引项 | **只有部分**搜索码值有索引项 |
| 空间开销 | 大 | 小 |
| 维护开销 | 大 | 小 |
| 查找速度 | 快（直接定位） | 慢（需要继续在数据块中找） |
| 使用限制 | 无限制 | **只能在聚集索引上使用** |

> **为什么辅助索引必须是稠密的？**（经典考题 2006B-Q7）
>
> 因为辅助索引的搜索码顺序与文件物理顺序不同。如果辅助索引用稀疏索引，两个相邻的索引项之间被"跳过"的搜索码值，它们的记录分散在磁盘的各个位置，无法像聚集稀疏索引那样按顺序一块块读。必须为每个搜索码值都建索引项才能保证能定位到所有记录。

**记忆口诀**：聚集可稀疏可稠密，辅助必须是稠密。

## 14.2.3 多级索引 (Multilevel Index)

**问题**：如果索引很大（放不进内存），访问索引本身也需要多次磁盘I/O。

**解决**：对索引再建索引（外层索引是内层索引的稀疏索引）。

```
外层索引 (sparse on 内层索引) → 内层索引 (dense/sparse on 数据文件) → 数据文件
      ↑ 小，能放入内存                    ↑ 可能放不下                    ↑ 很大
```

**思想**：不断在上一层建稀疏索引，直到最顶层索引能放入内存。

## 14.2.4 复合搜索码 (Composite Search Key)

搜索码可以包含多个属性，如 `(dept_name, salary)`。

```
索引项按字典序排列：
(CS, 60000) → ptr
(CS, 75000) → ptr
(CS, 90000) → ptr
(EE, 55000) → ptr
...
```

**查询能力**：
- `dept_name = 'CS'` → 可以用（前缀匹配）
- `dept_name = 'CS' AND salary = 75000` → 可以用（精确匹配）
- `dept_name = 'CS' AND salary > 60000` → 可以用（范围查询）
- `salary = 75000` → **不能直接用**（不是前缀）

# 14.3 🔴🔴🔴 B+树索引 (B+-Tree) [必考操作题]

**这是全章最核心的内容，必须能独立完成插入和删除操作！**

## 14.3.1 B+树结构

```
                    ┌─────────┐
                    │  根节点   │  ← 非叶节点（除非是唯一的节点）
                    └─┬───┬───┘
              ┌───────┘   └───────┐
        ┌────┴────┐         ┌────┴────┐
        │  内节点  │         │  内节点  │  ← 非叶节点
        └─┬──┬──┬─┘        └─┬──┬──┬─┘
     ┌────┘  │  └────┐      │  │  └────┐
  ┌──┴──┐ ┌──┴──┐ ┌──┴──┐ ┌──┴──┐ ...
  │叶节点│ │叶节点│ │叶节点│ │叶节点│     ← 叶节点（相连成链表）
  └─────┘ └─────┘ └─────┘ └─────┘
```

### 节点结构

**非叶节点**：包含 n 个孩子指针和 n-1 个搜索码值
```
[P1 | K1 | P2 | K2 | P3 | ... | Kn-1 | Pn]
```
P1指向 `<K1` 的子树，P2指向 `>=K1且<K2` 的子树，以此类推。

**叶节点**：包含 n-1 个搜索码值和对应的记录指针，最后一个指针指向下一个叶节点。
```
[K1,ptr | K2,ptr | ... | Kn-1,ptr | P_next]
```

### B+树属性（必须牢记！）

| 参数 | 含义 |
|------|------|
| **n** | 每个节点的最大指针数（阶/order） |
| 根节点 | 至少 2 个孩子（除非是叶节点） |
| 非根非叶节点 | 至少 ⌈n/2⌉ 个孩子，最多 n 个孩子 |
| 叶节点 | 至少 ⌈(n-1)/2⌉ 个值，最多 n-1 个值 |
| **所有路径从根到叶长度相同** | 完美平衡！ |

### 高度计算

```
高度 ≤ ⌈log_⌈n/2⌉(K)⌉

其中 K 是搜索码值的总数
n=100时，100万个搜索码的高度最多为4层
（vs 二叉树的约20层深度）
```

## 14.3.2 B+树查询

**等值查询**：
```
function find(v):
    从根节点开始
    while 当前节点不是叶节点:
        找到最小的 i 使得 v < Ki
        如果 Ki 不存在（v ≥ 所有Ki），走 Pn
        否则走 Pi
    在叶节点中查找 v
    if 找到: 返回记录指针
    else: 返回 not found
```

**范围查询**：找到起始值所在的叶节点，然后沿叶节点链表向后遍历。

## 14.3.3 🔴🔴 B+树插入（操作题核心！）

### 算法步骤

1. **找到目标叶节点**：用 `find(v)` 定位
2. **如果叶节点有空间**（值数量 < n-1）：直接插入，保持有序
3. **如果叶节点已满**（值数量 = n-1）：**分裂！**
   - 前 ⌈n/2⌉ 个值留在原节点
   - 剩余值放入新节点
   - 将新节点的**最小搜索码值**插入父节点
4. **如果父节点也满**：父节点也分裂，分裂向上传播
5. **如果根节点分裂**：创建新根，B+树高度+1

### 插入示例（n=4 的B+树）

```
初始B+树(n=4,最多3个值):
根/叶: [Adams | Brandt | Califieri]

插入 "Adams"（重复）：直接插入叶节点
→ [Adams | Adams | Brandt | Califieri] 分裂！
  前⌈4/2⌉=2个在原节点: [Adams | Adams]
  后2个在新节点: [Brandt | Califieri]
  父节点插入 "Brandt"（新节点的最小值）

新的B+树:
         [Brandt]
        /         \
[Adams|Adams]  [Brandt|Califieri]
```

### 关键规则总结

| 规则 | 说明 |
|------|------|
| 分裂点 | 叶节点：前 ⌈n/2⌉ 个；非叶节点：⌈n/2⌉ 个孩子指针 |
| 向父节点插入 | 新节点的**最小值**（叶节点）或**中间值上移后不保留**（非叶节点） |
| 根分裂 | 创建仅含1个值的新根，高度+1 |

## 14.3.4 🔴🔴 B+树删除（操作题核心！）

### 算法步骤

1. **找到目标叶节点**并删除目标值
2. **如果节点仍然满足最小值要求**：完成
3. **如果节点元素太少**（值数量 < ⌈(n-1)/2⌉）：
   - **优先与兄弟合并**：如果邻接兄弟的值能合并装入一个节点（总数 ≤ n-1）
   - **否则从兄弟借**：从兄弟移一个值过来（重分布/redistribution）
4. **合并导致父节点少一个孩子**：父节点也需要调整，向上传播
5. **如果根只剩一个孩子**：删除根，高度-1

### 删除示例（n=4）

```
        [Brandt]
       /         \
[Adams]  [Brandt|Califieri]

删除 "Adams"：
  叶节点变空，合并兄弟
  父节点 "Brandt" 失效
  → [Brandt|Califieri] 成为新根，高度-1

新B+树:
[Brandt|Califieri]  ← 既是根也是叶
```

### 非唯一搜索码的处理

如果搜索码不是唯一的（如多个教师的dept_name相同），将搜索码变成复合键 `(search_key, unique_id)`，unique_id可以是主码或内部记录ID。

## 14.3.5 B+树复杂度

所有操作（查找、插入、删除）都是 O(log_⌈n/2⌉(K)) 次 I/O。

## 14.3.6 B+树批量加载 (Bulk Loading)

如果已经有大量数据，逐个插入效率低。更好的做法：
1. 按搜索码排序所有数据
2. 自底向上构建B+树（先建满叶节点，再建父节点）

# 14.4 B树 vs B+树 [重点]

| | B+树 | B树 |
|---|---|---|
| 搜索码存储 | 重复存储（非叶和叶都存） | 每个搜索码只出现一次 |
| 数据指针 | **只在叶节点** | **非叶节点也有** |
| 叶节点链表 | **有**（支持范围查询） | **无** |
| 扇出(Fan-out) | 更大（非叶节点项更小） | 更小（非叶节点含数据指针） |
| 树高 | 更矮 | 更高 |
| 范围查询 | 高效（链表遍历） | 低效（需要遍历树） |
| 实现难度 | 简单 | 复杂 |

> **记忆**：B+树 = B树的增强版，"+"在叶节点链表支持范围查询，且扇出更大、树更矮。

# 14.5 🔴 散列 (Hashing) [重点]

## 14.5.1 静态散列 (Static Hashing)

**原理**：散列函数 h(k) 将搜索码值映射到桶(Bucket)地址。

**桶溢出(Bucket Overflow)处理**：
- 溢出链（Overflow Chaining）：溢出的记录放在溢出桶中，用链表连接

**缺点**：桶数量固定
- 数据增长→桶溢出严重→性能退化
- 数据缩减→大量空桶→空间浪费

## 14.5.2 动态散列 (Dynamic Hashing)

**可扩展散列 (Extendable Hashing)**：
- 使用散列值的前缀（i位）来索引桶地址表
- 桶分裂时增加使用的位数
- 多个散列值可以共享同一个桶

## 14.5.3 散列 vs 顺序索引 [必考]

| | 散列索引 | 顺序索引 (B+树) |
|---|---|---|
| **等值查询** | **更好**（直接定位桶） | 好（树遍历） |
| **范围查询** | **差**（桶之间无序） | **更好**（叶节点链表） |
| 维护 | 桶分裂/合并 | 节点分裂/合并 |
| 存储有序性 | **无** | **有** |

> **记忆口诀**：等值用散列，范围用B+树。这就是为什么PostgreSQL不推荐散列索引，SQL Server只支持B+树。

# 14.6 写优化索引 (Write-Optimized Indices) [重点]

## LSM树 (Log-Structured Merge Tree)

**核心思想**：将随机写变成顺序写。

```
插入流程：
  新数据 → L0（内存树，如红黑树）
  当L0满了 → 合并到L1（磁盘，有序）
  当L1满了 → 合并到L2（更大，有序）
  ...
```

### LSM树特点

| 优点 | 缺点 |
|------|------|
| 插入是顺序I/O（极快） | 查询需要搜索所有层 |
| 叶节点始终满（无碎片） | 数据被多次复制（写放大） |
| I/O次数比B+树少 | 后台合并占用资源 |

**使用系统**：Google BigTable, Apache Cassandra, MongoDB, LevelDB, MyRocks (MySQL)

## 缓冲树 (Buffer Tree)
- 每个内部节点有插入缓冲区
- 缓冲区满了才向下推数据
- 查询开销比LSM树小，但随机I/O更多
- 用于PostgreSQL GiST索引

# 14.7 🔴 位图索引 (Bitmap Index) [重点]

## 适用场景

属性**不同值很少**的情况：性别（男/女）、国家、收入等级等。

## 工作原理

```
instructor表 (ID, name, gender, income_level)

gender位图:
  male:   [1, 0, 1, 1, 0, ...]
  female: [0, 1, 0, 0, 1, ...]

income_level位图:
  L1: [1, 0, 0, 1, ...]
  L2: [0, 1, 1, 0, ...]
  L3: [0, 0, 0, 0, ...]
```

## 查询方法

```sql
-- 查男性且收入L1
SELECT * FROM instructor WHERE gender = 'male' AND income_level = 'L1';

-- 用位图操作：
male_bitmap AND L1_bitmap
→ [1, 0, 1, 1, 0, ...] AND [1, 0, 0, 1, ...] = [1, 0, 0, 1, ...]
→ 第1和第4条记录
```

## 优点
- **极度紧凑**：如果记录100字节，每个位图只有关系的1/800大小
- 布尔运算极快（AND/OR/NOT都是按位操作）
- 计数查询非常快

# 14.8 空间索引 (Spatial Index) [了解]

| 索引类型 | 特点 |
|------|------|
| **k-d树** | 轮流按不同维度划分空间 |
| **四叉树 (Quadtree)** | 每个节点将区域分成4等份 |
| **R树 (R-tree)** | B+树的N维扩展，边界框可重叠 |

### PostgreSQL PostGIS 空间查询 [可能考]

```sql
-- 找包含某个点的所有区域
SELECT name FROM cities WHERE ST_Contains(geom, ST_Point(121.5, 31.2));

-- 找距离某点10公里内的城市
SELECT name FROM cities 
WHERE ST_Distance(geom, ST_Point(121.5, 31.2)) < 10000;
```

**常用函数**：`ST_Contains()`, `ST_Distance()`, `ST_Intersects()`, `ST_Within()`
