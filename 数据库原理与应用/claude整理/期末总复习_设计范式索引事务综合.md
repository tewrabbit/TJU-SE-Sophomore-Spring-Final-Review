## A.1 E-R模型核心要素复习

### 符号速查表（画图必记）

| 图形 | 含义 |
|------|------|
| 矩形 (Rectangle) | 实体集 (Entity Set) |
| 菱形 (Diamond) | 联系集 (Relationship Set) |

| **双线**矩形/菱形 | 弱实体集/标识联系 |
| 虚线椭圆 | 派生属性 (Derived Attribute) |
| 双椭圆 | 多值属性 (Multivalued Attribute) |
| **箭头** (→) 指向 | 该实体在联系中为"一"方 (one) |
| 无箭头线 | "多"方 (many) |
| **双线** | Total Participation（完全参与） |
| 单线 | Partial Participation（部分参与） |
| ISA三角形 | 特化/泛化 |

### 基数约束 (Cardinality Constraints)

```
一对一 (1:1): 两边都有箭头
  instructor ← advisor → student

一对多 (1:N): 箭头指向"一"方
  department ← dept_of → instructor
  (一个系有很多教师，一个教师只属于一个系)

多对一 (N:1): 同1:N，方向反过来

多对多 (M:N): 两边都无箭头
  student ← takes → section
```

### ER转关系模式 — 7种情况 [必考操作]

| 情况 | 转换规则 |
|------|---------|
| **1. 强实体集** | 每个强实体 → 一张表。复合属性展平，派生属性不存 |
| **2. 弱实体集** | 建表，主码 = 标识实体主码 + 分辨符，外码引用标识实体 |
| **3. 多值属性** | **单独建表**！属性 = 实体主码 + 多值属性，主码 = 全部 |
| **4. M:N联系集（无箭头两边）** | 建表，主码 = 两个实体主码的并集 |
| **5. 1:N联系集（箭头指向N方有total参与）** | **不建表**！把"一"方主码作为"多"方实体的外码 |
| **6. 1:N联系集（partial参与或多方无total）** | 建表（或仍用外码+允许NULL） |
| **7. 1:1联系集** | 任意一边作为"多"方处理，把对方主码当外码加进来 |
| **N元联系集(N≥3)** | 如果没有箭头，建表，主码 = 所有参与实体主码的并集 |
| **弱实体联系** | 不单独建表（弱实体本身已经包含标识实体主码） |

### 特化/泛化的SQL实现

**方法1**：高层一张表 + 低层各一张表（只存特有属性+主码）
**方法2**：只建低层的表（各含高层全部属性）

## A.2 🔴🔴🔴 规范化完整操作指南（本章分值最重！）

### Step 1: 计算属性闭包 α⁺ [必考操作]

**算法**：
```
result := α
repeat
    for each FD β → γ in F:
        if β ⊆ result:
            result := result ∪ γ
until result 不再变化
return result
```

**用途**：
- 测试 α 是否是超码：α⁺ 包含R的所有属性 → 是超码
- 检验 FD α→β：计算α⁺，如果β ⊆ α⁺，则α→β ∈ F⁺
- 找候选码：从包含所有属性的最小子集开始尝试

### Step 2: 找候选码 [必考操作]

**方法**：
1. 不在任何FD右边出现的属性 → 必须属于所有候选码
2. 尝试用这些属性计算闭包
3. 如果不够，逐步添加其他属性，直到闭包包含R的所有属性
4. 最小化：去掉任何一个都不行的属性集合

### Step 3: 计算正则覆盖 Fc [必考操作]

**定义**：Fc 满足——
- Fc 中的每个FD都不含无关属性
- Fc 中每个FD的左部唯一
- Fc⁺ = F⁺

**算法**：
```
Fc = F
# 第一步：标准化左部（合并相同左部）
将 α→β1, α→β2 合并为 α→β1β2

# 第二步：去除左部无关属性
for each FD α→β in Fc:
    for each A in α:
        如果 β ⊆ (α-A)⁺ under Fc:
            从α中删除A（即A是无关的）

# 第三步：去除右部无关属性
for each FD α→β in Fc:
    for each B in β:
        计算 α⁺ under Fc（但把当前FD的β换成β-B）
        如果 B ∈ α⁺:
            从β中删除B

# 第四步：删除F⁺中合并产生的冗余FD
```

**⚠️ 易错点**：每次只能删除一个无关属性！删除一个后，其他属性的无关性可能改变，必须重新检查！

### Step 4: 判断范式级别

| 范式 | 判定条件 |
|------|---------|
| **2NF** | 每个非主属性完全函数依赖于候选码（不能部分依赖） |
| **3NF** | 对每个非平凡FD α→β，要么 α是超码，要么β中的每个属性都在**某个候选码**中 |
| **BCNF** | 对每个非平凡FD α→β，α必须是超码 |
| **4NF** | 对每个非平凡MVD α→→β，α必须是超码 |

**范式层级**：1NF ⊃ 2NF ⊃ 3NF ⊃ BCNF ⊃ 4NF

### Step 5: 🔴 BCNF分解算法 [必考]

**算法**：
```
result := {R}  -- 初始只有原关系R
while 存在R_i ∈ result 不是BCNF:
    找到R_i中违反BCNF的FD α→β
    (即α⁺不包含R_i的所有属性)
    result := (result - {R_i}) ∪ {α∪β, R_i - (β-α)}
    -- 注意：分解为 R_i1(α∪β) 和 R_i2(R_i - (β-α))
end while
```

**⚠️ 易错点**：
- 对分解后的子关系R_i测试BCNF时，不能只用F中的FD，必须用**F⁺中限制在R_i上的FD**（即要用属性闭包方法检查每个α）
- 分解总是无损的，但**不一定依赖保持**
- 检查是否无损：R₁∩R₂ → R₁ 或 R₁∩R₂ → R₂ 是否在F⁺中

### Step 6: 🔴 3NF分解算法 [必考]

**算法**：
```
Fc = 正则覆盖(F)
result := {}
for each α→β in Fc:
    result := result ∪ {α∪β}
-- ⚠️ 关键步骤！如果所有R_i都不包含R的任何候选码
-- 必须额外添加一个包含候选码的关系
if 没有任何 R_i ∈ result 包含R的候选码:
    选R的任意一个候选码key
    result := result ∪ {key}
-- 去除冗余（如果某个R_i是另一个R_j的子集，删除R_i）
```

**3NF优点**：总是无损连接+依赖保持
**3NF缺点**：可能有冗余（这是为了保持依赖的代价）

### Step 7: 多值依赖(MVD)与4NF

**MVD定义**：X→→Y 成立，当且仅当对R的任一关系r，在(X,Z)（Z=R-X-Y）上的每个值对应于一组Y值，这组值只依赖于X而与Z无关。

**平凡MVD**：Y⊆X 或 X∪Y=R

**4NF判定**：对每个非平凡MVD X→→Y，X必须是超码。

**4NF分解**：用违反的MVD分解，和BCNF同样的方法。每个MVD对应一个独立的表。

# Part B: 索引 (Ch14)

## B.1 索引类型对比速查

| 对比维度 | 选项A | 选项B |
|------|------|------|
| 与文件顺序 | 聚集索引(一致) | 辅助索引(不一致) |
| 索引项密度 | 稠密索引(每个值) | 稀疏索引(间隔) |
| 顺序 vs 散列 | 顺序索引(B+树) | 散列索引 |
| 树类型 | B+树 | B树 |

## B.2 B+树操作速查表

### 插入流程
```
find(v) 找到叶节点
  ├── 有空间 → 直接插入
  └── 已满(n-1个值) →
      分裂: 前⌈n/2⌉在原节点, 后半在新节点
      新节点最小值插入父节点
      如父节点满 → 继续分裂向上
      如根分裂 → 创建新根, 高度+1
```

### 删除流程
```
find(v) 找到并删除
  ├── 仍有≥⌈(n-1)/2⌉个值 → 完成
  └── 不足 →
      如果兄弟可合并(总数≤n-1) → 合并, 父节点删除一个键
      否则从兄弟借一个值(重分布), 更新父节点
      如需调整父节点 → 向上传播
      如根变为空 → 删除根, 高度-1
```

## B.3 散列vs顺序索引 — 一句话决策

> **等值查询多用散列，范围查询用B+树。**

# Part D: 查询处理 (Ch15)

## D.1 五种Join算法决策树

```
需要什么类型的Join?
├── 等值连接/自然连接
│   ├── 两个表都已排序 → 归并连接 (Merge-Join)
│   ├── 小表能装入内存 → 散列连接 (Hash-Join)
│   ├── 大表在连接属性上有索引 → 索引嵌套循环连接
│   └── 两个都大且无索引 → 块嵌套循环 + 先用小表做外层
└── 非等值连接(如r.A > s.B)
    ├── 有索引 → 索引嵌套循环
    └── 无索引 → 块嵌套循环连接
```

## D.2 Join代价速查（块传输次数）

| 算法 | 最坏代价 | 最优场景代价 |
|------|---------|------------|
| 嵌套循环 | n_r×b_s + b_r | — |
| 块嵌套循环 | b_r×b_s + b_r | — |
| 索引嵌套循环 | b_r + n_r×c | — |
| 归并连接 | b_r+b_s + sort cost | b_r+b_s (已排序) |
| 散列连接 | 3(b_r+b_s) | b_r+b_s (全内存) |

## D.3 物化 vs 流水线

| | 物化 | 流水线 |
|---|---|---|
| 中间结果 | 写磁盘 | 直接传递 |
| I/O | 高 | 低 |
| 适用 | 任意操作 | 非阻塞操作 |
| 阻塞操作 | 排序/散列连接/聚集 | 不能流水线 |

## E.2 冲突可串行化判定步骤

```
Step 1: 列出所有操作，标识访问的数据项和事务编号
Step 2: 找出所有冲突操作对（同数据项 + 至少一个写）
Step 3: 对每对冲突，画边 T_先 → T_后
Step 4: 检查优先图：
  - 无环 → 冲突可串行化，拓扑排序得串行顺序
  - 有环 → 不是冲突可串行化
```

# Part F: 综合案例 — 从设计到实现

## 案例：大学选课系统

假设要设计一个大学选课系统，需求如下：
- 学生有学号、姓名、系别
- 课程有课号、名称、学分
- 学生可以选多门课，每门课可以被多个学生选
- 每次选课需要记录成绩和选课学期
- 教师有工号、姓名、工资，属于一个系
- 一位教师可以教多门课，一门课可由多位教师教（不同学期）

### ER图分析
```
student(ID, name, dept_name) ———— takes ———— section(course_id, sec_id, semester, year)
                                    |
                                   grade (属性)
instructor(ID, name, dept_name, salary) ———— teaches ———— section
                                    
course(course_id, title, credits) ———— section
department(dept_name, building, budget) ———— student, instructor
```

- student-takes-section: M:N（一个学生选多个section，一个section有多个学生）
- instructor-teaches-section: M:N（类似）
- course-section: 1:N（一门课有多个section）
- department-student: 1:N
- department-instructor: 1:N

### 关系模式
```sql
department(dept_name, building, budget, PRIMARY KEY(dept_name))
student(ID, name, dept_name, tot_cred, 
        PRIMARY KEY(ID), FOREIGN KEY(dept_name) REFERENCES department)
instructor(ID, name, dept_name, salary,
           PRIMARY KEY(ID), FOREIGN KEY(dept_name) REFERENCES department)
course(course_id, title, credits, dept_name,
       PRIMARY KEY(course_id), FOREIGN KEY(dept_name) REFERENCES department)
section(course_id, sec_id, semester, year,
        PRIMARY KEY(course_id, sec_id, semester, year),
        FOREIGN KEY(course_id) REFERENCES course)
takes(ID, course_id, sec_id, semester, year, grade,
      PRIMARY KEY(ID, course_id, sec_id, semester, year),
      FOREIGN KEY(ID) REFERENCES student,
      FOREIGN KEY(course_id, sec_id, semester, year) REFERENCES section)
teaches(ID, course_id, sec_id, semester, year,
        PRIMARY KEY(ID, course_id, sec_id, semester, year),
        FOREIGN KEY(ID) REFERENCES instructor,
        FOREIGN KEY(course_id, sec_id, semester, year) REFERENCES section)
```

### B+树索引建议
```sql
-- 主码上自动有索引（聚集索引）
```

### 查询分析示例

```sql
-- Q: 查出CS系在2024年秋季所有课程的平均成绩
-- 查询计划分析：
SELECT C.course_id, C.title, AVG(T.grade)
FROM course C
JOIN section S ON C.course_id = S.course_id
JOIN takes T ON S.course_id = T.course_id 
    AND S.sec_id = T.sec_id 
    AND S.semester = T.semester 
    AND S.year = T.year
WHERE C.dept_name = 'CS' 
  AND S.semester = 'Fall' 
  AND S.year = 2024
GROUP BY C.course_id, C.title;

-- 分析：
-- 1. course上用dept_name索引选CS的课
-- 2. section上用(course_id, semester, year)复合索引
-- 3. takes上用(course_id, sec_id, semester, year)索引嵌套循环
-- 4. 散列连接或归并连接（看哪个表大）
-- 5. 按course_id分组做聚集
```
