# 17.1 🔴🔴 事务概念与ACID [必考]

## 事务是什么？

事务是访问并可能更新数据的一个**程序执行单元**。

### 经典示例：转账

```
T: 从账户A转$50到账户B

read(A)      -- 读A的余额
A := A - 50  -- 扣$50
write(A)     -- 写回A
read(B)      -- 读B的余额
B := B + 50  -- 加$50
write(B)     -- 写回B
```

**一致性约束**：A+B的总和保持不变，且A不能是负数。

## 🔴 ACID 四大特性 [必考必背]

| 字母 | 全称 | 含义 | 违反会导致... | 实现方式 |
|------|------|------|-------------|---------|
| **A** | Atomicity<br>原子性 | 要么全做，要么全不做 | 系统在A扣了钱但没给B加钱时崩溃→钱凭空消失 | 日志(Log)+恢复(Recovery) |
| **C** | Consistency<br>一致性 | 事务执行前后数据库保持一致状态 | 约束被破坏（如A变成负数） | 程序逻辑+完整性约束 |
| **I** | Isolation<br>隔离性 | 并发事务互不干扰，感觉不到其他事务存在 | 脏读、不可重复读、丢失修改 | 并发控制(Concurrency Control) |
| **D** | Durability<br>持久性 | 已提交的事务更改永久保留，即使系统崩溃 | 系统重启后提交的数据丢失 | 日志(Log)+非易失存储 |

> **真题**(2006B-Q9)：解释事务的ACID特性。
> 
> **答题要点**：先展开全称，再用转账例子说明每一点。

### 原子性与持久性的区别 [易错点]

| | 原子性 | 持久性 |
|---|---|---|
| 关注 | **事务执行过程中**失败 | **事务提交后**数据不丢 |
| 时间点 | 事务提交**前** | 事务提交**后** |
| 机制 | UNDO/回滚(Rollback) | REDO/重做 |

# 17.2 🔴 事务状态图 [必考]

```
                    ┌─────────┐
                    │  Active  │ ← 初始状态：事务正在执行
                    │ (活动的)  │
                    └────┬─────┘
                         │
               ┌─────────┴─────────┐
               ↓                   ↓
        ┌─────────────┐    ┌─────────────┐
        │   Partially  │    │   Failed    │ ← 无法继续正常执行
        │  Committed   │    │  (失败的)    │    (检测到死锁/违反约束/系统错误)
        │ (部分提交的)  │    └──────┬──────┘
        └──────┬──────┘           │
               │                  ↓
               │          ┌─────────────┐
               ↓          │   Aborted   │ ← 回滚完成，恢复到事务开始前状态
        ┌─────────────┐   │  (中止的)    │
        │  Committed   │   └──────┬──────┘
        │  (提交的)     │          │
        └─────────────┘           ↓
                          重新开始(Restart)
                          或终止(Kill)
```

### 五个状态的触发条件 [必考]

| 状态转移 | 触发条件 |
|------|------|
| Active → **Partially Committed** | 最后一条语句执行完毕 |
| Active → **Failed** | 事务执行出错/检测到死锁/违反约束/系统崩溃 |
| Partially Committed → **Committed** | 日志成功写入磁盘，更改持久化 |
| Partially Committed → **Failed** | 日志写入过程中硬件故障 |
| Failed → **Aborted** | 回滚(Rollback)完成，撤销所有更改 |
| Aborted → 重新开始 | 事务重启（如果错误不是永久性的） |

# 17.3 并发执行 [重点]

## 为什么要并发？

| 优点 | 说明 |
|------|------|
| **提高吞吐量(Throughput)** | CPU和磁盘可并行工作（一个事务等I/O时，另一个可用CPU） |
| **减少平均响应时间** | 短事务不用等长事务执行完 |

## 调度 (Schedule)

**定义**：指定了来自并发事务的指令按时间顺序执行的序列。

**要求**：一个事务内的指令顺序必须保持（不能把write(A)放到read(A)之前）。

```
串行调度 (Serial Schedule):
  T1全部指令 → T2全部指令
  问题：没错但慢，没利用并发

并发调度 (Concurrent Schedule):
  T1指令、T2指令交替执行
  目标：找到和某个串行调度效果相同的并发调度
```

# 17.4 🔴🔴🔴 冲突可串行化 (Conflict Serializability) [必考操作题]

## 冲突指令

两条指令**冲突**的条件（必须同时满足）：
1. 它们访问**同一个数据项**
2. 其中**至少有一条是write**

| 操作组合 | 是否冲突 | 原因 |
|------|---------|------|
| read(A), read(A) | **不冲突** | 两个读互不影响 |
| read(A), write(A) | **冲突** | 读到写前还是写后的值？ |
| write(A), read(A) | **冲突** | 同上 |
| write(A), write(A) | **冲突** | 最终写的是哪个值？ |

> **关键理解**：冲突意味着**顺序重要**。如果两条指令冲突，交换它们会改变结果或数据库状态。

## 冲突等价

两个调度是**冲突等价(Conflict Equivalent)**的，如果：
- 它们涉及同一组事务的同一组操作
- 每对冲突操作的顺序相同

即：可以通过交换**不冲突**的相邻操作来互相转换。

## 冲突可串行化

一个调度S是**冲突可串行化(Conflict Serializable)**的，当且仅当：
- S冲突等价于某个**串行调度**

**等价条件**：S的**优先图(Precedence Graph)无环**。

## 🔴 优先图判定法（操作题步骤）[必考]

### 步骤

1. **画节点**：每个事务一个节点
2. **画边**：对每对冲突操作，如果T_i的操作在T_j之前，画 T_i → T_j
3. **判断**：
   - 优先图**无环** → **是**冲突可串行化
   - 优先图**有环** → **不是**冲突可串行化

### 如何画边（核心规则）

对于每个数据项Q的每对冲突操作：
```
如果 T_i 的 read(Q) 在 T_j 的 write(Q) 之前 → T_i → T_j
如果 T_i 的 write(Q) 在 T_j 的 read(Q) 之前 → T_i → T_j
如果 T_i 的 write(Q) 在 T_j 的 write(Q) 之前 → T_i → T_j
```

**实用技巧**：只需检查相邻的访问相同数据的不同事务操作。

### 得到串行顺序

对无环优先图做**拓扑排序(Topological Sort)**，即可得到等价的串行顺序。

### 完整示例

```
调度 S:
T1: read(A), write(A)
T2: read(A), write(A), read(B), write(B)
T3: read(B), write(B)

具体交错顺序:
T1: read(A)   -- ①
T1: write(A)  -- ②
T2: read(A)   -- ③
T2: write(A)  -- ④
T3: read(B)   -- ⑤
T3: write(B)  -- ⑥
T2: read(B)   -- ⑦
T2: write(B)  -- ⑧

分析A上的冲突操作:
  T1的write(A)② vs T2的read(A)③ → T1→T2 (因为②在③之前)
  T1的write(A)② vs T2的write(A)④ → T1→T2

分析B上的冲突操作:
  T3的write(B)⑥ vs T2的read(B)⑦ → T3→T2 (因为⑥在⑦之前)
  T3的write(B)⑥ vs T2的write(B)⑧ → T3→T2

优先图: T1 → T2, T3 → T2
         T1 ──→ T2 ←── T3

无环 → 冲突可串行化
拓扑排序 → 串行顺序: T1, T3, T2 (或 T3, T1, T2)
```

> **真题**(2006C-Q10)：判断给定调度是否冲突可串行化，如果是请给出串行顺序。

# 17.5 视图可串行化 (View Serializability) [了解]

## 视图等价

两个调度S和S'是**视图等价(View Equivalent)**的，如果：
1. 初始读相同（每个数据项的第一个读来自同一个事务）
2. 读-写关系相同（每个数据项，每个事务读的是哪个事务写的）
3. 最终写相同（每个数据项的最后一个写来自同一事务）

## 视图可串行化

S是**视图可串行化**的，如果它视图等价于某个串行调度。

### 与冲突可串行化的关系

```
冲突可串行化 ⊂ 视图可串行化

即：所有冲突可串行化的调度都是视图可串行化的
    但反过来不一定！
```

**反例（视图可串行化但不是冲突可串行化的特征）**：包含**盲写(Blind Write)**——不读取就直接写入。

```
T1: write(A)   -- 盲写（没有先read(A)）
T2: write(A)   -- 也是盲写
```

> 注意：测试视图可串行化是**NP-complete**问题，实际系统中使用冲突可串行化。

# 17.6 🔴 可恢复性 (Recoverability) [重点]

## 可恢复调度 (Recoverable Schedule)

对于一个调度，如果T_j读取了T_i写入的数据，则T_i的提交必须在T_j的提交**之前**。

**为什么重要？** 如果T_i后来失败了需要回滚（UNDO），T_j读的"脏数据"也需要回滚。但如果T_j已经提交了，就无法回滚了（持久性）→ 矛盾！

## 级联回滚 (Cascading Rollback)

一个事务失败导致一系列其他事务必须回滚。

```
T1: write(A) → ... → T1失败, UNDO A
T2: read(A), write(B) → 读了T1的脏数据A, 现在也必须回滚
T3: read(B), write(C) → 读了T2的脏数据B, 也必须回滚
...
```

## 无级联调度 (Cascadeless Schedule)

T_j只能在T_i提交**之后**才能读取T_i写入的数据。

```
无级联调度 ⊆ 可恢复调度
（每个无级联调度都是可恢复的，反过来不一定）
```

## 隔离级别与可恢复性

理想情况是**无级联**的调度——这是实际数据库实现的目标。

# 17.7 🔴 隔离级别 (Isolation Levels) [必考]

SQL标准定义了四个隔离级别：

| 级别 | 英文 | 脏读(Dirty Read) | 不可重复读(Non-repeatable) | 幻读(Phantom) |
|------|------|:---:|:---:|:---:|
| **可串行化** | Serializable | 否 | 否 | 否 |
| **可重复读** | Repeatable Read | 否 | 否 | 是 |
| **已提交读** | Read Committed | 否 | 是 | 是 |
| **未提交读** | Read Uncommitted | 是 | 是 | 是 |

### 三个问题解释

| 问题 | 英文 | 现象 |
|------|------|------|
| **脏读** | Dirty Read | T1读了T2未提交的修改，T2后来回滚了 |
| **不可重复读** | Non-repeatable Read | T1读了一次，T2修改后提交，T1再读得到不同值 |
| **幻读** | Phantom | T1按条件查了一批记录，T2插入了符合条件的新记录并提交，T1再查发现多了记录 |

> **SQL默认隔离级别**：大多数数据库默认是Read Committed
> **Oracle使用快照隔离(Snapshot Isolation)**，不是SQL标准的四个之一

## SQL语法

```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- 或
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

# 17.8 并发控制的其他概念 [了解]

## 并发问题（除了脏读、不可重复读、幻读之外）

| 问题 | 说明 |
|------|------|
| **丢失修改(Lost Update)** | T1和T2同时读，T1改完写回被T2的写回覆盖 |
| **写偏斜(Write Skew)** | 两个事务读相同数据，但修改不同数据，违反全局约束 |

## 锁协议（从学长资料提及，可能在考题中出现）

| 锁协议 | 要求 |
|------|------|
| **两阶段锁(2PL)** | 增长阶段只能加锁不能解锁，缩减阶段只能解锁不能加锁 |
| **严格两阶段锁(Strict 2PL)** | 排他锁在事务提交后才释放 |
| **强两阶段锁(Rigorous 2PL)** | 所有锁在事务提交后才释放 |

## 死锁 vs 饥饿

| | 死锁(Deadlock) | 饥饿(Starvation) |
|---|---|---|
| 定义 | 互相等待对方持有的锁，谁都无法继续 | 一个事务永远得不到所需资源 |
| 涉及 | 至少两个事务 | 可能只是一个事务 |
| 检测 | 等待图(Wait-for Graph)中检测环 | — |
| 解决 | 选择牺牲者回滚 | 公平调度(FCFS) |

## 死锁预防方法

| 方法 | 原理 |
|------|------|
| 基于图的协议 | 规定加锁顺序，避免环 |
| 基于时间戳(Wait-Die / Wound-Wait) | 用时间戳决定等待还是回滚 |
| 锁超时(Lock Timeout) | 等太久就认为死锁，回滚 |
