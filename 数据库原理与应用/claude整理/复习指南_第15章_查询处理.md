```
SQL文本 → 语法解析 → 生成查询计划 → 优化 → 执行 → 返回结果
```

# 15.1 查询处理的基本步骤

```
┌─────────────┐
│ SQL语句文本   │
└──────┬──────┘
       ↓
┌─────────────┐
│ 解析与翻译    │  ← Parser: 检查语法, 翻译成关系代数表达式
│ (Parser)     │
└──────┬──────┘
       ↓
┌─────────────┐
│ 优化          │  ← Optimizer: 生成多个等价查询计划, 选代价最小的
│ (Optimizer)  │
└──────┬──────┘
       ↓
┌─────────────┐
│ 执行          │  ← Executor: 按查询计划逐步执行, 返回结果
│ (Evaluation) │
└─────────────┘
```

# 15.2 🔴 查询代价度量 [必考]

## 代价公式

```
总代价 = b_r × t_T + S × t_S

其中:
  b_r  = 传输的磁盘块数 (block transfers)
  t_T  = 传输一个块的时间
  S    = 磁盘寻道次数 (seeks)
  t_S  = 一次寻道的时间
```

### 典型参数值

| 参数 | HDD典型值 | SSD典型值 |
|------|-----------|----------|
| t_S (寻道) | 4 ms | 20-90 μs |
| t_T (传输) | 0.1 ms (4KB块) | 2-10 μs |

**关键理解**：HDD上 t_S ≫ t_T（寻道比传输贵40倍），所以优化主要目标是**减少寻道次数**。SSD上两者差距小得多。

**我们考试中通常忽略CPU代价**，只计算磁盘I/O代价。同时通常假设写操作的代价等于读操作的代价。

# 15.3 🔴 选择操作 (Selection) [重点]

## 各种选择算法的代价分析

假设关系r有 b_r 个块，每条记录等大小。

### A1: 线性扫描 (Linear Search)

扫描每个块，检查所有记录。

| 场景 | 代价 (块传输) | 寻道次数 |
|------|-------------|---------|
| 任意选择 | b_r | 1 |
| 码等值查询 (可提前停) | b_r / 2 (平均) | 1 |

### A2: 聚集索引 + 码等值查询

| 代价构成 | 公式 |
|------|------|
| 块传输 | h_i + 1 |
| 寻道 | h_i + 1 |

> h_i = B+树高度，+1是读实际数据块

### A3: 聚集索引 + 非码等值查询

| 代价构成 | 公式 |
|------|------|
| 块传输 | h_i + 1 + b |
| 寻道 | h_i + 1 + 1 |

> b = 包含匹配记录的块数（可以一次连续读多个块）

### A4: 辅助索引 + 等值查询

| 场景 | 块传输 | 寻道 |
|------|--------|------|
| 码等值查询 | h_i + 1 | h_i + 1 |
| 非码等值查询 | h_i + n | h_i + n |

> n = 匹配记录数。**每条匹配记录都可能导致一次寻道**（因为数据块不连续）！

**这就是辅助索引可能非常慢的原因**：如果有10000条匹配记录，就需要约10000次寻道。

### 复杂选择的处理

| 类型 | 策略 |
|------|------|
| **合取 (AND)** | ①用一个索引找到最小结果集，再测试其他条件<br>②用复合索引<br>③用多个索引取交集 |
| **析取 (OR)** | ①分别用索引找再求并集<br>②如果任何条件没有索引 → 线性扫描 |
| **取反 (NOT)** | 通常只能线性扫描 |

## Bitmap Index Scan (PostgreSQL)

结合了辅助索引和线性扫描的优点：
1. 先扫描辅助索引，标记包含目标数据的页（每个页1 bit）
2. 然后线性扫描，但**只读被标记的页**

这样避免了辅助索引对每条匹配记录都做一次寻道。

# 15.4 🔴 排序 (Sorting) [重点]

## 内存够用时
用快速排序(Quicksort)等标准算法。

## 内存不够用时：外部归并排序 (External Sort-Merge)

### 阶段1：创建有序归并段 (Create Sorted Runs)

```
读入 M 个块到内存 → 排序 → 写回磁盘 → 形成一个归并段
重复直到所有数据都被处理
共产生 ⌈b_r / M⌉ 个归并段
```

### 阶段2：归并 (Merge)

```
每次从 M-1 个归并段各读1块，归并输出到1个输出缓冲区
→ 共使用 M 个缓冲区
→ M-1 路归并

如果归并段数 ≥ M，需要多轮归并
每轮归并把归并段数减少为原来的 1/(M-1)
```

### 代价分析

总块传输 ≈ b_r × (2 × ⌈log_{M-1}(b_r / M)⌉ + 1)

> **记忆**：每轮读一次写一次 = 2×b_r，最后输出不用写回，所以+1不是+2

# 15.5 🔴🔴🔴 连接操作 (Join) [必考！]

这是全章最重要的内容，五种连接算法必须能说出各自的过程、代价和适用场景。

## 15.5.1 嵌套循环连接 (Nested-Loop Join)

```
for each tuple t_r in r:
    for each tuple t_s in s:
        if t_r 和 t_s 满足连接条件:
            输出 (t_r, t_s)
```

| 代价 | 公式 |
|------|------|
| 块传输(最坏) | n_r × b_s + b_r |
| 寻道 | n_r + b_r |

> n_r = r的元组数，b_s = s的块数

> **效率极低**：r中每一条记录都要扫描s一遍！

## 15.5.2 块嵌套循环连接 (Block Nested-Loop Join) [必考]

```
for each block B_r in r:
    for each block B_s in s:
        for each tuple t_r in B_r:
            for each tuple t_s in B_s:
                if 匹配: 输出
```

| 代价 | 公式 |
|------|------|
| 块传输(最坏) | b_r × b_s + b_r |
| 寻道 | 2 × b_r |

### 优化：用更多内存

如果有M个缓冲区：用M-2个块做外层，1个做内层，1个做输出。

| 代价 | 公式 |
|------|------|
| 块传输 | ⌈b_r / (M-2)⌉ × b_s + b_r |

**关键**：把较小的关系作为内层(inner)，可以减少内层扫描次数。

## 15.5.3 索引嵌套循环连接 (Indexed Nested-Loop Join) [重点]

```
for each tuple t_r in r:
    用 t_r 的连接属性值在 s 的索引中查找匹配记录
```

| 代价 | 公式 |
|------|------|
| 块传输 | b_r + n_r × c |
| 寻道 | 取决于索引 |

> c = 在s的索引中查找一条记录的代价

**适用**：当r很小而s很大且有索引时最佳。

## 15.5.4 🔴 归并连接 (Merge-Join) [必考]

```
步骤1: 将 r 和 s 都按连接属性排序
步骤2: 两个指针分别遍历r和s，遇到匹配的就输出

类似合并两个有序数组的归并过程
```

| 前提 | 两个关系必须按连接属性排序 |
|------|-------------------------|

| 代价 | 公式 |
|------|------|
| 排序 + 归并 | sort(r) + sort(s) + b_r + b_s |

**适用**：只适用于**等值连接和自然连接**，两个表已经排好序时最优。

## 15.5.5 🔴 散列连接 (Hash-Join) [必考]

```
步骤1 (Partition): 用相同散列函数h将r和s都分成n个分区
       → 分区r_i和s_i对应相同的散列值
步骤2 (Build + Probe): 
       for 每个分区i:
           将较小的s_i装入内存，建立散列索引
           然后扫描r_i，用散列索引查找匹配
```

### 关键概念

| 术语 | 含义 |
|------|------|
| **Build Input** | 较小的那个关系（构建散列表） |
| **Probe Input** | 较大的那个关系（探测查找） |
| **递归分区** | 如果一个分区放不进内存，需要再散列成更小的分区 |

### 代价

| 场景 | 块传输 |
|------|--------|
| 无递归 | 3(b_r + b_s) + 4×n_h |
| 全在内存 | b_r + b_s |

> 3(b_r + b_s) = 原始写分区(1次读1次写=2) + Build+Probe(读=1) = 3次

### 混合散列连接 (Hybrid Hash-Join)

将Build Input的第一个分区保留在内存中，减少一次写回。

## 五种连接算法对比总结 [必背]

| 算法 | 代价(块传输) | 适用条件 | 
|------|-------------|---------|
| **嵌套循环** | n_r×b_s + b_r | 任何条件，通用但最慢 |
| **块嵌套循环** | b_r×b_s + b_r | 任何条件，比嵌套循环好 |
| **索引嵌套循环** | b_r + n_r×c | s的连接属性上有索引 |
| **归并连接** | sort(r)+sort(s)+b_r+b_s | 等值连接，最好已有序 |
| **散列连接** | 3(b_r+b_s) | 等值连接，至少一个能装入内存 |

# 15.6 其他操作

## 去重 (Duplicate Elimination)
- **排序法**：排序后相同值相邻，扫描时跳过
- **散列法**：散列后同值在同一桶，桶内去重

## 投影 (Projection)
先投影，再去重（因为投影可能产生重复）。

## 聚集 (Aggregation)
- **排序法**：按GROUP BY属性排序，然后每组计算
- **散列法**：按GROUP BY属性散列，桶内计算
- **部分聚集优化**：先部分聚集再合并（如先算各组sum+count，最后算avg）

## 集合操作 (UNION / INTERSECT / EXCEPT)
变体版的归并连接或散列连接。

## 外连接 (Outer Join)
内连接 + 补充没有匹配的元组（用NULL填充）。归并连接和散列连接都可以修改来支持外连接。

# 15.7 🔴 表达式求值：物化 vs 流水线 [必考]

## 物化 (Materialization)

```
每个操作独立执行 → 中间结果写入磁盘 → 下一个操作读中间结果
```

| 优点 | 缺点 |
|------|------|
| 始终可用 | I/O代价高（中间结果写入磁盘） |

**优化**：双缓冲(Double Buffering)——一个缓冲区在写入时，另一个在计算下一批结果。

## 流水线 (Pipelining)

```
元组产生后立即传给下一个操作，不存中间结果
```

| 类型 | 说明 |
|------|------|
| **需求驱动 (Demand-Driven / Pull / Lazy)** | 顶层不断调用`next()`向底层要下一个元组 |
| **生产者驱动 (Producer-Driven / Push / Eager)** | 底层产生元组后向上推送给父操作 |

**实现方式**：迭代器模式 (Iterator Pattern)
- `open()` — 初始化
- `next()` — 返回下一个元组
- `close()` — 清理

## 阻塞操作 (Blocking Operations)

**不能产生任何输出直到消耗完所有输入**，因此不能用于流水线：
- 排序
- 散列连接
- 聚集（需要所有输入才能输出最终结果）

## 物化 vs 流水线对比

| | 物化 | 流水线 |
|---|---|---|
| I/O代价 | 高（写+读中间结果） | 低（不存盘） |
| 内存需求 | 低 | 高 |
| 适用性 | 任何操作 | 非阻塞操作之间 |
| 并行性 | 易实现 | 需协调 |
