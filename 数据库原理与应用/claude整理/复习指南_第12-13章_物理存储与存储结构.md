# 第12章：Physical Storage Systems（物理存储系统）

## 12.1 🔴 存储层次（Storage Hierarchy）[必考]

> 从上到下：速度递减，容量递增，价格递减，易失性递减

```
┌──────────────────────────────────────────────┐
│  一级存储 (Primary / Volatile)                │
│  ├── Cache（高速缓存）    最快/最贵/最小        │
│  └── Main Memory（主存）  DRAM                │
├──────────────────────────────────────────────┤
│  二级存储 (Secondary / On-line / Non-volatile) │
│  ├── Flash Memory（闪存） SSD                 │
│  └── Magnetic Disk（磁盘） HDD                │
├──────────────────────────────────────────────┤
│  三级存储 (Tertiary / Off-line)               │
│  ├── Optical Storage（光盘）                  │
│  └── Tape Storage（磁带）   最慢/最便宜/最大   │
└──────────────────────────────────────────────┘
```

**考试要点**：能说出每一层的代表介质、速度比较、是否易失。

## 12.2 存储接口与网络存储 [了解]

| 接口 | 速率 | 适用 |
|------|------|------|
| SATA | 6 Gbps | 消费级HDD/SSD |
| SAS | 12 Gbps | 企业级 |
| NVMe (PCIe) | 24+ Gbps | 高性能SSD |

| 存储网络 | 全称 | 说明 |
|------|------|------|
| **SAN** | Storage Area Network | 块级存储网络 |

## 12.3 🔴 磁盘结构与性能指标 [重点]

### 磁盘物理结构

```
磁盘由多个盘片(Platter)组成
每个盘片表面有同心圆形磁道(Track)
每个磁道被分成扇区(Sector) ← 读写的最小单位（通常512字节）
柱面(Cylinder)：所有盘片同一编号的磁道集合
```

### 磁盘性能核心公式 [必考]

```
访问时间 (Access Time) = 寻道时间 (Seek Time) + 旋转延迟 (Rotational Latency)
```

| 指标                            | 含义           | 典型值(HDD)                 |
| ----------------------------- | ------------ | ------------------------ |
| **寻道时间 (Seek Time)**          | 磁臂移动到目标磁道的时间 | 4-10 ms                  |
| **旋转延迟 (Rotational Latency)** | 目标扇区转到磁头下的时间 | 4-11 ms (5400-15000 RPM) |
| **总访问时间**                     | 两者之和         | 5-20 ms                  |
| **数据传输率**                     | 连续读写的速度      | 25-200 MB/s              |
| **IOPS**                      | 每秒I/O操作数     | 50-200 (HDD)             |
| **MTTF**                      | 平均故障时间       | 3-5年                     |

> **关键理解**：磁盘操作中最贵的是机械移动（寻道+旋转），而不是数据传输本身。这就是为什么数据库设计千方百计减少随机I/O。

### 磁盘臂调度（Elevator Algorithm / SCAN算法）[重点]

**问题**：多个I/O请求同时到达，磁臂应先去哪个磁道？

**电梯算法**：磁臂从最内圈扫描到最外圈（或反向），沿途处理所有请求，到头后折返。就像电梯一样，朝一个方向走到底再反向。

**目的**：最小化磁臂移动距离，提高吞吐量。

## 12.4 Flash / SSD [重点]

### NAND Flash 基本原理

| 操作 | 粒度 | 速度 |
|------|------|------|
| **读 (Read)** | 页(Page), 512B-4KB | 20-100 μs/页 |
| **写 (Write)** | 页 | 必须先擦除再写 |
| **擦除 (Erase)** | 擦除块(Erase block), 256KB-1MB | 2-5 ms |

**关键约束**：页只能写一次，必须擦除后才能重新写入。这是SSD的"写入放大"和性能衰减的根源。

### 关键技术

| 技术                                | 作用                                               |
| --------------------------------- | ------------------------------------------------ |
| **Flash Translation Layer (FTL)** | 逻辑页地址→物理页地址映射，隐藏擦除操作                             |
| **Wear Leveling（磨损均衡）**           | 均匀分布擦除次数，因为每个擦除块擦除10万-100万次后不可靠                  |
| **SSD性能**                         | 4KB随机读：1万-35万 IOPS；顺序读：400MB/s(SATA)-3GB/s(NVMe) |

### HDD vs SSD 对比 [重点]

| | HDD | SSD |
|---|---|---|
| 随机读 | 慢（机械寻道） | 快（无机械部件） |
| 顺序读写 | 中等 | 快 |
| 写入耐久 | 理论上无限 | 有限（擦除次数） |
| 价格/容量 | 便宜 | 贵 |
| 适合 | 大量顺序读写、大容量存储 | 随机读写、OLTP |

## 12.5 🔴🔴 RAID（冗余磁盘阵列）[必考]

### 为什么需要RAID？

多块磁盘一起用，但同时使用意味着故障概率也倍增（N块盘的MTTF = 单盘MTTF / N），所以需要**冗余**。

### RAID核心三目标
1. **高容量**（多块盘）
2. **高速度**（并行读写）
3. **高可靠性**（冗余容错）

### 两种冗余技术

| 技术 | 说明 | 写入放大 |
|------|------|---------|
| **镜像 (Mirroring)** | 每块盘完整复制一份 | 2倍写入 |
| **奇偶校验 (Parity)** | 用校验信息恢复数据 | 校验计算开销 |

### 分条 (Striping)
- **比特级分条**：1字节的8位分散在8块盘上
- **块级分条**：第i个块放在第(i mod n)块盘上

### RAID级别对比表（最常考！）

| RAID级别 | 技术 | 最少盘数 | 可靠性 | 空间效率 | 写性能 | 适用场景 |
|------|------|---------|------|---------|------|------|
| **RAID 0** | 块分条，无冗余 | 2 | 无容错 | 100% | 最好 | 数据安全不重要 |
| **RAID 1** | 镜像+块分条 | 2 | 高(可坏1块) | 50% | 最好 | 随机小更新、日志 |
| **RAID 4** | 块分条+专用校验盘 | 3 | 中(可坏1块) | 75%+ | 差(校验盘瓶颈) | 不常用 |
| **RAID 5** | 块分条+分散校验 | 3 | 中(可坏1块) | 75%+ | 中等 | 顺序大写入 |
| **RAID 6** | 块分条+双校验(P+Q) | 4 | 高(可坏2块) | 67%+ | 较低 | 数据安全极其重要 |

### RAID选择建议 [重点]
- RAID 0：数据不重要，只要速度
- RAID 1：随机/小更新多，写性能要求高（如OLTP日志）
- RAID 5：顺序/大写入多，存储成本敏感（如数据仓库）
- RAID 6：数据安全最优先，能容忍2块盘同时坏

> **易错点**：RAID 4不常用是因为**专用校验盘成为写瓶颈**，每次写都要更新校验盘。RAID 5解决了这个问题（分散校验）。

## 12.6 磁盘块访问优化 [了解]

| 技术 | 说明 |
|------|------|
| **缓冲 (Buffering)** | 在内存中缓存磁盘块 |
| **预读 (Read-ahead)** | 提前读取后续块 |
| **电梯算法** | 磁盘臂调度优化 |
| **连续分配** | 文件存放在连续的盘区(extent)，减少寻道 |
| **非易失写缓冲** | 用NVRAM加速写入 |
| **日志磁盘** | 顺序写入日志，再批量更新数据 |

## 12.7 磁带存储 [了解]

- 顺序访问（不能随机访问）
- 容量大（1-12TB/盘），价格低
- 用于归档和备份
- 磁带库（jukeboxes）可管理PB级数据

# 第13章：Data Storage Structures（数据存储结构）

## 13.1 文件组织基础

**层次关系**：
```
数据库 → 多个文件(File) → 多个记录(Record) → 多个字段(Field)
              ↓
         块(Block/Page) — 数据传输和存储分配的基本单元
```

## 13.2 定长记录 (Fixed-Length Records) [重点]

**简单方案**：第i条记录从第 n×(i-1) 字节处开始

**删除处理**：
| 方法 | 做法 | 优缺点 |
|------|------|--------|
| 移动后续记录 | 把删除位后的所有记录前移 | 简单但开销大 |
| 移动最后记录 | 把最后一条移到删除位置 | 快速但改变顺序 |
| **空闲链表 (Free List)** | 把删除位置加入空闲列表 | 最佳方案 |

## 13.3 🔴 变长记录与分槽页结构 (Slotted Page) [重点]

### 为什么需要变长记录？
- 多种记录类型存在同一文件中
- VARCHAR等变长字段
- 重复字段（如数组）

### 分槽页结构 (Slotted Page Structure)

```
┌─────────────────────────────────────┐
│ 页头(Header)                         │
│  ├── 记录数量                         │
│  ├── 空闲空间末尾位置                  │
│  ├── [记录1位置, 记录1大小]            │ ← 槽数组
│  ├── [记录2位置, 记录2大小]            │
│  └── ...                             │
├─────────────────────────────────────┤
│          ← 空闲空间 →                 │
├─────────────────────────────────────┤
│  记录3   记录2   记录1                │ ← 实际记录从页尾向前增长
└─────────────────────────────────────┘
```

**核心思想**：
- 记录可以在页内移动（整理碎片），指针指向**槽条目**而非记录本身
- 槽条目始终不变，只是更新位置和大小
- 对于BLOB/CLOB（超大对象），记录必须小于页大小，通常存储在文件系统的单独文件中

## 13.4 🔴 文件中记录的组织方式 [重点]

### 四种组织方式对比

| 方式 | 英文 | 说明 | 优点 | 缺点 |
|------|------|------|------|------|
| **堆文件** | Heap | 记录随意放在有空闲的地方 | 插入快 | 查找慢（需全表扫描） |
| **顺序文件** | Sequential | 按搜索码排序存储 | 按搜索码查询快 | 插入需要找位置，可能需要溢出块 |
| **多表聚集** | Multitable Clustering | 多个表的相关记录存在同一文件中 | 连接查询极快 | 单表扫描需要跳过无关记录 |
| **分区** | Partitioning | 按范围拆分到不同文件 | 管理灵活 | 跨分区查询复杂 |

### 堆文件 (Heap)

使用**空闲空间映射 (Free-Space Map)**：每个块一个条目，记录该块的空闲比例（如1=全空，0=全满）。插入时扫描映射找有足够空间的块。

### 顺序文件 (Sequential)

- 删除用指针链跳过
- 插入时如果当前块满了，放入溢出块(overflow block)
- 需要定期重组(reorganization)恢复物理顺序

### 多表聚集 (Multitable Clustering)

```
普通存储：
  department文件: [Comp.Sci | Taylor] [Biology | Watson] ...
  instructor文件: [10101 | Srinivasan] [12121 | Wu] ...

多表聚集存储（按dept_name聚集）：
  [Comp.Sci | Taylor | 10101 | Srinivasan]
  [Biology  | Watson | 12121 | Wu]
  -- department 和 instructor 有关联的数据存在同一块 --
```

**关键取舍**：连接查询快，但单表SELECT * FROM department慢（因为要把instructor的数据也读进来）。

### 分区 (Partitioning)

将表按条件拆成多个小表：
```sql
-- PostgreSQL示例
CREATE TABLE transaction_2018 (...) TABLESPACE fast_disk;
CREATE TABLE transaction_2019 (...) TABLESPACE archive_disk;
```

**好处**：减少每个分区的空闲空间管理开销，不同分区可用不同存储设备。

## 13.5 数据字典 (Data Dictionary / System Catalog) [了解]

存储**元数据（Metadata）**，包括：
- 关系名、属性名/类型/长度
- 视图定义
- 完整性约束（主码、外码、断言）
- 用户和权限信息
- 统计信息（行数、块数等）
- 物理文件组织信息
- 索引信息（索引名、类型、属性等）

> 也叫做 **System Catalog**，总是尽量放在主存中以便快速访问。

## 13.6 🔴 缓冲管理器 (Buffer Manager) [重点]

### 核心功能

缓冲区管理器负责在主存中分配缓冲空间，减少磁盘I/O。

### 关键概念

| 概念 | 说明 |
|------|------|
| **Pin（钉住）** | 块正在被使用，不能被淘汰。只有当 pin count = 0 时才能换出 |
| **共享/排他锁** | 并发控制：读可以共享，写需要排他 |
| **强制输出 (Forced Output)** | 为恢复目的强制将块写回磁盘 |

### 🔴 缓冲区替换策略 [必考]

| 策略 | 全称 | 做法 | 问题 |
|------|------|------|------|
| **LRU** | Least Recently Used | 淘汰最久未使用的块 | 顺序扫描时表现差（把所有常用块都挤出去） |
| **MRU** | Most Recently Used | 钉住当前处理块 | 适合顺序扫描 |
| **Toss-Immediate** | — | 处理完就立即释放 | 适合只用一次的查询 |

> **关键理解（经典考点）**：
> - LRU对顺序扫描**不好**！扫描把新块不断换入，把热点块全挤出去
> - 所以数据库系统对顺序扫描通常用**MRU或立即丢弃**，而不是LRU

### 其他相关技术

| 技术 | 说明 |
|------|------|
| 非易失写缓冲(NVRAM) | 用非易失RAM加速写入 |
| 日志磁盘(Log Disk) | 顺序写日志，再异步更新数据磁盘 |
| 日志文件系统(Journaling FS) | 文件系统层面保证一致性 |

## 13.7 🔴🔴 列式存储 (Column-Oriented Storage) [必考]

### 行存储 vs 列存储

```
行存储（Row Store）— 传统数据库：
块0: [ID=1, name=Alice,   dept=CS, salary=80K]
块1: [ID=2, name=Bob,     dept=EE, salary=70K]
块2: [ID=3, name=Charlie, dept=CS, salary=90K]

列存储（Column Store）：
ID列:   [1, 2, 3, ...]
name列: [Alice, Bob, Charlie, ...]
dept列: [CS, EE, CS, ...]
salary列: [80K, 70K, 90K, ...]
```

### 列存储的优缺点

| 优点 | 缺点 |
|------|------|
| **减少I/O**：只读需要的列 | **元组重建代价高**：读整行时需要访问多个列文件 |
| **压缩效果好**：同列数据类型相同 | **删除和更新代价高** |
| **CPU缓存友好**：连续同类型数据 | **解压缩有开销** |
| **支持向量化处理**：SIMD指令加速 | — |

### 适用场景

| 场景 | 推荐存储方式 |
|------|------------|
| **OLTP（事务处理）**：频繁读写单行 | **行存储** |
| **OLAP / 决策支持**：大表扫描，只选几列 | **列存储** |
| **混合** | 行列混合存储（如Oracle, SQL Server） |

### 列存储文件格式
- **ORC** (Optimized Row Columnar)：Hadoop生态
- **Parquet**：Spark生态，广泛使用

## 13.8 主存数据库 [了解]

- 数据直接放在内存中（不需要缓冲管理器）
- 列存储对主存数据库仍然有价值（压缩+向量化）
- 崩溃后需要从磁盘/副本恢复

## 易错点警示

1. **寻道时间 ≠ 访问时间**：访问时间 = 寻道时间 + 旋转延迟
2. **扇区是最小读写单位，块是数据传输基本单元**：两者的粒度概念不同
3. **RAID 0没有冗余！** 名字里有个"0"容易误以为是最低冗余，实际上是零冗余
4. **RAID 4的瓶颈是专用校验盘**，RAID 5通过分散校验解决了这个问题
5. **LRU对顺序扫描不好**——这是经典的反直觉考点
6. **列存储适合OLAP（分析查询），行存储适合OLTP（事务处理）**
7. **Slotted page中指针指向槽条目而不是记录本身**——记录可以移动，槽条目不变
