# 第一部分：进程概念与组成

## 3.1 进程概念

### 3.1.1 为什么引入进程

在早期的计算机系统中（如单道批处理系统），内存中同时只存在一道程序，CPU只能串行地执行指令。这种方式存在严重的资源浪费：当正在运行的程序等待I/O操作时，CPU必须空转等待，无法执行其他任何工作。这种**CPU与I/O设备速度不匹配**的矛盾导致了系统资源利用率极低。

为了解决这一问题，操作系统引入了**多道程序设计（Multiprogramming）**技术，允许多个程序同时进入内存并交替执行。然而，仅仅将多个程序放入内存是不够的——程序本身是一个静态的指令集合，无法描述其在内存中的运行状态、资源占用情况以及当前执行到的位置。因此，需要一种新的**动态概念**来刻画程序的执行过程，这就是**进程（Process）**。

引入进程的核心目的：
- **实现并发执行（Concurrent Execution）**：让多个程序能够交替使用CPU，逻辑上"同时"向前推进
- **提高资源利用率（Utilization）**：当某个进程因I/O而阻塞时，CPU可以立即切换到另一个可运行的进程，避免CPU空闲
- **提高系统吞吐量（Throughput）**：单位时间内完成的工作量增加
- **实现系统的正确管理**：为每个运行中的程序维护独立的执行状态、资源清单，保证互不干扰

### 3.1.2 进程的定义

进程的定义在不同的教材中有不同的表述方式，以下是最经典的定义：

> **进程（Process）**是**程序在某个数据集合上的一次执行过程**，是**系统进行资源分配和调度的独立单位**。

这个定义包含三个关键要素：
1. **程序（Program）**：提供可执行的指令代码
2. **数据集合（Data Set）**：程序运行所需的输入数据和工作区
3. **执行过程（Execution）**：程序被CPU逐条执行指令的动态过程

更准确地说，进程是一个**具有独立功能的程序在某个数据集合上的一次运行活动**。它不仅是程序的执行，更包含了程序执行时的**上下文环境**——包括寄存器的值、内存中的变量、打开的文件、当前的执行位置等全部状态信息。

### 3.1.3 进程与程序的对比

进程与程序是操作系统中最容易混淆的两个概念。它们之间有着本质的区别，同时又密切相关。下面从五个维度进行详细比较：

| 比较维度 | 进程（Process） | 程序（Program） |
|---|---|---|
| **本质属性** | **动态的**：进程是程序的一次执行过程，有生命周期——创建、运行、暂停、终止 | **静态的**：程序是一组有序指令的集合，存储在磁盘上，是一种静态的文本文件 |
| **存在时间** | **暂时的**：进程是"活"的，创建后存在，终止后消亡，寿命有限 | **永久的**：程序是"死"的，只要不删除就永久存在于存储介质上 |
| **组成结构** | 包含**程序段、数据段、PCB**三部分，其中PCB是进程存在的唯一标识 | 仅包含指令代码和数据，没有控制块 |
| **对应关系** | **多对多**：一个程序可以产生多个进程（如同时打开多个浏览器窗口），一个进程也可以包含多个程序（如调用execve()加载新程序） | **多对多**：同上 |
| **并发性** | **可并发执行**：多个进程可以交替使用CPU实现并发，这是操作系统的基本能力 | **无并发性**：程序本身是指令的静态集合，不具备执行特性，不能参与并发 |

**举例说明**：
- **同一程序对应多个进程**：用户打开了三个记事本窗口（notepad.exe），磁盘上只有一个notepad.exe程序文件，但内存中存在三个独立的notepad进程。每个进程有自己的数据区（编辑的不同文本内容）、独立的执行状态和窗口。
- **进程与程序的一对一关系**：在UNIX系统中，execve()系统调用可以将当前进程的地址空间完全替换为一个新程序。在这种情况下，进程的"身份"（PID、PCB等）保持不变，但执行的程序代码被替换了。
- **程序未被执行时**：一个C语言源代码经过编译链接生成的可执行文件（.exe文件），存储在磁盘上，此时它只是一个程序。只有当用户双击运行它时，操作系统才会为其创建进程。

**一个经典比喻**：
- **程序 ≈ 乐谱**：乐谱是静态的符号记录，可以长期保存
- **进程 ≈ 演奏**：演奏是乐谱的一次"执行"过程，有开始、有结束，每次演奏可能略有不同
- 同一份乐谱可以被不同乐队在不同时间演奏，产生多个"演奏进程"
- 一个乐队在演奏过程中可以更换乐谱（进程调用execve()）

### 3.1.4 进程的六大特征

进程作为现代操作系统的核心概念，具有以下六个基本特征：

#### 1. 并发性（Concurrency）
多个进程可以同时存在于内存中，并在宏观上同时向前推进。在单CPU系统中，这种"同时"实际上是微观上的交替执行——操作系统通过快速切换进程给用户造成"同时运行"的假象。并发性是进程与程序最重要的区别之一。

#### 2. 动态性（Dynamics）
进程是程序的一次执行过程，具有从创建到终止的生命周期：
- **创建**：系统为进程分配PCB、分配资源
- **执行**：进程在CPU上逐条执行指令
- **等待/阻塞**：进程因等待某事件（如I/O完成）而暂停
- **唤醒**：等待的事件发生，进程重新进入就绪状态
- **终止**：进程执行完毕或被强制结束，系统回收资源

#### 3. 独立性（Independence）
**进程是系统进行资源分配和CPU调度的独立单位**。每个进程拥有独立的：
- 地址空间（代码段、数据段、堆栈）
- 资源清单（打开的文件、I/O设备、信号量等）
- PCB（包含进程的所有状态信息）

进程之间默认互不干扰——一个进程的崩溃通常不会影响其他进程。

#### 4. 交互性（Interaction）
并发执行的进程之间并非完全隔离。它们可能：
- 需要**同步（Synchronization）**：合作进程需要按某种顺序执行
- 需要**互斥（Mutual Exclusion）**：竞争使用共享资源（如打印机、共享变量）
- 需要**通信（Communication）**：通过共享内存、消息传递等方式交换数据

这种交互性也是引入进程间通信（IPC）机制的根本原因。

#### 5. 异步性（Asynchrony）
每个进程按照各自独立的、不可预知的速度向前推进。由于：
- 进程会因I/O等待而阻塞
- CPU调度策略可能导致进程被随时抢占
- 系统负载波动影响执行速度

因此，**同一程序在同一数据上的两次运行，执行轨迹可能完全不同**。异步性要求操作系统必须提供同步机制来保证关键操作的顺序正确性。

#### 6. 结构性（Structure）
进程在内存中具有固定的结构组成。每个进程由三个部分组成：
- **PCB（Process Control Block）**：进程的控制信息
- **程序段（Program Segment）**：可执行的指令代码
- **数据段（Data Segment）**：进程运行所需的数据和工作区

这三个部分构成了进程在内存中的完整映像。

## 3.2 进程的组成

进程在内存中由三部分组成：PCB、程序段和数据段。其中PCB是最核心的部分，是进程存在的唯一标识。

### 3.2.1 进程控制块 PCB

#### PCB的概念与地位

**PCB（Process Control Block，进程控制块）** 是操作系统为每个进程维护的一个**数据结构**，用于存放进程的所有管理信息。PCB是操作系统中最重要的数据结构之一。

> **PCB是进程存在的唯一标识**——操作系统通过PCB来感知和管理进程。创建进程时创建PCB，终止进程时回收PCB。有PCB就有进程，没有PCB就没有进程。

从操作系统的角度看，只要PCB存在，进程就存在。即使进程的程序段和数据段被换出到磁盘交换区，只要PCB还在内存中，操作系统的调度程序仍然可以将该进程视为一个潜在的可运行实体（待其被换入后继续执行）。

#### PCB的完整内容

PCB包含的信息非常丰富，通常分为以下几个大类：

##### （1）进程标识信息
- **进程ID（PID, Process Identifier）**：每个进程有唯一的整数标识符，用于区分不同的进程。PID是进程的"身份证号"。
- **父进程ID（PPID, Parent PID）**：记录创建本进程的父进程的PID，用于构建进程家族树。
- **用户标识（UID, User ID）**：标识进程属于哪个用户，用于安全性检查和资源配额管理。

##### （2）处理器状态信息（上下文信息）
当进程被暂停执行时（如发生中断或进程切换），CPU的寄存器内容必须保存到PCB中，以便后续恢复执行。这些信息包括：
- **程序计数器（PC, Program Counter）**：指向下一条要执行的指令地址
- **程序状态字（PSW, Program Status Word）**：包含条件码（零标志、进位标志等）、中断允许/禁止位、CPU运行模式（用户态/内核态）
- **通用寄存器**：累加器、变址寄存器、栈指针等
- **栈指针（SP, Stack Pointer）**：指向进程内核栈或用户栈的栈顶

##### （3）进程调度信息
- **进程状态（Process State）**：就绪、运行、阻塞、挂起等
- **优先级（Priority）**：进程的调度优先级，可以是静态的（创建时设定）或动态的（运行时调整）
- **调度相关指针**：指向就绪队列、阻塞队列中前后PCB的指针
- **等待事件**：进程正在等待的事件标识
- **已等待时间**：用于计算优先级老化等调度决策

##### （4）内存管理信息
- **代码段指针**：指向进程代码段在内存中的基地址
- **数据段指针**
- **页表/段表基址**：用于地址转换的内存管理数据结构
- **界限寄存器值**：用于内存保护

##### （5）文件管理信息
- **打开文件描述符表**：进程当前打开的所有文件列表
- **当前工作目录**
- **根目录**

##### （6）I/O状态信息
- **已分配的设备列表**
- **未完成的I/O请求**
- **I/O设备的状态**

##### （7）记账信息
- **CPU总使用时间**
- **已使用的实际时间（墙上时钟时间）**
- **时间限制**
- **进程标识号**
- **I/O操作计数**

#### PCB的存储：PCB表

操作系统为所有进程维护一个**PCB表（PCB Table）**，存放在系统空间中（内核区域）。PCB表的组织形式决定了操作系统管理进程的效率。

**PCB表的特点**：
- PCB表位于内核空间，用户程序无法直接访问
- PCB表的大小决定了系统能同时容纳的最大进程数
- PCB表的每一项（或每条链表的一个节点）对应一个PCB

#### PCB的组织方式

PCB在内存中的组织方式主要有两种：

##### 方式一：链表方式（Linked List）

将相同状态进程的PCB通过指针链接成一个链表。常见的有：
- **就绪队列（Ready Queue）**：所有处于就绪状态的进程PCB组成的链表
- **阻塞队列（Blocked Queue）**：一个或多个阻塞队列（可按等待事件分类）
- **运行队列（Running Queue）**：单CPU系统中只有一个运行进程，不需要队列

**优点**：
- 插入和删除操作灵活，只需修改指针
- 队列长度可动态变化
- 可以实现优先级调度（按优先级排序的链表）

**缺点**：
- 查询特定PID的PCB时可能需要遍历链表

##### 方式二：索引表方式（Index Table）

系统建立一张或多张索引表，索引表的每一项指向一个PCB。常用的索引表包括：
- **就绪索引表**
- **阻塞索引表**

**优点**：
- 通过索引可以直接定位到PCB，访问速度快
- 适合静态分配PCB数组的情况

**缺点**：
- 需要预先分配固定大小的索引表空间
- 插入和删除需要维护索引表

> **实际系统通常采用混合方式**：对于就绪队列，用链表方便动态插入和删除；对于进程表（所有PCB的集合），用数组方便按PID快速查找。

### 3.2.2 程序段（Program Segment）

程序段是进程的**代码区**，存放的是：
- 可执行的机器指令代码
- 程序的只读数据（如字符串常量、const变量）
- 程序入口地址

**代码重入（Reentrant Code）**：
- **可重入代码（Reentrant Code）**也称为**纯代码（Pure Code）**，是指能被多个进程同时共享而不产生副作用的代码
- 可重入代码在执行过程中不会修改自身——所有可变数据存放在数据段中
- 允许多个进程共享同一个代码段副本，从而节省内存
  - 例如：多个用户同时运行vi编辑器，可以在内存中只保留一份vi代码段副本
  - 每个进程有自己的数据段和PCB，但代码段指向同一块物理内存

### 3.2.3 数据段（Data Segment）

数据段存放的是进程运行期间需要的所有**可变数据**：
- **全局变量**
- **静态变量**（static变量）
- **堆空间**（动态分配的内存，如malloc/new）
- **栈空间**（函数调用的参数、返回地址、局部变量）
- **用户打开的文件描述符表**
- **I/O缓冲区**

数据段是**进程私有的**——不同进程的数据段彼此隔离，这正是操作系统内存保护的基础。一个进程不能直接访问另一个进程的数据段（除非通过共享内存等特殊机制）。

### 3.2.4 进程在内存中的映像

一个进程在内存中的典型布局（从低地址到高地址）：

```
+------------------+ 高地址
|      栈段        | ← 栈指针（向下增长）
|   (局部变量、    |
|    函数参数、    |
|    返回地址)     |
|   ↓ ↓ ↓ ↓       |
|   -------        |
|   ↑ ↑ ↑ ↑       |
|      堆段        | 堆（向上增长）
|   (动态分配)     |
|------------------|
|    数据段        |
|   (全局/静态变量) |
|------------------|
|    代码段        | ← 程序计数器PC
|   (只读指令)     |
+------------------+ 低地址（0x00000000）
```

其中，数据段又分为：
- **初始化数据段（.data）**：已初始化的全局变量和静态变量
- **未初始化数据段（.bss）**：未初始化的全局变量和静态变量（在程序加载时被清零）
- **只读数据段（.rodata）**：字符串常量、const全局变量

## 3.3 进程状态与转换

### 3.3.1 进程的基本三状态模型

在教材中最经典的进程三状态模型包括：

#### （1）就绪状态（Ready）
- 进程已获得除CPU之外的所有所需资源
- 只要被调度程序选中，就可以立即在CPU上执行
- 在单CPU系统中，同一时刻可能有多个进程处于就绪状态
- 就绪进程被组织成**就绪队列（Ready Queue）**

#### （2）运行状态（Running）
- 进程已获得CPU，正在执行指令
- 在单CPU系统中，任何时刻**最多只有一个**进程处于运行状态
- 在多CPU系统中，可以有多个进程同时运行（每个CPU上一个）

#### （3）阻塞状态（Blocked / Waiting）
- 进程因等待某事件发生而暂停执行
  - 如等待I/O完成、等待信号量、等待子进程结束
- 即使分配CPU给阻塞进程，它也无法继续执行
- 阻塞进程通常按等待事件分类组织成多个**阻塞队列（Blocked Queue）**
  - 如：等待磁盘读写的队列、等待打印机就绪的队列、等待信号量的队列

### 3.3.2 三状态转换图及转换分析

```
                              ┌─────────────┐
                              │             │
              ┌──────────────→│   Running   │──┐
              │               │   (运行)    │  │
              │               └─────────────┘  │
              │                    ↑           │
              │                    │           ↓
              │               ┌─────────────┐  │
              │               │    Ready    │  │
              │               │   (就绪)    │←─┘
              │               └─────────────┘
              │                    ↑
              │                    │
              │               ┌─────────────┐
              └───────────────│   Blocked   │
                              │   (阻塞)    │
                              └─────────────┘
```

上面的图中，进程状态的转换用箭头表示，每个箭头代表一种可能的转换。但实际在仅有三种基本状态时，完整的转换关系如下：

#### 转换1：就绪 → 运行（Ready → Running）
- **触发条件**：进程调度程序从就绪队列中选择一个进程，为其分配CPU
- **发生频率**：非常频繁（毫秒级别），每次时钟中断或其他调度事件都可能触发
- **性质**：由操作系统的调度程序**主动发起**

#### 转换2：运行 → 就绪（Running → Ready）
- **触发条件**：
  - **时间片用完**：在分时系统中，当前进程的时间片耗尽
  - **被更高优先级进程抢占**：优先级更高的进程变为就绪，操作系统剥夺当前进程的CPU
  - **时钟中断**：每次时钟中断触发调度程序重新评估
- **性质**：**被动**的——当前进程并不是自己愿意放弃CPU，而是被操作系统强制剥夺
- **关键点**：这是实现分时系统和优先级调度的核心机制

#### 转换3：运行 → 阻塞（Running → Blocked）
- **触发条件**：
  - 进程请求I/O操作（如read()、write()）
  - 进程等待某个事件发生（如wait()等待子进程结束）
  - 进程申请资源但资源暂时不可用（如请求已被占用的锁）
  - 进程主动调用block()原语
- **性质**：**唯一一个由进程主动触发的转换**——进程因自身需要而自愿放弃CPU
- **关键点**：进程此时即使有CPU也无法继续执行，所以主动让出CPU

#### 转换4：阻塞 → 就绪（Blocked → Ready）
- **触发条件**：
  - I/O操作完成——设备控制器向CPU发送中断信号
  - 等待的事件发生——其他进程释放了信号量
  - 进程所申请的资源变为可用
- **性质**：**被动的**——进程被外部事件"唤醒"
- **关键点**：阻塞进程被唤醒后不能直接进入运行状态，必须经过就绪队列排队等待CPU调度

#### 关于状态转换的重要结论

**错误的转换**（在经典三状态模型中不可能发生）：
1. **阻塞 → 运行**（Blocked → Running）：阻塞进程必须先变为就绪，再被调度执行。阻塞进程没有获得CPU的资格。
2. **就绪 → 阻塞**（Ready → Blocked）：就绪进程什么都不缺只缺CPU，没有任何理由阻塞。只有运行时才能主动阻塞。

**哪个转换是进程主动发起的？**
- 正确答案：**只有 运行→阻塞 是进程主动发起的**
- 运行→就绪 是被操作系统剥夺CPU
- 就绪→运行 是被调度程序选中
- 阻塞→就绪 是被外部事件唤醒

### 3.3.3 五状态扩展模型

在实际的操作系统中，为了更准确地描述进程的完整生命周期，引入了**创建状态（New）**和**终止状态（Terminated）**，形成五状态模型：

#### 创建状态（New）
- **定义**：进程正在被创建的过程中，尚未进入就绪队列
- **发生过程**：
  1. 系统调用创建新进程（如fork()）
  2. 操作系统为新进程分配PCB
  3. 初始化PCB的各个字段
  4. 分配必要的内存资源
  5. 当初始化完成后，进程进入就绪状态
- **特点**：此时进程正处在"孕育"阶段，尚未准备好运行

#### 终止状态（Terminated / Zombie）
- **定义**：进程已停止执行，但PCB尚未被完全回收
- **发生过程**：
  1. 进程执行完毕或收到终止信号
  2. 操作系统停止进程执行
  3. 释放大部分资源（内存、打开的文件等）
  4. PCB保留（包含退出状态码等信息），等待父进程收集
  5. 父进程调用wait()后，PCB被完全释放
- **特点**：进程已"死亡"，但还留下PCB这个"遗体"供父进程查验

#### 完整的五状态转换图

```
        ┌─────────────┐
        │    New      │
        │   (创建)    │
        └──────┬──────┘
               │ admit（创建完成，进入就绪队列）
               ↓
        ┌─────────────┐      schedule      ┌─────────────┐
   ┌───→│   Ready     │───────────────────→│   Running   │───┐
   │    │   (就绪)    │←───────────────────│   (运行)    │   │
   │    └─────────────┘   timeout/preempt  └──────┬──────┘   │
   │                                              │          │
   │         ┌─────────────┐                      │          │
   │         │   Blocked   │←─────────────────────┘          │
   │         │   (阻塞)    │    wait for event               │
   │         └──────┬──────┘   (等待事件，如I/O)             │
   │                │                                        │
   │                │ event occurs（事件发生，如I/O完成）      │
   └────────────────┘                                        │
                                                              │
                     ┌─────────────┐                          │
                     │ Terminated  │←─────────────────────────┘
                     │  (终止)     │   exit（进程自行退出或被杀死）
                     └─────────────┘
```

#### 各状态转换的详细说明

| 转换 | 说明 | 触发机制 |
|---|---|---|
| New → Ready | 创建完成，进程可以被调度执行 | 初始化完成后由OS自动转换 |
| Ready → Running | 进程获得CPU开始执行 | 调度程序选择 |
| Running → Ready | 进程被剥夺CPU（时间片用完或高优先级抢占） | 调度程序决定，被动 |
| Running → Blocked | 进程等待某事件（I/O、锁、子进程等） | 进程自身系统调用，主动 |
| Running → Terminated | 进程执行完毕 | exit()系统调用或致命错误 |
| Blocked → Ready | 等待的事件发生 | 中断（如I/O完成）或其他进程的唤醒操作 |

### 3.3.4 不可能的状态转换

在五状态模型中，以下状态转换是**不可能发生的**：

1. **Blocked → Running**：阻塞进程必须先变为就绪才能运行。阻塞进程缺少的是某事件而非CPU，即使给它CPU也无法执行。

2. **Ready → Blocked**：就绪进程什么都不缺，只缺CPU。不可能在拥有CPU之前就进入等待状态。

3. **Blocked → Terminated**：通常不能直接让阻塞进程终止——必须先唤醒再终止（或特殊情况下在阻塞状态终止，但需要特殊处理）。

4. **New → Running**：新创建的进程必须先进入就绪队列排队，不能直接"插队"运行（实时系统除外，但通常也有初始化过程）。

**面试经典问题**：为什么不可能从阻塞状态直接进入运行状态？

**答**：阻塞状态意味着进程正在等待某个事件的发生（如I/O完成、信号量被释放）。即使分配CPU给它，它也无法继续执行——因为等待的条件尚未满足。因此，阻塞进程必须先被唤醒（转入就绪状态），等到CPU调度程序选中它时，才能进入运行状态。这保证了CPU不会被分配给"无法执行"的进程。

### 3.3.5 挂起状态（Suspended States）

#### 为什么需要挂起状态？

引入挂起状态的主要原因是**内存资源有限**。当系统中进程数量过多，内存容纳不下所有进程时，操作系统需要将部分进程**换出（swap out）**到磁盘交换区，腾出内存空间给其他进程使用。这就是**中程调度（Medium-term Scheduling）**的核心任务。

挂起状态与阻塞状态的根本区别在于：
- **阻塞**：进程等待事件，但进程仍**在内存中**
- **挂起**：进程被换出到磁盘（**不在内存中**），无论它在等待事件还是就绪

#### 引入挂起后的五状态

引入挂起状态后，将原来的就绪和阻塞分别拆分为"活动"和"静止"两个版本：

1. **活动就绪（Active Ready / Ready）**：进程在内存中就绪，随时可被调度执行
2. **静止就绪（Suspended Ready / Ready-Suspend）**：进程在磁盘中就绪（挂起就绪），需要先换入内存才能被调度
3. **活动阻塞（Active Blocked / Blocked）**：进程在内存中等待事件
4. **静止阻塞（Suspended Blocked / Blocked-Suspend）**：进程在磁盘中等待事件（被挂起的阻塞进程）
5. **运行（Running）**：正在执行的进程（不可能被挂起）

#### 包含挂起的七状态模型

```
                        ┌──────────────┐
                        │     New      │
                        └──────┬───────┘
                               │ admit
                               ↓
   ┌───────────────────────────────────────────────────────┐
   │               ┌─────────────────┐                     │
   │               │  Active Ready   │                     │
   │               │  (活动就绪)      │←───────────┐        │

   │                        │ schedule            │        │

   │      │        │    (运行)       │            │        │
   │      │        └────────┬────────┘            │        │
   │      │                 │                     │        │
```

#### 挂起状态相关的转换

| 转换 | 描述 | 触发时机 |
|---|---|---|
| 活动就绪 → 静止就绪 | 就绪进程被换出到磁盘 | 内存不足，中程调度 |
| 活动阻塞 → 静止阻塞 | 阻塞进程被换出到磁盘 | 内存不足，优先级低 |
| 静止就绪 → 活动就绪 | 挂起就绪进程被换入内存 | 内存有空闲，中程调度 |
| 静止阻塞 → 静止就绪 | 挂起阻塞进程等待的事件发生 | I/O完成等（事件发生时进程仍在磁盘上） |
| 静止阻塞 → 活动阻塞 | 挂起阻塞进程被换入内存 | 内存有空闲，但等待的事件尚未发生 |
| 静止就绪 → 运行 | 理论上不行——必须先换入变为活动就绪 | — |

关于**静止阻塞 → 静止就绪**的转换需要特别注意：当在磁盘上等待事件（如I/O）的挂起阻塞进程所等待的事件发生时，I/O设备仍然可以向CPU发送中断，操作系统处理中断后将该进程的状态从静止阻塞改为静止就绪。但因为该进程仍然在磁盘上（不在内存中），所以不能立即被调度执行——需要先被换入内存（变为活动就绪）。

### 3.3.6 进程队列（Process Queues）

操作系统维护多个队列来管理不同状态的进程：

#### （1）作业队列（Job Queue）
- 系统中所有进程的集合（包括新创建、就绪、运行、阻塞、终止等所有状态）
- 保存所有进程的PCB
- 实际上就是进程表（Process Table）

#### （2）就绪队列（Ready Queue）
- 所有处于就绪状态的进程PCB组成的链表
- 通常按照优先级排序，队首指向优先级最高的就绪进程
- 包含活动就绪队列和静止就绪队列（如果支持挂起）

#### （3）设备队列（Device Queue）
- 每个I/O设备维护一个队列
- 等待使用该设备的进程PCB链接到队列中
- 设备队列中的进程处于阻塞状态

#### （4）队列图（Queueing Diagram）

```
                  ┌─────────┐
   admit          │ Ready   │
   ─────────────→ │ Queue   │───→ schedule ──→ CPU ──→ exit
                  └─────────┘                    │
                                                 │ I/O request
                                                 ↓
                                         ┌──────────────┐
                                         │ Device Queue  │
                                         │ (等待某磁盘)   │
                                         └──────┬───────┘
                                                │ I/O completed
                                                │ (interrupt)
                                                ↓
                                            (回到Ready Queue)
```

该图展示了进程在系统中的典型流转路径：
1. 新进程被接纳（admit）后加入就绪队列
2. 被调度程序选中后获取CPU执行
3. 执行过程中可能请求I/O操作，进入设备队列阻塞
4. I/O完成后通过中断被唤醒，回到就绪队列
5. 进程执行完毕后退出系统

## 3.4 进程调度

### 3.4.1 三级调度

操作系统的调度功能分为三个层次，分别在不同的时间尺度上管理进程和资源：

#### （1）长程调度（Long-term Scheduling）—— 作业调度

**作用对象**：作业（Job）

**功能**：
- 从磁盘的**后备队列（Job Pool）**中选择作业加载到内存
- 决定哪些作业可以进入系统变为进程
- 控制**多道程序度（Degree of Multiprogramming）**——内存中同时存在的进程数量

**发生频率**：
- **低**（分钟级别甚至更长）
- 当一个作业运行完毕或系统有空闲内存时触发

**关键决策**：
- 选择CPU密集型和I/O密集型作业的合理混合
  - 如果全是CPU密集型→I/O设备空闲，CPU也可能因为缺乏可切换的进程而效率下降
  - 如果全是I/O密集型→CPU利用率低（大部分时间在等待I/O）
  - 好的混合：CPU密集型进程运行时触发I/O密集型进程，使两者都能高效率工作

**典型实现**：
- 批处理系统中显式存在作业调度
- 在分时系统中（如UNIX/Linux），长程调度通常与用户登录、程序启动融合在一起

#### （2）中程调度（Medium-term Scheduling）—— 交换调度

**作用对象**：进程

**功能**：
- 将进程在内存和磁盘交换区之间换入换出（swap in / swap out）
- 处理内存超载问题
- 引入挂起状态

**发生频率**：
- **中等**（秒级别或分钟级别）
- 当内存总需求量超过可用内存时触发

**关键决策**：
- 选择哪个进程被换出
  - 通常选择阻塞状态且优先级低的进程
  - 有时也会选择就绪进程（为更重要的进程腾出空间）
- 预判进程何时会被再次使用

#### （3）短程调度（Short-term Scheduling）—— CPU调度

**作用对象**：进程（严格说是线程）

**功能**：
- 从就绪队列中选择下一个要执行的进程
- 分配CPU给它

**发生频率**：
- **非常高**（毫秒级别甚至更低）
- 每次时钟中断、I/O中断、系统调用返回时都可能触发

**关键目标**：
- 选择"最佳"的进程运行以优化系统性能
- 不同调度算法追求不同的目标（详见3.8节）

#### 三级调度的对比总结

| 特性 | 长程调度（作业调度） | 中程调度（交换调度） | 短程调度（CPU调度） |
|---|---|---|---|
| **决策频率** | 低 | 中等 | 非常高 |
| **时间尺度** | 分钟/秒 | 秒 | 毫秒/微秒 |
| **控制内容** | 作业能否进入系统 | 进程是否在内存中 | 哪个进程获得CPU |
| **控制对象** | 作业 | 进程（换入换出） | 进程（分配CPU） |
| **主要目标** | 控制多道程序度 | 缓解内存压力 | CPU利用率和响应速度 |
| **典型算法** | FCFS、优先级 | 页面置换算法 | FCFS、SJF、RR等 |

### 3.4.2 进程控制原语

**原语（Primitive）**是操作系统内核中实现特定功能的程序段，其特点是**执行过程不可被中断**（即原子性）。进程控制原语是操作系统对进程进行管理的基本工具。

操作系统通过进程控制原语实现对进程生命周期的全面控制。以下是核心的进程控制原语：

#### （1）进程创建原语 create()

**功能**：创建一个新进程

**步骤**：
1. 为新进程分配唯一的PID
2. 分配PCB空间
3. 初始化PCB（状态设为"创建"或"就绪"）
4. 分配必要的内存资源（代码段、数据段、栈等）
5. 将进程插入就绪队列
6. 返回新进程的PID

#### （2）进程撤销原语 terminate() / exit()

**功能**：终止一个进程

**步骤**：
1. 根据PID找到目标PCB
2. 回收进程占用的所有资源（内存、文件、设备等）
3. 如果进程有子进程，处理子孙进程（挂起或终止）
4. 将进程状态设为"终止"
5. 通知父进程（发送SIGCHLD信号）
6. 如果父进程已终止，则将进程托付给init进程（或处理为孤儿进程）

#### （3）进程阻塞原语 block()

**功能**：进程主动进入阻塞状态

**步骤**：
1. 保存进程的CPU上下文（寄存器到PCB）
2. 将进程状态从"运行"改为"阻塞"
3. 将PCB插入相应的阻塞队列
4. 转进程调度程序（选择新的就绪进程运行）

#### （4）进程唤醒原语 wakeup()

**功能**：将阻塞进程变为就绪状态

**步骤**：
1. 从阻塞队列中找到目标PCB
2. 将进程状态从"阻塞"改为"就绪"
3. 将PCB从阻塞队列移到就绪队列
4. （可选）如果唤醒的进程优先级高于当前运行进程，触发重新调度

#### （5）挂起原语 suspend()

**功能**：将进程从内存换出到磁盘

**步骤**：
1. 检查进程状态：
   - 如果是活动就绪 → 变为静止就绪
   - 如果是活动阻塞 → 变为静止阻塞
2. 将进程的内存映像保存到磁盘交换区
3. 释放进程占用的内存空间
4. 更新PCB中的内存地址信息

#### （6）激活原语 resume() / activate()

**功能**：将挂起进程换入内存

**步骤**：
1. 根据PCB中的信息在内存中分配空间
2. 从磁盘交换区读取进程映像到内存
3. 更新PCB中的内存信息
4. 将进程状态从静止改为活动
5. 如果进程变为活动就绪，插入就绪队列

### 3.4.3 进程的创建

#### 创建事件

以下事件会触发新进程的创建：

1. **用户登录（User Login）**：
   - 用户在终端输入用户名和密码
   - 系统验证通过后，创建用户会话进程（如shell进程）
   - 例如：Linux中的sshd为每个SSH登录创建一个shell进程

2. **作业调度（Job Scheduling）**：
   - 在批处理系统中，当作业调度程序选择了一个新作业
   - 为该作业创建进程

3. **提供服务（Service Provision）**：
   - 用户程序请求某种服务，操作系统创建服务进程响应
   - 例如：打印请求 → 创建打印守护进程处理

4. **应用请求（Application Request）**：
   - 正在运行的进程通过系统调用（如fork()）主动创建子进程
   - 这是最典型的进程创建方式

#### UNIX/Linux系统的进程创建

**fork()系统调用**

```c
#include <unistd.h>
pid_t fork(void);
```

**fork()的语义**：
- 创建调用进程（父进程）的一个副本——**子进程**
- 子进程获得父进程的**完整拷贝**：
  - 代码段（通常共享，写时复制优化）
  - 数据段（写时复制，Copy-on-Write）
  - 堆栈
  - 打开的文件描述符表
  - 信号处理设置
  - 环境变量
- fork()的返回值：
  - **父进程**：返回子进程的PID（正数）
  - **子进程**：返回0
  - **错误**：返回-1

**fork()的典型用法**：

```c
pid_t pid = fork();
if (pid < 0) {
    // 创建失败
    perror("fork failed");
    exit(1);
} else if (pid == 0) {
    // 子进程代码
    printf("I am child, PID = %d\n", getpid());
    // 子进程可能调用exec族函数加载新程序
    execlp("/bin/ls", "ls", "-l", NULL);
} else {
    // 父进程代码
    printf("I am parent, child PID = %d\n", pid);
    wait(NULL);  // 等待子进程结束
}
```

**写时复制（Copy-on-Write, COW）**：
- fork()在创建子进程时并不立即复制父进程的所有内存页
- 父子进程共享物理内存页，但页表被标记为"只读"
- 当任一进程试图写入共享页时，触发页面异常
- 内核在异常处理中复制该页，为每个进程创建独立的副本
- COW大大减少了fork()的开销——如果子进程立即调用exec()，则根本不需要复制父进程的任何页面

**execve()系统调用**

```c
#include <unistd.h>
int execve(const char *pathname, char *const argv[], char *const envp[]);
```

**execve()的语义**：
- 用pathname指定的新程序**替换**当前进程的地址空间
- 当前进程的代码段、数据段、堆栈被完全替换
- 但进程的**PID不变**，PCB大部分信息保留（如打开的文件描述符默认保留）
- 如果exec成功，它不会返回（因为原来的代码已经不存在了）
- 如果exec失败，返回-1

**fork() + exec() 模式**：
在UNIX/Linux中，创建新进程的标准模式是先fork()再exec()：
1. fork()创建一个子进程（是父进程的副本）
2. 子进程调用exec()加载并运行新程序
3. 父进程可以选择等待子进程结束（wait()）或继续并发执行

#### 创建进程时的资源选项

操作系统在创建新进程时，关于资源分配有以下策略：

1. **父进程与子进程共享所有资源**：
   - 子进程直接使用父进程的地址空间（此时它们不是独立的，类似于线程）
   - 实际系统中很少使用这种方式

2. **子进程共享父进程的部分资源**：
   - 如UNIX中子进程继承父进程的文件描述符
   - 但地址空间独立（通过写时复制或完全复制）
   - 这是最常用的方式

3. **父进程与子进程完全不共享资源**：
   - 子进程获得全新的资源分配
   - 效率较低但隔离性最好

#### 创建进程时的执行选项

1. **父进程和子进程并发执行**：
   - fork()返回后，父子进程同时存在于就绪队列中
   - 由调度程序决定谁先运行
   - 这是UNIX的默认行为

2. **父进程等待子进程结束**：
   - 父进程调用wait()系统调用，进入阻塞状态
   - 等待子进程终止后再继续执行
   - 适用于需要子进程结果才能继续的情况

### 3.4.4 进程的终止

#### 终止原因

进程终止的原因可以分为四大类：

**1. 正常退出（Normal Exit）**：
- 进程执行完main()函数中的最后一条语句
- 进程调用exit()系统调用
- C语言中main()函数的return语句会隐式调用exit()
- 退出状态码0通常表示成功，非0表示错误

**2. 错误退出（Error Exit）**：
- 进程检测到自身无法处理的错误
- 例如：输入文件不存在、权限不足、配置错误
- 进程主动调用exit(非0)终止

**3. 致命错误（Fatal Error）**：
- 进程在执行过程中发生了无法恢复的错误
- 例如：段错误（Segmentation Fault）、除零错误、非法指令
- 由操作系统捕获异常并强制终止进程
- 常见的信号：SIGSEGV（段错误）、SIGFPE（浮点异常）、SIGILL（非法指令）

**4. 被其他进程终止（Killed by Another Process）**：
- 一个进程向另一个进程发送终止信号
- 最典型的是kill()系统调用（UNIX/Linux）
- 常见信号：
  - SIGKILL（9）：强制终止，不可被捕获或忽略
  - SIGTERM（15）：请求终止，进程可以捕获并执行清理
  - SIGHUP（1）：终端断开，通常导致shell终止其子进程

#### 僵尸进程（Zombie Process）

**定义**：已经终止执行但其PCB尚未被父进程回收的进程

**产生过程**：
1. 子进程调用exit()终止
2. 操作系统释放子进程的大部分资源（内存、打开的文件等）
3. 子进程的PCB保留（包含退出状态码）
4. 子进程状态变为"终止"，向父进程发送SIGCHLD信号
5. 如果父进程没有调用wait()/waitpid()，子进程变为僵尸进程

**僵尸进程的危害**：
- 占用系统进程表的一个表项
- 虽然不占用CPU和内存资源，但PCB占用内核内存
- 如果系统中存在大量僵尸进程，会导致进程表满，无法创建新进程

**避免僵尸进程的方法**：
- 父进程应始终调用wait()/waitpid()等待子进程结束
- 父进程可以注册SIGCHLD信号处理函数
- 父进程可以忽略SIGCHLD信号（UNIX自动回收子进程）

#### 孤儿进程（Orphan Process）

**定义**：父进程先于子进程终止的进程

**处理方式**：
- 操作系统中所有孤儿进程被**init进程（PID=1）收养**
- init进程成为这些孤儿进程的新父进程
- init进程会定期调用wait()回收已终止的孤儿进程
- 从而避免孤儿进程变成僵尸进程

**重要区别**：
- **僵尸进程**：进程已终止但PCB未被回收（父进程还在但没有wait）
- **孤儿进程**：父进程已终止但进程本身可能还在运行（被init收养）
- 孤儿进程本身不是问题——init进程会妥善管理它们

#### 级联终止（Cascading Termination）

在某些操作系统（如早期UNIX）中，当父进程终止时，其所有子进程也被强制终止。这是因为：
- 子进程通常依赖于父进程提供的某些服务或资源
- 父进程终止后，子进程可能无法正常运行
- 级联终止可以保证系统的一致性

现代UNIX/Linux系统通常不采用级联终止——父进程终止后，子进程成为孤儿进程，由init进程收养并继续运行。

## 3.5 上下文切换（Context Switch）

### 3.5.1 什么是上下文切换

**上下文切换（Context Switch）**是指操作系统将CPU从正在执行的进程切换到另一个进程的过程。这个过程涉及保存当前进程的CPU状态（上下文）并恢复另一个进程的先前保存的状态。

**进程的上下文**包括：
- CPU寄存器的值（通用寄存器、程序计数器PC、程序状态字PSW）
- 进程的内核栈
- 内存管理信息（页表基址、地址空间ID等）
- 浮点寄存器状态

### 3.5.2 上下文切换的完整步骤

当操作系统决定进行进程切换时（如时间片用完、更高优先级的进程变为就绪），执行以下步骤：

#### 步骤一：确定切换原因
- 通常是时钟中断处理程序发现当前进程的时间片已用完
- 或者某个阻塞队列中的进程被唤醒，优先级高于当前进程
- 或者当前进程主动调用block()进入休眠

#### 步骤二：保存当前进程的CPU上下文
1. 将程序计数器（PC）保存到进程的内核栈或PCB中
2. 将所有通用寄存器的值保存到PCB的寄存器保存区
3. 保存程序状态字（PSW）
4. 保存浮点寄存器和其他特殊寄存器的状态

#### 步骤三：更新当前进程的PCB状态
1. 将进程状态从"运行"改为"就绪"（或"阻塞"）
2. 记录进程当前的统计信息（CPU使用时间等）
3. 更新PCB中的其他相关字段

#### 步骤四：将当前进程的PCB移到适当的队列
- 如果进程被抢占：放入就绪队列
- 如果进程主动阻塞：放入相应的阻塞队列
- 如果进程终止：放入终止队列，等待父进程收集

#### 步骤五：选择下一个要执行的进程
- 调用**CPU调度程序（Scheduler）**
- 从就绪队列中选择一个进程
- 调度算法决定了选择哪个进程（FCFS、SJF、RR等）

#### 步骤六：恢复下一个进程的上下文
1. 从新进程的PCB中读取之前保存的程序计数器（PC）
2. 恢复所有通用寄存器的值
3. 恢复程序状态字（PSW）
4. 恢复浮点寄存器状态

#### 步骤七：更新内存管理单元（MMU）
- 如果新进程属于不同地址空间，需要切换页表
- 刷新TLB（Translation Lookaside Buffer，快表）
- 这可能是上下文切换中最耗时的操作之一

#### 步骤八：执行新进程的第一条指令
- 程序计数器已经指向新进程的下一条指令
- CPU从新进程的地址空间取指令，开始执行

### 3.5.3 上下文切换的开销

**上下文切换是纯粹的开销（Pure Overhead）**——在切换过程中，CPU没有执行任何有用的应用程序指令。

#### 上下文切换时间的主要组成

| 开销来源 | 说明 | 典型耗时 |
|---|---|---|
| 寄存器保存/恢复 | 保存和恢复数十个寄存器的值 | 微秒级 |
| TLB刷新 | 切换地址空间时需刷新快表 | 数十到数百微秒 |
| 缓存（Cache）失效 | 新进程的数据不在缓存中，需要重新加载 | 微秒级 |
| 内核栈切换 | 切换到新进程的内核栈 | 微秒级 |
| 调度程序执行 | 运行调度算法选择下一个进程 | 微秒级 |
| 模式切换 | 从用户态切换到内核态再切换回来 | 微秒级 |

**典型的上下文切换时间**：
- 现代操作系统：**1~10微秒**（取决于硬件和OS优化）
- 每次切换约耗费几百到几千个CPU周期

### 3.5.4 上下文切换 vs 模式切换

这是一个非常重要的区别，也是常见的考点：

| 比较维度 | 上下文切换（Context Switch） | 模式切换（Mode Switch） |
|---|---|---|
| **定义** | 从一个进程切换到另一个进程 | 在用户态和内核态之间切换（同一进程内） |
| **是否切换进程** | 是 | 否 |
| **性能开销** | 大（数十到数百微秒） | 较小（约1微秒） |
| **触发原因** | 时间片用完、I/O等待、进程抢占 | 系统调用、异常、中断 |
| **TLB影响** | 影响（需要刷新或标记） | 不影响（仍在同一地址空间） |
| **缓存影响** | 缓存失效 | 缓存命中 |

**举例说明**：
- **系统调用**（如read()、write()）：只需从用户态切换到内核态处理请求，然后返回用户态——是模式切换，不是上下文切换。进程没有被切换，只是CPU运行模式改变了。
- **时间片用完**：进程A时间片到，调度程序选择进程B运行——这是上下文切换。先保存A的状态，再恢复B的状态。

### 3.5.5 影响上下文切换时间的因素

1. **硬件支持力度**：
   - 是否有专门的指令批量保存/恢复寄存器（如x86的PUSHA/POPA）
   - 是否有硬件支持多组寄存器（如SPARC的寄存器窗口）

2. **内存速度**：
   - 内存访问速度直接影响PCB读写速度
   - 高速缓存（Cache）可以显著减少切换时间

3. **寄存器数量**：
   - 架构寄存器越多，保存/恢复的开销越大
   - 但寄存器越多，程序执行效率越高——这是一个权衡

4. **调度算法的复杂度**：
   - 简单调度算法（如FCFS）调度决策时间短
   - 复杂调度算法（如多级反馈队列）决策时间较长

5. **TLB和Cache大小**：
   - TLB/Cache越大，切换后重新加载的命中率越高
   - 某些架构通过给TLB项添加地址空间标识（ASID）避免完全刷新

## 3.6 进程间关系与协作

### 3.6.1 进程间关系分类

系统中的进程之间存在多种关系：

#### （1）无关进程（Independent Processes）
- 进程之间没有任何共享资源或数据交换
- 各自独立执行，互不影响
- 操作系统可以对它们进行独立的调度和资源管理

#### （2）父子关系（Parent-Child Relationship）
- 父进程通过fork()或类似机制创建子进程
- 子进程继承父进程的某些资源（环境变量、文件描述符等）
- 父进程可以wait()等待子进程结束，获取子进程的退出状态
- 子进程的终止会向父进程发送SIGCHLD信号

#### （3）兄弟关系（Sibling Relationship）
- 同一父进程创建的多个子进程之间的关系
- 兄弟进程之间通常彼此独立
- 可以互相发送信号（如果有权限）
- 通过父进程留下的共享资源（如共享内存、管道）进行通信

#### （4）合作关系（Cooperating Processes）
- 进程之间通过某种机制交互或共享数据
- 需要满足特定的顺序约束或互斥访问要求
- 必须使用进程间通信（IPC, Inter-Process Communication）机制

### 3.6.2 进程协作的好处

进程间的协作虽然在管理和同步方面引入了复杂性，但带来了显著的好处：

1. **信息共享（Information Sharing）**：
   - 多个进程可以访问共享数据库或文件
   - 如数据库管理系统中的多个查询进程同时访问同一数据库
   - 避免了数据冗余和不一致

2. **计算加速（Computation Speedup）**：
   - 将一个大型任务分解为多个子任务并行执行
   - 在多核/多处理器系统上可以实现真正的并行加速
   - 例如：视频渲染时，将一帧画面分成多个区域并行渲染
   - Amdahl定律指出：加速比受限于串行部分的比例

3. **模块化（Modularity）**：
   - 将复杂系统分解为多个功能独立的进程
   - 每个进程负责单一功能，便于开发和维护
   - 例如：Web服务器将监听连接、处理请求、日志记录分离到不同进程

4. **便利性（Convenience）**：
   - 用户可同时运行多个程序（浏览器、音乐播放器、文本编辑器）
   - 操作系统通过并发机制使这种使用方式成为可能

### 3.6.3 进程间的约束关系

在进程协作中，存在两种约束关系：

#### （1）直接约束 —— 同步（Synchronization）
- **定义**：合作进程之间需要按某种顺序执行
- **场景**：
  - 进程A产生数据，进程B消费数据——B必须在A之后执行
  - 生产者-消费者问题：缓冲区满时生产者必须等待，缓冲区空时消费者必须等待
- **解决**：信号量（Semaphore）、条件变量（Condition Variable）、管程（Monitor）

#### （2）间接约束 —— 互斥（Mutual Exclusion）
- **定义**：多个进程竞争使用同一共享资源
- **场景**：
  - 两个进程同时向打印机输出——输出内容会混杂
  - 两个进程同时修改同一个全局变量——数据不一致
  - 两个进程同时访问同一文件——内容可能损坏
- **特点**：资源本身不可共享（或共享会导致冲突）
- **解决**：互斥锁（Mutex）、信号量（Semaphore）、临界区（Critical Section）
- **临界区（Critical Section）**：进程中访问临界资源的代码段
- **临界资源（Critical Resource）**：一次只允许一个进程使用的资源

## 3.7 线程

### 3.7.1 为什么需要线程

#### 传统进程模型的局限性

在传统的操作系统中，**进程既是资源分配的基本单位，也是CPU调度的基本单位**。这种"双重身份"带来了以下问题：

1. **进程切换的开销大**：
   - 每次进程切换涉及地址空间切换（刷新TLB、页表切换）
   - 上下文的保存和恢复开销显著
   - 频繁切换会严重影响系统性能

2. **进程间通信开销大**：
   - 进程地址空间相互隔离
   - IPC需要内核介入（消息传递、共享内存设置等）
   - 通信延迟较高

3. **并行粒度不够细**：
   - 同一进程内部的不同任务无法并发执行
   - 例如：一个文本编辑器在用户输入的同时进行拼写检查和自动保存——如果只有进程模型，需要三个独立的进程，通信复杂

#### 线程的引入

**线程（Thread）**的引入将进程的两个职能分离开来：
- **进程**：只负责**资源拥有**（Resource Ownership）——管理内存地址空间、文件、I/O设备等
- **线程**：负责**CPU调度执行**（Scheduling/Execution）——是CPU调度的基本单位

这种分离使得：
- 同一进程内的多个线程可以共享进程的资源（地址空间、文件等）
- 线程切换的开销远小于进程切换
- 线程间通信更加方便（通过共享内存）
- 可以实现更细粒度的并发

**现实世界的类比**：
- **进程 ≈ 工厂**：工厂拥有土地、厂房、设备（资源）
- **线程 ≈ 工人**：工人在工厂中干活（执行），共享工厂的资源
- 一个工厂可以有多个工人，工人之间可以方便地共享工具和材料
- 不同工厂之间的资源共享（IPC）需要额外的运输和协调

### 3.7.2 线程的定义

> **线程（Thread）**，又称**轻量级进程（Lightweight Process, LWP）**，是**CPU调度的基本单位**。

线程是进程内部的一条执行路径（执行流），是程序执行的最小单元。

**线程的组成**：
- **线程ID（Thread ID）**：每个线程有唯一的标识
- **程序计数器（PC）**：记录线程当前执行的指令地址
- **寄存器集合**：线程运行时使用的CPU寄存器值
- **栈指针（Stack Pointer）**：每个线程有自己的栈空间
- **线程控制块（TCB, Thread Control Block）**：包含线程的状态信息

**线程共享的进程资源**：
- 代码段（指令代码）
- 数据段（全局变量、静态变量）
- 堆空间（动态分配的内存）
- 打开的文件描述符表
- 信号处理函数
- 当前工作目录
- 用户和组ID

**线程私有的资源**：
- 线程ID
- 程序计数器（PC）
- 寄存器集合
- 栈空间（每个线程有独立的函数调用栈）
- 栈指针（SP）
- 错误码（errno，在POSIX线程中每个线程独立）
- 信号掩码（Signal Mask，每个线程可以设置不同的信号阻塞集合）
- 线程局部存储（TLS, Thread-Local Storage）

### 3.7.3 线程控制块 TCB

与进程的PCB类似，每个线程有一个**线程控制块（TCB）**来管理线程的状态信息。

**TCB的内容**：
- 线程ID（TID）
- 线程状态（就绪、运行、阻塞）
- 程序计数器（PC）
- CPU寄存器集合
- 栈指针（指向线程自己的栈）
- 指向所属进程PCB的指针
- 线程优先级
- 线程私有数据的指针（TLS）

### 3.7.4 进程 vs 线程：完整对比表

以下是进程与线程的八维度对比表（极其重要，常考）：

| 维度 | 进程（Process） | 线程（Thread） |
|---|---|---|
| **1. 基本身份** | 资源分配的基本单位 | CPU调度的基本单位（轻量级进程） |
| **2. 地址空间** | 每个进程有独立的地址空间（代码段、数据段、堆栈） | 同一进程内的所有线程共享进程的地址空间 |
| **3. 资源拥有** | 拥有独立的资源（内存、文件、I/O设备、信号等） | 仅拥有少量必要的资源（栈、寄存器、TCB） |
| **4. 切换开销** | 大——需要切换地址空间、刷新TLB、保存/恢复所有寄存器 | 小——同一进程内切换只需保存/恢复PC和寄存器，地址空间不变 |
| **5. 通信方式** | 需要通过IPC机制（管道、消息队列、共享内存、信号、Socket等），需要内核介入 | 可以直接读写共享内存和全局变量，无需内核介入（但需同步机制） |
| **6. 独立性** | 高度独立——一个进程的崩溃通常不影响其他进程 | 低独立性——一个线程的崩溃可能导致整个进程终止（如段错误） |
| **7. 系统开销** | 创建/撤销/切换均需操作系统参与，开销大 | 用户级线程的创建/切换可在用户空间完成，开销极小 |
| **8. 并发性** | 不同进程可以在多CPU上真正并行 | 同一进程的多个线程可以在多CPU上真正并行（内核级线程） |

**不同场景下的选择建议**：

| 场景 | 适合使用进程 | 适合使用线程 |
|---|---|---|
| 需要强隔离性（安全性、稳定性要求高） | 进程（一个崩溃不影响另一个） | 不推荐 |
| 需要大量并发且频繁切换 | 不推荐 | 线程（切换开销小） |
| 需要共享大量数据 | 不推荐（IPC复杂） | 线程（天然共享地址空间） |
| 多核计算密集任务 | 都可以 | 线程更优（共享数据方便） |
| Web服务器处理大量请求 | 都可以（Apache使用进程） | 线程更优（Nginx、Node.js） |

### 3.7.5 线程的实现模型

线程的实现有三种主要模型，它们在**创建、调度、切换**等方面有本质区别：

#### 模型一：用户级线程（User-Level Threads, ULT）

**实现方式**：
- 线程的管理完全在**用户空间**完成，操作系统内核**不知道**线程的存在
- 由用户空间的**线程库（Thread Library）**负责线程的创建、调度、同步和销毁
- 典型库：POSIX Pthreads（用户模式）、Java线程（JVM管理）、GNU Portable Threads

**工作方式**：
```
           ┌──────────────────────────────────┐
           │        用户空间（User Space）        │
           │                                    │
           │  ┌─────┐  ┌─────┐  ┌─────┐        │
           │  │ T1  │  │ T2  │  │ T3  │ ← 用户级线程 │
           │  └──┬──┘  └──┬──┘  └──┬──┘        │
           │     └────┬───┘───────┘             │
           │          │                          │
           │    ┌─────┴─────┐                   │
           │    │ 线程库     │  ← 调度在用户空间  │
           │    └─────┬─────┘                   │
           ├──────────┼──────────────────────────┤
           │    内核  │  (内核只看到一个进程)      │
           │    ┌─────┴─────┐                   │
           │    │ 内核调度   │  ← 调度单位是进程   │
           │    └─────┬─────┘                   │
           └──────────┼──────────────────────────┘
                      │
                      CPU
```

**ULT的优点**：
1. **线程切换快**（不需要模式切换到内核态）
   - 线程库在用户空间保存/恢复寄存器，只需几条指令
   - 比内核级线程切换快1~2个数量级
2. **定制调度策略**：应用程序可以为自己的线程实现任何调度算法
3. **跨平台兼容**：可以在不支持线程的操作系统上实现
4. **创建/销毁开销小**：不需要系统调用

**ULT的致命缺点**：
1. **一个线程阻塞 = 整个进程所有线程阻塞**：
   - 当线程T1发起系统调用（如read()）时，内核将**整个进程**（包括T1、T2、T3）阻塞
   - 即使T2、T3准备就绪，也无法运行——因为它们在内核看来属于同一个进程
   - 这是ULT最大的问题
2. **不能利用多核CPU**：
   - 内核只按进程分配CPU
   - 即使系统有多个CPU核心，同一进程的多个用户线程仍然只能在一个核心上交替执行
   - 无法实现真正并行（True Parallelism）
3. **线程调度受限于进程调度**：
   - 用户线程调度是"二级调度"——先由内核调度进程（时间片分配给进程），再由线程库在线程间分配
   - 如果时间片分配给整个进程，一个进程中线程数量的增加不会获得更多CPU时间

**适用场景**：
- 要求线程切换速度极快的应用
- 不需要真正的并行执行
- 线程不会频繁阻塞（不执行大量I/O操作）
- 较早的系统或嵌入式系统

#### 模型二：内核级线程（Kernel-Level Threads, KLT）

**实现方式**：
- 线程的管理完全由**操作系统内核**负责
- 内核维护每个线程的TCB，负责线程的创建、调度和切换
- 用户程序通过系统调用接口操作线程
- 典型系统：Windows（全内核线程）、Linux（从2.6内核开始使用NPTL——Native POSIX Threads Library）

**工作方式**：
```
                      ┌──────────────────┐
                      │    用户空间        │
                      │  ┌──┐ ┌──┐ ┌──┐  │
                      │  │T1│ │T2│ │T3│  │ ← 内核级线程（对用户可见）
                      │  └──┘ └──┘ └──┘  │
                      ├──────────────────┤
                      │    内核空间        │
                      │  └──┘ └──┘ └──┘  │ ← 内核调度单位是线程
                      │  T1  T2  T3      │
                      │  每个线程有独立TCB │
                      └──────────────────┘
```

**KLT的优点**：
1. **真正并行**：同一进程的多个线程可以在不同CPU核心上同时执行
2. **一个线程阻塞不影响其他线程**：当一个线程因I/O阻塞时，内核可以调度该进程的其他线程（或其他进程的线程）运行
3. **内核本身也可以多线程化**：内核代码可以在多处理器上并行执行

**KLT的缺点**：
1. **线程切换开销大**：需要模式切换（用户态↔内核态），涉及系统调用
2. **创建/销毁开销大**：每次都需要系统调用，由内核分配/回收内核资源
3. **灵活性低**：线程调度策略由内核决定，应用程序无法定制
4. **对内核的依赖性高**：要求操作系统内核支持多线程

**适用场景**：
- 多核/多处理器系统
- 需要真正并行处理的应用
- I/O密集型应用（线程阻塞不会导致整个进程挂起）
- 现代通用操作系统（Windows、Linux、macOS）

#### 模型三：组合模型（Combined / Hybrid Model）

**实现方式**：
- 用户级线程和内核级线程相结合
- 用户线程在用户空间创建和管理
- 多个用户线程**多路复用**到较少数量的内核线程上
- 引入**轻量级进程（LWP, Lightweight Process）**作为中间结构

**工作方式**：
```
           ┌─────────────────────────────────────┐
           │          用户空间                      │
           │  ┌─T1─┐  ┌─T2──T3─┐  ┌─T4─T5─T6─┐  │ ← 用户级线程
           │  │    │  │        │  │           │  │
           │  └─┐  └──┐  └──┐  └──┐  └────────┘  │
           │     │     │     │     │              │
           │  ┌──LWP1┐ ┌LWP2┐ ┌ LWP3 ┐           │ ← LWP
           │  └──────┘ └────┘ └──────┘           │
           ├──────────────────────────────────────┤
           │         内核空间                       │
           │  ┌──KLT1──┐ ┌──KLT2──┐ ┌──KLT3──┐   │ ← 内核线程
           │  └────────┘ └────────┘ └────────┘   │
           ├──────────────────────────────────────┤
           │              CPU                       │
           └──────────────────────────────────────┘
```

**LWP的作用**：
- LWP是用户级线程和内核级线程之间的**桥梁**
- 每个LWP对应一个内核级线程（在经典组合模型中）
- 用户级线程通过LWP访问内核资源
- 当用户线程进行系统调用时，通过LWP传递到内核线程

**组合模型的优点**：
1. 灵活性强：可以根据需要配置用户线程和LWP的比例
2. 部分并行：多个LWP可以在多核上并行执行
3. 阻塞不影响全部：一个LWP上的用户线程阻塞时，可以切换到另一个LWP

**组合模型的缺点**：
1. 实现复杂（调度逻辑复杂）
2. 管理开销增加
3. LWP数量需要精心设计

### 3.7.6 线程模型的具体实现

#### 多对一模型（Many-to-One）

**对应**：多个用户级线程映射到一个内核级线程

```
用户空间： T1  T2  T3  T4  (用户级线程)
            \   \  |  /
             \  \ | /
              \ \|/
内核空间：    [KLT1]          (一个内核线程)
```

**等价于**：纯用户级线程实现（在操作系统层面，进程只有单个控制线程）

**代表系统**：
- 早期的UNIX系统
- Green threads（Java早期的线程实现）
- GNU Portable Threads

**特点**：
- 一个用户线程阻塞 → 所有用户线程阻塞
- 无法实现多核并行
- 线程切换极快（用户空间）
- 本质上在现代系统中已被淘汰

#### 一对一模型（One-to-One）

**对应**：每个用户级线程映射到一个独立的内核级线程

```
用户空间： T1     T2     T3     T4
            |      |      |      |
内核空间： KLT1   KLT2   KLT3   KLT4
```

**等价于**：纯内核级线程

**代表系统**：
- Linux（NPTL——Native POSIX Threads Library，自2.6内核起）
- Windows（从Windows NT起，所有线程都是内核线程）
- macOS

**特点**：
- 真正并行（多核上同时执行）
- 一个线程阻塞不影响其他线程
- 线程切换需要内核介入，开销相对较大
- 可能的限制：系统能创建的最大线程数受内核资源限制

**Linux NPTL的优化**：
- Linux 2.6内核引入NPTL，使Linux线程性能大幅提升
- clone()系统调用：比传统fork()更灵活地控制资源共享
- 线程创建速度接近进程创建速度的10倍
- 线程切换优化（使用分段栈等）

#### 多对多模型（Many-to-Many）

**对应**：多个用户级线程映射到多个内核级线程（用户线程数 ≥ 内核线程数）

```
用户空间： T1  T2  T3  T4  T5  (用户级线程)
            \  / \  / \  /
             \/   \/   \/
内核空间：  KLT1 KLT2 KLT3    (内核线程)
```

**代表系统**：
- Solaris（早期版本，Solaris 9之前）
- IRIX

**特点**：
- 结合了ULT和KLT的优点：
  - 可以创建大量用户线程（不受内核资源限制）
  - 阻塞不会导致整个进程阻塞（还有其他的内核线程）
  - 可以实现一定程度的并行
- 实现最复杂
- 需要LWP作为中间桥梁

**应用现状**：
- 由于实现复杂度和现代操作系统对一对一模型的高效支持，多对多模型在主流系统中已不太常见
- Solaris从9开始转向两级模型

#### 两级模型（Two-Level Model）

**对应**：多对多模型的变体——允许一个用户线程绑定到一个特定的内核线程

```
用户空间： T1  T2  T3  T4  T5  T6  (用户级线程)
            |  / \   |   / \   |
            | /   \  |  /   \  |
内核空间： KLT1  KLT2  KLT3  KLT4  (内核线程)
```

**代表系统**：
- Solaris 9及以上
- HP-UX

**特点**：
- 兼具多对多的灵活性
- 允许"绑定"（Bounding）——关键线程可以独占一个内核线程
- 比纯粹的多对多模型更灵活

### 3.7.7 线程池（Thread Pool）

#### 为什么需要线程池

在多线程应用中，经常需要处理大量短期的并发任务（如Web服务器处理HTTP请求）。如果为每个请求都创建一个新的线程，会产生以下问题：
- **创建线程的开销大**：需要系统调用，分配内核资源
- **线程数量不受限制**：可能耗尽系统资源
- **创建/销毁的延迟**：影响响应时间

#### 线程池的概念

**线程池（Thread Pool）**在进程启动时创建一定数量的线程，这些线程处于**空闲等待**状态。当有任务到来时：
1. 从线程池中取出一个空闲线程
2. 将任务分配给该线程执行
3. 线程执行完任务后，不销毁，而是回到线程池中继续等待新任务

```
           ┌─────────────────────────────────────┐
           │            线程池                      │
           │                                       │
           │   ┌─────┐  ┌─────┐  ┌─────┐          │
           │   │ T1  │  │ T2  │  │ T3  │  ← 空闲线程 │
           │   └─────┘  └─────┘  └─────┘          │
           │      ↑         ↑                      │
           │      │ 分配任务 │                      │
           │      ├─────────┤                      │
           │   ┌─────┐                             │
           │   │任务1 │  → T1执行中                  │
           │   └─────┘                             │
           │   ┌─────┐                             │
           │   │任务2 │  → T2执行中                  │
           │   └─────┘                             │
           └─────────────────────────────────────┘
```

#### 线程池的优点

1. **限制线程数量**：避免创建过多线程消耗系统资源
2. **减少创建/销毁开销**：线程复用，节省频繁创建和销毁的开销
3. **加快响应速度**：任务到达时已有现成的线程可用，无需等待创建
4. **统一管理**：可以集中管理所有线程的生命周期和调度

#### 线程池的设计考虑

- **池大小（Pool Size）**：线程池中线程的数量
  - 太小 → 任务队列积压，响应慢
  - 太大 → 线程之间竞争CPU和内存资源
  - 理想值取决于CPU核心数、任务类型（CPU密集型 vs I/O密集型）
- **任务队列**：待处理任务的缓冲队列
- **动态调整**：某些系统支持动态调整池大小（如Java的ThreadPoolExecutor）

### 3.7.8 线程取消（Thread Cancellation）

**线程取消**是指在线程完成其预定工作之前强制终止它的行为。

#### 取消方式

##### （1）异步取消（Asynchronous Cancellation）

- **定义**：一个线程立即终止目标线程
- **实现**：调用pthread_cancel()后，立即向目标线程发送取消请求
- **风险**：目标线程可能在执行重要操作（如写入文件、获取锁、分配内存）时被终止
  - 可能导致资源泄漏（锁未释放、内存未回收）
  - 可能导致数据不一致（文件写入一半）
  - 可能导致死锁（持锁线程被取消，其他线程永远无法获取该锁）

##### （2）延迟取消（Deferred Cancellation）

- **定义**：目标线程在**取消点（Cancellation Point）**检查取消请求并自行终止
- **实现**：
  1. 目标线程设置取消类型为延迟取消
  2. 其他线程调用pthread_cancel()发送请求
  3. 目标线程正常执行，在到达取消点时检查是否有取消请求
  4. 如果有取消请求，线程在取消点安全地终止
- **取消点**：通常是可能导致阻塞的系统调用
  - read()、write()、sleep()、wait()、printf()等
  - 可以通过pthread_testcancel()手动插入取消点

**比较**：

| 特性 | 异步取消 | 延迟取消 |
|---|---|---|
| 响应速度 | 快（立即终止） | 慢（直到下一个取消点） |
| 安全性 | 低（可能资源泄漏） | 高（在安全点终止） |
| 实现复杂度 | 简单 | 较复杂（需要定义取消点） |
| 推荐使用 | 不推荐 | 推荐（POSIX标准建议） |

### 3.7.9 线程局部存储（TLS, Thread-Local Storage）

**TLS**是指每个线程拥有其私有数据的副本，虽然变量名相同，但不同线程访问的是不同的内存位置。

#### 为什么需要TLS

在传统的多线程编程中，全局变量被所有线程共享：
```c
int counter = 0;  // 所有线程共享同一个counter

void* thread_func(void* arg) {
    counter++;  // 线程不安全！多个线程同时修改
    return NULL;
}
```

但有些时候，我们需要某些数据是**线程专有的**——例如：
- errno变量：每个线程应独立记录自己的错误码
- 随机数种子（rand()的种子状态）
- 线程特定的配置数据

#### TLS的实现

```c
// C11标准
_Thread_local int tls_var = 0;

// GCC/Clang扩展
__thread int tls_var = 0;

// POSIX pthreads API
pthread_key_t key;
pthread_key_create(&key, NULL);  // 创建TLS键
pthread_setspecific(key, value); // 设置线程私有值
void* val = pthread_getspecific(key); // 获取线程私有值
```

**TLS的工作原理**：
- 编译器为每个TLS变量分配一个"槽位"（slot index）
- 在运行时，每个线程维护一个TLS数组
- 访问TLS变量时，通过线程自己的TLS数组和槽位索引定位到实际数据
- 不同线程的相同TLS变量名指向不同的物理地址

### 3.7.10 多线程程序中的 fork() 和 exec()

当多线程程序调用fork()时，涉及复杂的问题——子进程应该复制所有线程还是只复制调用线程？

#### POSIX标准的规定

**fork()在多线程程序中**：
- **只复制调用线程（calling thread）**到子进程中
- 其他线程在子进程中不再存在
- 这可能导致问题：
  - 如果其他线程正在持有锁，子进程中这些锁永远不会被释放
  - 如果其他线程正在修改数据，数据可能处于不一致状态

**安全使用fork()的注意事项**：
```c
// 安全做法：在fork前获取所有锁，fork后立即释放
pthread_atfork(prepare_func, parent_func, child_func);
// prepare_func：fork前调用，获取所有锁
// parent_func：fork后父进程中调用，释放所有锁
// child_func：fork后子进程中调用，释放所有锁
```

**exec()在多线程程序中**：
- **替换整个进程地址空间**——所有线程（包括调用线程）被销毁
- 新程序从main()开始执行，只有一个初始线程
- 不需要担心遗留线程的问题（全部消失了）

### 3.7.11 调度器激活（Scheduler Activations）

**调度器激活**是一种用于**多对多模型**的**上行调用（Upcall）机制**，用于实现内核和用户级线程库之间的通信。

#### 为什么需要调度器激活

在纯用户级线程实现中，内核不知道线程的存在，导致：
- 当一个线程阻塞时，内核阻塞整个进程（所有用户线程都不能运行）
- 无法实现真正的并行

调度器激活通过以下方式解决这个问题：

#### 工作机制

```
步骤1：用户进程有4个用户线程，映射到2个内核线程（LWP）

步骤2：用户线程T1进行系统调用（如read()），导致阻塞
       → 内核检测到LWP1上的T1阻塞
       → 内核通过上行调用通知用户级线程库："T1阻塞了，但我已经为你保留了一个处理器"

步骤3：线程库收到上行调用后：
       → 保存T1的上下文
       → 将T2调度到LWP1上（使用保留的处理器）

步骤4：当T1等待的I/O完成时：
       → 内核再次通过上行调用通知线程库："T1的I/O完成了"
       → 线程库可以选择：
          a) 立即切换回T1执行
          b) 继续执行T2，将T1放入就绪队列
```

#### 上行调用（Upcall）

- **上行调用 vs 系统调用**：
  - 系统调用：用户程序 → 内核（低到高）
  - 上行调用：内核 → 用户空间（高到低，通知事件）
- 上行调用在用户空间执行，由线程库注册的处理函数处理
- 处理上行调用时，线程库可以重新调度用户线程

#### 调度器激活的优点

1. 内核知道用户线程的存在（"激活"了用户线程")
2. 当一个用户线程阻塞时，内核可以通知线程库
3. 线程库可以在阻塞发生时及时调度其他用户线程
4. 可以更好地支持真正的并行

## 3.8 CPU调度

### 3.8.1 CPU调度概述

**CPU调度（CPU Scheduling）**是操作系统的基本功能，它决定**就绪队列中哪个进程（或线程）获得CPU的使用权**。

#### CPU调度发生的时机

CPU调度（短程调度）在以下四种情况下发生：

1. **进程从运行状态切换到阻塞状态**（如I/O请求）
   - 调度类型：**非抢占式（Non-preemptive）**
2. **进程从运行状态切换到就绪状态**（如时间片用完）
   - 调度类型：**抢占式（Preemptive）**
3. **进程从阻塞状态切换到就绪状态**（如I/O完成）
   - 调度类型：抢占式
4. **进程终止**（正常退出或被杀死）
   - 调度类型：非抢占式

#### 非抢占式调度 vs 抢占式调度

| 特性 | 非抢占式（Non-preemptive） | 抢占式（Preemptive） |
|---|---|---|
| **定义** | 进程一旦获得CPU，就保持CPU直到它主动释放（终止或阻塞） | 操作系统可在任何时候剥夺进程的CPU，时间片用完是典型情况 |
| **调度时机** | 进程主动释放CPU | 任何时刻（时钟中断、I/O中断等） |
| **优点** | 简单，开销小，不需要时钟中断硬件 | 公平性好，响应时间短 |
| **缺点** | 不公平，一个长进程可长时间独占CPU | 复杂，需要处理同步问题（如内核数据结构的竞态条件） |
| **典型系统** | 早期的批处理系统 | 现代分时系统、实时系统 |

#### 调度程序（Dispatcher）

**调度程序**是短程调度中的最后一个模块，负责：
1. **切换上下文**（保存当前进程的状态，恢复新进程的状态）
2. **切换到用户模式**
3. **跳转到新进程程序计数器（PC）指定的地址开始执行**

**调度延迟（Dispatch Latency）**：调度程序停止一个进程并启动另一个进程所花费的时间。

```
                            ←  调度延迟  →
  当前进程执行完毕 ──→ 保存上下文 ──→ 运行调度程序 ──→ 恢复新进程的上下文 ──→ 新进程开始执行
```

### 3.8.2 调度准则（Scheduling Criteria）

不同的调度算法追求不同的性能目标。以下是评价调度算法的主要指标：

#### 核心调度指标

**1. CPU利用率（CPU Utilization）**
- **定义**：CPU处于忙碌状态的时间占总时间的百分比
- **公式**：CPU利用率 = (CPU忙碌时间) / (总时间) × 100%
- **理想值**：40% ~ 90%（轻负载），接近100%（重负载）
- **含义**：CPU利用率越高，说明CPU越"忙碌"，通常意味着系统资源利用越充分

**2. 吞吐量（Throughput）**
- **定义**：单位时间内完成的进程数量
- **公式**：吞吐量 = 完成的进程数 / 总时间
- **单位**：进程数/秒（或任务数/小时）
- **含义**：衡量系统的工作效率——每单位时间能完成多少工作
- **注意**：长进程的吞吐量低于短进程的吞吐量

**3. 周转时间（Turnaround Time）**
- **定义**：从进程提交到进程完成所经过的总时间
- **公式**：
  - 周转时间 = 完成时间 - 提交时间（到达时间）
  - 或者：周转时间 = CPU执行时间 + 等待时间 + I/O时间 + 其他开销
- **含义**：衡量一个进程从"出生"到"死亡"的总时间
- **用户关心的指标**：周转时间越短，用户感觉执行越快

**4. 等待时间（Waiting Time）**
- **定义**：进程在就绪队列中等待CPU的总时间
- **公式**：等待时间 = 周转时间 - CPU执行时间（近似，忽略I/O时间等）
- **含义**：纯"排队"时间
- **注意**：对调度算法来说，等待时间是最需要关注的指标——因为CPU执行时间是固定的（对于给定程序），调度算法能减少的只有等待时间

**5. 响应时间（Response Time）**
- **定义**：从提交请求到产生第一个响应所经过的时间
- **含义**：交互式系统中最重要的指标
- **区别**：
  - 周转时间 = 完成全部工作的时间（批处理关注）
  - 响应时间 = 产生第一个输出的时间（交互式关注）

#### 各指标之间的关系

- **最大化**：CPU利用率、吞吐量
- **最小化**：周转时间、等待时间、响应时间

#### 公式汇总

| 指标 | 公式 | 说明 |
|---|---|---|
| CPU利用率 | CPU忙碌时间 / 总时间 | 越高越好 |
| 吞吐量 | 完成进程数 / 总时间 | 越高越好 |
| 周转时间 | 完成时间 - 到达时间 | 越短越好 |
| 等待时间 | 周转时间 - CPU执行时间（近似） | 越短越好 |
| 响应时间 | 首次响应时间 - 提交时间 | 越短越好 |

### 3.8.3 调度算法详解

以下对每一种调度算法进行详细分析，并给出甘特图（Gantt Chart）示例和计算方法。

为便于说明，考虑以下进程集合：

| 进程 | 到达时间（Arrival Time） | CPU执行时间（Burst Time） |
|---|---|---|
| P1 | 0 | 6 |
| P2 | 2 | 8 |
| P3 | 4 | 7 |
| P4 | 6 | 3 |
| P5 | 7 | 5 |

#### 1. 先来先服务（FCFS, First-Come, First-Served）

**原理**：
- 最简单的调度算法
- 按照进程到达就绪队列的顺序依次执行
- **非抢占式**——一个进程获得CPU后一直执行到结束或阻塞

**实现**：
- 使用FIFO（先进先出）队列
- 新进程到达时插入队尾
- 从队首取出进程执行

**甘特图**（按上述进程集合）：

```
P1        P2              P3              P4   P5
|---------|---------------|---------------|----|-----|
0         6              14              21   24    29
```

**计算各项指标**：

| 进程 | 到达时间 | 执行时间 | 开始时间 | 完成时间 | 周转时间 | 等待时间 |
|---|---|---|---|---|---|---|
| P1 | 0 | 6 | 0 | 6 | 6 | 0 |
| P2 | 2 | 8 | 6 | 14 | 12 | 4 |
| P3 | 4 | 7 | 14 | 21 | 17 | 10 |
| P4 | 6 | 3 | 21 | 24 | 18 | 15 |
| P5 | 7 | 5 | 24 | 29 | 22 | 17 |

- **平均周转时间** = (6 + 12 + 17 + 18 + 22) / 5 = 75/5 = **15**
- **平均等待时间** = (0 + 4 + 10 + 15 + 17) / 5 = 46/5 = **9.2**

**优缺点分析**：

**优点**：
- 实现简单（只需一个队列）
- 公平（按到达顺序服务）

**缺点**：
- **护航效应（Convoy Effect）**：当一个长进程先获得CPU时，所有短进程必须在其后排队等待
  - 如上例中，P1（执行时间6）虽然不长，但它之后的长进程P2（8）和P3（7）导致后面的短进程P4（3）和P5（5）等待了很长时间
  - 极端情况：一个执行100ms的长进程会导致100个执行1ms的短进程等待100ms
- 平均等待时间长（尤其是短进程可能等待很久）
- 不适合分时系统（一个长进程占据CPU，用户交互响应极慢）

#### 2. 短作业优先（SJF, Shortest-Job-First）—— 非抢占式

**原理**：
- 选择**CPU执行时间最短**的就绪进程执行
- **非抢占式**——一旦获得CPU，就执行到结束或阻塞
- 理论上**平均等待时间最小**（已被证明）

**实现**：
- 就绪队列按CPU执行时间排序（短进程在前）
- 当CPU空闲时，选择执行时间最短的进程

**甘特图**：
```
P1        P4   P5        P3              P2
|---------|----|---------|---------------|---------------|
0         6    9         14              21              29
```

**调度过程解释**：
- t=0时，就绪队列只有P1，执行P1
- t=6时，P1结束，此时就绪队列中有P2(8)、P3(7)、P4(3)、P5(5)
  - 最短的是P4(3)，执行P4到t=9
- t=9时，最短的是P5(5)，执行P5到t=14
- t=14时，最短的是P3(7)，执行P3到t=21
- t=21时，只剩P2(8)，执行P2到t=29

**计算各项指标**：

| 进程 | 到达时间 | 执行时间 | 开始时间 | 完成时间 | 周转时间 | 等待时间 |
|---|---|---|---|---|---|---|
| P1 | 0 | 6 | 0 | 6 | 6 | 0 |
| P2 | 2 | 8 | 21 | 29 | 27 | 19 |
| P3 | 4 | 7 | 14 | 21 | 17 | 10 |
| P4 | 6 | 3 | 6 | 9 | 3 | 0 |
| P5 | 7 | 5 | 9 | 14 | 7 | 2 |

- **平均周转时间** = (6 + 27 + 17 + 3 + 7) / 5 = 60/5 = **12**
- **平均等待时间** = (0 + 19 + 10 + 0 + 2) / 5 = 31/5 = **6.2**

对比FCFS：平均等待时间从9.2降到6.2——可见SJF确实优于FCFS。

**优缺点分析**：

**优点**：
- 平均等待时间最小（理论最优——对于非抢占式调度）
- 适合批处理系统（短任务快速完成）

**致命缺点**：
- **需要预知执行时间**：操作系统不可能精确知道一个进程需要执行多久
  - 解决方案：**指数平均法预测**：τ(n+1) = α·t(n) + (1-α)·τ(n)
  - 其中t(n)是第n次实际执行时间，τ(n)是第n次预测值，α是平滑系数
- **可能造成长进程饥饿（Starvation）**：
  - 如果不断有短进程到达，长进程永远得不到CPU
  - 极端情况：长进程可能永远无法执行
- **不适用于分时系统**：不能保证响应时间

#### 3. 最短剩余时间优先（SRTF, Shortest Remaining Time First）—— 抢占式SJF

**原理**：
- SJF的**抢占式**版本
- 当一个新进程到达时，比较新进程的执行时间和当前进程的**剩余执行时间**
- 如果新进程的执行时间 < 当前进程的剩余时间，则**抢占**当前进程

**实现**：
- 每次有进程到达或进程结束时检查最短剩余时间
- 需要时钟中断支持

**甘特图**：
```
P1   P2        P4   P5        P3              P2
|----|---------|----|---------|---------------|---------------|
0    2         6    9         14              21              29
```

**调度过程解释**：
- t=0，P1到达，执行P1（剩余6）
- t=2，P2到达，P1剩余4，P2=8，4<8 → 不抢占，继续执行P1
- t=4，P3到达，P1剩余2，P3=7，2<7 → 不抢占
- t=6，P1结束，此时就绪队列：P2(8)、P3(7)、P4(3)
  - P4最短(3)，执行P4
- t=6，P4到达 → 但实际上P4在t=6刚刚被调度
- t=7，P5到达，此时P4剩余2，P5=5，2<5 → 不抢占
- t=9，P4结束，此时就绪队列：P2(8)、P3(7)、P5(5)
  - P5最短(5)，执行P5
- t=14，P5结束，就绪队列：P2(8)、P3(7)
  - P3最短(7)，执行P3
- t=21，P3结束，执行P2(8)
- t=29，P2结束

**等等——这里需要重新分析！**

**正确的SRTF调度过程**：

- t=0，P1到达，执行P1
- t=2，P2到达，P1剩余4，P2=8 → P1剩余更短，继续执行P1
- t=4，P3到达，P1剩余2，P3=7 → P1剩余更短，继续执行P1
- t=6，P1完成，此时就绪队列中有P2(8)、P3(7)、P4(3)（P4在t=6刚刚到达）
  - 选择P4(3)执行
- t=7，P5到达，P4剩余2，P5=5 → P4剩余更短，继续执行P4
- t=9，P4完成，此时就绪队列：P2(8)、P3(7)、P5(5)
  - 选择P5(5)执行
- t=14，P5完成，此时就绪队列：P2(8)、P3(7)
  - 选择P3(7)执行
- t=21，P3完成，执行P2(8)
- t=29，P2完成

甘特图：
```
P1        P4   P5        P3              P2
|---------|----|---------|---------------|---------------|
0         6    9         14              21              29
```

等一下——这个结果和非抢占SJF是一样的！因为在这个例子中，新进程到达时实际上都没有比当前正在执行的进程的剩余时间更短。让我换一个例子。

**新例子——展示SRTF优于SJF**：

| 进程 | 到达时间 | 执行时间 |
|---|---|---|
| P1 | 0 | 7 |
| P2 | 2 | 4 |
| P3 | 4 | 1 |
| P4 | 5 | 4 |

**非抢占SJF的甘特图**：
```
P1              P3   P2        P4
|---------------|----|---------|----|
0               7    8         12   16
```
- t=0，P1开始执行（此时就绪队列只有P1）
- t=7，P1结束，就绪队列：P2(4)、P3(1)、P4(4)→ 最短是P3(1)
- t=8，P3结束，最短是P2(4) → 执行P2
- t=12，P2结束，执行P4
- t=16，P4结束

平均等待时间：
- P1: 0
- P2: (7 - 2) = 5
- P3: (7 - 4) = 3
- P4: (12 - 5) = 7
- 平均： (0 + 5 + 3 + 7) / 4 = 15/4 = 3.75

**抢占比SJF（SRTF）的甘特图**：
```
P1   P2   P3   P2        P4
|----|----|----|---------|----|
0    2    4    5         9    13
```

**调度过程**：
- t=0，P1到达，执行P1（剩余7）
- t=2，P2到达，P1剩余5，P2=4 → 5>4 → **抢占！P1被切换出去，P2开始执行**
- t=4，P3到达，P2剩余2，P3=1 → 2>1 → **抢占！P2被切换出去，P3开始执行**
- t=5，P3完成，此时就绪队列：P1(剩余5)、P2(剩余2)、P4(4)
  - P2剩余最短(2)，执行P2
- t=7，P2完成，此时就绪队列：P1(剩余5)、P4(4)
  - P4最短(4)，执行P4
- t=11，P4完成，执行P1(剩余5)
- t=16，P1完成

**修正后的SRTF甘特图**：
```
P1   P2   P3   P2        P4         P1
|----|----|----|---------|----------|----|
0    2    4    5         7          11   16
```

平均等待时间：
- P1: (0-0) + (11-2) = 9
- P2: (2-2) + (5-4) = 1
- P3: (4-4) = 0
- P4: (7-5) = 2
- 平均： (9 + 1 + 0 + 2) / 4 = 12/4 = **3.0**

**结论**：SRTF（3.0）的平均等待时间小于非抢占SJF（3.75），证明了抢占式短作业优先在平均等待时间上优于非抢占式。

**优缺点分析**：

**优点**：
- 平均等待时间在抢占式调度中**理论最优**
- 对短进程的响应更快

**缺点**：
- 需要预知执行时间（同SJF）
- 长进程可能**饥饿**（如果不断有更短的进程到达）
- **抢占开销**更大（比SJF更多次上下文切换）
- 不能保证响应时间（同SJF）

#### 4. 优先级调度（Priority Scheduling）

**原理**：
- 每个进程被分配一个优先级
- 调度时选择**优先级最高**的进程执行
- 可以是非抢占式或抢占式

**优先级确定方式**：
- **静态优先级（Static Priority）**：创建时确定，运行期间不变
  - 优点：实现简单
  - 缺点：长进程可能饥饿
- **动态优先级（Dynamic Priority）**：运行时可调整
  - 优点：可以防止饥饿
  - 缺点：实现复杂

**典型的优先级范围**：
- UNIX：nice值（-20到19，数值越小优先级越高）
- Windows：32个优先级等级
- 实时系统：0（最高）~ 255（最低）

**甘特图示例**（非抢占式优先级，数字越小优先级越高）：

| 进程 | 到达时间 | 执行时间 | 优先级 |
|---|---|---|---|
| P1 | 0 | 5 | 3 |
| P2 | 1 | 3 | 1（最高） |
| P3 | 2 | 4 | 4 |
| P4 | 3 | 2 | 2 |

```
P2        P1              P4   P3
|---------|---------------|----|----|
1         4               9    11   15
```

等一下，非抢占式优先级调度的正确过程：
- t=0，P1到达，只有P1，执行P1
- t=1，P2到达（优先级1 > P1的优先级3），但非抢占式→不能抢占，继续执行P1直到结束
- t=5，P1结束，此时就绪队列：P2(3,优先级1)、P3(4,优先级4)、P4(2,优先级2)
  - P2优先级最高(1)，执行P2
- t=8，P2结束，此时：P3(4,优先级4)、P4(2,优先级2)
  - P4优先级更高(2)，执行P4
- t=10，P4结束，执行P3
- t=14，P3结束

```
P1              P2        P4   P3
|---------------|---------|----|----|
0               5         8    10   14
```

平均等待时间：
- P1: 0
- P2: (5-1) = 4
- P3: (10-2) = 8
- P4: (8-3) = 5
- 平均： (0+4+8+5)/4 = 17/4 = 4.25

**饥饿（Starvation）问题**：
- **定义**：低优先级进程永远得不到CPU的情况
- **原因**：如果系统中不断有高优先级进程到达，低优先级进程永远无法被执行
- **解决方案——老化（Aging）**：
  - 在等待过程中逐渐提高低优先级进程的优先级
  - 例如：每等待15分钟，优先级提升1级
  - 最终低优先级进程的优先级会升高到足以被调度执行

#### 5. 轮转调度（Round Robin, RR）

**原理**：
- 每个进程获得一个固定的**时间片（Time Quantum / Time Slice）**
- 就绪队列按FCFS方式排队
- 从队首取进程执行一个时间片
  - 如果进程在时间片内结束 → 正常退出
  - 如果时间片用完进程未结束 → 进程被剥夺CPU，放回就绪队列队尾
- **抢占式**算法

**时间片大小的影响**：
- **时间片太大 → 退化为FCFS**
  - 如果时间片大于所有进程的执行时间，RR=FCFS
  - 响应时间变差（交互用户等待时间长）
- **时间片太小 → 上下文切换开销太大**
  - 系统花费大量时间在上下文切换上
  - CPU有效利用率下降
- **典型值**：10~100ms（现代OS常用10~50ms）
- **经验法则**：时间片应远大于上下文切换时间（通常要大100倍以上）

**时间片 vs 上下文切换的关系**：

```
假设时间片 = 10ms, 上下文切换 = 0.1ms
利用率 = 10 / (10+0.1) = 99%

假设时间片 = 1ms, 上下文切换 = 0.1ms  
利用率 = 1 / (1+0.1) = 91%

假设时间片 = 0.1ms, 上下文切换 = 0.1ms
利用率 = 0.1 / (0.1+0.1) = 50% ← 过半时间浪费在切换上！
```

**甘特图示例**（时间片=4）：

| 进程 | 到达时间 | 执行时间 |
|---|---|---|
| P1 | 0 | 6 |
| P2 | 2 | 8 |
| P3 | 4 | 7 |
| P4 | 6 | 3 |
| P5 | 7 | 5 |

**调度过程**：
- t=0: P1到达，执行P1 [0~4]，P1剩余2
- t=2: P2到达，加入队尾
- t=4: P1时间片到，P1回到队尾。执行P2（队首）[4~8]，P2剩余4
- t=4: P3到达
- t=6: P4到达
- t=7: P5到达
- t=8: P2时间片到，P2回到队尾。执行P3 [8~12]，P3剩余3
- t=12: P3时间片到，P3回到队尾。执行P4 [12~15]（P4只需3，在时间片内完成）
- t=15: P4结束。执行P5 [15~19]，P5剩余1
- t=19: P5时间片到，P5回到队尾。执行P1 [19~21]（P1只需2，完成）
- t=21: P1结束。执行P2 [21~25]（P2剩余4，完成）
- t=25: P2结束。执行P3 [25~28]（P3剩余3，完成）
- t=28: P3结束。执行P5 [28~29]（P5剩余1，完成）

```
P1   P2   P3   P4   P5   P1   P2   P3   P5
|----|----|----|----|----|----|----|----|----|
0    4    8    12   15   19   21   25   28   29
```

**计算各项指标**：

| 进程 | 到达 | 执行 | 完成 | 周转 | 等待 |
|---|---|---|---|---|---|
| P1 | 0 | 6 | 21 | 21 | 15 |
| P2 | 2 | 8 | 25 | 23 | 15 |
| P3 | 4 | 7 | 28 | 24 | 17 |
| P4 | 6 | 3 | 15 | 9 | 6 |
| P5 | 7 | 5 | 29 | 22 | 17 |

- **平均周转时间** = (21+23+24+9+22)/5 = 99/5 = **19.8**
- **平均等待时间** = (15+15+17+6+17)/5 = 70/5 = **14**

**对比不同调度算法的性能**：

| 算法 | 平均周转时间 | 平均等待时间 |
|---|---|---|
| FCFS | 15 | 9.2 |
| SJF（非抢占） | 12 | 6.2 |
| RR (q=4) | 19.8 | 14 |

注意：RR的周转时间和等待时间比FCFS和SJF都差——这是因为时间片导致频繁切换。但RR的**响应时间**远优于FCFS和SJF。

**RR的响应时间计算**：
- 最坏情况响应时间 = (n-1) × q，其中n是就绪进程数，q是时间片
- 假设就绪队列有5个进程，时间片=4ms
- 最坏响应时间 = 4 × 4 = 16ms（远好于FCFS的最坏情况）
- 如果时间片=100ms，最坏响应时间=400ms——仍然可以接受用于交互式系统

#### 6. 多级队列调度（Multilevel Queue）

**原理**：
- 将就绪队列分成多个独立的队列
- 每个队列有自己的调度算法
- 进程被**永久分配**到某个队列（基于进程类型）
- 需要决定队列间的调度方式

**典型的分层**：

```
┌─────────────────────────────┐
│    队列0：系统进程（最高优先级）│ ← FCFS 或 RR
│    （如内核线程）              │
├─────────────────────────────┤
│    队列1：交互式进程          │ ← RR
│    （如shell、GUI应用）        │
├─────────────────────────────┤
│    队列2：批处理进程          │ ← FCFS
│    （如编译、后台计算）         │
├─────────────────────────────┤
│    队列3：实时进程（最低优先级）│ ← 优先级调度
│    （但在实际系统中实时进程优先） │
└─────────────────────────────┘
```

**队列间调度方式**：
1. **固定优先级抢占（Fixed Priority Preemptive）**：
   - 高优先级队列先执行
   - 高优先级队列有进程时，低优先级队列不执行
   - 优点：简单
   - 缺点：低优先级可能饥饿

2. **时间片轮转（Time Slicing）**：
   - 每个队列获得一定比例的CPU时间
   - 例如：队列0获得50%，队列1获得30%，队列2获得20%
   - 防止低优先级饥饿

**优点**：
- 根据进程类型量身定制调度策略
- 适合已知进程分类的系统

**缺点**：
- 进程一旦分配不能移动（不灵活）
- 如果进程分类错误，性能会受影响
- 缺乏适应性（不能根据进程行为变化调整）

#### 7. 多级反馈队列调度（Multilevel Feedback Queue）

**原理**：
- 多个就绪队列，每层优先级从高到低
- **进程可以在队列之间移动**（这是与多级队列的最大区别）
- 调度策略：上层队列使用RR（较小时间片），下层队列使用RR（较大时间片）或FCFS

**经典的三级反馈队列**：

```
┌─────────────────────────────────────────────────────┐
│  队列0（最高优先级）                                 │
│  RR q=8ms                                           │
│  新进程先进入此队列                                  │
├─────────────────────────────────────────────────────┤
│  队列1                                              │
│  RR q=16ms                                          │
│  时间片在队列0未用完的进程移入此队列                    │
├─────────────────────────────────────────────────────┤
│  队列2（最低优先级）                                  │
│  FCFS                                               │
│  时间片在队列1未用完的进程移入此队列                    │
└─────────────────────────────────────────────────────┘
```

**调度规则**：
1. 新进程进入队列0（最高优先级，时间片8ms）
2. 队列0采用RR，时间片8ms
3. 如果进程在8ms内完成 → 正常退出
4. 如果进程在8ms内未完成 → 移动到队列1
5. 队列1采用RR，时间片16ms
6. 如果进程在16ms内完成 → 正常退出
7. 如果进程在16ms内未完成 → 移动到队列2
8. 队列2采用FCFS（或大时间片RR）
9. **只有当高优先级队列为空时，才执行低优先级队列**
10. 如果有进程进入队列0，正在执行的队列1/2进程被抢占

**多级反馈队列的参数**：
一个完整的多级反馈队列调度器需要定义以下参数：
1. 队列数量
2. 每层队列的调度算法
3. 队列间的优先级关系（如何调度队列）
4. 进程何时升级（移到更高级队列）
5. 进程何时降级（移到更低级队列）
6. 进程刚进入时放在哪个队列

**优点（为什么是最全面的调度算法）**：
- **对短进程友好**：短进程在顶层队列快速完成，响应时间短
- **对I/O密集型进程友好**：I/O密集型进程往往在时间片用完前就阻塞了→留在高层队列
- **对长进程公平**：长进程虽然被降到低层队列，但最终仍能获得CPU
- **自适应**：进程行为变化时（如从交互式变为批处理），会自动调整到合适的队列
- **结合了多种算法的优点**：RR的响应速度 + FCFS的低开销

**缺点**：
- **实现复杂**：需要维护多个队列
- **参数选择困难**：队列数量、时间片大小、升级/降级策略需要精心设计
- **可能饥饿**：如果不断有短进程进入系统，底层队列的长进程可能被饿死

**多级反馈队列在实践中的地位**：
- 被许多现代操作系统采用（或类似变体）
- Linux的完全公平调度器（CFS, Completely Fair Scheduler）在概念上类似
- Windows也使用基于优先级的抢占式多级调度

### 3.8.4 实时调度（Real-Time Scheduling）

#### 实时系统概述

实时系统要求任务在**确定的时间限制内**完成。根据严格程度分为两类：

**硬实时系统（Hard Real-Time）**：
- 必须在截止时间（Deadline）前完成任务
- 超过截止时间的后果是灾难性的
  - 例：汽车安全气囊必须在碰撞后10ms内弹出
  - 例：飞机控制系统必须在50ms内响应操纵指令
- 需要精确的时间保证和可调度性分析

**软实时系统（Soft Real-Time）**：
- 偶尔超过截止时间可以接受，但性能会下降
- 超过截止时间不会导致灾难
  - 例：视频播放器错过一帧——画面卡顿但不会崩溃
  - 例：在线游戏服务器的延迟波动

#### 实时调度算法

##### （1）速率单调调度（RMS, Rate Monotonic Scheduling）

**原理**：
- **静态优先级**调度（优先级在任务创建时确定，运行期间不变）
- 优先级分配原则：**周期越短（速率越高）的任务，优先级越高**
- 前提：任务是**周期性的**，且每个任务的CPU执行时间是固定的

**示例**：
| 任务 | 周期（T） | 执行时间（C） | 利用率（C/T） |
|---|---|---|---|
| P1 | 50ms | 10ms | 20% |
| P2 | 100ms | 25ms | 25% |
| P3 | 200ms | 40ms | 20% |

优先级：P1 > P2 > P3（周期越短优先级越高）

**可调度性条件**：

对于n个周期性任务，RMS的**可调度（Schedulable）条件**是：

n(2^(1/n) - 1) ≥ Σ(C_i / T_i)

其中C_i是任务i的执行时间，T_i是任务i的周期。

对于大规模n，该条件趋近于ln2 ≈ 69.3%

**上例检查**：
Σ(C_i / T_i) = 0.2 + 0.25 + 0.2 = 0.65

3 × (2^(1/3) - 1) ≈ 3 × (1.26 - 1) = 0.78

0.65 < 0.78 → **可调度** ✓

##### （2）最早截止时间优先（EDF, Earliest Deadline First）

**原理**：
- **动态优先级**调度
- 截止时间（Deadline）越近的任务，优先级越高
- 在每次调度时，选择截止时间最近的就绪任务执行

**EDF vs RMS**：

| 特性 | RMS | EDF |
|---|---|---|
| 优先级类型 | 静态（固定） | 动态（每次调度变化） |
| 可调度性上限 | ~69%（大n时） | 100%（理论上） |
| 实现复杂度 | 简单 | 较复杂（需要跟踪每个任务的截止时间） |
| 过载行为 | 可预测（低优先级任务先失败） | 不可预测（所有任务可能都失败） |
| 适用场景 | 任务数固定，周期稳定 | 任务数可变，周期多变 |

**EDF的可调度性条件**：
Σ(C_i / T_i) ≤ 1（利用率不超过100%）

这意味着EDF可以100%利用CPU仍然保证所有任务在截止时间前完成（理论上）。

### 3.8.5 优先级反转与继承（Priority Inversion and Inheritance）

#### 优先级反转的定义

**优先级反转**是指一个**低优先级**进程持有了一把锁，导致**高优先级**进程被阻塞等待，而**中优先级**进程（不需要该锁）却可以抢占低优先级进程，导致高优先级进程的延迟不可预测。

#### Mars Pathfinder 经典案例

1997年，NASA的Mars Pathfinder火星探测器在着陆后几天，系统开始不断重启。原因是**优先级反转**触发了看门狗定时器：

**场景**：
- 高优先级任务：总线管理任务（需要频繁运行）
- 中优先级任务：天气预报任务（长时间运行，不与总线任务共享数据）
- 低优先级任务：数据收集任务（持有共享互斥锁）

**问题过程**：
1. 低优先级任务获取互斥锁，开始收集数据
2. 高优先级任务被唤醒，需要获取同一把锁→被阻塞
3. 中优先级任务（不需要锁）开始运行，抢占低优先级任务
4. 低优先级任务无法释放锁（被中优先级抢占）
5. 高优先级任务一直等待锁→无法运行
6. 高优先级任务超时→系统看门狗触发重启

**解决方案——优先级继承（Priority Inheritance）**：

**原理**：
- 当低优先级任务持有高优先级任务所需的锁时
- 低优先级任务**临时继承**高优先级任务的优先级
- 低优先级任务以高优先级运行，尽快释放锁
- 锁释放后，低优先级任务恢复原来的优先级

**Mars Pathfinder的修复**：
- 工程师上传了修复代码，启用了优先级继承协议
- 问题不再发生
- 这也是优先级继承协议最著名的案例

**优先级继承 vs 优先级天花板**：

| 协议 | 机制 | 实现复杂度 |
|---|---|---|
| **优先级继承（Priority Inheritance）** | 临时提升持有锁的任务的优先级 | 较简单 |
| **优先级天花板（Priority Ceiling）** | 每个锁有一个"天花板优先级"，代表可能请求该锁的最高优先级。任何任务要获取锁，其优先级必须高于所有已锁定资源的当前天花板 | 较复杂，但可避免死锁 |

### 3.8.6 多处理器调度（Multiprocessor Scheduling）

#### 多处理器调度的挑战

在多CPU/多核系统中，调度问题变得更加复杂：
- 哪些CPU执行哪些进程？
- 如何平衡负载？
- 如何管理缓存亲和性？

#### 处理器亲和性（Processor Affinity）

**定义**：进程倾向于在同一个CPU上运行的特性。

**为什么需要处理器亲和性**：
- 当进程在CPU上运行时，其数据被加载到该CPU的Cache中
- 如果进程被迁移到另一个CPU，其Cache数据无效（Cache Miss）
- 频繁迁移会导致性能下降（重新加载Cache数据）

**软亲和性（Soft Affinity）**：
- 操作系统尽量让进程在同一个CPU上运行
- 但不保证（负载均衡时可能迁移）

**硬亲和性（Hard Affinity）**：
- 进程明确指定只能在某个CPU子集上运行
- Linux：sched_setaffinity()系统调用

#### 负载均衡（Load Balancing）

**定义**：在多处理器系统中，将工作负载均匀分布到所有CPU上。

**为什么需要负载均衡**：
- 即使有亲和性，也要避免"某些CPU繁忙，某些CPU空闲"的情况
- 目标：最大化系统吞吐量

**推迁移（Push Migration）**：
- 一个守护进程定期检查CPU的负载
- 将任务从负载高的CPU推到负载低的CPU
- 例如：Linux的load_balance()函数

**拉迁移（Pull Migration）**：
- 空闲的CPU主动从繁忙的CPU的就绪队列中"拉"任务
- 例如：Linux的idle_balance()函数

#### 对称多处理（SMP）调度

**对称多处理（SMP, Symmetric Multi-Processing）**中，所有CPU是对等的：

**调度的两种方式**：

1. **每个CPU维护自己的就绪队列（Per-CPU Ready Queue）**：
   - 优点：缓存友好，扩展性好，竞争少
   - 缺点：负载可能不均衡
   - Linux采用此方式

2. **全局就绪队列（Global Ready Queue）**：
   - 所有进程放在一个全局队列中
   - 任何一个CPU空闲时从全局队列取进程
   - 优点：天然负载均衡
   - 缺点：竞争（需要加锁），缓存不友好

### 3.8.7 调度算法的选择指南

| 系统类型 | 主要目标 | 推荐算法 |
|---|---|---|
| 批处理系统 | 高吞吐量、短周转时间 | SJF、SRTF |
| 交互式系统（分时系统） | 短响应时间、公平 | RR、多级反馈队列 |
| 实时系统 | 满足截止时间 | RMS、EDF |
| 通用操作系统 | 兼顾所有指标 | 多级反馈队列 |
| Web服务器 | 高并发、短响应 | 多级反馈队列 + 线程池 |

**现代操作系统的实际选择**：
- **Linux**：完全公平调度器（CFS），基于红黑树的多级反馈队列变体
- **Windows**：基于优先级的抢占式多级反馈队列
- **macOS/iOS**：基于Mach的多级反馈队列
- **Solaris**：多级反馈队列 + 调度类框架

---

以下是在学习进程管理时最容易出错的10+个知识点：

### 易错点1：阻塞 → 运行 是不可能的状态转换

**错误理解**：认为阻塞状态的进程被唤醒（I/O完成）后可以直接进入运行状态。

**正确理解**：阻塞进程被唤醒后进入**就绪状态**，必须排队等待调度程序分配CPU才能进入运行状态。阻塞→就绪→运行，不能跳跃。

**记忆方法**：阻塞队列中所有进程都缺少"事件"，不是缺少CPU。事件发生后，它们不再缺少事件，但还需要等待CPU，所以先进就绪队列。

### 易错点2：运行 → 就绪 是被动的，运行 → 阻塞 是主动的

**错误理解**：认为所有状态转换都是被动的。

**正确理解**：
- **运行→就绪**：被动的（时间片用完或被更高优先级进程抢占）
- **运行→阻塞**：**主动的**（进程自己调用block()，请求I/O等）
- 这是进程三状态中**唯一一个由进程主动触发的转换**

**记忆方法**：进程主动"放弃"CPU只有一种情况——它需要等待某资源/事件。其他情况都是被剥夺。

### 易错点3：PCB是进程的唯一标识符，而不是PID

**错误理解**：认为PID是进程存在的唯一标识。

**正确理解**：虽然PID很重要，但**PCB才是进程存在的唯一标识**。操作系统通过PCB感知进程的存在。一个进程无论有没有PID（创建过程中），只要有PCB就代表它存在。终止的进程（僵尸进程）仍然保留PCB。

### 易错点4：上下文切换 ≠ 模式切换

**错误理解**：认为系统调用导致上下文切换。

**正确理解**：
- **系统调用**：仅模式切换（用户态↔内核态），不切换进程，不改变地址空间
- **时间片用完**：上下文切换（切换进程，切换地址空间，保存/恢复寄存器）
- 模式切换的开销远小于上下文切换

### 易错点5：系统调用期间也可能发生上下文切换

**错误理解**：认为系统调用过程中不会发生进程切换。

**正确理解**：虽然普通系统调用只是模式切换，但在以下情况下系统调用可能导致上下文切换：
- 进程请求I/O操作（如read()）→ 进入阻塞 → 需要切换到其他进程
- 进程调用exit() → 进程终止 → 需要切换到其他进程
- 系统调用中唤醒更高优先级进程 → 可能触发抢占

### 易错点6：用户级线程的一个线程阻塞 = 所有线程都阻塞

**错误理解**：认为用户级线程中每个线程独立执行，一个阻塞其他还能运行。

**正确理解**：在**用户级线程（ULT）实现**中，内核只认识进程，不认识线程。当一个用户线程发起系统调用（如I/O请求），内核将**整个进程**阻塞——这意味着该进程的所有线程都无法运行，即使其他线程是就绪的。

而在**内核级线程（KLT）实现**中，内核认识每个线程，一个线程阻塞不影响其他线程。

### 易错点7：SJF需要预知未来，SRTF也需要

**错误理解**：认为SJF/SRTF在实际系统中完全可行。

**正确理解**：SJF和SRTF都需要预知每个进程的CPU执行时间（Burst Time），这在现实中是不可能的。操作系统只能通过**预测**（指数平均法）来估算，但预测并不准确。因此实际系统中没有纯粹的SJF/SRTF实现。

**实际问题**：
- 指数平均预测：τ(n+1) = α·t(n) + (1-α)·τ(n)
- α=0.5的常见选择意味着最近和历史的权重各半
- 预测误差可能导致调度次优

### 易错点8：RR的平均周转时间不一定比FCFS好

**错误理解**：认为RR是"更好的"调度算法，各项指标都应该优于FCFS。

**正确理解**：RR的主要优势是**响应时间**短，而不是周转时间。事实上：
- RR的**周转时间**通常比FCFS长（因为上下文切换增加了开销）
- RR的**等待时间**通常比FCFS长（同一进程可能被多次中断/恢复）
- RR的**响应时间**远优于FCFS（这就是RR的用途所在）

### 易错点9：多级反馈队列的参数选择极其关键

**错误理解**：认为只要使用多级反馈队列就能自动获得好的性能。

**正确理解**：多级反馈队列的性能**高度依赖于参数设置**：
- 队列数量：太少了不够灵活，太多了管理复杂
- 每层时间片大小：太小了切换太多，太大了退化
- 升降级策略：升降太快或太慢都会影响性能
- 这些参数需要根据系统负载特征调整

### 易错点10：僵尸进程和孤儿进程的区别

**错误理解**：混淆僵尸进程和孤儿进程。

**正确理解**：
| 性质 | 僵尸进程（Zombie） | 孤儿进程（Orphan） |
|---|---|---|
| **状态** | 已终止但PCB未被回收 | **仍在运行**（未终止） |
| **父进程** | 父进程仍在但不wait | 父进程已终止 |
| **资源占用** | PCB（很小） | 正常占用所有资源 |
| **新父进程** | 无变化 | 被init进程收养 |
| **危害** | 少量，但大量积累会耗尽进程表 | 基本无害（init会管理） |
| **终止** | 父进程必须wait()或自身终止 | init进程在其结束时自动回收 |

### 易错点11：长程/中程/短程调度的混淆

**错误理解**：搞混三级调度的作用对象和频率。

**正确理解**：
- **长程调度（作业调度）**：控制作业进入内存的时机 → 控制多道程序度 → 频率低（分钟级）
- **中程调度（交换调度）**：进程换入换出 → 缓解内存压力 → 频率中等（秒级）
- **短程调度（CPU调度）**：选择哪个进程获得CPU → 核心调度 → 频率极高（毫秒级）

### 易错点12：时间片太小的后果分析

**错误理解**：认为时间片越小，调度越公平，响应越快，因此越小越好。

**正确理解**：时间片过小导致**上下文切换开销占CPU比例过高**：
- 如果时间片=1ms，上下文切换=0.5ms，那么1/3的CPU时间浪费在切换上
- CPU有效利用率 = q / (q + switch_time)
- 现代OS通常选择10~100ms的时间片，将切换开销控制在1%以内

# 第五章：本章常考题型

---

以下是本章在考试中常见的题型及解题思路：

## 题型一：状态转换题

**典型题目**：
> 进程在运行过程中可能发生哪些状态转换？分别说明触发条件。哪些转换是主动的，哪些是被动的？

**解题思路**：
1. 画出三状态或五状态图
2. 逐一列出所有可能的转换
3. 标明触发条件和主动性

**参考答案要点**：
- 五状态：New→Ready→Running↔Ready, Running→Blocked, Blocked→Ready, Running→Terminated
- 主动转换：仅Running→Blocked（进程自己请求I/O或等待资源）
- 被动转换：其他所有转换

## 题型二：调度算法计算题

**典型题目**：
> 给定一组进程的到达时间和CPU执行时间，计算FCFS、SJF（非抢占）、SRTF（抢占式SJF）、RR的周转时间和等待时间。

**解题步骤**：
1. **画甘特图**：按时间轴画出每个进程的CPU占用区间
2. **填表**：计算每个进程的完成时间、周转时间、等待时间
3. **求平均**

**重要技巧**：
- 甘特图是得分关键——图片清晰、标注时间点
- SJF/SRTF中如果执行时间相同，按PID顺序执行
- RR中别忘了新到达的进程按FCFS规则放在队尾

## 题型三：计算题——等待时间和周转时间

**重点公式**：
- 周转时间 = 完成时间 - 到达时间
- 等待时间 = 周转时间 - 执行时间（注意：这里的"执行时间"是CPU burst，不是总时间）

**常见陷阱**：
- I/O时间的影响：如果题目中给出了I/O时间，等待时间 = 周转时间 - CPU执行时间 - I/O时间
- 上下文切换时间：如果题目给出上下文切换时间，需要在甘特图中体现

## 题型四：进程与线程的对比

**典型题目**：
> 比较进程和线程在以下方面的区别：地址空间、切换开销、通信方式、系统资源占用。

**答题框架**（按8维度对比表的思路）：
1. 基本身份（资源分配 vs 调度单位）
2. 地址空间（独立 vs 共享）
3. 资源拥有（全面 vs 少量）
4. 切换开销（大 vs 小）
5. 通信方式（内核IPC vs 共享内存）
6. 独立性（高 vs 低）
7. 系统开销（大 vs 小）
8. 并发性（进程级 vs 线程级并行）

## 题型五：用户级线程 vs 内核级线程

**典型题目**：
> 说明用户级线程和内核级线程的区别，并分析各自的优缺点。

**答题要点**：
1. 管理主体（用户库 vs 内核）
2. 切换开销（小 vs 大）
3. 阻塞影响（全进程阻塞 vs 单线程阻塞）
4. 多核利用（不能 vs 能）
5. 系统调用开销（无 vs 有）

## 题型六：多级反馈队列分析题

**典型题目**：
> 描述多级反馈队列调度算法的原理，画出三级反馈队列的示意图，并分析其优点。

**答题要点**：
1. 画出三级反馈队列示意图
2. 描述进程在各层之间的移动规则
3. 分析对短进程、I/O进程、长进程的影响
4. 总结：自适应性、响应时间、公平性

## 题型七：结合PV操作的进程同步题（跨章节）

**注意**：进程同步的PV操作题虽然在第四章中详细讲解，但考试中常结合本章的进程状态转换一起考察。

**典型命题**：
> "在生产者-消费者问题中，说明进程状态如何变化：消费者发现缓冲区为空时从运行状态转换为什么状态？"

**分析**：
- 消费者发现缓冲区为空 → 调用P(empty) → 信号量<0 → 进入阻塞队列
- 状态转换：**Running → Blocked**
- 生产者放入产品后 → 调用V(empty) → 唤醒消费者 → 消费者从Blocked变为Ready

## 题型八：上下文切换开销计算题

**典型题目**：
> 如果上下文切换需要2ms，时间片大小为20ms、50ms、100ms时，分别计算CPU用于上下文切换的开销占比。

**计算**：
- 每个时间片需要2次上下文切换（切换进来+切换出去）
- 总开销 = 切换时间 × 2 = 4ms
- 开销占比 = 4 / (20+4) = 16.7% (q=20ms)
- 开销占比 = 4 / (50+4) = 7.4% (q=50ms)
- 开销占比 = 4 / (100+4) = 3.8% (q=100ms)

## 题型九：实时调度分析题

**典型题目**：
> 三个周期性任务，周期分别为50ms、100ms、200ms，执行时间分别为12ms、20ms、35ms。问该任务集在RMS下是否可调度？

**解题**：
1. 计算总利用率：U = 12/50 + 20/100 + 35/200 = 0.24 + 0.20 + 0.175 = 0.615
2. 计算RMS可调度上限：n=3时，3×(2^(1/3)-1) ≈ 3×0.26 ≈ 0.78
3. 因为0.615 < 0.78 → **可调度**

## 题型十：概念辨析题

**常见辨析题目**：
1. "PCB和PID哪个是进程存在的唯一标识？"
2. "系统调用是否一定会导致上下文切换？"
3. "用户级线程和内核级线程哪个更轻量？"
4. "挂起和阻塞有什么区别？"
5. "僵尸进程和孤儿进程哪个危害更大？"

# 附录：重要概念速查表

| 概念 | 核心要点 |
|---|---|
| 进程 | 程序的一次执行过程，资源分配的基本单位 |
| PCB | 进程控制块，进程存在的唯一标识，包含所有状态信息 |
| 就绪状态 | 拥有除CPU外所有资源 |
| 阻塞状态 | 等待事件发生（I/O、锁等） |
| 上下文切换 | 保存旧进程状态，恢复新进程状态，是纯粹的开销 |
| 模式切换 | 用户态↔内核态，不切换进程，开销小 |
| 线程 | CPU调度的基本单位，轻量级进程 |
| ULT | 用户级线程，用户库管理，切换快但一个阻塞全阻塞 |
| KLT | 内核级线程，内核管理，真正的并行 |
| FCFS | 先来先服务，简单但可能导致护航效应 |
| SJF | 短作业优先，平均等待时间理论最优，但需预测 |
| SRTF | 抢占式短作业优先，平均等待时间理论最优 |
| RR | 轮转调度，时间片是关键参数，保证响应时间 |
| 多级反馈队列 | 最全面的调度算法，自适应 |
| RMS | 速率单调调度，静态优先级，周期越短优先级越高 |
| EDF | 最早截止时间优先，动态优先级 |
