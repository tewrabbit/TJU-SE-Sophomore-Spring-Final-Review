# 第五章 内存管理（Memory Management）

## 本章概述

内存管理是操作系统的核心功能之一，负责对内存资源进行分配、回收和保护。本章从程序装入与链接开始，逐步深入到连续分配、分页、分段、段页式存储管理，最后讨论虚拟存储器、页面置换算法、抖动以及内核内存管理等高级主题。本章是操作系统课程中最重要的章节之一，也是各类考试的必考内容。

## 5.1 程序装入与链接

### 5.1.1 从源程序到内存执行的完整流程

一个用户程序要从磁盘上的源代码变成内存中正在执行的进程，需要经历以下三个阶段：

**（1）编译（Compile）**
由编译器（Compiler）将用户编写的源代码（如C、C++、Java等）编译成若干个目标模块（Object Module）。每个目标模块中的地址是相对地址（Relative Address），也称为逻辑地址或虚拟地址，通常从0开始编址。

例如：假设有两个源文件 main.c 和 utils.c，经过编译后分别生成 main.o 和 utils.o 两个目标模块。每个目标模块都有自己的地址空间，都是从0开始。

**（2）链接（Link）**
由链接器（Linker）将编译得到的一组目标模块以及它们所需要的库函数链接在一起，形成一个完整的装入模块（Load Module），也称为可执行文件。链接过程主要解决模块间的符号引用问题，将各个模块中的相对地址转换为整个程序的相对地址（即逻辑地址空间中的地址）。

**（3）装入（Load）**
由装入器（Loader）将链接后形成的装入模块装入内存中，此时需要将逻辑地址转换为物理地址（Physical Address），即内存中的实际地址。

### 5.1.2 地址绑定的三个时机

地址绑定（Address Binding）是指将程序中的逻辑地址转换为物理地址的过程。根据绑定发生的时间不同，可以分为以下三种：

**（1）编译时地址绑定（Compile-Time Binding）**
如果程序在编译时就知道它将驻留在内存中的哪个位置，那么编译器可以生成绝对代码（Absolute Code）。例如，MS-DOS的.COM文件格式就是在编译时绑定地址的。

- 优点：简单，不需要硬件支持
- 缺点：如果起始地址发生变化，必须重新编译程序
- 适用场景：单道程序环境，或者嵌入式系统中

**（2）装入时地址绑定（Load-Time Binding）**
如果编译器在编译时不知道程序将驻留在内存中的哪个位置，那么编译器必须生成可重定位代码（Relocatable Code）。在装入时，装入器根据内存的当前使用情况，将逻辑地址转换为物理地址。

- 优点：程序可以在内存中的不同位置执行，增加了灵活性
- 缺点：程序一旦装入内存后就不能再移动位置
- 适用场景：多道程序环境，但不支持交换

**（3）运行时地址绑定（Execution-Time Binding）**
如果进程在运行过程中可以在内存中移动（例如，由于交换或紧凑），那么必须将地址绑定推迟到运行时。这种方案需要硬件支持，通常使用基址寄存器（Base Register）和界限寄存器（Limit Register）来实现。

- 优点：程序可以在执行过程中移动，支持交换和紧凑
- 缺点：需要硬件支持，地址转换会增加开销
- 适用场景：现代操作系统（如Windows、Linux）均采用此方式

### 5.1.3 链接方式详解

**（1）静态链接（Static Linking）**
在程序运行之前，将所有目标模块和所需的库函数链接成一个完整的装入模块，以后不再拆开。

- 优点：程序运行时不需要额外的链接开销，运行速度快
- 缺点：
  - 浪费空间：如果多个程序使用同一个库函数，每个程序的装入模块中都包含该库函数的副本
  - 更新困难：如果库函数需要更新，所有链接了该库的程序都必须重新链接
- 典型例子：C语言中的静态库（.a文件，.lib文件）

**（2）装入时动态链接（Dynamic Linking at Load Time）**
在装入程序时，边装入边链接。也就是说，在程序被装入内存时，如果发现它调用了某个动态链接库中的函数，则立即将该动态链接库装入内存并完成链接。

- 优点：多个程序可以共享同一个动态库的同一份副本，节省内存空间
- 缺点：装入时间较长，如果程序运行中不需要的动态库也会被装入
- 典型例子：早期的Windows DLL（动态链接库）

**（3）运行时动态链接（Dynamic Linking at Runtime）**
在程序运行过程中，只有当真正需要调用某个动态链接库中的函数时，才将该动态库装入内存并完成链接。这种方式的灵活性最高，也是现代操作系统普遍采用的方式。

- 优点：
  - 延迟装入（Lazy Loading）：只装入真正需要的模块
  - 共享代码：多个进程可以共享同一个动态库的物理内存页面
  - 便于更新：只需替换动态库文件，无需重新链接程序
- 缺点：第一次调用动态库中的函数时会有延迟
- 典型例子：Linux的共享对象（.so文件），Windows的DLL文件
- 关键机制：动态链接库中的代码必须是位置无关代码（Position-Independent Code, PIC），这样不同进程才能共享同一份物理副本

### 5.1.4 装入方式详解

**（1）绝对装入方式（Absolute Loading）**
在编译时就知道程序将驻留在内存中的哪个位置，编译器生成绝对地址的代码。装入器只需将程序装入到指定的内存位置即可，不需要进行地址转换。

- 适用条件：程序中的地址是绝对地址（物理地址）
- 优点：实现简单，装入速度快
- 缺点：灵活性极差，如果内存起始地址变化，必须重新编译
- 典型应用：早期操作系统、嵌入式固件

**（2）可重定位装入方式（Static Relocation / Relocatable Loading）**
装入器根据内存的当前使用情况，将装入模块中的逻辑地址全部转换为物理地址。这个转换过程是在装入时一次性完成的，之后程序在运行过程中不能移动。

- 适用条件：目标模块使用相对地址（从0开始）
- 地址转换公式：物理地址 = 逻辑地址 + 装入起始地址
- 优点：程序可以装入到内存中的任何空闲区域，提高了内存利用率
- 缺点：程序装入后不能移动，不支持交换和紧凑
- 注意点：静态重定位需要修改程序中的每个地址引用，因此程序代码在装入后不能被多个进程共享

**（3）动态运行时装入方式（Dynamic Relocation）**
将地址转换推迟到程序运行时进行。这需要硬件支持，通常使用一对寄存器：基址寄存器（Base Register）和界限寄存器（Limit Register）。

- 基址寄存器：存放进程在内存中的起始物理地址
- 界限寄存器：存放进程的长度（或结束地址），用于越界检查
- 地址转换公式：物理地址 = 逻辑地址 + 基址寄存器中的值
- 优点：
  - 程序可以在执行过程中移动
  - 支持交换和紧凑
  - 多个进程可以共享同一段代码（只需设置不同的基址寄存器）
- 缺点：每次内存访问都需要进行地址转换，增加了硬件开销
- 注意：界限寄存器中的值是与逻辑地址进行比较还是与物理地址进行比较，取决于具体实现。通常，CPU先检查逻辑地址是否小于界限寄存器中的值，然后再进行基址加法。

**典型数值计算示例：**
假设某进程被装入到物理地址从10000开始的位置，该程序的逻辑地址空间大小为20000字节。
- 界限寄存器 = 20000（逻辑地址空间大小）
- 基址寄存器 = 10000
- 当CPU访问逻辑地址 3584 时：
  1. 检查：3584 < 20000 → 合法
  2. 物理地址 = 3584 + 10000 = 13584
- 当CPU访问逻辑地址 25000 时：
  1. 检查：25000 >= 20000 → 越界，产生地址越界中断

## 5.2 连续分配方式

连续分配（Contiguous Allocation）是指为每个用户程序分配一片连续的内存空间。连续分配是最早使用的内存管理方式，按实现方式的不同分为固定分区分配和动态分区分配。

### 5.2.1 固定分区分配（Fixed Partitioning）

固定分区分配是最简单的多道程序内存管理方式。它在系统初始化时将内存划分为若干个固定大小的区域（分区），每个分区中只能装入一道程序。

**（1）分区大小**
- 相等分区（Equal-Sized Partitions）：所有分区大小相同
  - 缺点：任何程序大小不能超过一个分区的大小；小程序会浪费大量空间
  - 适用场景：程序大小已知且固定的环境
- 不等分区（Unequal-Sized Partitions）：分区大小不同
  - 优点：大程序可以用大分区，小程序可以用小分区，减少浪费
  - 分配策略：可以为每个分区维护一个队列，或者维护一个全局队列

**（2）内部碎片（Internal Fragmentation）**
内部碎片是指分配给进程的内存块中，未被进程使用的部分。由于分区大小固定，当进程大小小于分区大小时，剩余的空白部分无法被其他进程使用，造成浪费。

例如：分区大小为256KB，一个程序只需要200KB，那么分区内剩余的56KB就是内部碎片。

**（3）固定分区的优缺点**
- 优点：
  - 实现简单，开销小
  - 操作系统管理方便
- 缺点：
  - 内部碎片严重，内存利用率低
  - 分区总数固定，限制了多道程序的道数
  - 程序大小不能超过最大分区的大小
- 现在几乎不再使用，仅用于某些专用系统或嵌入式系统

### 5.2.2 动态分区分配（Dynamic Partitioning）

动态分区分配（也称为可变分区分配）在进程装入时动态地创建分区，使分区的大小正好等于进程的大小。

**（1）基本原理**
初始时，整个用户空间是一个大空闲块。当进程到达时，操作系统从空闲块中分割出一块正好适合（或稍大）的区域分配给进程。当进程结束时，释放其占用的分区，这些分区可能与其他空闲分区合并。

**（2）外部碎片（External Fragmentation）**
外部碎片是指所有空闲块的总和虽然足够大，但每个空闲块都比较小，无法满足某个进程的需求。外部碎片是动态分区分配的主要问题。

例如：内存总空闲空间为50MB，但被分成了10个5MB的碎片，此时如果需要分配一个10MB的进程，则无法满足。

**（3）紧凑（Compaction）**
解决外部碎片的一种方法是紧凑（也称为紧缩，Compaction）。通过移动已分配分区的内容，将所有空闲块合并成一个连续的大空闲块。

- 优点：可以有效消除外部碎片
- 缺点：
  - 紧凑需要移动大量数据，开销很大
  - 紧凑必须在运行时进行，需要动态重定位硬件的支持
  - 紧凑的时机难以确定（是立即紧凑还是等待碎片积累到一定程度再紧凑）
- 注意：紧凑只能在支持动态重定位的系统中使用（即程序在运行时可以移动）

**（4）动态分区的优缺点**
- 优点：
  - 分区大小和数量可变，灵活性强
  - 没有内部碎片（或很少，因为分配时的剩余部分通常很小）
- 缺点：
  - 存在外部碎片问题
  - 分配和释放的管理比固定分区复杂
  - 需要紧凑来解决碎片问题，但紧凑开销大

### 5.2.3 分区分配算法（Partition Allocation Algorithms）

下面详细分析四种常用的动态分区分配算法。

#### 首次适应算法（First-Fit, FF）

**算法描述：**
将空闲分区按地址递增的顺序（即从内存低地址到高地址）链接成一个空闲分区链（Free List）。当需要分配一个大小为S的分区时，从链首开始顺序查找，找到第一个大小大于等于S的空闲分区。如果该分区大小等于S，则直接分配；如果大于S，则从中分割出S大小的区域分配给进程，剩余部分仍留在空闲链中。

**数据结构：**
- 空闲分区链：按起始地址递增顺序排列
- 每个空闲分区节点：包含起始地址、大小、指向下一个节点的指针

**分配示例：**
假设空闲分区链为：A(0, 50KB) → B(100KB, 30KB) → C(200KB, 60KB)
- 请求分配 40KB：从头开始，找到A(50KB)，50 >= 40，从A中分割40KB，A变为(40, 10KB)
- 请求分配 50KB：从头开始，A现在是(40, 10KB)，10 < 50 → 跳过；B(100, 30KB)，30 < 50 → 跳过；C(200, 60KB)，60 >= 50 → 分配
- 请求分配 8KB：从头开始，A(40, 10KB)，10 >= 8 → 分配，A变为(48, 2KB)

**优点：**
- 算法简单，实现容易
- 尽量利用低地址空间，高地址空间保留了大块空闲区（当有大进程到达时，可以直接使用高地址的大块）

**缺点：**
- 低地址空间会被分割成越来越小的碎片
- 每次查找从头开始，随着低地址碎片增多，查找时间变长

#### 循环首次适应算法（Next-Fit, NF）

**算法描述：**
循环首次适应算法是首次适应算法的一种变体。它不是每次从链首开始查找，而是从上一次查找结束的位置开始继续查找。当查找到链尾时，回到链首继续查找。

**数据结构：**
- 与首次适应算法相同的空闲分区链（按地址递增）
- 额外的指针：记录上次查找结束的位置

**分配示例：**
假设空闲分区链为：A(0, 50KB) → B(100KB, 30KB) → C(200KB, 60KB)
上次分配在B结束。
- 请求分配 40KB：从上一次结束位置(B)之后开始查找，找到C(200KB, 60KB)，60 >= 40 → 从C分割40KB，C变为(240, 20KB)
- 请求分配 10KB：从上一次结束位置(C)之后开始查找，到链尾后回到A，找到A(0, 50KB)，50 >= 10 → 从A分割10KB，A变为(10, 40KB)

**优点：**
- 空闲分区分布更均匀，低地址空间不会出现过多的小碎片
- 查找速度比首次适应算法快一些

**缺点：**
- 高地址空间的大块空闲区域也可能被分割成碎片
- 总体性能通常比首次适应算法略差

#### 最佳适应算法（Best-Fit, BF）

**算法描述：**
将空闲分区按大小递增的顺序链接成空闲分区链。当需要分配大小为S的分区时，从链首开始查找，找到第一个大小大于等于S的空闲分区（这个分区是所有满足条件的分区中大小最小的）。

**数据结构：**
- 空闲分区链：按分区大小递增顺序排列

**分配示例：**
假设空闲分区为：A(40KB), B(30KB), C(60KB)，按大小排序：B(30KB) → A(40KB) → C(60KB)
- 请求分配 25KB：从链首开始，B(30KB)，30 >= 25 → 分配，B变为(5KB)，插入到空闲链中
- 请求分配 10KB：现在空闲链为：B(5KB) → D(10KB新) → A(40KB) → C(60KB)，继续查找→D(10KB)，10 >= 10 → 分配

**优点：**
- 每次分配时选择最合适的（最小的满足要求的分区），保留大块空闲区
- 从全局来看，可以产生最大的内存利用率（理论上是这样）

**缺点：**
- 每次分配后产生的剩余部分很小，容易形成大量无法使用的小碎片
- 查找时需要遍历链表（如果按大小排序的话可以使用更高效的数据结构，但管理复杂度增加）

#### 最坏适应算法（Worst-Fit, WF）

**算法描述：**
将空闲分区按大小递减的顺序链接成空闲分区链。当需要分配大小为S的分区时，从链首开始查找，第一个分区（即最大的空闲分区）就是被选中的分区。

**数据结构：**
- 空闲分区链：按分区大小递减顺序排列

**分配示例：**
假设空闲分区为：A(40KB), B(30KB), C(60KB)，按大小排序：C(60KB) → A(40KB) → B(30KB)
- 请求分配 25KB：从链首开始，C(60KB) → 分配，C变为(35KB)，插入到链中适当位置

**优点：**
- 每次分配后剩余的分区仍然可能较大，可以满足后续分配
- 产生的外部碎片相对较小

**缺点：**
- 最大的空闲分区被快速消耗，当有大进程需要大块连续空间时可能无法满足
- 也需要遍历链表

#### 四种算法的综合比较

| 算法 | 空闲链排序方式 | 查找策略 | 碎片特点 | 分配速度 | 大进程满足能力 |
|------|---------------|---------|---------|---------|--------------|
| FF | 按地址递增 | 第一个满足的 | 低地址小碎片多 | 中等 | 高（高地址保留大块） |
| NF | 按地址递增 | 循环从上一次结束 | 碎片分布均匀 | 较快 | 中等 |
| BF | 按大小递增 | 最小的满足的 | 大量微小碎片 | 较慢 | 较高 |
| WF | 按大小递减 | 最大的 | 较少外部碎片 | 较快 | 低 |

**结论：** 在实际系统中，首次适应算法（FF）通常被认为综合性能最好。最佳适应算法（BF）虽然理论上内存利用率最高，但产生的微小碎片使得实际利用率并不高。

### 5.2.4 内存扩充技术（Memory Expansion Techniques）

连续分配方式的一个严重限制是进程必须全部装入内存才能执行。为了突破这个限制，出现了两种内存扩充技术：覆盖（Overlay）和交换（Swapping）。

#### 覆盖技术（Overlay）

**基本原理：**
覆盖技术允许在同一个进程中，将程序的不同部分在时间上交替地占用同一块内存区域。程序员需要将程序划分为若干功能独立的模块，并将那些不会同时执行的模块分配到同一块覆盖区。

**实现方式：**
- 常驻部分（Root/Resident）：始终驻留在内存中的代码
- 覆盖区（Overlay Area）：多个覆盖模块共享的内存区域
- 覆盖管理器（Overlay Manager）：负责在需要时将对应的覆盖模块从磁盘装入覆盖区

**覆盖结构示例：**
假设一个程序由以下模块组成：
- 主模块A（调用B或C，但不同时调用）
- B模块（调用D）
- C模块（调用E和F）
- D、E、F模块（互不调用）

覆盖结构：
- 常驻部分：A（固定占用内存）
- 覆盖区1：B和C共享（不同时执行）
- 覆盖区2：D、E、F共享（不同时执行）

内存占用 = A + max(B,C) + max(D,E,F)，而不是A+B+C+D+E+F

**优点：**
- 可以在较小的内存中运行较大的程序
- 不需要硬件的特殊支持

**缺点：**
- 必须由程序员手工划分覆盖结构，增加了编程负担
- 覆盖结构的设计非常复杂，且与程序的逻辑结构紧密相关
- 覆盖结构的修改会导致程序的大规模重写

#### 交换技术（Swapping）

**基本原理：**
交换技术是指将暂时不能运行的进程（或进程的一部分）从内存换出到磁盘上的对换区（Swap Area），以释放内存空间，同时将准备运行的进程从磁盘换入内存。

**实现方式：**
- 对换区（Swap Area）：磁盘上的一块专门区域，用于存放被换出的进程映像
- 交换过程：
  1. 当内存空间不足时，操作系统选择一个或多个进程换出到磁盘
  2. 换出的进程变为就绪状态（挂起就绪），等待再次被换入
  3. 当被换出的进程满足运行条件时，操作系统将其换入内存
- 交换与进程调度的关系：交换技术引入了中级调度（Medium-Term Scheduling）

**交换的类型：**
- 整体交换（Whole Process Swapping）：将整个进程的地址空间全部换出/换入
- 部分交换（Partial Swapping）：只交换进程的一部分（如页面或段），这是虚拟存储器的前身

**交换的优缺点：**
- 优点：
  - 增加了多道程序的道数，提高了CPU利用率
  - 允许运行比物理内存更大的程序（与覆盖配合）
- 缺点：
  - 交换操作本身需要I/O时间，开销较大
  - 需要确定合适的交换时机和交换对象
  - 需要动态重定位硬件的支持

#### 覆盖 vs 交换的详细比较

| 比较维度 | 覆盖技术 | 交换技术 |
|---------|---------|---------|
| 作用范围 | 同一进程内部 | 不同进程之间 |
| 控制方式 | 程序员控制 | 操作系统控制 |
| 目标 | 减少单个进程的内存需求 | 增加多道程序道数 |
| 对程序员要求 | 高（需要了解程序结构） | 低（透明） |
| 硬件需求 | 低 | 需要动态重定位支持 |
| 灵活性 | 低（覆盖结构固定） | 高 |
| 现代使用 | 很少使用 | 部分使用（UNIX/Linux） |

## 5.3 分页存储管理

### 5.3.1 基本概念与原理

分页存储管理（Paging）是一种离散分配方式，它将进程的逻辑地址空间划分为若干个大小相等的部分，称为页面或页（Page），同时将物理内存也划分为同样大小的部分，称为页框或帧（Frame）。页面和帧的大小通常为2的幂次，最常见的页面大小是4KB。

**核心思想：**
- 程序以页为单位离散地装入到内存中不连续的帧中
- 通过页表（Page Table）实现从页号到帧号的映射
- 逻辑上相邻的页面在物理上不一定相邻

**页面大小选择：**
页面大小通常选择为2的幂次，这是为了便于硬件进行地址转换。常见的页面大小有：
- 4KB（最常用，如x86、ARM架构的默认页面大小）
- 8KB
- 16KB
- 2MB（大页，Huge Pages）
- 1GB（巨页，Gigantic Pages）

页面大小的选择对系统性能有重要影响（将在5.10节详细讨论）。

**逻辑地址结构：**
```
+----------------+------------------+
|  页号 p        |  页内偏移 d      |
+----------------+------------------+
```

对于一个32位逻辑地址空间，页面大小为4KB（2^12字节）：
- 页内偏移d：12位（低12位），可寻址4KB
- 页号p：20位（高20位），可表示2^20 = 1M个页

**地址分解计算：**
- 页号 p = 逻辑地址 / 页面大小
- 页内偏移 d = 逻辑地址 % 页面大小

**示例：**
逻辑地址 = 16385，页面大小 = 4KB = 4096
- p = 16385 / 4096 = 4（取整）
- d = 16385 % 4096 = 1

所以，逻辑地址16385对应页号4，页内偏移1。

### 5.3.2 地址转换过程

分页系统的地址转换由硬件（内存管理单元，MMU）自动完成，基本流程如下：

1. 从CPU发出的逻辑地址中提取页号p和页内偏移d
2. 以页号p为索引，在页表中查找对应的页表项
3. 从页表项中取出帧号f
4. 物理地址 = f × 页面大小 + d

**详细地址转换示例：**
假设：
- 页面大小 = 4KB = 4096字节
- 某进程的页表：页0→帧2, 页1→帧5, 页2→帧8, 页3→帧1
- CPU访问逻辑地址 9000

步骤1：提取页号p和偏移d
- p = 9000 / 4096 = 2
- d = 9000 % 4096 = 9000 - 2×4096 = 9000 - 8192 = 808

步骤2：在页表中查找页2，得到帧号f = 8

步骤3：计算物理地址
- 物理地址 = 8 × 4096 + 808 = 32768 + 808 = 33576

### 5.3.3 页表详解

**页表的基本概念：**
页表（Page Table）是每个进程独有的一种数据结构，用于记录进程的每个逻辑页面对应哪个物理帧。页表存放在内存中（属于进程上下文的一部分），由操作系统维护，由MMU进行查询。

**页表寄存器：**
- 页表基址寄存器（Page Table Base Register, PTBR）：指向当前进程页表在内存中的起始地址
- 页表长度寄存器（Page Table Length Register, PTLR）：指示当前进程页表的大小（页表项数量）
- 进程切换时，操作系统会更新PTBR和PTLR

**页表项（Page Table Entry, PTE）的结构：**
每个页表项通常包含以下字段（以32位系统为例，每个页表项占4字节）：

| 字段 | 位数 | 说明 |
|------|------|------|
| 帧号（Frame Number） | 20位 | 物理帧的编号（对于4KB页面，可寻址2^20×4KB=4GB物理内存） |
| 存在位（Present/Absent） | 1位 | 该页是否在内存中（1=在内存，0=不在内存或在磁盘上） |
| 保护位（Protection） | 2位 | 读写权限（00=无权限，01=只读，10=读写，11=可执行） |
| 修改位（Modified/Dirty） | 1位 | 该页被装入后是否被修改过（1=修改过，换出时需要写回磁盘） |
| 访问位（Referenced/Accessed） | 1位 | 该页最近是否被访问过（1=访问过，用于页面置换算法） |
| 缓存禁用位（Caching Disabled） | 1位 | 是否禁止对该页进行高速缓存 |
| 保留位 | 剩余位 | 供将来使用或操作系统自定义 |

**注意：** 不同架构的页表项格式不同，但核心字段大同小异。

### 5.3.4 转换检测缓冲区（TLB）

**问题引出：**
由于页表存放在内存中，每次访问内存中的指令或数据时，都需要先访问内存中的页表来获取物理地址，然后再访问内存中的指令或数据。这意味着每次内存访问实际上需要两次物理内存访问（一次读页表，一次读数据），性能下降一倍。

**TLB的基本原理：**
TLB（Translation Look-aside Buffer），也称为快表，是一种位于CPU内部的小型高速缓存（通常使用内容可寻址存储器CAM实现，也称为关联存储器Associative Memory）。TLB中存放近期使用过的页表项，可以大大加快地址转换速度。

**TLB的查找过程：**
1. CPU发出逻辑地址
2. 硬件首先在TLB中查找是否有匹配的页号
3. 如果TLB命中（TLB Hit）：直接从TLB中取出帧号，计算物理地址
4. 如果TLB未命中（TLB Miss）：
   - 去内存中的页表查找
   - 将找到的页表项加入TLB（如果TLB已满，需要替换）
   - 计算物理地址
5. 访问物理内存

**有效访问时间（Effective Access Time, EAT）的计算：**

设：
- TLB查找时间 = ε（很小，通常0.1~1纳秒）
- 内存访问时间 = t（通常10~100纳秒）
- TLB命中率 = α（0 ≤ α ≤ 1）

情况1：TLB命中
- 时间 = ε + t（查找TLB + 一次内存访问）

情况2：TLB未命中
- 时间 = ε + t + t（查找TLB + 访问页表 + 访问内存）
- 或 ε + 2t（简化写法，因为ε相对于t很小）

有效访问时间公式：
EAT = α × (ε + t) + (1 - α) × (ε + 2t)
    = αε + αt + (1-α)ε + 2(1-α)t
    = ε + αt + 2t - 2αt
    = ε + 2t - αt
    = ε + t(2 - α)

**EAT计算实例：**
假设TLB查找时间 ε = 20ns，内存访问时间 t = 100ns，TLB命中率 α = 80%：
EAT = 20 + 100 × (2 - 0.8) = 20 + 100 × 1.2 = 20 + 120 = 140ns

如果TLB命中率提高到 α = 99%：
EAT = 20 + 100 × (2 - 0.99) = 20 + 100 × 1.01 = 20 + 101 = 121ns

如果TLB命中率 α = 99.9%：
EAT = 20 + 100 × (2 - 0.999) = 20 + 100 × 1.001 = 20 + 100.1 = 120.1ns

作为对比，没有TLB的情况下每次内存访问需要200ns（ε + 2t）。使用TLB并达到99%命中率时，有效访问时间降至约121ns，性能提升了约65%。

**TLB的局限性：**
- TLB大小有限（通常只有几十到几百个条目），无法容纳整个页表
- 进程切换时需要刷新TLB（或使用地址空间标识符ASID来区分不同进程的TLB条目）
- TLB未命中时的开销很大（需要访问内存中的页表）

### 5.3.5 多级页表（Hierarchical/Multi-Level Page Tables）

**为什么需要多级页表？**

对于32位系统，页面大小4KB，页表项大小4B：
- 共需要2^20 = 1,048,576个页表项
- 每个页表项4B，页表总大小 = 1,048,576 × 4B = 4MB
- 每个进程都需要一个4MB的页表（且必须是连续的）
- 如果系统中有100个进程，仅页表就需要400MB内存——这是巨大的浪费

此外，大多数进程实际上只使用了地址空间的一小部分（如代码段、数据段、堆、栈），但页表必须为整个地址空间（0~2^32-1）建立映射。

多级页表的思想是：不一次性分配所有页表，而是按需分配，通过多级索引来减少页表占用的内存。

#### 二级页表（Two-Level Paging）

**地址结构（32位系统，4KB页面）：**
```
+----------------+----------------+------------------+
| 页目录索引 p1  | 页表索引 p2    | 页内偏移 d       |
|  10位           |  10位          |  12位             |
+----------------+----------------+------------------+
```

- 页目录（Page Directory）：2^10 = 1024个页目录项，每个指向一个二级页表
- 页表（Page Table）：2^10 = 1024个页表项，每个指向一个物理帧
- 页内偏移：2^12 = 4096 = 4KB

**地址转换过程：**
1. 从逻辑地址中提取p1（高10位）、p2（中间10位）、d（低12位）
2. 以p1为索引，在页目录中查找二级页表的起始地址（页目录的物理地址由CR3寄存器给出）
3. 以p2为索引，在二级页表中查找帧号
4. 物理地址 = 帧号 × 页面大小 + d

**详细转换示例：**
假设：
- 逻辑地址 = 0x003AB014（十六进制）
- 二进制：0000 0000 0011 1010 1011 0000 0001 0100
  - p1（高10位）：0000 0000 00 = 0
  - p2（中间10位）：11 1010 1011 = 939
  - d（低12位）：0000 0001 0100 = 20

转换步骤：
1. CR3 → 页目录物理地址
2. 页目录[0] → 二级页表的物理地址
3. 二级页表[939] → 帧号（假设为 0x1A）
4. 物理地址 = 0x1A × 0x1000 + 0x014 = 0x1A014

**空间节省原理：**
假设一个进程只使用了4个页面（代码、数据、堆、栈各1页），二级页表只需要：
- 1个页目录（4KB） + 4个页表（每个4KB）= 20KB
- 而不使用二级页表时需要4MB
- 实际系统中，一个进程通常使用几十到几百个页面，二级页表可以节省大量内存

#### 三级和四级页表

对于64位系统（如x86-64），地址空间极大（2^64字节），使用二级页表仍然会产生巨大的页表开销。x86-64使用四级页表（也称为四层分页）：

**地址结构（x86-64，4KB页面）：**
```
+----------------+----------------+----------------+----------------+------------------+
| PML4索引 p1    | 目录指针 p2   | 页目录索引 p3 | 页表索引 p4   | 页内偏移 d       |
|  9位            |  9位          |  9位           |  9位          |  12位             |
+----------------+----------------+----------------+----------------+------------------+
```

- PML4（Page Map Level 4）：2^9 = 512个条目
- 目录指针（Page Directory Pointer）：2^9 = 512个条目
- 页目录（Page Directory）：2^9 = 512个条目
- 页表（Page Table）：2^9 = 512个条目
- 共索引2^9 × 2^9 × 2^9 × 2^9 × 2^12 = 2^48字节 = 256TB的虚拟地址空间

### 5.3.6 散列页表（Hashed Page Table）

**适用场景：**
对于64位地址空间，使用多级页表也会导致大量的页表层次（x86-64是4级，IA-64可能需要7级），每次地址转换需要多次内存访问，开销巨大。散列页表通过虚拟页号进行哈希，可以快速找到对应的页表项。

**基本原理：**
- 散列页表中的每个条目包含三个字段：虚拟页号、映射的帧号、指向下一个元素的指针（链地址法解决冲突）
- 地址转换时，对虚拟页号进行哈希，得到散列页表中的索引
- 遍历该索引处的链表，查找匹配的虚拟页号
- 如果找到，取出对应的帧号；如果未找到，产生缺页异常

**优点：**
- 对于稀疏的大地址空间特别有效
- 可以减少页表的层次

**缺点：**
- 散列冲突会降低查找效率
- 不适合小地址空间的进程

### 5.3.7 反置页表（Inverted Page Table, IPT）

**传统页表的问题：**
- 每个进程维护一个页表，按虚拟页号索引
- 如果有N个进程，每个进程有M个虚拟页面，则需要N×M个页表项
- 页表占用的内存总量与进程数量成正比

**反置页表的基本思想：**
- 整个系统只有一个页表
- 页表中的每个条目对应物理内存中的一个帧，而不是进程中的一个页
- 条目内容为：（进程ID，虚拟页号）
- 页表的大小 = 物理帧的总数，与进程数量无关

**地址转换过程：**
1. 给定逻辑地址（进程ID pid，虚拟页号 p，页内偏移 d）
2. 在反置页表中搜索匹配（pid, p）的条目
3. 该条目在表中的位置就是帧号f
4. 物理地址 = f × 页面大小 + d

**反置页表的搜索问题：**
反置页表需要顺序搜索来找到匹配的（pid, p）对，搜索时间与物理内存大小成正比。为了提高搜索速度，通常与散列页表结合使用：对（pid, p）进行哈希，得到在反置页表中的近似位置。

**优点：**
- 页表大小与物理内存大小成正比，与进程数量无关
- 节省了大量内存（特别是当物理内存远小于虚拟地址空间时）

**缺点：**
- 地址转换需要散列查找，比传统页表复杂
- 难以实现共享内存（多个进程共享同一个物理帧时，反置页表中只能记录一个进程的映射关系）
- 每个地址转换都需要搜索，即使使用了散列技术，性能也不如传统页表

**谁在使用反置页表：**
IBM PowerPC、HP PA-RISC、Sun UltraSPARC等架构使用了反置页表或类似技术。x86架构使用的是传统多级页表。

### 5.3.8 共享页面（Shared Pages）

**基本原理：**
多个进程可以通过共享页面来共享同一段代码或数据，从而节省内存空间。共享通常通过让不同进程的页表中的不同条目指向同一个物理帧来实现。

**可重入代码（Reentrant Code）：**
可重入代码（也称为纯代码，Pure Code）是一种在执行的期间不会自我修改的代码。多个进程可以共享同一份可重入代码的物理副本。

具体实现：
- 进程A的页表：页3→帧10
- 进程B的页表：页5→帧10
- 两个进程的不同页号映射到同一个物理帧10
- 物理帧10中存放的是可重入的共享代码
- 每个进程的私有数据（如栈、堆）不能共享，必须使用不同的物理帧

**写时复制（Copy-on-Write, CoW）：**

写时复制是一种优化技术，用于在创建进程（fork）时减少不必要的复制。其基本原理如下：

1. 当父进程调用fork()创建子进程时，子进程不是立即获得父进程所有页面的副本
2. 而是让子进程的页表指向父进程的物理帧，同时将这些页面的权限设置为只读
3. 所有页面都被标记为"写时复制"
4. 如果父进程或子进程都不修改这些共享页面，那么它们始终共享同一份物理副本——避免了不必要的复制
5. 当某个进程试图修改一个共享页面时，会触发写保护异常
6. 操作系统在异常处理程序中：
   a. 分配一个新的物理帧
   b. 将原页面的内容复制到新帧
   c. 修改该进程的页表，使其指向新帧
   d. 将新帧的权限设置为可读写
   e. 恢复执行引起异常的指令

**CoW的性能优势：**
- fork()创建子进程的开销大大降低（只需要复制页表，而不需要复制所有物理页面）
- 子进程exec()新程序时，如果立即执行exec()，则几乎不需要复制任何页面
- 只有当进程真正需要修改共享页面时，才进行页面复制

**实际的CoW实现（以Linux为例）：**
- Linux为每个物理帧维护一个引用计数（通过struct page结构体中的_count字段）
- 当引用计数大于1时，该帧被标记为写时复制
- 当某个进程试图写入时，如果引用计数大于1，则执行复制；如果等于1，则直接写入（无需复制）
- vfork()系统调用进一步优化：父进程在子进程exec()之前被阻塞，子进程直接使用父进程的地址空间，不复制任何内容（甚至不复制页表）

### 5.3.9 分页管理的优缺点总结

**优点：**
- 没有外部碎片（所有帧大小相同，分配和回收时不存在碎片问题）
- 支持虚拟存储器（不需要将进程的全部页面都装入内存）
- 可以实现页面共享（多个进程共享同一段代码）
- 对程序员完全透明，不需要程序员干预

**缺点：**
- 存在少量内部碎片（进程的最后一页可能没有完全填满）
- 页表占用内存空间（多级页表可以缓解，但不能完全消除）
- 地址转换增加了时间开销（TLB可以缓解）

## 5.4 分段存储管理

### 5.4.1 为什么需要分段

分页管理虽然解决了外部碎片和离散分配的问题，但它有一个重要缺点：分页是基于物理的划分（将地址空间划分为大小相等的页面），而不是基于逻辑的划分。这在以下几个方面存在不足：

**（1）方便编程**
在分段系统中，程序员可以将程序按照逻辑功能划分为多个段（如主程序段MAIN、数据段DATA、栈段STACK等），每个段的长度由程序逻辑决定，不需要强制使用固定的页面大小。

**（2）信息共享**
在分页系统中，共享代码时要求共享的代码必须恰好占据整数个页面，且页面边界可能与代码的逻辑边界不一致。在分段系统中，共享是以段为单位的，一个逻辑段可以自然地被多个进程共享。

**（3）信息保护**
以段为单位进行保护比以页为单位更自然。例如，代码段可以设置为只读执行，数据段可以设置为读写，而栈段设置为读写但不执行。

**（4）动态增长**
某些数据结构（如堆、栈）在运行过程中会动态增长。在分段系统中，段的大小可以动态变化（只要不超过硬件限制），而在分页系统中，堆和栈的增长处理更加复杂。

**（5）动态链接**
在分段系统中，动态链接是以段为单位进行的。当一个程序调用某个动态链接库的函数时，只需将对应的段装入内存即可。这种以段为单位的动态链接比以页为单位的链接更加自然。

### 5.4.2 基本概念与原理

**分段的基本思想：**
- 将进程的逻辑地址空间按程序自身的逻辑关系划分为若干个段（Segment）
- 每个段都有自己的名字（通常用段号表示）和长度
- 每个段的地址空间从0开始连续编址
- 段的长度由程序逻辑决定，不同段的长度可以不同
- 段被离散地装入到物理内存的不同区域

**逻辑地址结构：**
```
+----------------+------------------+
|  段号 s        |  段内偏移 d      |
+----------------+------------------+
```

**段表（Segment Table）：**
- 每个进程有一个段表
- 每个段表项包含：
  - 段基址（Base）：该段在物理内存中的起始地址
  - 段长度（Limit）：该段的长度（字节数）
  - 保护信息（Protection）：该段的访问权限（读、写、执行）

**关键寄存器：**
- 段表基址寄存器（Segment Table Base Register, STBR）：指向当前进程段表的起始地址
- 段表长度寄存器（Segment Table Length Register, STLR）：指示当前进程段表的条目数量

### 5.4.3 地址转换过程

分段系统的地址转换由硬件（MMU）自动完成，基本流程如下：

1. 从CPU发出的逻辑地址中提取段号s和段内偏移d
2. 检查段号s是否合法（s < STLR）
3. 从段表基址 + s × 段表项大小处读取段表项
4. 检查段内偏移d是否小于段长度Limit
   - 如果 d >= Limit：产生越界异常（段错误，Segmentation Fault）
5. 如果 d < Limit：物理地址 = 段基址Base + d

**详细地址转换示例：**
假设某进程的段表如下：
| 段号 | 段基址 | 段长度 | 保护信息 |
|------|--------|--------|---------|
| 0    | 10000  | 8000   | 只读     |
| 1    | 25000  | 15000  | 读写     |
| 2    | 50000  | 10000  | 读写     |
| 3    | 70000  | 12000  | 只读可执行 |

**示例1：访问逻辑地址 (1, 5000)**
1. 段号=1，段内偏移=5000
2. 检查：5000 < 15000（段长度）→ 合法
3. 物理地址 = 25000 + 5000 = 30000
4. 检查保护：段1允许读写 → 合法（如果是写操作的话）

**示例2：访问逻辑地址 (0, 8500)**
1. 段号=0，段内偏移=8500
2. 检查：8500 >= 8000（段长度）→ 段错误！
3. 产生越界异常，进程被终止（或触发信号处理）

**示例3：对段0进行写操作**
1. 保护检查：段0是只读的 → 保护错误！
2. 产生保护异常

### 5.4.4 分页 vs 分段详细比较

| 比较维度 | 分页（Paging） | 分段（Segmentation） |
|---------|---------------|-------------------|
| **划分依据** | 物理划分，固定大小 | 逻辑划分，大小可变 |
| **对程序员** | 透明（程序员不知道也不关心） | 可见（程序员需要了解段结构） |
| **地址空间** | 一维线性地址空间 | 二维地址空间（段号+偏移） |
| **大小** | 页面大小固定（由硬件决定） | 段长度可变（由程序逻辑决定） |
| **碎片** | 只有内部碎片（很小） | 存在外部碎片（需要通过紧凑解决） |
| **共享** | 共享比较复杂（需要页面边界对齐） | 共享方便（以自然段为单位） |
| **保护** | 以页为单位，保护粒度不自然 | 以段为单位，保护更自然 |
| **动态增长** | 处理复杂（需要处理多页边界） | 处理简单（只需修改段表项的长度） |
| **动态链接** | 以页为单位，实现复杂 | 以段为单位，实现自然 |
| **硬件支持** | 页表、TLB | 段表、STBR/STLR |
| **存储管理方式** | 离散分配，无外部碎片 | 离散分配，有外部碎片 |

**两者结合：** 现代操作系统（如Linux、Windows）实际上都采用了分段和分页的组合（即段页式存储管理），但分段功能通常被弱化（所有段的基址都设为0，限制为整个地址空间），主要使用分页功能。

## 5.5 段页式存储管理

### 5.5.1 基本概念

段页式存储管理（Segmentation with Paging）结合了分段和分页两种方式的优点：
- **分段**：按程序的逻辑结构划分，方便编程、共享和保护
- **分页**：将内存划分为固定大小的帧，消除外部碎片

**段页式的基本思想：**
- 先将进程按照逻辑结构划分为若干个段
- 再将每个段划分为若干个固定大小的页面
- 内存仍然划分为固定大小的帧
- 以页为单位离散地分配内存

### 5.5.2 地址结构

段页式的逻辑地址包含三个部分：
```
+----------------+----------------+------------------+
|  段号 s        |  页号 p        |  页内偏移 d      |
+----------------+----------------+------------------+
```

例如，在一个32位系统中：
- 段号：8位（最多256个段）
- 页号：12位（每段最多4096页）
- 页内偏移：12位（页面大小4KB）

这样，一个进程的虚拟地址空间最大为：2^8 × 2^12 × 2^12 = 2^32 = 4GB

### 5.5.3 地址转换过程

段页式的地址转换需要多次查表，具体流程如下：

1. 从逻辑地址中提取段号s、页号p、页内偏移d
2. 以段号s为索引，从段表中查找该段对应的页表基址和段长度
   - 如果s >= STLR → 段号越界
   - 如果p >= 段长度（页数）→ 页号越界
3. 从段表项中得到页表基址
4. 以页号p为索引，从页表中查找帧号
5. 物理地址 = 帧号 × 页面大小 + 页内偏移d

**图示地址转换流程：**
```
逻辑地址 → [s | p | d]
           ↓
        段表(STBR指向) → 段表项[s] → 页表基址
                                       ↓
                                    页表 → 页表项[p] → 帧号
                                                         ↓
                                         物理地址 = 帧号 × 页大小 + d
```

**没有TLB时的内存访问次数：**
- 第1次：访问段表，获取页表基址
- 第2次：访问页表，获取帧号
- 第3次：访问物理内存，获取指令/数据
- 总共：3次内存访问

**有TLB时（TLB命中）：**
- TLB中同时缓存了段信息+页信息
- 1次TLB查找 + 1次内存访问 = 2次
- 有效访问时间依然可以用公式 EAT = ε + t(2 - α) 计算（但有细微差别）

### 5.5.4 详细数值示例

假设系统采用段页式存储管理：
- 段号：4位（最多16段）
- 页号：10位（每段最多1024页）
- 页内偏移：12位（页面大小4KB）
- 段表、页表项均占4字节

某进程段表如下：
| 段号 | 页表大小（页数） | 页表基址 |
|------|-----------------|---------|
| 0    | 256             | 10000   |
| 1    | 512             | 20000   |

段0的页表（部分）：
| 页号 | 帧号 |
|------|------|
| 0    | 50   |
| 1    | 75   |
| 2    | 120  |

**示例：访问逻辑地址（0, 2, 500）**
- 段号=0，页号=2，页内偏移=500
- 段表[0] → 页表基址=10000，页表大小=256
- 检查：2 < 256 → 合法
- 访问页表：从内存地址10000 + 2×4 = 10008读取页表项
- 页表项[2] → 帧号=120
- 物理地址 = 120 × 4096 + 500 = 491520 + 500 = 492020

### 5.5.5 段页式管理的优缺点

**优点：**
- 兼具分段和分页的优点：逻辑清晰（段） + 没有外部碎片（页）
- 方便共享和保护（以段为单位）
- 支持虚拟存储器

**缺点：**
- 增加了管理复杂度
- 地址转换需要两次查表（段表和页表），开销更大
- 段表和页表都需要占据内存空间
- 如果没有TLB，每次内存访问需要3次内存访问

**实际应用：**
- x86架构的保护模式支持段页式存储管理
- 但现代操作系统（如Linux）实际上将分段功能最小化（使用"平坦模型"，所有段的基址为0，限制为4GB），主要使用分页功能
- 这种做法的好处是简化了内存管理，同时兼容不同CPU架构

## 5.6 虚拟存储器

### 5.6.1 虚拟存储器的基本概念

**定义：**
虚拟存储器（Virtual Memory）是一种内存管理技术，它允许进程执行时只将部分程序装入内存，从而在逻辑上扩充内存容量。虚拟存储器将用户程序的观点（巨大、连续、独立的虚拟地址空间）与物理内存的现实（有限、离散、共享的物理地址空间）分离开来。

**虚拟存储器的特征：**

（1）**离散性（Discreteness）**
进程的地址空间被离散地装入到物理内存中。这是虚拟存储器的基础，如果没有离散分配，就无法实现部分装入。

（2）**部分交换性（Partial Swapping）**
进程在运行时只装入部分内容到内存，其余内容在需要时再从磁盘调入。这与传统的交换技术（必须整体换入换出）不同。

（3）**大空间性（Large Virtual Space）**
用户可以使用的虚拟地址空间大于物理内存的大小。例如，在32位系统中，每个进程拥有4GB的虚拟地址空间，而物理内存可能只有128MB。

**虚拟存储器的理论基础：局部性原理**

虚拟存储器之所以可行，是因为程序执行时表现出局部性原理（Principle of Locality）：

**时间局部性（Temporal Locality）：**
- 最近被访问的指令或数据很可能在不久的将来再次被访问
- 例如：循环体中的代码、频繁使用的全局变量
- 原因：程序中存在大量的循环操作、频繁使用的变量和函数

**空间局部性（Spatial Locality）：**
- 最近被访问的地址附近的地址很可能在不久的将来被访问
- 例如：顺序执行的指令、数组的顺序访问
- 原因：程序通常顺序执行指令，数据通常以数组或结构体的形式连续存储

**局部性程度的衡量：**
- 工作集（Working Set）：进程在一段时间内频繁访问的页面集合
- 局部性越好，工作集越小，虚拟存储器的性能越好

### 5.6.2 请求分页（Demand Paging）

**基本思想：**
请求分页（Demand Paging）是虚拟存储器最常用的实现方式。当一个进程试图访问某个页面时，如果该页面不在内存中，则操作系统会从磁盘将该页面调入内存。

**纯请求分页（Pure Demand Paging）：**
- 进程启动时，没有任何页面在内存中
- CPU立即去取第一条指令，导致缺页异常
- 操作系统将包含第一条指令的页面调入内存
- 继续执行，可能会再次触发缺页异常
- 最终，进程需要的页面会逐渐被调入内存
- 这被称为"按需调页"——只有在真正需要时才调入页面

**硬件支持：**
请求分页需要特定的硬件支持：
1. **页表**：每个页表项包含一个存在位（valid/invalid bit）
   - valid = 1：该页在内存中，页表项有效
   - valid = 0：该页不在内存中（可能在磁盘上），页表项无效
2. **二级存储（磁盘）**：存放不在内存中的页面
   - 交换区（Swap Area）：用于存放进程的换出页面
   - 文件系统：可执行文件可以直接从文件系统调入
3. **指令重启（Instruction Restart）**：缺页异常处理后能重新执行被中断的指令

### 5.6.3 缺页异常处理（Page Fault Handling）

缺页异常（Page Fault）是请求分页系统的核心机制。下面是缺页异常处理的完整步骤：

**缺页处理流程（共8步）：**

1. **检查合法性**：操作系统检查进程的PCB（进程控制块）中的内部表，确认这次访问是否合法（即该地址是否在进程的虚拟地址空间中）。

2. **异常处理**：
   - 如果访问的是非法地址（不存在的虚拟地址）→ 终止进程（段错误，Segmentation Fault）
   - 如果访问的是合法地址但页面不在内存中 → 继续以下步骤

3. **查找空闲帧**：操作系统在物理内存中查找一个空闲帧（Free Frame）
   - 如果有空闲帧 → 直接使用
   - 如果没有空闲帧 → 启动页面置换算法，选择一个页面换出

4. **调度磁盘I/O**：操作系统调度磁盘操作，将需要的页面从磁盘读入选定的帧中
   - 查找页面在磁盘上的位置（交换区或文件系统）
   - 启动DMA传输

5. **等待磁盘I/O**：在等待磁盘操作完成期间，CPU可以调度其他进程执行（上下文切换）
   - 当前进程被阻塞（等待页面就绪）
   - 调度器选择另一个就绪进程运行

6. **磁盘中断处理**：磁盘I/O完成时，产生中断
   - 操作系统处理中断
   - 页面已经从磁盘读入内存中的帧

7. **更新页表**：操作系统更新进程的页表，将新调入的页面对应的页表项设置为有效（valid=1）
   - 填入帧号
   - 设置访问位、修改位等

8. **重新执行指令**：操作系统返回到引起缺页异常的指令处，重新执行该指令
   - 由于页面现在已经在内存中，这次执行不会再次触发缺页异常

**指令重启问题（Instruction Restart Problem）：**

某些指令在缺页异常发生时可能已经部分执行完毕，这意味着不能简单地从指令的起始处重新执行。

**典型问题场景：**
考虑一条块移动指令（如x86的MOVS指令，或M68K的MOVEM指令），该指令将内存中连续的一块数据从源地址复制到目的地址。这条指令在执行过程中需要访问多个内存地址：
- 先读取指令本身
- 再读取源地址的内容
- 最后写入目的地址

如果这条指令在执行过程中跨越了多个页面，其中一些页面在内存中，一些不在，那么当缺页异常发生时，部分数据已经移动（被覆盖），部分还没有。

**解决方案：**
- **方法1**：微程序（Microcode）记录每条指令的中间状态，缺页异常发生后从断点处恢复执行
  - 优点：恢复粒度精确
  - 缺点：微程序设计复杂
- **方法2**：预先检查指令所需的所有页面是否都在内存中，如果都在才执行
  - 优点：实现简单
  - 缺点：降低性能（增加了预检查的开销）
- **方法3**：使用临时寄存器保存被覆盖的数据，异常处理后恢复
  - 现代操作系统通常采用这种方法

**缺页异常与TLB缺页的区别：**
- TLB缺页（TLB Miss）：页表项在内存中但不在TLB中，需要访问内存中的页表
- 真正的缺页（Page Fault）：页表项中的存在位为0，页面不在内存中，需要从磁盘调入

### 5.6.4 页面置换（Page Replacement）

**为什么需要页面置换：**
在请求分页系统中，当发生缺页异常时，如果内存中没有空闲帧，那么必须从当前占用内存的页面中选择一个换出（写到磁盘交换区），以腾出空间给新页面。

**基本流程：**
1. 查找所需页面在磁盘上的位置
2. 查找一个空闲帧：
   a. 如果有空闲帧 → 使用它
   b. 如果没有空闲帧 → 使用页面置换算法选择一个"牺牲"帧（Victim Frame）
3. 将牺牲帧的内容写到磁盘（如果该帧被修改过/Dirty Frame），更新该页的页表项（存在位置0）
4. 将所需的页面从磁盘读入牺牲帧，更新页表项（存在位置1，帧号改变）
5. 重新执行引起缺页异常的指令

**脏页（Dirty Page）问题：**
- 如果一个页面被修改过（Dirty Bit = 1），在换出时必须将其内容写回磁盘
- 如果一个页面没有被修改过（Dirty Bit = 0），在换出时可以直接覆盖（因为磁盘上的副本仍然有效）
- 这个优化可以减少不必要的磁盘写操作

**页面置换的实现难点：**
- 需要选择合适的置换算法，使得缺页率尽可能低
- 置换算法的选择对系统性能影响极大
- 没有一种算法在所有情况下都是最优的（下一节将详细讨论各种置换算法）

### 5.6.5 虚拟存储器的性能分析

**有效访问时间（Effective Access Time）的完整计算：**

设：
- ma = 内存访问时间（memory access time）
- pf = 缺页概率（page fault probability），0 ≤ pf ≤ 1
- pft = 缺页处理时间（page fault time），包括：
  - 磁盘寻道时间 + 旋转延迟 + 传输时间
  - 以及缺页处理的开销

有效访问时间：
EAT = (1 - pf) × ma + pf × pft

**典型数值计算：**
假设：
- ma = 200ns（0.2μs）
- 磁盘访问时间 = 25ms（25000μs）
  - 其中寻道时间约10ms，旋转延迟约5ms，传输时间约10ms
- 缺页处理的其他开销 = 5ms

pft = 25ms + 5ms = 30ms = 30,000,000ns

如果缺页率 pf = 0.01（每100次访问发生一次缺页）：
EAT = 0.99 × 200 + 0.01 × 30,000,000
    = 198 + 300,000
    = 300,198ns ≈ 0.3ms

有效访问时间比普通内存访问慢了1500倍！

如果缺页率 pf = 0.0001（每10000次访问发生一次缺页）：
EAT = 0.9999 × 200 + 0.0001 × 30,000,000 = 199.98 + 3000 = 3199.98ns ≈ 3.2μs

可见，缺页率必须控制在一个极低的水平（低于10^-5），虚拟存储器才能有可接受的性能。这也是为什么局部性原理如此重要——良好的局部性意味着较低的缺页率。

## 5.7 页面置换算法（全文最详细部分）

页面置换算法（Page Replacement Algorithm）是虚拟存储器的核心。当发生缺页且内存已满时，操作系统必须选择一个页面换出。不同的置换策略会极大地影响系统性能。下面逐一详细分析各种置换算法。

### 5.7.1 页面置换算法的一般框架

所有页面置换算法都遵循以下框架：
1. 当CPU访问某个页面时，硬件检查该页是否在内存中（通过页表的存在位）
2. 如果在（Page Hit）→ 正常访问
3. 如果不在（Page Fault）→ 
   - 如果内存有空闲帧 → 直接调入
   - 如果内存没有空闲帧 → 调用置换算法选择牺牲页面
     - 如果牺牲页被修改过 → 写回磁盘
     - 更新牺牲页的页表项
   - 调入新页面，更新页表项
4. 重新执行引起缺页的指令

**缺页率（Page Fault Rate）的计算：**
缺页率 = 缺页次数 / 总的内存访问次数（或页面引用次数）

**页面引用串（Page Reference String）：**
页面引用串是进程执行过程中访问的页面号序列。例如：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2, 0, 1, 7, 0, 1

### 5.7.2 先进先出算法（FIFO）

**算法描述：**
FIFO（First-In, First-Out）算法是最简单的页面置换算法。它维护一个页面队列，将新调入的页面加入队尾，每次置换时选择队首的页面（即最早进入内存的页面）换出。

**实现方式：**
- 使用一个队列数据结构
- 页面调入时加入队尾
- 置换时从队首取出

**数据结构：**
- 队列（或循环队列）
- 队列中存放页面号

**详细示例：**
页面引用串：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2, 0, 1, 7, 0, 1
内存帧数：3个

| 时间 | 引用 | 内存状态（队首→队尾） | 缺页 |
|------|------|---------------------|------|
| 1    | 7    | [7]                  | 缺页 |
| 2    | 0    | [7, 0]               | 缺页 |
| 3    | 1    | [7, 0, 1]            | 缺页 |
| 4    | 2    | [0, 1, 2]            | 缺页（换出7） |
| 5    | 0    | [0, 1, 2]            | 命中 |
| 6    | 3    | [1, 2, 3]            | 缺页（换出0） |
| 7    | 0    | [2, 3, 0]            | 缺页（换出1） |
| 8    | 4    | [3, 0, 4]            | 缺页（换出2） |
| 9    | 2    | [0, 4, 2]            | 缺页（换出3） |
| 10   | 3    | [4, 2, 3]            | 缺页（换出0） |
| 11   | 0    | [2, 3, 0]            | 缺页（换出4） |
| 12   | 3    | [2, 3, 0]            | 命中 |
| 13   | 2    | [2, 3, 0]            | 命中 |
| 14   | 1    | [3, 0, 1]            | 缺页（换出2） |
| 15   | 2    | [0, 1, 2]            | 缺页（换出3） |
| 16   | 0    | [0, 1, 2]            | 命中 |
| 17   | 1    | [0, 1, 2]            | 命中 |
| 18   | 7    | [1, 2, 7]            | 缺页（换出0） |
| 19   | 0    | [2, 7, 0]            | 缺页（换出1） |
| 20   | 1    | [7, 0, 1]            | 缺页（换出2） |

缺页次数：15次
缺页率：15/20 = 75%

**Belady异常（Belady's Anomaly）：**

Belady异常是指：在某些情况下，增加分配给进程的物理帧数，反而会导致更多的缺页。这是FIFO算法特有的现象。

**Belady异常示例：**
页面引用串：1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5

**3个帧时：**
| 引用 | 内存（FIFO队列） | 缺页 |
|------|-----------------|------|
| 1    | [1]              | 缺页 |
| 2    | [1, 2]           | 缺页 |
| 3    | [1, 2, 3]        | 缺页 |
| 4    | [2, 3, 4]        | 缺页(1出) |
| 1    | [3, 4, 1]        | 缺页(2出) |
| 2    | [4, 1, 2]        | 缺页(3出) |
| 5    | [1, 2, 5]        | 缺页(4出) |
| 1    | [1, 2, 5]        | 命中 |
| 2    | [1, 2, 5]        | 命中 |
| 3    | [2, 5, 3]        | 缺页(1出) |
| 4    | [5, 3, 4]        | 缺页(2出) |
| 5    | [5, 3, 4]        | 命中 |

缺页次数：9次

**4个帧时：**
| 引用 | 内存（FIFO队列） | 缺页 |
|------|-----------------|------|
| 1    | [1]              | 缺页 |
| 2    | [1, 2]           | 缺页 |
| 3    | [1, 2, 3]        | 缺页 |
| 4    | [1, 2, 3, 4]     | 缺页 |
| 1    | [1, 2, 3, 4]     | 命中 |
| 2    | [1, 2, 3, 4]     | 命中 |
| 5    | [2, 3, 4, 5]     | 缺页(1出) |
| 1    | [3, 4, 5, 1]     | 缺页(2出) |
| 2    | [4, 5, 1, 2]     | 缺页(3出) |
| 3    | [5, 1, 2, 3]     | 缺页(4出) |
| 4    | [1, 2, 3, 4]     | 缺页(5出) |
| 5    | [2, 3, 4, 5]     | 缺页(1出) |

缺页次数：10次

**结论：** 帧从3个增加到4个，缺页次数从9次增加到10次，Belady异常发生。

**为什么Belady异常会发生？**
因为FIFO算法不基于页面的实际使用模式，而是基于页面进入内存的时间。增加帧数后，页面在内存中存活的时间变长，但FIFO无法区分哪些页面正在被使用，哪些不再被使用。一个页面可能因为进入内存早而被置换，但实际上它可能是一个频繁使用的页面。

### 5.7.3 最优置换算法（OPT / MIN）

**算法描述：**
OPT（Optimal Page Replacement）算法选择未来最长时间不会被访问的页面进行置换。也就是说，它选择在将来引用串中出现在最靠后的那个页面。

**性质：**
- 理论上最优的算法（缺页率最低）
- 不可实现——因为无法预知未来的页面访问序列
- 主要用于理论分析，作为衡量其他算法性能的基准（Belady最优算法）

**实现方式（模拟）：**
- 对于每个当前在内存中的页面，查看它在未来引用串中下一次出现的位置
- 选择出现位置最远的页面（或者永远不再出现的页面）作为牺牲页面

**详细示例：**
页面引用串：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2, 0, 1, 7, 0, 1
内存帧数：3个

| 时间 | 引用 | 内存状态 | 缺页 | 说明 |
|------|------|---------|------|------|
| 1    | 7    | [7]      | 缺页 | |
| 2    | 0    | [7, 0]   | 缺页 | |
| 3    | 1    | [7, 0, 1] | 缺页 | |
| 4    | 2    | [7, 0, 2] | 缺页 | 替换1，因为1在将来最远（位置14） |
| 5    | 0    | [7, 0, 2] | 命中 | |
| 6    | 3    | [3, 0, 2] | 缺页 | 替换7，因为7在将来最远（位置18） |
| 7    | 0    | [3, 0, 2] | 命中 | |
| 8    | 4    | [3, 0, 4] | 缺页 | 替换2，因为2不再出现 |
| 9    | 2    | [3, 0, 2] | 缺页 | 替换4，因为4不再出现 |
| 10   | 3    | [3, 0, 2] | 命中 | |
| 11   | 0    | [3, 0, 2] | 命中 | |
| 12   | 3    | [3, 0, 2] | 命中 | |
| 13   | 2    | [3, 0, 2] | 命中 | |
| 14   | 1    | [3, 0, 1] | 缺页 | 替换2，因为2在将来最远（位置13已过→不再出现） |
| 15   | 2    | [3, 0, 2] | 缺页 | 替换1，因为1不再出现 |
| 16   | 0    | [3, 0, 2] | 命中 | |
| 17   | 1    | [3, 1, 2] | 缺页 | 替换0，0不再出现；3还有1次(位置20后) |
| 18   | 7    | [7, 1, 2] | 缺页 | 替换3，3不再出现 |
| 19   | 0    | [7, 0, 2] | 缺页 | 替换1, 1不再出现 |
| 20   | 1    | [7, 0, 1] | 缺页 | 替换2, 2不再出现 |

缺页次数：13次
缺页率：13/20 = 65%

**对比：** 在相同条件下，FIFO产生了15次缺页，OPT只有13次。OPT是最优的。

### 5.7.4 最近最久未使用算法（LRU）

**算法描述：**
LRU（Least Recently Used）算法选择最长时间没有被使用的页面进行置换。LRU基于"过去的使用情况可以预测未来的使用情况"这一假设，利用"最近的历史"来近似"未来的趋势"。

**与OPT的关系：**
LRU可以说是OPT的"反向实现"——OPT看未来，LRU看过去。如果程序的局部性良好，LRU的性能非常接近OPT。

**实现方式：**

**方法1：计数器实现（Counter-based）**
- 每个页表项关联一个使用时间字段
- CPU中有一个逻辑时钟或计数器，每次内存访问时+1
- 每次访问页面时，将当前时钟值复制到该页面的使用时间字段
- 置换时，扫描所有页面，找到使用时间最小的页面（即最近最久未使用）

**方法2：栈实现（Stack-based）**
- 维护一个栈，存储所有在内存中的页面号
- 每次访问某个页面时，将该页面从栈中移除并压入栈顶
- 栈底始终是最近最久未使用的页面
- 置换时选择栈底的页面

**详细示例（计数器方法）：**
页面引用串：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2, 0, 1, 7, 0, 1
内存帧数：3个

| 时间 | 引用 | 内存状态及上次使用时间 | 缺页 |
|------|------|---------------------|------|
| 1    | 7    | [7(1)]               | 缺页 |
| 2    | 0    | [7(1), 0(2)]         | 缺页 |
| 3    | 1    | [7(1), 0(2), 1(3)]   | 缺页 |
| 4    | 2    | [0(2), 1(3), 2(4)]   | 缺页（替换7，7的使用时间t=1最小） |
| 5    | 0    | [0(5), 1(3), 2(4)]   | 命中，更新时间 |
| 6    | 3    | [0(5), 2(4), 3(6)]   | 缺页（替换1，1的使用时间t=3最小） |
| 7    | 0    | [0(7), 2(4), 3(6)]   | 命中 |
| 8    | 4    | [0(7), 3(6), 4(8)]   | 缺页（替换2，2的使用时间t=4最小） |
| 9    | 2    | [0(7), 4(8), 2(9)]   | 缺页（替换3，3的使用时间t=6最小） |
| 10   | 3    | [0(7), 2(9), 3(10)]  | 缺页（替换4，4的使用时间t=8最小） |
| 11   | 0    | [0(11), 2(9), 3(10)] | 命中 |
| 12   | 3    | [0(11), 2(9), 3(12)] | 命中 |
| 13   | 2    | [0(11), 2(13), 3(12)]| 命中 |
| 14   | 1    | [0(11), 3(12), 1(14)]| 缺页（替换2，2的使用时间t=13最小） |
| 15   | 2    | [0(11), 1(14), 2(15)]| 缺页（替换3，3的使用时间t=12最小） |
| 16   | 0    | [0(16), 1(14), 2(15)]| 命中 |
| 17   | 1    | [0(16), 1(17), 2(15)]| 命中 |
| 18   | 7    | [1(17), 2(15), 7(18)]| 缺页（替换0，0的使用时间t=16最小） |
| 19   | 0    | [2(15), 7(18), 0(19)]| 缺页（替换1，1的使用时间t=17最小） |
| 20   | 1    | [7(18), 0(19), 1(20)]| 缺页（替换2，2的使用时间t=15最小） |

缺页次数：14次
缺页率：14/20 = 70%

**结果对比（对于该引用串，3个帧）：**
- OPT：13次缺页（最优）
- LRU：14次缺页（非常接近最优）
- FIFO：15次缺页（最差）

**LRU算法的优缺点：**
- 优点：
  - 性能接近OPT
  - 没有Belady异常（LRU属于栈算法，栈算法不会出现Belady异常）
- 缺点：
  - 实现代价高昂：计数器方法需要扫描所有页面，栈方法需要维护栈结构
  - 硬件支持复杂：需要每次内存访问时更新时间戳或栈

### 5.7.5 LRU近似算法

由于精确的LRU实现代价太高，实际系统中使用各种近似方法来实现LRU效果。

#### 引用位算法（Reference Bit Algorithm）

**基本原理：**
- 每个页表项关联一个引用位（Reference Bit）
- 初始时，所有引用位为0
- 当页面被访问时，硬件将引用位设置为1
- 操作系统定期（如每次时钟中断时）清零所有引用位
- 置换时，选择引用位为0的页面（说明在上一个周期中没有被访问过）

**优点：**
- 实现简单，只需要一个引用位
- 硬件支持简单

**缺点：**
- 粒度较粗（只能区分"被访问过"和"未被访问过"，不能精确知道访问顺序）
- 如果所有页面的引用位都是1，选择哪个？

#### 第二次机会算法（Second-Chance Algorithm / Clock Algorithm）

**算法描述：**
第二次机会算法（也称时钟算法，Clock Algorithm）是引用位算法的一个改进。它将所有页面组织成一个循环队列（像时钟的盘面），每个页面有一个引用位。

**具体实现：**
1. 所有页面排列成循环队列，每个页面有一个引用位R
2. 维护一个指针，指向当前页面
3. 当需要置换时，检查指针指向的页面：
   a. 如果R=0（未被访问），选择该页面作为牺牲页面
   b. 如果R=1（已被访问），将R置为0，指针移动到下一个页面，重复检查和修改
4. 如果在循环一周后所有页面的R都是1（都被清零了），那么选择当前页面（即最初的那个页面）

**直观理解：**
每个页面有一个"第二次机会"——如果它在上一个周期被访问过（R=1），它就可以再存活一个周期（被清零后留在内存中）。只有当它在下一个周期仍然没有被访问时（R=0），才会被置换。

**详细示例：**
页面引用串：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2, 0, 1, 7, 0, 1
内存帧数：3个

假设初始时钟指针指向帧0。

| 步骤 | 引用 | 内存状态(R位)及指针位置 | 缺页 | 操作 |
|------|------|----------------------|------|------|
| 1    | 7    | 帧0:[7,R=1] 指针→帧1     | 缺页 | 装入7 |
| 2    | 0    | 帧0:[7,R=1] 帧1:[0,R=1] 指针→帧2 | 缺页 | 装入0 |
| 3    | 1    | 帧0:[7,R=1] 帧1:[0,R=1] 帧2:[1,R=1] | 缺页 | 装入1 |
| 4    | 2    | 全部已满，指针→帧0 | 缺页 | 指针扫过帧0(7,R=1→R=0),帧1(0,R=1→R=0),帧2(1,R=1→R=0),回到帧0(7,R=0)→替换7为2,帧0:[2,R=1],指针→帧1 |
| 5    | 0    | 帧0:[2],帧1:[0],帧2:[1] | 命中 | 0在帧1, 设置R=1 |
| 6    | 3    | 指针→帧1 | 缺页 | 检查帧1(0,R=1→R=0), 指针→帧2; 检查帧2(1,R=0)→替换为3, 帧2:[3,R=1], 指针→帧0 |
| 7    | 0    | 帧0:[2],帧1:[0],帧2:[3] | 命中 | 0在帧1, 设置R=1 |
| 8    | 4    | 指针→帧0 | 缺页 | 检查帧0(2,R=0)→替换为4, 帧0:[4,R=1], 指针→帧1 |
| 9    | 2    | 指针→帧1 | 缺页 | 检查帧1(0,R=1→R=0),指针→帧2; 检查帧2(3,R=0)→替换为2, 帧2:[2,R=1], 指针→帧0 |
| 10   | 3    | 指针→帧0 | 缺页 | 检查帧0(4,R=0)→替换为3, 帧0:[3,R=1], 指针→帧1 |
| ... | ... | ... | ... | ... |

**第二次机会算法的改进——增强型第二次机会算法（Enhanced Second-Chance Algorithm）：**

使用两个位：引用位（Reference）和修改位（Modify），组合成4种情况：

| (R, M) | 描述 | 是否优先被置换 |
|--------|------|--------------|
| (0, 0) | 未使用且未修改 | 最优先被置换 |
| (0, 1) | 未使用但已修改 | 较优先（置换前需要写回磁盘） |
| (1, 0) | 已使用但未修改 | 较不优先 |
| (1, 1) | 已使用且已修改 | 最不优先（可能很快会再被用，且置换开销大） |

算法流程（时钟扫描）：
1. 从指针位置开始，扫描查找(R=0, M=0)的页面，不修改任何位
2. 如果找不到，再扫描查找(R=0, M=1)的页面，扫描时将所有(R=1)的页面置为(R=0)
3. 如果还找不到，回到第一步（此时所有页面的R位都是0了）
4. 重复上述过程直到找到牺牲页面

**优点：** 比简单时钟算法更精细，考虑了页面是否被修改（即置换代价）

### 5.7.6 最不常用算法（LFU）和最常用算法（MFU）

#### LFU（Least Frequently Used）

**算法描述：**
LFU算法选择访问次数最少的页面进行置换。每个页面维护一个计数器，每次被访问时计数器加1，置换时选择计数值最小的页面。

**实现方式：**
- 每个页表项关联一个计数器
- 每次访问时，计数器加1
- 置换时，选择计数器值最小的页面

**问题：**
- 一个页面在进程初期被大量使用（计数器很高），之后不再使用，但计数器仍然很高，不会被置换
- 解决：计数器定期右移（衰减）

#### MFU（Most Frequently Used）

**算法描述：**
MFU算法选择访问次数最多的页面进行置换。其背后逻辑是：访问次数最少的页面可能是刚刚调入、很快会再次使用的页面。

**实现方式：**
- 与LFU类似，但选择计数器值最大的页面

**实际效果：**
MFU在实际测试中性能通常不如LRU和LFU，基本不被使用。

### 5.7.7 页面置换算法的综合评价

| 算法 | 实现复杂度 | 性能 | Belady异常 | 实际采用情况 |
|------|-----------|------|-----------|------------|
| FIFO | 极简单 | 差 | 有 | 很少单独使用 |
| OPT | 不可实现 | 最优（基准） | 无 | 仅作为理论参考 |
| LRU | 高 | 接近最优 | 无 | 部分系统使用近似LRU |
| Clock | 简单 | 较好 | 无 | 广泛使用（Linux、Windows等） |
| 增强型Clock | 较简单 | 更好 | 无 | 广泛使用 |
| LFU | 中等 | 中等 | 无 | 较少使用 |

**实际系统中的使用：**
- Linux：使用LRU的近似算法（基于引用位的老化算法），结合了LFU和LRU的思想
- Windows：使用Clock算法（称为"Clock页面置换"）
- Solaris：使用Clock算法

## 5.8 帧分配策略

### 5.8.1 最少帧数

**概念：**
每个进程需要至少一定数量的物理帧才能执行。这个最小数量取决于指令集架构——某些指令在执行时需要访问多个页面，所有涉及的页面必须在内存中。

**最少帧数的确定：**
- 假设一条指令最多涉及N个页面（如块移动指令需要源地址一个页面、目的地址一个页面，加上指令本身一个页面，共3个页面）
- 最少帧数 ≥ N
- 如果系统分配给进程的帧数少于最少帧数，进程会因为频繁缺页而无法正常运行

**实际值：**
- x86架构：最少帧数通常为3~4个
- 有些架构可能更多（如某些RISC架构的复杂指令可能需要更多页面）

### 5.8.2 分配算法

**（1）平均分配（Equal Allocation）**
- 将所有物理帧平均分配给所有进程
- 每个进程获得的帧数 = 总帧数 / 进程数
- 优点：公平
- 缺点：大进程和小进程获得相同的帧数，不合理

**（2）按比例分配（Proportional Allocation）**
- 根据进程的大小（虚拟地址空间的大小）或优先级，按比例分配帧数
- 按大小分配：进程i的分配数 = (进程i的大小 / 所有进程总大小) × 总帧数
- 按优先级分配：高优先级进程获得更多帧

**按比例分配示例：**
假设总帧数 = 128，进程A大小 = 100页，进程B大小 = 50页，进程C大小 = 50页
- 总页数 = 200
- 进程A分配：100/200 × 128 = 64帧
- 进程B分配：50/200 × 128 = 32帧
- 进程C分配：50/200 × 128 = 32帧

**（3）优先级分配（Priority-Based Allocation）**
- 使用比例分配，但比例不是基于大小，而是基于优先级
- 高优先级进程获得更多帧，有助于其更快运行

### 5.8.3 全局置换 vs 局部置换

**局部置换（Local Replacement）：**
- 每个进程只从自己已经分配到的帧中选择牺牲页面
- 进程的帧数固定
- 优点：一个进程的缺页行为不会影响其他进程的帧数
- 缺点：如果某个进程的局部性很差，即使其他进程有空闲帧，它也只能在自己的帧中置换，无法利用其他进程的空闲帧

**全局置换（Global Replacement）：**
- 从整个系统的所有帧中选择牺牲页面（不考虑帧属于哪个进程）
- 进程的帧数可以动态变化（系统回收空闲帧或为进程增加帧）
- 优点：灵活性高，可以更有效地利用物理内存
- 缺点：一个进程的缺页行为会影响其他进程的帧数，导致系统行为难以预测

**全局置换的典型问题：**
- 进程A发生缺页，选择进程B的一个帧换出
- 进程B下次访问该帧时又发生缺页，选择进程A的一个帧换出
- 这种"互相踢皮球"的现象会导致系统效率下降

**实际系统如何选择：**
- 大多数实际操作系统（如Linux、Windows）使用全局置换，因为它更灵活
- 但为了防止一个进程过度占用内存，会结合工作集模型进行控制

### 5.8.4 非均匀内存访问（NUMA）与帧分配

在多处理器系统中，不同的处理器访问不同内存区域的速度可能不同（NUMA架构）。帧分配在NUMA系统中需要额外考虑：
- 尽量将进程的帧分配在与该进程运行的CPU相同的内存节点上
- 避免跨节点内存访问（开销较大）

## 5.9 抖动（Thrashing）——极其详细

### 5.9.1 抖动的定义

抖动（Thrashing），中文也称为颠簸或系统颠簸，是指操作系统中由于过度使用页面置换而导致系统性能急剧下降的现象。

**抖动的典型表现：**
- CPU利用率急剧下降（大部分时间花在等待I/O上）
- 磁盘I/O活动急剧增加（频繁的页面换入换出）
- 系统吞吐量显著下降
- 用户感觉到系统响应缓慢

**抖动的后果：**
在极端情况下，系统可能进入一种"死锁"状态——CPU永远在等待缺页处理的完成，而缺页处理需要CPU调度才能进行。虽然系统没有真正死锁，但有效工作量为零。

### 5.9.2 抖动的成因——详细分析

抖动的根本原因是：系统中运行的进程数（多道程序度，Degree of Multiprogramming）超过了物理内存所能容纳的范围，导致所有进程的工作集之和大于物理内存的总帧数。

**抖动的形成过程（详细）：**

1. **初始状态**：系统中有多个进程在并发执行，多道程序度为N

2. **进程递增**：操作系统为了充分利用CPU，增加多道程序度（创建一个新进程或唤醒一个挂起进程）

3. **内存压力增大**：新进程需要分配帧，导致空闲帧减少，每个已有进程的可用帧数可能被回收

4. **缺页率上升**：每个进程的可用帧数减少，导致工作集无法完全驻留在内存中，缺页率上升

5. **CPU等待I/O**：缺页率上升意味着进程经常在等待页面从磁盘调入，CPU利用率开始下降

6. **操作系统误判**：操作系统看到CPU利用率下降，误判为"CPU空闲"，于是增加更多进程以提高CPU利用率（这是一个致命的错误判断！）

7. **恶性循环**：
   - 更多进程 → 更少的内存分配给每个进程 → 更高的缺页率 → 更多的I/O等待 → CPU利用率更低 → 操作系统继续增加进程 → ...

8. **系统崩溃**：最终，系统绝大部分时间用于在内存和磁盘之间交换页面，几乎不做任何有用的工作

**一个数值例子说明抖动的成因：**

假设：
- 物理内存总帧数 = 100帧
- 每个进程的工作集大小 = 20帧
- 缺页处理时间 = 25ms
- 页面在内存中可以平均运行 200ms 才触发一次缺页（当工作集完整时）

情况1：多道程序度 = 4
- 工作集总需求 = 4 × 20 = 80帧 < 100帧 → 可以容纳
- 每个进程的工作集都在内存中，缺页率低
- CPU利用率高

情况2：多道程序度 = 6
- 工作集总需求 = 6 × 20 = 120帧 > 100帧 → 无法容纳
- 至少20帧的工作集内容必须被换出
- 每个进程频繁缺页
- CPU利用率急剧下降

### 5.9.3 工作集模型（Working-Set Model）

工作集模型由Denning于1968年提出，是解决抖动问题的理论基础。

**工作集的定义：**
进程在时间区间[t-Δ, t]内访问过的页面集合，记作W(t, Δ)，其中Δ是工作集窗口（Working-Set Window），是一个固定的时间间隔。

**工作集的特性：**
- 工作集大小|W|是进程当前局部性的度量
- 工作集大小随时间变化（当进程进入新的执行阶段时，工作集会变化）
- 如果进程的局部性好，|W|相对稳定
- 如果进程进入一个新的代码段（如函数调用），工作集可能快速变化（称为"过渡期"）

**工作集窗口Δ的选择：**
- Δ太小时，工作集不能全面反映进程的内存需求
- Δ太大时，工作集包含太多已经不使用的页面，失去意义
- 一般的经验值：Δ ≈ 10,000 ~ 100,000次内存访问
- 实际系统中，Δ是固定值，通过经验确定

**工作集模型的工作原理：**
1. 操作系统为每个进程记录其工作集（近似地，通过检查引用位）
2. 定期计算所有进程的工作集大小之和D = Σ|Wi|
3. 检查D是否超过物理内存总帧数m：
   - 如果D > m：说明内存无法容纳所有进程的工作集，选择挂起（Suspend）一个或多个进程
   - 如果D <= m：所有进程的工作集可以驻留内存，不需要挂起

**工作集近似实现：**
- 使用页表的引用位
- 每隔Δ次内存访问（或每隔固定时间），检查所有页面的引用位
- 如果引用位=1：该页在当前工作集中（设置为1并保留）
- 如果引用位=0：该页不在当前工作集中
- 定期清零引用位，开始新的周期

**工作集模型的局限性：**
- 需要为每个进程维护工作集信息，开销较大
- Δ的选择对性能影响很大，但难以确定最优值
- 工作集突变的处理（如进程进入一个新的执行阶段时）

### 5.9.4 缺页频率控制（Page-Fault Frequency, PFF）

缺页频率控制是另一种防止抖动的方法，比工作集模型更容易实现。

**基本原理：**
- 为每个进程设置缺页率的上限和下限
- 监控每个进程的实际缺页率
- 如果缺页率超过上限：说明该进程的帧数太少，需要分配更多帧
- 如果缺页率低于下限：说明该进程的帧数太多，可以回收一部分帧
- 如果无法为某个进程分配更多帧（因为总帧数已用完），则挂起该进程

**PFF的实现：**
```
缺页率计算：
    缺页率 = 缺页次数 / 单位时间（或每N次内存访问）

PFF控制逻辑：
    if (缺页率 > 上限) {
        if (有空闲帧) {
            分配给该进程更多帧;
        } else {
            挂起该进程（换出全部页面）;
        }
    } else if (缺页率 < 下限) {
        从该进程回收一些帧;
    }
```

**PFF的优点：**
- 实现简单，只需要监控缺页频率
- 不需要维护工作集信息
- 可以有效防止抖动

**PFF的缺点：**
- 上下限的设定需要经验
- 缺页率的波动可能导致频繁的帧调整

### 5.9.5 工作集 vs PFF 比较

| 方面 | 工作集模型 | PFF控制 |
|------|-----------|--------|
| 监控内容 | 页面使用模式 | 缺页频率 |
| 实现复杂度 | 较高（需要记录工作集） | 较低 |
| 控制精度 | 较高 | 中等 |
| 适用性 | 理论研究好 | 实际系统更常用 |
| 关键参数 | 窗口大小Δ | 缺页率上下限 |

### 5.9.6 防止抖动的其他策略

**（1）调整多道程序度**
直接控制系统中的进程数量，确保活动进程的总内存需求不超过物理内存。这是最直接的防止抖动的方法。

**（2）局部置换策略**
使用局部置换（每个进程只能替换自己的帧）可以防止一个进程的抖动扩散到其他进程。但局部置换不能防止单个进程的抖动。

**（3）优先权置换**
高优先级进程可以置换低优先级进程的帧。可以确保重要进程不会因为抖动而影响性能。

**（4）I/O互锁（I/O Interlocking）**
一个正在等待I/O操作的页面（即正在进行DMA传输的页面）不能被置换。如果置换这样的页面，会导致I/O数据写入错误的帧。

## 5.10 其他考虑

### 5.10.1 预调页（Prepaging）

**基本思想：**
预调页是指在进程真正需要某个页面之前，预先将其调入内存。目的是减少进程启动时或进入新的执行阶段时的大量缺页。

**预调页的时机：**
- 进程创建时：预先将程序代码的初始页面调入内存
- 进程从挂起状态恢复时：预先将进程的工作集调入内存
- 进程进入新的执行阶段时

**预调页的实现：**
- 使用进程的历史信息（如上次运行时的页面使用情况）
- 或者使用程序的语义信息（如编译器提供的提示）

**预调页 vs 请求调页：**
- 预调页：在需要之前调入，可以减少缺页次数
- 请求调页：在需要时才调入，可以避免调入不需要的页面

**预调页的收益分析：**
预调页是否有效取决于实际调入的页面中真正被使用的比例：
- 如果预调的页面大部分被使用（如进程恢复时的工作集预调），则收益很大
- 如果预调的页面大部分没有被使用（如过度预调），则浪费了磁盘I/O带宽

### 5.10.2 页面大小的选择

页面大小是内存管理系统设计中的一个重要参数，它与系统性能密切相关。

**小页面的优点：**
- 内部碎片少（最后一页浪费的空间少）
- 更精确的工作集控制（页面粒度小，不会包含太多不需要的内容）
- 可以减少I/O传输量（每次缺页只调入一个较小的页面）

**小页面的缺点：**
- 页表大（每个进程需要更多的页表项）
  - 例如：4KB页面时，4GB地址空间需要1M个页表项
  - 8KB页面时，4GB地址空间需要512K个页表项
- 磁盘I/O次数可能增加（每次缺页调入的页面小，但次数多）
- TLB覆盖范围小（每个TLB条目只映射4KB，TLB能够覆盖的总内存大小有限）

**大页面的优点：**
- 页表小（页表项数量少）
- TLB覆盖范围大（每个TLB条目映射更大的内存）
  - 4KB页面，64个TLB条目映射256KB
  - 2MB页面，64个TLB条目映射128MB（512倍提升！）
- 磁盘I/O效率高（一次读入更多数据，顺序I/O吞吐量大）

**大页面的缺点：**
- 内部碎片大（特别是对于小进程）
- 可能调入不需要的数据（浪费I/O带宽和内存）

**页面大小的折中：**
- 传统页面大小：4KB（x86、ARM等主流架构的默认设置）
- 大页面（Huge Pages）：2MB（x86-64支持），用于数据库、大型科学计算等
- 巨页面（Gigantic Pages）：1GB（x86-64支持），用于虚拟化场景

**影响页面大小选择的关键因素：**
1. **页表大小**：页面越大，页表越小
2. **TLB覆盖范围**：页面越大，TLB覆盖范围越大，TLB命中率越高
3. **内存碎片**：页面越大，内部碎片越严重
4. **I/O效率**：页面越大，I/O效率越高（磁道内寻道时间相对更少）
5. **程序结构**：不同程序有不同的访问模式

**局部性对页面大小的影响：**
- 如果一个程序具有良好的空间局部性，大页面可以获得更好的性能（TLB命中率高）
- 如果一个程序的空间局部性差（如随机访问大数组），小页面更合适（不会调入大量无用数据）

### 5.10.3 程序结构与分页性能

程序的设计和数据的组织方式会对分页系统的性能产生显著影响。下图展示了一个典型的问题：

**行优先 vs 列优先访问二维数组：**

假设有一个 1024 × 1024 的整型数组，每个整数占4字节，页面大小4KB（可存放1024个整数），数组按行优先存储在内存中。

**情况1：行优先方式访问（外层循环按行，内层循环按列）**

```
for (i = 0; i < 1024; i++)
    for (j = 0; j < 1024; j++)
        x = array[i][j];
```

- 访问顺序：array[0][0], array[0][1], ..., array[0][1023], array[1][0], ...
- 由于数组按行存储，array[0][0]到array[0][1023]在同一个页面中（同一个4KB页面）
- 每访问一行只引起一次缺页（第一次访问该行时）
- 总共缺页次数：1024次（每行一次）

**情况2：列优先方式访问（外层循环按列，内层循环按行）**

```
for (j = 0; j < 1024; j++)
    for (i = 0; i < 1024; i++)
        x = array[i][j];
```

- 访问顺序：array[0][0], array[1][0], ..., array[1023][0], array[0][1], ...
- 由于数组按行存储，array[0][0]和array[1][0]不在同一个页面中（它们相距4096字节）
- 每访问一个元素（几乎）都引起一次缺页
- 总共缺页次数：1024 × 1024 = 1,048,576次

**结论：**
行优先方式比列优先方式的缺页次数少1000倍！这就是程序结构对分页性能的巨大影响。

**其他影响分页性能的程序设计原则：**
1. **提高空间局部性**：
   - 尽量顺序访问数组元素（而不是跳跃访问）
   - 将相关数据组织在临近的内存位置

2. **提高时间局部性**：
   - 对同一数据重复使用（而不是每次都重新计算或从磁盘读取）
   - 使用缓存（Cache）技术

3. **减少页面跨越**：
   - 热数据（频繁访问的数据）尽量集中存放
   - 避免在循环中访问跨越多个页面的数据

## 5.11 内核内存管理

### 5.11.1 内核内存管理的特殊性

操作系统的内核也需要管理自己的内存（为内核数据结构分配内存），但内核内存管理与用户进程的内存管理存在显著差异：

1. **内核代码不能使用缺页**：内核通常运行在"禁止缺页"的模式下（中断上下文或自旋锁保护的区域中），不能发生缺页异常
2. **内存分配必须快速**：内核的内存分配路径很短，不能有复杂的查找和回收
3. **内存碎片控制**：内核内存分配对碎片更加敏感
4. **物理连续性**：某些硬件设备需要连续物理内存（如DMA缓冲区）

因此，内核采用专门的内存管理机制。

### 5.11.2 伙伴系统（Buddy System）

**基本思想：**
伙伴系统是内核中最常用的物理页面分配算法。它将物理内存按照2的幂次大小进行管理，以页（通常是4KB）为最小分配单元。

**数据结构：**
- 维护一个链表数组（free_area），每个链表对应一个大小的块（2^0页, 2^1页, 2^2页, ..., 2^MAX_ORDER页）
- 例如：
  - free_area[0]：大小为1页（4KB）的空闲块列表
  - free_area[1]：大小为2页（8KB）的空闲块列表
  - free_area[2]：大小为4页（16KB）的空闲块列表
  - ...
  - free_area[9]：大小为512页（2MB）的空闲块列表

**分配过程：**
假设需要分配一个大小为2^n页的块：
1. 在free_area[n]中查找是否有空闲块
2. 如果有，直接分配
3. 如果没有，在free_area[n+1]中查找：
   a. 如果找到一个大块（大小为2^(n+1)页），将其一分为二（分裂）成两个伙伴（Buddy）
   b. 一个伙伴（大小为2^n页）分配给请求者
   c. 另一个伙伴（大小为2^n页）放入free_area[n]中
4. 如果free_area[n+1]也没有，继续在free_area[n+2]中查找，依此类推

**回收过程：**
释放一个大小为2^n页的块时：
1. 检查其伙伴（Buddy，与它相邻的相同大小的块）是否空闲
2. 如果是空闲的，将两个块合并成一个大小为2^(n+1)页的大块
3. 继续检查这个大块的伙伴是否空闲，进行更大规模的合并
4. 最终将尽量大的空闲块放入对应的free_area链表中

**伙伴的定义：**
两个大小为2^n页的块互为伙伴，如果它们是由同一个大小为2^(n+1)页的块分裂得到的。具体来说，如果块A的起始地址为P，块B的起始地址为P + 2^n × 页面大小，则A和B是伙伴。

**伙伴系统的数值示例：**

假设初始时系统有一个1MB的空闲块（256页，假设页面大小4KB），即free_area[8]中有1个块。

分配请求：需要32KB（8页，即2^3页）

1. 在free_area[3]中查找 → 没有
2. 在free_area[4]中查找 → 没有
3. ... 直到free_area[8]中找到1MB块
4. 将1MB块一分为二：512KB + 512KB，将512KB放入free_area[7]
5. 将512KB块一分为二：256KB + 256KB，将256KB放入free_area[6]
6. 将256KB块一分为二：128KB + 128KB，将128KB放入free_area[5]
7. 将128KB块一分为二：64KB + 64KB，将64KB放入free_area[4]
8. 将64KB块一分为二：32KB + 32KB，将32KB（8页）分配给请求者，将另一个32KB放入free_area[3]

**伙伴系统的优缺点：**

- 优点：
  - 快速合并和分裂（只需要位操作即可找到伙伴）
  - 有效减少外部碎片
  - 内存回收时自动合并，减少碎片化

- 缺点：
  - 内部碎片仍然存在（分配的块必须是2的幂次）
  - 小块分配（如几十字节）非常浪费

**实际应用：**
- Linux内核的物理页面分配器使用伙伴系统（称为Buddy Allocator）
- Linux的free_area有11个条目（MAX_ORDER = 11），从2^0到2^10页（4KB到4MB）

### 5.11.3 SLAB分配器（Slab Allocator）

**为什么需要SLAB分配器？**

伙伴系统以页为单位分配内存，但内核中经常需要分配很小的内存对象（如几十字节）。如果直接使用伙伴系统分配，会产生巨大的内部碎片（分配一个48字节的对象需要4KB的页面，浪费了99%的空间）。

此外，内核中某些对象（如task_struct、inode等）被频繁地创建和销毁，如果每次分配和释放都调用伙伴系统，开销很大。

**SLAB的基本思想：**
- 为每种内核对象类型维护一个专门的缓存（Cache）
- 每个缓存由若干块（Slab）组成，每个Slab中包含若干个相同大小的对象
- 对象被创建时从缓存中分配，销毁时归还到缓存（不释放给伙伴系统）
- 这是一种"对象缓存"（Object Caching）机制

**SLAB的三种状态：**
- 完全满（Full Slab）：所有对象都在使用
- 部分满（Partial Slab）：部分对象在使用，部分空闲
- 完全空（Empty Slab）：所有对象都空闲

**SLAB分配流程：**
1. 请求分配某个类型的对象
2. 在对应的缓存中查找：部分满 → 有空闲对象
3. 如果找到：从部分满的Slab中分配一个对象
4. 如果没找到：从完全空的Slab中分配（如果有的话）
5. 如果完全空的Slab也没有：向伙伴系统申请新页面创建新的Slab

**SLAB回收流程：**
1. 释放对象到其所属的Slab中
2. 如果Slab变为完全空：
   - 可以选择立即释放给伙伴系统，也可以保留作为备用（取决于缓存配置）
   - 如果保留，下次分配时可以直接使用

**SLAB的着色（Slab Coloring）：**
- 不同的Slab中的对象起始地址不同（称为"颜色"不同）
- 目的是使不同Slab中的同一个缓存行（Cache Line）对应不同的物理地址
- 可以提高CPU缓存的利用率，减少缓存冲突

**SLUB分配器：**
- SLUB是Linux内核中SLAB的简化替代品
- SLUB简化了SLAB的复杂管理逻辑，性能更好
- 从Linux 2.6.23开始，SLUB成为默认的分配器
- SLUB的基本思想与SLAB类似，但去除了每个CPU的本地对象缓存等复杂机制

**SLAB/SLUB与进程内存分配的关系：**
- SLAB/SLUB负责分配内核对象（如task_struct、inode、dentry、mm_struct等）
- 用户进程的内存分配（如malloc/free）不直接使用SLAB，但用户进程的控制块等内核对象会使用SLAB

**典型的内核对象大小（64位系统）：**
| 内核对象 | 大小（字节） |
|---------|------------|
| task_struct | ~2000 |
| inode | ~600 |
| dentry | ~200 |
| mm_struct | ~800 |
| vm_area_struct | ~200 |

## 本章易错点（至少10条）

**易错点1：页号和页框号的区别**
- 页号（Page Number）是逻辑地址空间中的编号
- 帧号/页框号（Frame Number）是物理地址空间中的编号
- 页表的作用就是将页号映射为帧号，不要混淆

**易错点2：内部碎片 vs 外部碎片**
- 内部碎片：分配给进程的内存块内部未被使用的部分（固定分区和分页中的最后一页）
- 外部碎片：空闲内存块太小，无法满足分配需求（动态分区和分段）
- 分页管理只有内部碎片（最后一页），没有外部碎片
- 分段管理只有外部碎片，没有内部碎片（如果分配时严格按需分配）

**易错点3：TLB命中时的有效访问时间计算**
- 正确公式：EAT = ε + αt + (1-α)(ε + t + t) = ε + 2t - αt
- 常见错误：忘记加上TLB查找时间ε，或者忘记TLB未命中时需要两次内存访问

**易错点4：Belady异常只发生在FIFO算法中**
- 只有FIFO（及其变体）会出现Belady异常
- LRU和OPT（以及所有栈算法）不会出现Belady异常
- 栈算法：内存集N是内存集N-1的子集

**易错点5：OPT算法不可实现**
- OPT需要知道未来的页面引用，无法实现
- 只能作为理论最优基准（Belady最优算法）
- 不要误以为OPT可以在实际系统中使用

**易错点6：LRU的实现代价很高**
- 精确的LRU需要每次访问时更新硬件状态（时间戳或栈）
- 实际系统都使用LRU的近似算法（如Clock算法）
- 考试中可能会问为什么实际系统不使用精确LRU

**易错点7：分页和分段地址空间的维度不同**
- 分页：一维地址空间（页号 + 偏移，页号是一个连续的整数序列）
- 分段：二维地址空间（段号 + 偏移，不同段的地址空间是独立的）
- 在分段中，(0, 100) 和 (1, 100) 是两个完全不同的地址

**易错点8：段页式的地址转换需要访问多少次内存**
- 无TLB：3次（段表→页表→物理内存）
- 有TLB且命中：1次（直接从TLB获得物理地址）或说1次TLB查找+1次内存访问
- 易错：有人以为只需要2次（段表→页表），忘记了最后需要访问物理内存

**易错点9：覆盖和交换的区别**
- 覆盖：同一进程内，不同模块共享同一块内存区域（时间上交替）
- 交换：不同进程之间，整个进程的换入换出（空间上竞争）
- 覆盖需要程序员参与，交换由操作系统自动完成

**易错点10：缺页率对系统性能的巨大影响**
- 即使缺页率只有0.01（1%），系统性能也会下降上千倍
- 缺页率必须低于10^-5才能保证系统性能可接受
- 公式：EAT = (1-p)×ma + p×pft，其中pft是毫秒级，ma是纳秒级

**易错点11：工作集窗口Δ的选择**
- Δ太小：工作集不能反映真实的内存需求
- Δ太大：工作集包含太多已经不使用的页面
- Δ的选择对工作集模型的有效性至关重要

**易错点12：伙伴系统的最小分配单元不是1字节，而是1页**
- 伙伴系统以页为最小分配单位
- SLAB/SLUB在伙伴系统之上进行更细粒度的分配
- 不要混淆伙伴系统和SLAB分配器的职责范围

**易错点13：LRU vs LFU vs MFU的区别**
- LRU：最近最久未使用（基于时间，看上次使用时间的远近）
- LFU：最不常用（基于频率，看总使用次数）
- MFU：最常用（基于频率，看总使用次数最多）
- 三个名字很相似，不要记混

**易错点14：边界确定问题**
- 段页式中段号和页号的位数分配不是固定的
- 考试中可能会给具体的位数分配，需要根据位数计算地址空间大小
- 例如：段号8位、页号10位、偏移12位 → 最大段数256，每段最大页数1024，页面大小4KB

## 本章常考题型

### 题型一：地址转换计算

**典型题目：**
某系统采用分页存储管理，页面大小为4KB，已知某进程的页表如下（页号从0开始），请将逻辑地址转换为物理地址。

页表：
| 页号 | 帧号 |
|------|------|
| 0    | 2    |
| 1    | 5    |
| 2    | 8    |
| 3    | 1    |

逻辑地址：16385

**解题步骤：**
1. 页面大小 = 4KB = 4096
2. 页号 p = 16385 / 4096 = 4（取整），页内偏移 d = 16385 % 4096 = 1
3. 如果页表中没有页号4 → 缺页异常
4. 如果页表中有页号4 → 物理地址 = 帧号 × 4096 + 1

### 题型二：TLB有效访问时间计算

**典型题目：**
某系统TLB查找时间为20ns，内存访问时间为100ns。如果TLB命中率为90%，请计算有效访问时间（EAT）。

**解题过程：**
EAT = 20 + 100 × (2 - 0.9) = 20 + 100 × 1.1 = 20 + 110 = 130ns

**变体：**
如果TLB命中率为98%，EAT = 20 + 100 × (2 - 0.98) = 20 + 100 × 1.02 = 20 + 102 = 122ns

### 题型三：页面置换算法计算（必考重点）

**典型题目：**
给定页面引用串：3, 2, 1, 0, 3, 2, 4, 3, 2, 1, 0, 4，内存有3个帧，请分别计算FIFO、LRU、OPT的缺页次数。

**解题步骤：**
1. 分别绘制三个算法的置换过程表
2. 统计缺页次数
3. 按缺页次数从少到多排列：OPT < LRU < FIFO（一般情况下）

### 题型四：Belady异常证明

**典型题目：**
证明FIFO算法可能出现Belady异常，用引用串1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5，分别用3个帧和4个帧计算缺页次数。

**解题步骤：**
1. 3个帧时缺页9次
2. 4个帧时缺页10次
3. 帧数增加，缺页反而增加，证明Belady异常存在

### 题型五：多级页表地址转换

**典型题目：**
某系统采用二级页表，页面大小为4KB，页目录有10位，页表有10位，页内偏移有12位。给定逻辑地址0x003AB014，进行地址转换。

**解题步骤：**
1. 将十六进制地址转换为二进制
2. 按10位、10位、12位分割
3. 分别得到页目录索引、页表索引、页内偏移
4. 按二级页表的转换流程进行转换

### 题型六：分段地址转换

**典型题目：**
某系统采用分段存储管理，段表如下：
| 段号 | 基址 | 长度 |
|------|------|------|
| 0    | 2000 | 1000 |
| 1    | 5000 | 2000 |
| 2    | 8000 | 3000 |

请判断以下地址是否合法，如果合法给出物理地址：
(1) (0, 500)    (2) (1, 2500)    (3) (2, 1500)

**解题过程：**
(1) 500 < 1000 → 合法，物理地址 = 2000 + 500 = 2500
(2) 2500 >= 2000 → 不合法，段错误
(3) 1500 < 3000 → 合法，物理地址 = 8000 + 1500 = 9500

### 题型七：工作集计算

**典型题目：**
给定页面引用串：1, 2, 3, 2, 4, 1, 3, 2, 1, 4, 5, 1, 2, 3，工作集窗口Δ=4，请计算时间t=6时的工作集。

**解题过程：**
时间t=6，窗口[t-Δ+1, t] = [3, 6]
引用串位置3到6的页面：3, 2, 4, 1
工作集W = {1, 2, 3, 4}
工作集大小 = 4

### 题型八：缺页率对有效访问时间的影响

**典型题目：**
某系统内存访问时间为200ns，缺页处理时间为25ms。请计算缺页率分别为0.1、0.01、0.001、0.0001时的有效访问时间。

**解题过程：**
pf=0.1：EAT = 0.9×200 + 0.1×25000000 = 180 + 2500000 = 2500180ns ≈ 2.5ms
pf=0.01：EAT = 0.99×200 + 0.01×25000000 = 198 + 250000 = 250198ns ≈ 0.25ms
pf=0.001：EAT = 0.999×200 + 0.001×25000000 = 199.8 + 25000 = 25199.8ns ≈ 25μs
pf=0.0001：EAT = 0.9999×200 + 0.0001×25000000 = 199.98 + 2500 = 2699.98ns ≈ 2.7μs

结论：缺页率必须低于0.001（即0.1%）系统性能才可接受，低于0.0001才接近无缺页性能。

### 题型九：伙伴系统的分配和合并

**典型题目：**
初始时有一个1MB的空闲内存块（页面大小为4KB），请使用伙伴系统完成以下操作并给出最终的内存状态：
(1) 分配64KB
(2) 分配128KB
(3) 释放64KB
(4) 分配32KB

**解题过程：**
初始：1MB块

(1) 分配64KB = 16页 = 2^4页
- 1MB → 分裂：512KB + 512KB（将512KB加入free_area[7]）
- 512KB → 分裂：256KB + 256KB（将256KB加入free_area[6]）
- 256KB → 分裂：128KB + 128KB（将128KB加入free_area[5]）
- 128KB → 分裂：64KB + 64KB（分配第一个64KB，第二个64KB加入free_area[4]）

(2) 分配128KB = 32页 = 2^5页
- free_area[5]中有128KB块 → 直接分配

(3) 释放64KB
- 释放的64KB加入free_area[4]
- 检查其伙伴（另一个64KB）：也在free_area[4]中 → 合并为128KB，放入free_area[5]

(4) 分配32KB = 8页 = 2^3页
- 在free_area[3]中查找 → 没有
- 在free_area[4]中查找 → 没有
- 在free_area[5]中查找 → 有128KB
- 128KB → 分裂：64KB + 64KB（将64KB加入free_area[4]）
- 64KB → 分裂：32KB + 32KB（分配第一个32KB，第二个32KB加入free_area[3]）

### 题型十：段页式地址转换

**典型题目：**
某系统采用段页式存储管理，页面大小为2KB，段号占4位，页号占10位，页内偏移占11位。已知段表如下：
| 段号 | 页表大小（页数） | 页表基址 |
|------|-----------------|---------|
| 0    | 8               | 1000    |
| 1    | 4               | 2000    |

段0的页表（从地址1000开始，每个页表项占4字节）：
| 页号 | 帧号 |
|------|------|
| 0    | 10   |
| 1    | 15   |
| 2    | 20   |

请将逻辑地址（0, 1, 500）转换为物理地址。

**解题过程：**
1. 段号=0，页号=1，偏移=500
2. 段表[0]：页表基址=1000，页表大小=8页
3. 检查：页号1 < 8 → 合法
4. 页表项地址 = 1000 + 1 × 4 = 1004
5. 从地址1004读取页表项 → 帧号=15
6. 物理地址 = 15 × 2048 + 500 = 30720 + 500 = 31220

## 本章总结

内存管理是操作系统的核心功能，涉及的内容广泛而深入。从最基本的连续分配到先进的分页分段虚拟存储器，每种技术都有其适用场景和优劣。本章的关键知识点归纳如下：

**必须掌握的核心概念：**
1. 地址绑定（编译时、装入时、运行时）
2. 静态链接与动态链接的区别
3. 连续分配的四种算法（FF、NF、BF、WF）
4. 内部碎片与外部碎片的成因和区别
5. 分页管理的地址转换机制
6. TLB的原理和有效访问时间计算
7. 多级页表的工作原理
8. 分段管理的地址转换和越界检查
9. 段页式管理的三层地址转换
10. 虚拟存储器的核心概念（请求分页、缺页异常）
11. 五种页面置换算法（FIFO、OPT、LRU、Clock、增强型Clock）
12. Belady异常的成因和证明
13. 抖动的成因和解决方案（工作集模型、PFF）
14. 伙伴系统和SLAB分配器的原理

**重要公式：**
1. EAT(TLB) = ε + t(2 - α)
2. EAT(缺页) = (1-p) × ma + p × pft
3. 物理地址(分页) = 帧号 × 页面大小 + 页内偏移
4. 物理地址(分段) = 段基址 + 段内偏移

**常见考试陷阱：**
- 计算页号时忘记取整运算
- 混淆段号和页号
- 忘记多级页表的地址转换需要多次访存
- 对缺页率的数量级缺乏直观认识
- 混淆各种置换算法的特点和适用场景
