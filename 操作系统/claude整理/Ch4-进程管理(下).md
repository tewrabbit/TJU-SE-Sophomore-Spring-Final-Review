# 第四章 进程管理（下）：进程同步与死锁

> 本章是操作系统课程的核心难点，涉及进程间的协作与资源管理两大主题。进程同步解决多个进程如何有序访问共享资源的问题，死锁解决多个进程因竞争资源而永久阻塞的问题。

---

## 4.1 临界区问题（Critical Section Problem）

### 4.1.1 临界资源（Critical Resource）

**临界资源（Critical Resource）** 是指一次只允许一个进程访问的资源，也称为**互斥资源**。多个进程如果同时访问临界资源，会导致数据不一致或状态错误。

**临界资源的常见例子：**

| 资源类型 | 具体例子 | 说明 |
|---------|---------|------|
| 硬件设备 | 打印机 | 同时打印多个文档会导致内容交错 |
| 硬件设备 | 磁带机 | 不能同时执行多个读写命令 |
| 软件数据 | 共享变量 | 多个进程同时修改导致数据不一致 |
| 软件数据 | 文件 | 同时写入会导致内容混乱 |
| 数据结构 | 队列、链表 | 同时修改破坏指针完整性 |

**关于临界资源的两个重要直觉：**

1. 临界资源不一定是硬件——**共享变量、共享数据结构**也是临界资源，而且是考试重点。
2. 把互斥访问的代码段叫作"临界区"容易产生混淆，需要明确区分"临界资源"和"临界区"两个概念。

### 4.1.2 临界区（Critical Section）

**临界区（Critical Section）** 是一段代码，这段代码**访问（读或写）临界资源**。多个进程的临界区之间需要互斥执行。

**关键澄清（极易混淆的概念）：**

> ❌ **错误理解**：临界区是"实现互斥的代码"。
> ✅ **正确理解**：临界区是"访问临界资源的代码"。
>
> 换言之：程序中所有访问共享变量、共享数据结构的代码片段，都称为临界区。而实现互斥的代码（如 P/V 操作、锁）是**同步机制**，不是临界区本身。

**举例说明：**

```c
// 共享变量 counter 是临界资源
int counter = 0;

// 进程 A
void process_A() {
    // 非临界区
    printf("doing other work\n");
    
    // 临界区 —— 访问临界资源 counter
    counter++;
    
    // 非临界区
    printf("done\n");
}

// 进程 B
void process_B() {
    // 非临界区
    printf("doing other work\n");
    
    // 临界区 —— 访问同一个临界资源 counter
    counter++;
    
    // 非临界区
    printf("done\n");
}
```

如果 A 和 B 的临界区不互斥，可能出现以下执行顺序导致错误：
- A 读取 counter(=0) → B 读取 counter(=0) → A 写入 counter(=1) → B 写入 counter(=1)
- 最终结果是 1 而不是 2，这就是典型的**竞态条件**。

### 4.1.3 竞态条件（Race Condition）

**定义：** 多个进程并发访问和操作同一组数据，执行结果与访问的特定顺序有关，称为竞态条件。

**竞态条件的本质：** "检查然后设置"（check-then-act）操作不是原子的。一个进程在"检查"和"设置"之间，另一个进程插入了操作。

**经典竞态条件场景：**

```
时间 → 进程 A                    进程 B
t1    load counter → reg1
t2                                 load counter → reg2
t3    reg1 + 1 → reg1
t4                                 reg2 + 1 → reg2
t5    reg1 → counter (=1)
t6                                 reg2 → counter (=1)
```

两个进程都对 counter 做了 +1，但最终结果是 1 而不是 2。这是因为"counter++"实际上对应三条机器指令（读、加、写），这三条指令不是原子的。

### 4.1.4 解决临界区问题的四项原则

任何一个解决临界区问题的方案，都必须满足以下四个条件：

#### 原则一：空闲让进（Mutual Exclusion / Idle-Allow-Entry）

当没有进程在临界区时，任何请求进入临界区的进程应当被允许立即进入。不能因为某个进程不在临界区而阻止其他进程进入。

**通俗理解：** 锁是空闲的，谁想进来就让谁进来。

#### 原则二：忙则等待（Progress / Busy-Wait）

当已有进程在临界区时，其他企图进入临界区的进程必须等待。这是**互斥**的直接体现。

**通俗理解：** 有人在里面，外面的人等着。

#### 原则三：有限等待（Bounded Waiting）

任何一个请求进入临界区的进程，必须在有限时间内获准进入，不能无限期等待。

**通俗理解：** 不能让人等到天荒地老——必须保证不会出现**饥饿（Starvation）**现象。

#### 原则四：让权等待（Yield-Wait / No Busy Waiting）

当一个进程无法进入临界区时，应当释放 CPU（即主动阻塞/睡眠），而不是占用 CPU 不断循环检查（忙等待），以避免 CPU 资源的浪费。

**通俗理解：** 进不去就别在那干等，把 CPU 让给其他能执行的进程。

> **考试提示：** 四项原则中，前三条（空闲让进、忙则等待、有限等待）是所有互斥方案必须满足的基本要求。第四条（让权等待）是理想要求——纯软件方案和硬件方案都无法满足让权等待，需要信号量等高级抽象才能实现。

## 4.2 软件互斥方法

软件互斥方法是指仅通过编程语言层面的逻辑（如变量设置和检查）来实现进程互斥，不依赖任何特殊的硬件指令或操作系统支持。

> **共同缺点：** 所有纯软件方法都存在**忙等待（busy waiting）**问题，不满足"让权等待"原则。

### 4.2.1 算法一：单标志法（Single Flag / Turn Variable）

**核心思想：** 设置一个共享变量 turn，表示"轮到谁进入临界区"。turn = i 表示进程 i 可以进入。

**完整代码：**

```c
// 共享变量
int turn = 0;  // 初始化为 0，表示进程 0 先进入

// 进程 Pi (i = 0 或 1)
void Pi() {
    do {
        // 非临界区
        while (turn != i);  // 等待轮到自己
        
        // 临界区
        
        turn = 1 - i;       // 将 turn 设为对方
        
        // 非临界区
    } while (1);
}
```

**正确性分析：**

| 原则 | 是否满足 | 说明 |
|------|---------|------|
| 忙则等待 | ✅ | turn 同时只能取一个值，只有一个进程能通过检查 |
| 空闲让进 | ❌ | 当 turn = 0 但 P0 不在临界区时，P1 也无法进入（turn 是 0） |
| 有限等待 | ✅ | 每次退出临界区后 turn 交给对方，不会无限等待 |
| 让权等待 | ❌ | while 循环是忙等待 |

**根本缺陷：强制交替进入（Forced Alternation）**

这个算法的最大问题是**违背了"空闲让进"原则**。假设 turn = 0：
- P0 不想进入临界区（在做其他事）
- P1 想进入临界区
- 但 turn = 0，P1 被阻塞
- 即使临界区空闲，P1 也无法进入

这就像是两个人过一扇门，规定必须轮流——即使一方不想过门，另一方也不能连续过两次。这严重降低了系统效率。

### 4.2.2 算法二：双标志先检查法（Double Flag, Check-Then-Set）

**核心思想：** 每个进程设置自己的标志 flag[i] = true 表示"我想进入临界区"。在进入临界区前，先检查对方是否也想进入。

**完整代码：**

```c
// 共享变量
bool flag[2] = {false, false};  // flag[i] = true 表示进程 i 想进入

// 进程 Pi (i = 0 或 1)
void Pi() {
    do {
        // 非临界区
        
        // 检查：对方是否想进入？
        while (flag[1 - i]);    // 如果对方想进入，我就等待
        
        // 设置：我想进入
        flag[i] = true;
        
        // 临界区
        
        flag[i] = false;        // 退出时清除标志
        
        // 非临界区
    } while (1);
}
```

**为什么失败——竞态条件导致双双进入：**

致命问题在于"检查"和"设置"不是原子的。考虑以下执行序列：

```
时间 → P0                           P1
t1    while (flag[1]) → false
t2                                    while (flag[0]) → false
t3    flag[0] = true
t4                                    flag[1] = true
t5    进入临界区                       进入临界区
```

**两个进程同时进入了临界区！** 违反了"忙则等待"原则。

这就是典型的**检查-设置非原子性问题**。两个进程都检查到对方不想进入（flag 都是 false），然后都设置自己的标志，结果双双进入临界区。

### 4.2.3 算法三：双标志后检查法（Double Flag, Set-Then-Check）

**核心思想：** 先设置自己的标志再检查对方，试图解决算法二的竞态条件问题。

**完整代码：**

```c
// 共享变量
bool flag[2] = {false, false};

// 进程 Pi (i = 0 或 1)
void Pi() {
    do {
        // 非临界区
        
        // 设置：我想进入
        flag[i] = true;
        
        // 检查：对方是否也想进入？
        while (flag[1 - i]);
        
        // 临界区
        
        flag[i] = false;        // 退出时清除标志
        
        // 非临界区
    } while (1);
}
```

**为什么失败——死锁（Deadlock-Like）僵局：**

```
时间 → P0                           P1
t1    flag[0] = true
t2                                    flag[1] = true
t3    while (flag[1]) → true，等待      while (flag[0]) → true，等待
```

**两个进程都永远无法进入临界区！** 因为没有进程愿意先退出循环（先放弃自己的标志）。

这就是一个**类似死锁**的状态。每个进程都持有资源（自己的 flag 为 true）并等待对方释放资源（对方的 flag 变为 false）。严格来说，这不是死锁（因为死锁的定义涉及资源，而这里只是忙等待），但从效果上看完全相同——双方都无法继续执行。

### 4.2.4 算法四：Peterson 算法（Peterson's Solution）

**核心思想：** 结合 turn 变量和 flag 数组。每个进程在进入临界区前先设置自己的 flag 为 true，然后将 turn 让给对方，再检查对方是否也想进入且 turn 是否是自己。

**完整代码：**

```c
// 共享变量
bool flag[2] = {false, false};
int turn;

// 进程 Pi (i = 0 或 1)
void Pi() {
    do {
        // 非临界区
        
        flag[i] = true;         // 声明我想进入
        turn = 1 - i;           // 把机会让给对方
        
        // 等待：对方不想进 或者 轮到我了
        while (flag[1 - i] && turn == (1 - i));
        
        // 临界区
        
        flag[i] = false;        // 退出时清除标志
        
        // 非临界区
    } while (1);
}
```

**为什么 Peterson 算法是正确的？**

理解 Peterson 算法的关键在于分析 while 循环的条件：

```
while (flag[1-i] && turn == (1-i));
```

这个循环的意思是：**只有当对方也想进入（flag[1-i] 为 true）且轮到了对方（turn == 1-i）时，我才等待。**

#### 证明一：满足忙则等待（互斥）

**反证法：** 假设 P0 和 P1 同时进入了临界区。

当 P0 进入临界区时，它一定执行了 `flag[0] = true` 且 `turn = 1`。
当 P1 进入临界区时，它一定执行了 `flag[1] = true` 且 `turn = 0`。

turn 是共享变量，最后一次写入决定 turn 的值。如果两个进程都进入了临界区，turn 必须是 0 和 1 同时成立，这不可能。

更精确地说：
- 假设 P0 先设置 turn = 1，然后 P1 设置 turn = 0
- 那么 turn = 0
- 此时 P0 检查条件：flag[1] 为 true，turn == 1 为 false → P0 可以进入
- 但 P1 检查条件：flag[0] 为 true，turn == 0 为 true → P1 等待
- 所以 P1 无法进入

- 反之，如果 P1 先设置 turn = 0，然后 P0 设置 turn = 1
- 那么 turn = 1
- P0 等待，P1 进入

**无论如何，只有一个进程能进入临界区。** ✅

#### 证明二：满足空闲让进（Progress）

如果 P0 不想进入临界区（flag[0] = false），但 P1 想进入：
- P1 设置 flag[1] = true，turn = 0
- P1 检查条件：flag[0] 为 false，所以条件不成立 → P1 直接进入临界区
- 即使临界区空闲，P1 也能进入 ✅

#### 证明三：满足有限等待（Bounded Waiting）

假设 P0 在临界区中，P1 想进入：
- P1 设置 flag[1] = true，turn = 0
- P1 检查条件：flag[0] 为 true，turn == 0 为 true → P1 等待

当 P0 退出临界区时：
- P0 设置 flag[0] = false
- 此时 P1 的 while 条件中 flag[0] 为 false → P1 进入临界区
- P1 最多等待 P0 执行一次临界区的时间 ✅

**Peterson 算法的价值：**
- 这是第一个（也是最重要的）纯软件互斥算法
- 证明了互斥问题可以在纯软件层面解决
- 思想被用于许多高级同步原语的设计中
- 考试中常被要求写出完整代码并分析正确性

### 4.2.5 软件互斥方法的局限性

所有纯软件互斥方法共同面临的问题：

1. **忙等待（Busy Waiting）：** 进程在等待进入临界区时不断循环检查条件，浪费 CPU 时间。
2. **适用性有限：** 通常只适用于两个进程，扩展到 N 个进程时复杂度剧增。
3. **依赖内存一致性：** 在多核/多处理器系统中，由于缓存一致性协议，flag 数组的修改可能不会立即被其他处理器看到，导致算法失效。
4. **不能用于操作系统内核：** 实际操作系统中不采用纯软件方法实现互斥。

因此，现代系统和编程语言通常使用**硬件互斥方法**或**高级同步抽象（信号量、管程）**来实现互斥。

## 4.3 硬件互斥方法

硬件互斥方法利用处理器的特殊指令或机制来实现互斥，其核心优势在于**原子性**——硬件保证操作不可中断。

### 4.3.1 Test-and-Set（TSL）指令

**定义：** TSL 是一条原子指令，它执行两个操作：读取指定内存单元的值，然后将该内存单元设置为非零值（通常是 1）。这两个操作作为一个不可分割的整体执行。

**伪代码描述：**

```c
// TSL 指令的原子语义（硬件实现，不可中断）
int TestAndSet(int *lock) {
    int old = *lock;   // 读取锁的当前值
    *lock = 1;         // 将锁设置为 1（锁定状态）
    return old;        // 返回旧值
}
```

> **关键：** 上面的代码是**原子的**。如果两个进程同时执行 TSL，硬件保证它们串行执行。

**使用 TSL 实现互斥：**

```c
// 共享变量
int lock = 0;  // 0 表示未锁定，1 表示已锁定

// 进程 Pi
void Pi() {
    do {
        // 非临界区
        
        // 进入区：原子检查并上锁
        while (TestAndSet(&lock) == 1);  // 如果 lock 原来是 1，循环等待
        
        // 临界区
        
        // 退出区：解锁
        lock = 0;
        
        // 非临界区
    } while (1);
}
```

**工作原理分析：**

1. 初始状态：lock = 0（临界区空闲）
2. 进程 A 执行 TSL：读取 lock (=0)，设置 lock = 1，返回 0
3. while 条件为假（==1 不成立），A 进入临界区
4. 此时若进程 B 执行 TSL：读取 lock (=1)，设置 lock = 1，返回 1
5. while 条件为真，B 循环等待（忙等待）
6. A 退出临界区设置 lock = 0
7. B 的下一次 TSL：读取 lock (=0)，设置 lock = 1，返回 0，B 进入临界区

**TSL 的优点：**
- 实现简单
- 适用于任意数量的进程
- 支持多处理器系统（由硬件总线锁定保证原子性）

**TSL 的缺点：**
- **忙等待：** 等待进程浪费 CPU 时间
- **可能饥饿：** 无法保证哪个等待进程获得锁
- **可能死锁：** 如果低优先级进程持有锁，高优先级进程忙等待，导致优先级反转

### 4.3.2 Compare-and-Swap（CAS）指令

**定义：** CAS 是一条原子指令，它将内存位置的值与预期值比较，如果相等则更新为新值。

**伪代码描述：**

```c
// CAS 指令的原子语义（硬件实现，不可中断）
int CompareAndSwap(int *ptr, int expected, int new_value) {
    int old = *ptr;
    if (old == expected) {
        *ptr = new_value;    // 只有当当前值等于期望值时，才更新
    }
    return old;              // 返回实际旧值
}
```

**使用 CAS 实现互斥：**

```c
// 共享变量
int lock = 0;  // 0 表示未锁定

// 进程 Pi
void Pi() {
    do {
        // 非临界区
        
        // 进入区：只有当 lock == 0 时，才将其设为 1
        while (CompareAndSwap(&lock, 0, 1) != 0);  // 如果返回不是 0，说明 lock 不是空闲
        
        // 临界区
        
        // 退出区：解锁
        lock = 0;
        
        // 非临界区
    } while (1);
}
```

**CAS 与 TSL 的比较：**

| 特性 | TSL | CAS |
|-----|-----|-----|
| 原子操作 | 读 + 写 | 比较 + 条件写 |
| 实现互斥的方式 | 总是设为 1，检查旧值 | 仅在空闲时设为 1 |
| 语义 | 无条件写入 | 条件写入 |
| 在现代 CPU 中的存在 | 较少（x86 的 BTS） | 非常普遍（x86 的 CMPXCHG） |

**CAS 的典型应用场景：**

CAS 不仅是互斥工具，更是**无锁数据结构（Lock-Free Data Structures）**的基础。许多并发算法（如无锁栈、无锁队列）依赖 CAS 实现。

```c
// 无锁计数器（CAS 的典型应用）
int atomic_increment(int *counter) {
    int old;
    do {
        old = *counter;
    } while (CompareAndSwap(counter, old, old + 1) != old);
    return old + 1;
}
```

### 4.3.3 中断禁用（Interrupt Disable / Enable）

**核心思想：** 在进入临界区前关中断，在退出后开中断。关中断期间不会发生进程调度，从而保证临界区不会被打断。

```c
// 进程 Pi
void Pi() {
    do {
        // 非临界区
        
        // 进入区：关中断
        disable_interrupts();
        
        // 临界区（不会被打断）
        
        // 退出区：开中断
        enable_interrupts();
        
        // 非临界区
    } while (1);
}
```

**为什么只在单处理器系统有效？**

- 在**单处理器**系统中，进程调度由时钟中断触发。关中断后，时钟中断被屏蔽，当前进程不会被调度走，因此临界区是安全的。
- 在**多处理器**系统中，一个处理器关中断不影响其他处理器。进程 B 可以在另一个处理器上运行并访问同一临界资源。

**中断禁用的优缺点：**

| 优点 | 缺点 |
|------|------|
| 实现非常简单 | 只适用于单处理器 |
| 无忙等待 | 关中断可能导致关键事件丢失（如时钟） |
| | 长临界区会严重影响系统响应性 |
| | 用户程序通常不被允许关中断（特权指令） |

**实际应用：** 中断禁用主要用于操作系统内核内部，用于保护非常短的临界区（如修改调度器数据结构）。

### 4.3.4 硬件互斥方法的综合评价

| 评价维度 | 评价 |
|---------|------|
| 实现简单性 | ✅ 简单，代码量小 |
| 多进程支持 | ✅ 适用于任意数量进程 |
| 多处理器支持 | ✅ TSL/CAS 支持（中断禁用不支持） |
| 用户程序可用 | ❌ 中断禁用不行（特权指令）；TSL/CAS 可以作为系统调用暴露 |
| 忙等待 | ❌ 都有忙等待问题 |
| 饥饿 | ❌ 可能发生（取决于调度器） |
| 死锁 | ❌ 可能发生（如优先级反转场景） |
| 满足让权等待 | ❌ 不满足 |

## 4.4 信号量机制（Semaphore）

### 4.4.1 信号量的基本概念

**信号量（Semaphore）** 是 Dijkstra 在 1965 年提出的同步机制，是现代操作系统中最核心的同步抽象。

#### 信号量的数据结构

```c
typedef struct {
    int value;               // 信号量的值
    struct process *queue;   // 等待队列（指向等待在该信号量上的进程）
} semaphore;
```

#### 信号量的物理含义

信号量的值可以理解为资源的数量：

| value 的值 | 物理含义 |
|-----------|---------|
| value > 0 | 有 value 个资源可用 |
| value = 0 | 没有可用资源，也没有进程等待 |
| value < 0 | 没有可用资源，有 |value| 个进程在等待 |

**示例理解：**
- S.value = 3：有 3 个可用资源（如 3 台打印机）
- S.value = 0：无可用资源
- S.value = -2：没有可用资源，且有 2 个进程在等待

#### 两种信号量类型

**二进制信号量（Binary Semaphore）：** value 只能取 0 或 1。
- 用于互斥控制（类似锁）
- 初始值通常为 1

**计数信号量（Counting Semaphore）：** value 可以取任意非负整数值。
- 用于资源计数
- 初始值为可用资源的数量

### 4.4.2 P 操作（wait / down / procure / passeren）

**定义：** P 操作从信号量上请求资源。如果资源可用则分配，否则进程阻塞。

P 是荷兰语 "Proberen"（测试）的缩写。在 POSIX 中称为 `sem_wait`。

**伪代码：**

```c
void P(semaphore *S) {
    S->value--;                 // 请求一个资源
    if (S->value < 0) {
        // 没有可用资源，将当前进程加入等待队列并阻塞
        add_to_queue(S->queue, current_process);
        block(current_process); // 阻塞当前进程（主动放弃 CPU）
    }
}
```

**执行过程分析：**

| 调用前 S.value | S.value-- 后 | 结果 |
|---------------|-------------|------|
| 1 | 0 | 资源可用，进程继续执行 |
| 0 | -1 | 无资源，进程阻塞 |

### 4.4.3 V 操作（signal / up / release / verhogen）

**定义：** V 操作释放一个资源给信号量。如果有进程在等待，唤醒其中一个。

V 是荷兰语 "Verhogen"（增加）的缩写。在 POSIX 中称为 `sem_post`。

**伪代码：**

```c
void V(semaphore *S) {
    S->value++;                 // 释放一个资源
    if (S->value <= 0) {
        // 有进程在等待，从队列中取出一个并唤醒
        process *p = remove_from_queue(S->queue);
        wakeup(p);              // 唤醒该进程（放入就绪队列）
    }
}
```

**执行过程分析：**

| 调用前 S.value | S.value++ 后 | 结果 |
|---------------|-------------|------|
| -1 | 0 | 有 1 个进程等待，唤醒它 |
| 0 | 1 | 无进程等待，资源数增加 |
| 1 | 2 | 无进程等待，资源数增加 |

### 4.4.4 两种实现方式对比

#### 方式一：忙等待实现（Spinlock 风格）

```c
// 注意：这种实现必须依赖 TSL/CAS 保护对 value 的访问
void P(semaphore *S) {
    while (TestAndSet(&S->lock) == 1);  // 用 TSL 保护 value 的修改
    S->value--;
    S->lock = 0;
    if (S->value < 0) {
        // 忙等待：不断循环
        while (S->value < 0) {
            // 释放锁让其他进程操作
            S->lock = 0;
            // 等待... 然后重新尝试
            while (TestAndSet(&S->lock) == 1);
        }
    } else {
        S->lock = 0;
    }
}
```

**缺点：** CPU 时间浪费在循环等待上。

#### 方式二：阻塞实现（Block/Wakeup 风格）

```c
void P(semaphore *S) {
    disable_interrupts();       // 关中断保护对信号量数据结构的访问
    S->value--;
    if (S->value < 0) {
        // 将当前进程加入等待队列
        add_to_queue(S->queue, current_process);
        block(current_process); // 阻塞，进程让出 CPU
        // 被唤醒后继续
    }
    enable_interrupts();
}

void V(semaphore *S) {
    disable_interrupts();
    S->value++;
    if (S->value <= 0) {
        // 有进程在等待
        process *p = remove_from_queue(S->queue);
        wakeup(p);              // 将 p 放入就绪队列
    }
    enable_interrupts();
}
```

**优点：** 无忙等待，符合"让权等待"原则。
**问题：** P/V 操作本身的临界区（修改 value 和操作队列）需要保护。

#### 临界区保护问题

P 操作中修改 value 和操作队列入队列，V 操作中修改 value 和操作队列出队列——这些操作本身也是临界区。如何保护？

**保护 P/V 操作内部临界区的三种方法：**

1. **关中断（单处理器有效）：** 在单处理器系统中，P 和 V 操作内部关中断。
2. **使用硬件指令（多处理器有效）：** 用 TSL 或 CAS 保护 value 的修改，但等待队列操作仍可能需要额外保护。
3. **使用自旋锁（多处理器有效）：** 用自旋锁保护信号量数据结构。

> **考试提示：** 信号量的实现本身就是一个小型的"临界区问题"。value 和 queue 是临界资源，P/V 操作中访问它们的代码段是临界区，需要用更低级的同步机制（硬件指令或关中断）来保护。

### 4.4.5 用信号量实现互斥

这是信号量最基本的应用。将一个信号量初始化为 1，用它包裹临界区。

```c
// 全局信号量
semaphore mutex = 1;  // 初始值为 1，表示互斥锁

// 进程 Pi
void Pi() {
    do {
        // 非临界区
        
        P(&mutex);      // 进入区：请求进入临界区
        
        // 临界区（互斥访问）
        
        V(&mutex);      // 退出区：释放临界区
        
        // 非临界区
    } while (1);
}
```

**执行流程分析：**

```
初始状态：mutex.value = 1

场景一：无竞争
P0 执行 P(&mutex): mutex.value = 0, 进入临界区
P0 执行 V(&mutex): mutex.value = 1, 退出

场景二：有竞争
P0 执行 P(&mutex): mutex.value = 0, 进入临界区
P1 执行 P(&mutex): mutex.value = -1, P1 被阻塞
P0 执行 V(&mutex): mutex.value = 0, 唤醒 P1
P1 进入临界区
P1 执行 V(&mutex): mutex.value = 1, 退出
```

### 4.4.6 用信号量实现同步（前驱关系）

信号量还可以用于强制操作的执行顺序，这是比互斥更广泛的应用。

**基本模式：** 信号量初始化为 0，前驱操作后执行 V，后继操作前执行 P。

#### 单个前驱关系示例

```c
semaphore sync = 0;  // 初始化为 0

void Process_A() {
    // 操作 A
    printf("A executed");
    V(&sync);         // 通知 B：A 已完成
}

void Process_B() {
    P(&sync);         // 等待 A 完成
    // 操作 B（必须在 A 之后执行）
    printf("B executed");
}
```

**执行流程分析：**

```
场景一：A 先执行
A 执行 printf("A executed")
A 执行 V(&sync): sync.value = 1
B 执行 P(&sync): sync.value = 0, B 继续
B 执行 printf("B executed")

场景二：B 先执行
B 执行 P(&sync): sync.value = -1, B 阻塞
A 执行 printf("A executed")
A 执行 V(&sync): sync.value = 0, 唤醒 B
B 执行 printf("B executed")
```

**可见，无论 A 和 B 的执行顺序如何，B 总是在 A 之后执行。**

#### 复杂前驱关系

考虑以下执行顺序约束：
```
     S1
    /  \
   S2  S3
    \  /
     S4
```
条件：S1 → S2, S1 → S3, S2 → S4, S3 → S4

```c
semaphore s12 = 0, s13 = 0, s24 = 0, s34 = 0;

void do_S1() { /* S1 */ V(&s12); V(&s13); }
void do_S2() { P(&s12); /* S2 */ V(&s24); }
void do_S3() { P(&s13); /* S3 */ V(&s34); }
void do_S4() { P(&s24); P(&s34); /* S4 */ }
```

### 4.4.7 P/V 操作的使用规则

#### 规则一：成对使用

- **互斥场景：** 同一进程内，P 和 V 必须成对出现。忘记 V 会导致死锁。
- **同步场景：** 一个进程中 P，另一个进程中 V，也必须成对。

```c
// ❌ 错误：缺少 V
P(&mutex);
counter++;
// 忘记 V(&mutex); 导致其他进程永远无法进入临界区

// ✅ 正确：成对
P(&mutex);
counter++;
V(&mutex);
```

#### 规则二：互斥时同一进程，同步时不同进程

- **互斥：** P 和 V 在同一个进程中出现，用于保护该进程访问临界资源。
- **同步：** P 在一个进程中，V 在另一个进程中，用于保证执行顺序。

#### 规则三：P 操作的顺序至关重要

在需要多个 P 操作的场景中，P 的顺序决定了是否会发生死锁。

```c
// ❌ 危险写法：P 的顺序可能导致死锁
void Process_A() {
    P(&S1);     // 先请求 S1
    P(&S2);     // 再请求 S2
    // 临界区
    V(&S1);
    V(&S2);
}

void Process_B() {
    P(&S2);     // 先请求 S2
    P(&S1);     // 再请求 S1
    // 临界区
    V(&S2);
    V(&S1);
}
```

**如果 A 拿到 S1 后，B 拿到 S2，则 A 等待 S2（B 持有），B 等待 S1（A 持有）——死锁！**

**V 操作的顺序不影响正确性（但通常与 P 相反）。**

#### 规则四：同一个进程内，P/V 操作只能嵌套不能交叉

```c
// ✅ 正确：嵌套
P(&S1);
    P(&S2);
        // 临界区
    V(&S2);
V(&S1);

// ❌ 错误：交叉（可能导致死锁）
P(&S1);
P(&S2);
V(&S1);
V(&S2);
// 这种写法违反嵌套规则，虽然不一定会死锁，但通常表示设计有误
```

> **考试核心：** 信号量是本章最重要的考点。需要熟练掌握：信号量的数据结构、P/V 操作的语义、用信号量实现互斥和同步、P/V 操作的正确使用规则。

## 4.5 管程（Monitor）

### 4.5.1 管程的基本概念

**管程（Monitor）** 是 1973 年由 Brinch Hansen 和 Hoare 提出的一种高级同步结构。它将共享数据、对共享数据的操作以及同步逻辑封装在一起。

**为什么需要管程？**

信号量虽然功能强大，但分散在各处，容易出错：
- 忘记 P 或 V 导致互斥失败或死锁
- P/V 顺序错误导致死锁
- 全局信号量难以管理

管程通过**编译器强制互斥**来解决这些问题。

### 4.5.2 管程的结构

```c
monitor MonitorName {
    // 1. 共享数据结构（临界资源）
    int shared_data;
    
    // 2. 条件变量（条件变量）
    condition cond;
    
    // 3. 初始化代码
    void init() {
        shared_data = 0;
    }
    
    // 4. 操作函数（方法）
    void operation1() {
        // 对共享数据的操作
        // 编译器自动保证互斥
    }
    
    void operation2() {
        // 对共享数据的操作
    }
}
```

**管程的四个组成部分：**

| 组成部分 | 说明 |
|---------|------|
| 共享数据结构 | 管程保护的临界资源，只能通过管程内部函数访问 |
| 条件变量 | 用于实现同步（进程等待特定条件） |
| 初始化代码 | 初始化共享数据结构和条件变量 |
| 操作函数集合 | 对共享数据的所有操作，编译器保证互斥执行 |

### 4.5.3 管程的互斥机制

**核心特性：** 编译器（或运行时）自动保证同一时刻只有一个进程在管程内执行。

```c
// 管程中的函数被调用时，编译器自动插入互斥代码
// 这些对程序员是透明的
void operation1() {
    // 编译器插入：P(&monitor_mutex) —— 自动互斥
    // 实际的函数代码
    // 编译器插入：V(&monitor_mutex) —— 自动释放
}
```

**管程互斥 vs 信号量互斥：**

| 对比项 | 信号量互斥 | 管程互斥 |
|--------|-----------|---------|
| 互斥代码位置 | 分散在各个进程中（P/V） | 集中在管程中，编译器自动插入 |
| 互斥保证方式 | 程序员手动保证 | 编译器自动保证 |
| 错误可能性 | 高（容易遗漏 P/V） | 低（编译器保证） |
| 结构化程度 | 低（信号量是全局的） | 高（数据和操作封装在一起） |

### 4.5.4 条件变量（Condition Variables）

**条件变量**是管程内用于实现同步的机制。当进程在管程内发现无法继续执行时（如缓冲区为空），它在条件变量上等待。

#### wait() 操作

```c
void wait(condition *cond) {
    // 1. 将当前进程加入 cond 的等待队列
    // 2. 释放管程的互斥锁（让其他进程可以进入管程）
    // 3. 阻塞当前进程
}
```

#### signal() 操作

```c
void signal(condition *cond) {
    // 如果 cond 的等待队列不为空：
    //   唤醒等待队列中的一个进程
    // 否则：
    //   什么也不做（区别于信号量的 V 操作——不累积信号）
}
```

**条件变量 vs 信号量的关键区别：**

| 特性 | 信号量 | 条件变量 |
|------|-------|---------|
| 有值 | ✅ 有计数器值 | ❌ 没有值，只有队列 |
| V/signal 时无等待进程 | value 增加（累积） | 信号丢失（不累积） |
| 初始值 | 任意非负整数 | 始终为 0（无条件变量值） |
| 用途 | 互斥 + 同步 | 仅用于同步（管程内） |

### 4.5.5 Mesa 语义 vs Hoare 语义

这是管程实现中的两种不同的条件变量语义。

#### Hoare 语义（信号并等待，Signal-and-Wait）

```c
// Hoare 语义：signal 后，信号者阻塞，等待者立即执行
void signal(condition *cond) {
    if (cond->queue is not empty) {
        // 唤醒一个等待进程
        wakeup(one process from cond->queue);
        // 信号者自己阻塞（让出管程使用权）
        block(current_process);
    }
}
```

**特点：**
- signal 立即将管程控制权交给被唤醒的进程
- 信号者进入阻塞状态
- 被唤醒的进程运行时，条件一定成立（因为信号者交出控制权前保证了条件）
- 实现复杂，需要额外的切换

#### Mesa 语义（信号并继续，Signal-and-Continue）

```c
// Mesa 语义：signal 后，信号者继续执行，等待者仅被移至就绪队列
void signal(condition *cond) {
    if (cond->queue is not empty) {
        // 将等待进程从条件变量队列移至就绪队列
        move_to_ready_queue(one process from cond->queue);
    }
    // 信号者继续在管程中执行
}
```

**特点：**
- signal 只是通知等待进程"条件可能已满足"
- 信号者继续执行，不交出管程控制权
- 被唤醒的进程需要重新竞争管程使用权
- 被唤醒后需要**重新检查条件**（因为从 signal 到被唤醒之间，条件可能再次改变）

#### 两种语义的对比

| 特性 | Hoare 语义 | Mesa 语义 |
|------|-----------|-----------|
| signal 后行为 | 信号者阻塞 | 信号者继续 |
| 被唤醒进程何时执行 | 立即 | 重新竞争管程 |
| 条件保证 | ✅ 条件一定成立 | ❌ 条件可能已变 |
| 实现复杂度 | 高 | 低 |
| 上下文切换次数 | 多（两次额外切换） | 少 |
| 使用方式 | 不需要重新检查 | 需要 while 循环重新检查 |

**实际系统中的使用：**

| 系统/语言 | 使用的语义 |
|-----------|-----------|
| Java (`synchronized` + `wait()/notify()`) | Mesa 语义 |
| POSIX 条件变量 (`pthread_cond_wait/signal`) | Mesa 语义 |
| C# (`lock` + `Monitor.Wait/Pulse`) | Mesa 语义 |
| Python (`threading.Condition`) | Mesa 语义 |

**Mesa 语义下的正确使用模式：**

```c
// Mesa 语义下必须使用 while 循环重新检查条件
void consumer() {
    monitor.lock();
    while (buffer.is_empty()) {    // 不是 if！是 while！
        cond.wait();               // 等待缓冲区不为空
    }
    buffer.remove_item();          // 条件满足，取出数据
    monitor.unlock();
}
```

**如果使用 `if` 而不是 `while`**，当被唤醒的进程获得管程控制权时，缓冲区可能又被其他进程清空了，导致取出空数据（或越界）。

### 4.5.6 管程 vs 信号量对比总结

| 对比维度 | 信号量 | 管程 |
|---------|-------|------|
| 抽象层次 | 低（接近硬件） | 高（接近问题描述） |
| 互斥保证 | 程序员手动 P/V | 编译器自动保证 |
| 同步方式 | P/V 操作 | 条件变量的 wait/signal |
| 数据结构 | 单一计数器 + 队列 | 共享数据 + 条件变量 + 操作函数 |
| 封装性 | 差（全局分散） | 好（数据和操作封装） |
| 错误防护 | 差 | 好 |
| 灵活性 | 高（可以构造各种同步方案） | 受管程结构限制 |
| 适用范围 | 所有操作系统 | 高级语言和并发库 |
| 对程序员要求 | 高（容易出错） | 低（结构清晰） |

## 4.6 自旋锁（Spinlock）

### 4.6.1 自旋锁的定义

**自旋锁（Spinlock）** 是一种忙等待的锁机制。当一个线程尝试获取已被其他线程持有的自旋锁时，它会在循环中不断检查锁是否已释放，而不是阻塞睡眠。

**自旋锁的基本实现（依赖 TSL）：**

```c
// 自旋锁的数据结构
typedef struct {
    int locked;  // 0 = 未锁定, 1 = 已锁定
} spinlock;

// 初始化
void spinlock_init(spinlock *lock) {
    lock->locked = 0;
}

// 加锁（忙等待）
void spinlock_lock(spinlock *lock) {
    while (TestAndSet(&lock->locked) == 1) {
        // 空循环，不断尝试获取锁
        // 在 x86 上通常插入 PAUSE 指令优化
    }
}

// 解锁
void spinlock_unlock(spinlock *lock) {
    lock->locked = 0;
}
```

### 4.6.2 自旋锁的适用场景

**应当使用自旋锁的情况：**

1. **临界区非常短**（通常只有几十条指令）
   - 如果临界区执行时间比一次上下文切换时间还短，阻塞比自旋更浪费
2. **多处理器系统**
   - 一个处理器自旋等待时，另一个处理器可能在临界区内，很快释放锁
   - 单处理器下自旋锁意义不大（自旋时没有其他进程能释放锁，除非中断处理程序）
3. **中断处理程序中**
   - 中断上下文不能睡眠（不能使用互斥锁）
4. **实时系统**
   - 自旋锁不会导致优先级反转（但仍有其他问题）

**不应当使用自旋锁的情况：**

1. **临界区很长**——浪费大量 CPU 时间
2. **单处理器系统**——自旋等待时，持有锁的进程无法运行释放锁
3. **严重争用场景**——大量 CPU 时间浪费在自旋上

### 4.6.3 自旋锁 vs 互斥锁（Mutex）

| 对比维度 | 自旋锁（Spinlock） | 互斥锁（Mutex） |
|---------|------------------|----------------|
| 等待方式 | 忙等待（循环检查） | 阻塞（睡眠，让出 CPU） |
| CPU 消耗 | 高（自旋时占满 CPU） | 低（睡眠时不占用 CPU） |
| 上下文切换 | 无 | 有（阻塞和唤醒各一次） |
| 临界区要求 | 必须非常短 | 可以是任意长度 |
| 单处理器是否可用 | 可用但效果差 | ✅ 推荐使用 |
| 多处理器是否可用 | ✅ 可用 | ✅ 可用 |
| 中断上下文使用 | ✅ 可用 | ❌ 不可用（不能睡眠） |
| 是否满足让权等待 | ❌ 不满足（忙等待） | ✅ 满足（阻塞等待） |

**内核中的选择原则：**
- 临界区短 → 自旋锁
- 临界区长 → 互斥锁
- 不确定 → 互斥锁（更安全）

### 4.6.4 自旋锁的变种

**读写自旋锁（Read-Write Spinlock）：**
- 允许多个读者同时持有读锁
- 写者必须独占锁

**排队自旋锁（Ticket Spinlock）：**
- 保证 FIFO 顺序，避免饥饿
- 每个等待的 CPU 获得一个"票号"，按票号顺序获取锁

## 4.7 屏障同步（Barrier）

### 4.7.1 屏障的定义

**屏障（Barrier）** 是一种同步机制，用于强制多个进程（或线程）在某个汇合点同步。只有当所有参与者都到达屏障后，每个参与者才能继续执行。

**形象理解：** 就像跑步比赛——所有运动员都到达终点后，才能开始下一轮比赛。先到的必须等待后到的。

### 4.7.2 屏障的使用场景

屏障最常见的应用是**并行计算**中的分阶段计算：

```
并行阶段 1:          ┌── 进程1 ──┐
                     ├── 进程2 ──┤
                     ├── 进程3 ──┤
                     └── 进程4 ──┘
                     屏障（全部完成后）
并行阶段 2:          ┌── 进程1 ──┐
                     ├── 进程2 ──┤
                     ├── 进程3 ──┤
                     └── 进程4 ──┘
```

**典型场景：**
- 并行排序（奇偶交换排序的每一轮）
- 并行矩阵乘法（计算分块后的结果汇总）
- 迭代数值计算（每轮迭代结束后同步）

### 4.7.3 屏障的实现

**使用信号量实现屏障：**

```c
// N 个进程的屏障实现
int count = 0;           // 已到达屏障的进程数
semaphore mutex = 1;     // 保护 count
semaphore barrier = 0;   // 屏障信号量，初始为 0

// 进程 Pi (i = 0, 1, ..., N-1)
void worker() {
    while (1) {
        // 阶段 1 的工作
        do_phase1_work();
        
        // 到达屏障
        P(&mutex);
        count++;
        if (count == N) {
            // 我是最后一个到达的
            // 唤醒所有等待进程
            for (int j = 0; j < N; j++) {
                V(&barrier);   // 释放 N 次，让所有进程通过
            }
            count = 0;         // 重置计数器
            V(&mutex);
        } else {
            V(&mutex);
            P(&barrier);       // 等待其他进程
        }
        
        // 阶段 2 的工作
        do_phase2_work();
    }
}
```

**另一种实现（反转屏障，Reversing Barrier）：**

上面的实现中，最后一个进程执行了 N 次 V 操作。但这种方式在大型系统中不够高效。更常用的实现是**反转屏障（Sense-Reversing Barrier）**，通过交替使用两个屏障来避免重置竞争。

### 4.7.4 屏障与信号量的关系

| 对比维度 | 屏障 | 信号量 |
|---------|------|-------|
| 同步类型 | 多对多同步（全部到达才继续） | 一对一通知 |
| 使用模式 | 所有进程等待所有进程 | 一个进程通知另一个进程 |
| 状态含义 | 达到某一执行阶段 | 资源计数 |
| 通用性 | 专用同步机制 | 通用同步原语 |

## 4.8 经典同步问题

本节是所有操作系统教材中的核心内容，也是考试的必考内容。每个问题都需要理解问题描述、掌握完整伪代码、知道关键设计决策的原因。

### 4.8.1 生产者-消费者问题（Producer-Consumer Problem）

#### 问题描述

有一个固定大小的缓冲区（有界缓冲区，Bounded Buffer），容量为 N 个槽位：
- **生产者进程**：生产数据项，放入缓冲区
- **消费者进程**：从缓冲区取出数据项，消费

**约束条件：**
1. 缓冲区满时，生产者不能放入数据（必须等待）
2. 缓冲区空时，消费者不能取出数据（必须等待）
3. 生产者和消费者对缓冲区的访问必须互斥

#### 完整解决方案

```c
// 有界缓冲区（Bounded Buffer）
#define N 100        // 缓冲区容量

// 共享数据结构
int buffer[N];       // 缓冲区数组
int in = 0;          // 下一个放入位置
int out = 0;         // 下一个取出位置

// 信号量声明
semaphore mutex = 1; // 缓冲区互斥信号量
semaphore empty = N; // 空槽位数量（可用空间）
semaphore full = 0;  // 满槽位数量（可用数据）
```

**生产者进程：**

```c
void producer() {
    int item;
    while (1) {
        // 生产数据项
        item = produce_item();
        
        // 1. 请求空槽位（同步）
        P(&empty);
        
        // 2. 请求互斥访问缓冲区（互斥）
        P(&mutex);
        
        // 3. 临界区：放入缓冲区
        buffer[in] = item;
        in = (in + 1) % N;
        
        // 4. 释放互斥锁
        V(&mutex);
        
        // 5. 通知消费者有数据可用（同步）
        V(&full);
    }
}
```

**消费者进程：**

```c
void consumer() {
    int item;
    while (1) {
        // 1. 请求满槽位（同步：有数据可取）
        P(&full);
        
        // 2. 请求互斥访问缓冲区（互斥）
        P(&mutex);
        
        // 3. 临界区：取出数据
        item = buffer[out];
        out = (out + 1) % N;
        
        // 4. 释放互斥锁
        V(&mutex);
        
        // 5. 通知生产者有空槽位（同步）
        V(&empty);
        
        // 消费数据项
        consume_item(item);
    }
}
```

#### 为什么 P 操作的顺序至关重要？

**为什么在生产者中先 P(&empty) 再 P(&mutex)？**

如果交换为 P(&mutex) 先于 P(&empty)：

```c
// ❌ 错误顺序：可能导致死锁
void producer() {
    P(&mutex);      // 先获取互斥锁
    P(&empty);      // 再检查空槽位
    // ... 放入数据
    V(&mutex);
    V(&full);
}
```

考虑以下场景：
1. 缓冲区已满（empty = 0）
2. 生产者 P(&mutex) 成功：mutex = 0
3. 生产者 P(&empty)：empty 已为 0，P 操作后 empty = -1，生产者阻塞
4. 消费者：P(&full) 成功，但 P(&mutex) 被阻塞（mutex = 0）
5. **死锁！** 生产者等待消费者消费（释放 empty），消费者等待生产者释放 mutex

**正确顺序的保证：** 先检查资源是否可用（同步 P），再获取互斥锁。这样可以确保不会持有互斥锁而阻塞在资源等待上。

**V 操作的顺序不影响正确性：** V(&mutex) 和 V(&full) 的顺序可以互换。

#### 多生产者和多消费者

```c
// 当有多个生产者和多个消费者时，代码完全相同！
// 因为 mutex 保证了缓冲区的互斥访问
// empty 和 full 保证了同步条件

void producer_i() {
    int item;
    while (1) {
        item = produce_item();
        P(&empty);
        P(&mutex);
        // 临界区
        buffer[in] = item;
        in = (in + 1) % N;
        V(&mutex);
        V(&full);
    }
}

void consumer_j() {
    int item;
    while (1) {
        P(&full);
        P(&mutex);
        // 临界区
        item = buffer[out];
        out = (out + 1) % N;
        V(&mutex);
        V(&empty);
        consume_item(item);
    }
}
```

**重要：** 这里的 mutex 通过信号量实现（而不是自旋锁），意味着多生产者多消费者是安全的，不会因为 P 操作阻塞而进入死锁（因为正确顺序保证不会持有 mutex 去等同步信号量）。

### 4.8.2 读者-写者问题（Readers-Writers Problem）

#### 问题描述

一个数据对象（如文件、数据库记录）被多个进程共享：
- **读者（Reader）：** 只读取数据，不修改数据
- **写者（Writer）：** 读取并修改数据

**约束条件：**
1. 多个读者可以同时读取（读-读不互斥）
2. 只允许一个写者写入（写-写互斥）
3. 读者和写者互斥（读-写互斥）

#### 第一类读者-写者问题（读者优先，Readers-Priority）

**策略：** 只要有一个读者在读，就允许新的读者进入。写者可能被无限期推迟（写者饥饿）。

```c
// 共享变量
int readcount = 0;       // 当前正在读的读者数
semaphore rmutex = 1;    // 保护 readcount
semaphore wmutex = 1;    // 控制对数据对象的写访问
```

**读者进程：**

```c
void reader() {
    while (1) {
        // 1. 更新 readcount
        P(&rmutex);
        if (readcount == 0) {
            // 第一个读者：阻止写者进入
            P(&wmutex);
        }
        readcount++;
        V(&rmutex);
        
        // 2. 读取数据（多个读者可以同时读）
        read_data();
        
        // 3. 更新 readcount
        P(&rmutex);
        readcount--;
        if (readcount == 0) {
            // 最后一个读者：允许写者进入
            V(&wmutex);
        }
        V(&rmutex);
    }
}
```

**写者进程：**

```c
void writer() {
    while (1) {
        // 请求写访问
        P(&wmutex);
        
        // 写入数据
        write_data();
        
        // 释放写访问
        V(&wmutex);
    }
}
```

**执行流程分析：**

```
场景：R1 正在读，R2 请求进入，W1 请求进入
时间 →
R1: P(rmutex) → readcount=0 → P(wmutex) → readcount=1 → V(rmutex) → 正在读
R2: P(rmutex) → readcount=1（不为 0，不 P(wmutex)) → readcount=2 → V(rmutex) → 正在读
W1: P(wmutex) → 阻塞（wmutex 被 R1 持有）
R1: 读完 → P(rmutex) → readcount=1 → V(rmutex)
R2: 读完 → P(rmutex) → readcount=0 → V(wmutex) → V(rmutex)
W1: P(wmutex) 成功 → 写入数据 → V(wmutex)
```

**写者饥饿问题：**

如果读者不断到来，readcount 永远不为 0，写者永远无法获取 wmutex：

```
R1 进入读 → R2 进入读 → R1 退出 → R2 仍在读 → R3 进入读 → ...
                                      readcount 从未降到 0
                                      W1 一直在等待 wmutex（饿死）
```

**这是读者优先方案的天然缺陷：** 在高负载下，写者可能永远得不到执行。

#### 第二类读者-写者问题（写者优先，Writers-Priority）

**策略：** 一旦有写者请求写入，阻止新的读者进入。已有读者不受影响，但新读者必须在写者完成后才能开始。

```c
// 共享变量
int readcount = 0;        // 正在读的读者数
int writecount = 0;       // 等待或正在写的写者数
semaphore rmutex = 1;     // 保护 readcount
semaphore wmutex = 1;     // 保护 writecount
semaphore R = 1;          // 读者准入控制（写者优先的关键）
semaphore W = 1;          // 写者互斥
```

**读者进程：**

```c
void reader() {
    while (1) {
        // 1. 检查是否允许读者进入
        P(&R);              // 如果有写者在等待，此操作可能阻塞
        P(&rmutex);
        if (readcount == 0) {
            P(&W);          // 第一个读者阻止写者写入
        }
        readcount++;
        V(&rmutex);
        V(&R);              // 释放读者准入控制
        
        // 2. 读取数据
        read_data();
        
        // 3. 更新计数器
        P(&rmutex);
        readcount--;
        if (readcount == 0) {
            V(&W);          // 最后一个读者释放写者
        }
        V(&rmutex);
    }
}
```

**写者进程：**

```c
void writer() {
    while (1) {
        // 1. 声明写者请求
        P(&wmutex);
        writecount++;
        if (writecount == 1) {
            P(&R);          // 第一个写者阻止新读者进入
        }
        V(&wmutex);
        
        // 2. 请求写入
        P(&W);              // 与读者和其他写者互斥
        write_data();
        V(&W);
        
        // 3. 释放写者请求
        P(&wmutex);
        writecount--;
        if (writecount == 0) {
            V(&R);          // 最后一个写者释放读者准入
        }
        V(&wmutex);
    }
}
```

**写者优先的工作原理：**

关键在信号量 R。当一个写者请求进入时：
1. writecount++，如果这是第一个写者，执行 P(&R)
2. P(&R) 使信号量 R 变为 0
3. 任何试图进入的读者在 P(&R) 时被阻塞（等待 R 变为 1）
4. 只有当所有写者完成（writecount 变为 0），才会执行 V(&R)
5. V(&R) 唤醒在 R 上等待的读者

**这样，一旦有写者请求写入，所有后续读者必须等待直到写者完成，实现了写者优先。**

#### 两种方案的对比

| 特性 | 读者优先 | 写者优先 |
|------|---------|---------|
| 读者并发 | ✅ 高 | ✅ 较高（仍允许并发读） |
| 写者饥饿 | ❌ 可能发生 | ✅ 不会 |
| 读者饥饿 | ✅ 不会 | ❌ 可能发生（写者持续请求时） |
| 公平性 | 偏向读者 | 偏向写者 |
| 适用场景 | 读多写少，写可接受延迟 | 写需要及时响应 |

> **考试提示：** 读者-写者问题需要熟练掌握两种方案的代码。重点理解：读者优先中存不存在写者优先的特性，以及写者优先中 R 信号量的作用机制。

### 4.8.3 哲学家就餐问题（Dining Philosophers Problem）

#### 问题描述

5 个哲学家围坐在一张圆桌旁，每两个哲学家之间有一只筷子（共 5 只筷子）。哲学家的生活循环：
1. **思考**（不需要任何资源）
2. **饥饿**（想要吃饭）
3. **吃饭**（需要两只筷子：左边的和右边的）

**约束条件：**
- 哲学家只能拿自己左右两边的筷子
- 筷子是互斥资源（一次只能被一个哲学家使用）
- 拿筷子时不能等待——如果拿不到两只筷子，就一只都不拿（或者放下已经拿到的）

#### 基础实现（可能死锁的方案）

```c
#define N 5                // 哲学家数量
semaphore chopstick[5];    // 每只筷子是一个信号量，初始为 1

void philosopher(int i) {  // i = 0, 1, 2, 3, 4
    while (1) {
        // 思考
        think();
        
        // 饥饿：拿起左筷子
        P(&chopstick[i]);
        
        // 拿起右筷子
        P(&chopstick[(i + 1) % N]);
        
        // 吃饭
        eat();
        
        // 放下左筷子
        V(&chopstick[i]);
        
        // 放下右筷子
        V(&chopstick[(i + 1) % N]);
    }
}
```

**为什么会导致死锁？**

假设所有 5 个哲学家同时饥饿，每个人都拿起了左筷子：
- 哲学家 0：拿了 chopstick[0]
- 哲学家 1：拿了 chopstick[1]
- 哲学家 2：拿了 chopstick[2]
- 哲学家 3：拿了 chopstick[3]
- 哲学家 4：拿了 chopstick[4]

此时，所有 chopstick 信号量的值都为 0。每个哲学家在 P(&chopstick[(i+1)%5]) 时都被阻塞，等待右边的筷子被放下。但右边的筷子被右边的哲学家拿着，右边的哲学家也在等右边的筷子……形成了**循环等待**，满足死锁的四个必要条件，死锁发生。

#### 解决方案一：限制最多 4 个哲学家同时就餐

```c
#define N 5
semaphore chopstick[5];    // 每只筷子，初始为 1
semaphore room = 4;        // 房间容量，最多允许 4 个哲学家同时就餐

void philosopher(int i) {
    while (1) {
        think();
        
        P(&room);           // 请求进入餐厅（最多 4 人）
        
        P(&chopstick[i]);   // 拿左筷子
        P(&chopstick[(i + 1) % N]);  // 拿右筷子
        
        eat();
        
        V(&chopstick[i]);   // 放左筷子
        V(&chopstick[(i + 1) % N]);  // 放右筷子
        
        V(&room);           // 离开餐厅
    }
}
```

**为什么有效？**

最多允许 4 个哲学家进入餐厅，意味着最多只有 4 个人在竞争 5 只筷子。**至少有一个哲学家能拿到两只筷子**（因为 4 个人最多占用 4 只筷子，剩下 1 只筷子搭配某一个哲学家的另外一只筷子，可以组成一对）。

更严谨地说：当 4 个哲学家拿起左筷子后，第 5 只左筷子空闲，拿着这只筷子的哲学家可以再拿到右筷子（右筷子就是第 5 只空闲筷子）。

#### 解决方案二：奇偶不对称拿筷子

```c
#define N 5
semaphore chopstick[5];    // 每只筷子，初始为 1

void philosopher(int i) {
    while (1) {
        think();
        
        if (i % 2 == 0) {
            // 偶数哲学家：先拿左边，再拿右边
            P(&chopstick[i]);
            P(&chopstick[(i + 1) % N]);
        } else {
            // 奇数哲学家：先拿右边，再拿左边
            P(&chopstick[(i + 1) % N]);
            P(&chopstick[i]);
        }
        
        eat();
        
        V(&chopstick[i]);
        V(&chopstick[(i + 1) % N]);
    }
}
```

**为什么有效？**

偶数哲学家先左后右，奇数哲学家先右后左。这打破了循环等待——因为两个相邻的哲学家不会同时持有对方需要的筷子然后互相等待。

考虑哲学家 0（偶）和哲学家 1（奇）：
- 0 先拿 chopstick[0]（左）
- 1 先拿 chopstick[1]（在不对称方案中是 1 的右筷子）
- 0 需要 chopstick[1]（右），1 需要 chopstick[0]（左）
- 但 chopstick[1] 已经被 1 拿了，chopstick[0] 已经被 0 拿了
- 等等，这看起来还是死锁？

实际上，我们需要更仔细地分析。不对称方案的核心在于：
- 哲学家 0（偶数）：拿 0，再拿 1
- 哲学家 1（奇数）：拿 1，再拿 0
- 哲学家 2（偶数）：拿 2，再拿 3
- 哲学家 3（奇数）：拿 3，再拿 4
- 哲学家 4（偶数）：拿 4，再拿 0

注意哲学家 0 和哲学家 4：都是偶数，都先左后右。但 4 的左是 chopstick[4]，右是 chopstick[0]。这可能造成 0（持有 0，等待 1）和 4（持有 4，等待 0）之间的循环。但此时 0 的左边是哲学家 4，右边是哲学家 1。

实际上不对称方案的正确性有争议——它打破了圆环上的对称性，但在某些情况下仍可能死锁。更准确的版本通常认为：**只要让一个哲学家以不同的顺序拿筷子，就能打破循环等待条件**。如果 4 个偶数的哲学家都先左后右，1 个奇数的哲学家先右后左，则不可能形成完整的循环等待链。

**实际效果：** 这种方案在实践中有效是因为改变了资源请求顺序，破坏了死锁的"循环等待"条件。

#### 解决方案三：使用管程（或服务员/仲裁者）

```c
monitor DiningPhilosophers {
    enum { THINKING, HUNGRY, EATING } state[5];
    condition self[5];       // 每个哲学家有自己的条件变量
    
    void init() {
        for (int i = 0; i < 5; i++)
            state[i] = THINKING;
    }
    
    void pickup(int i) {
        state[i] = HUNGRY;
        test(i);             // 尝试让哲学家 i 吃饭
        if (state[i] != EATING)
            self[i].wait();  // 如果不能吃，等待
    }
    
    void putdown(int i) {
        state[i] = THINKING;
        test((i + 4) % 5);   // 检查左边邻居能否吃饭
        test((i + 1) % 5);   // 检查右边邻居能否吃饭
    }
    
    void test(int i) {
        // 如果 i 饥饿，且左右邻居都不在吃，允许 i 吃饭
        if (state[i] == HUNGRY &&
            state[(i + 4) % 5] != EATING &&
            state[(i + 1) % 5] != EATING) {
            state[i] = EATING;
            self[i].signal();  // 通知 i 可以吃（Mesa 语义下只是唤醒）
        }
    }
}
```

**哲学家进程：**

```c
void philosopher(int i) {
    while (1) {
        think();
        DiningPhilosophers.pickup(i);  // 通过管程申请筷子
        eat();
        DiningPhilosophers.putdown(i); // 通过管程释放筷子
    }
}
```

**管程方案的优势：**
- 不会死锁（管程保证互斥，test 函数决定何时允许吃饭）
- 不会饥饿（在正确的实现下，每个哲学家最终都能吃到）
- 结构清晰

#### 三种解决方案的对比

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 限 4 人 | 至少一人有两根筷子 | 简单易懂 | 降低并发度 |
| 不对称拿筷 | 打破循环等待 | 不限制并发 | 编码稍复杂 |
| 管程方案 | 集中式资源分配 | 最安全、最通用 | 需要管程支持 |

## 4.9 进程通信（Inter-Process Communication, IPC）

### 4.9.1 IPC 概述

**进程通信（IPC）** 是指进程之间交换数据和信息的机制。进程之间是相互独立的（每个进程有独立的地址空间），需要通过操作系统提供的机制进行通信。

**IPC 的三种主要模式：**

| 通信模式 | 特点 | 例子 |
|---------|------|------|
| 共享内存（Shared Memory） | 进程共享一块内存区域 | POSIX 共享内存 |
| 消息传递（Message Passing） | 通过内核交换消息 | 消息队列、套接字 |
| 管道（Pipe） | 字节流通信 | 匿名管道、命名管道 |

### 4.9.2 共享内存（Shared Memory）

**基本原理：** 两个或多个进程共享一块物理内存区域，一个进程写入的数据可以被其他进程立即看到。

**两种层次：**

#### 基于共享数据结构的共享内存（低级）

```c
// 共享一个数据结构（如生产者-消费者中的缓冲区）
// 所有进程通过共享数据结构（如环形缓冲区）进行通信
// 需要同步机制（信号量）保证互斥

// 共享数据
struct shared_buffer {
    int data[100];
    int in, out;
};

// 进程 A 向缓冲区写入
struct shared_buffer *buf = get_shared_memory(...);
buf->data[buf->in] = item;
buf->in = (buf->in + 1) % 100;

// 进程 B 从缓冲区读取
struct shared_buffer *buf = get_shared_memory(...);
item = buf->data[buf->out];
buf->out = (buf->out + 1) % 100;
```

**特点：** 共享数据的格式是固定的（由数据结构决定），灵活性低。

#### 基于共享内存区域的共享内存（高级）

操作系统提供系统调用，允许一个进程创建一块共享内存区域，其他进程可以将其附加到自己的地址空间。

**POSIX 共享内存示例：**

```c
// 进程 A（创建并写入）
int fd = shm_open("/myshm", O_CREAT | O_RDWR, 0666);
ftruncate(fd, 4096);
void *ptr = mmap(NULL, 4096, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
strcpy(ptr, "Hello from process A");

// 进程 B（读取）
int fd = shm_open("/myshm", O_RDWR, 0666);
void *ptr = mmap(NULL, 4096, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
printf("%s", (char *)ptr);
```

**特点：** 灵活，可以共享任意格式和数量的数据。

**共享内存的优缺点：**

| 优点 | 缺点 |
|------|------|
| 数据传输速度快（不需要系统调用） | 需要同步机制（信号量等） |
| 数据量大时效率高 | 存在竞态条件风险 |
| 用户体验好（像访问本地内存） | 编程复杂（需处理同步） |

### 4.9.3 消息传递（Message Passing）

**基本原理：** 进程通过发送和接收消息进行通信，消息通过操作系统内核传递。

**两种通信方式：**

#### 直接通信（Direct Communication）

发送方和接收方**显式命名对方**：

```c
// 发送消息给进程 Q
send(Q, message);

// 从进程 P 接收消息
receive(P, message);

// 或者不对称版本：
send(P, message);    // P 发送消息
receive(id, message); // id 保存发送方进程的 ID
```

**特点：**
- 通信链路自动建立
- 一条链路只连接两个进程
- 显式命名对方

#### 间接通信（Indirect Communication）

通过**邮箱（Mailbox）/端口（Port）**进行通信：

```c
// 发送消息到邮箱 M
send(M, message);

// 从邮箱 M 接收消息
receive(M, message);
```

**特点：**
- 通过共享邮箱通信
- 一个邮箱可以被多个进程共享
- 一个进程可以有多个邮箱
- 通信链路在进程中共享邮箱时自动建立

**邮箱的同步方式：**

- **阻塞发送/阻塞接收：** 发送方等待接收方接收，接收方等待消息到达
- **非阻塞发送/阻塞接收：** 发送方发送后继续执行，接收方等待消息到达
- **非阻塞发送/非阻塞接收：** 双方都不等待

#### 消息传递的缓冲

| 缓冲策略 | 容量 | 发送方行为 | 接收方行为 |
|---------|------|-----------|-----------|
| 零容量（无缓冲） | 0 | 阻塞直到接收方接收 | 阻塞直到有消息（同步） |
| 有界容量 | N | 队列满时阻塞 | 队列空时阻塞 |
| 无界容量 | ∞ | 永不阻塞 | 队列空时阻塞 |

### 4.9.4 管道通信（Pipe）

#### 匿名管道（Anonymous Pipe）

**特点：**
- 单向（半双工）通信
- 只能用于父子进程或有亲缘关系的进程之间
- 数据以字节流形式读写
- 管道的生命周期与进程绑定——所有进程关闭后，管道自动销毁
- 通常通过 `pipe()` 系统调用创建

```c
// 匿名管道创建（UNIX/POSIX）
int fd[2];
pipe(fd);    // fd[0] = 读端, fd[1] = 写端

pid_t pid = fork();
if (pid == 0) {
    // 子进程：从管道读取
    close(fd[1]);  // 关闭写端
    char buf[100];
    read(fd[0], buf, sizeof(buf));
    close(fd[0]);
} else {
    // 父进程：向管道写入
    close(fd[0]);  // 关闭读端
    write(fd[1], "Hello child!", 12);
    close(fd[1]);
}
```

**管道在 Shell 中的应用：**

```bash
# ls 的标准输出连接到 wc 的标准输入
ls | wc -l
# 等价于：
# pipe(fd)
# fork: 子进程 exec wc -l (标准输入从 fd[0] 读)
# fork: 子进程 exec ls (标准输出写到 fd[1])
```

#### 命名管道（Named Pipe / FIFO）

**特点：**
- 支持双向通信
- 可用于任意进程之间（不需要亲缘关系）
- 以文件系统中的一个特殊文件存在
- 需要显式删除（`unlink` 或 `rm`）

```c
// 创建命名管道（FIFO）
mkfifo("/tmp/myfifo", 0666);

// 进程 A（写入）
int fd = open("/tmp/myfifo", O_WRONLY);
write(fd, "Hello", 5);
close(fd);

// 进程 B（读取）
int fd = open("/tmp/myfifo", O_RDONLY);
read(fd, buf, 5);
close(fd);

// 删除命名管道
unlink("/tmp/myfifo");
```

#### 管道的实现原理

管道在内核中实现为一个**环形缓冲区（circular buffer）**：

```
管道（内核缓冲区，通常 4KB 或 64KB）
┌───┬───┬───┬───┬───┬───┬───┬───┐
│   │ D │ A │ T │ A │   │   │   │
└───┴───┴───┴───┴───┴───┴───┴───┘
  ↑                       ↑
读指针                  写指针
```

- 写进程向管道写入数据（追加到写指针位置）
- 读进程从管道读取数据（从读指针位置）
- 缓冲区满时写进程阻塞
- 缓冲区空时读进程阻塞
- 所有读端关闭后，写进程收到 SIGPIPE 信号
- 所有写端关闭后，读进程 read 返回 0（EOF）

### 4.9.5 套接字通信（Socket）

**基本原理：** 套接字是通信端点的抽象，通过网络在不同主机的进程之间进行通信（也可以在同一主机上）。

#### 两种主要类型

**流套接字（Stream Socket, SOCK_STREAM）：**
- 基于 TCP 协议
- 面向连接，可靠的字节流
- 保证按序、不丢失、不重复

**数据报套接字（Datagram Socket, SOCK_DGRAM）：**
- 基于 UDP 协议
- 无连接，不可靠
- 消息可能丢失、乱序

#### 客户端-服务器模型

```c
// 服务端
int server_fd = socket(AF_INET, SOCK_STREAM, 0);
bind(server_fd, &address, sizeof(address));
listen(server_fd, 5);              // 监听连接请求
int client_fd = accept(server_fd, NULL, NULL);  // 接受连接
read(client_fd, buffer, 1024);     // 读取客户端发送的数据
write(client_fd, "Hello client", 12);
close(client_fd);

// 客户端
int sock = socket(AF_INET, SOCK_STREAM, 0);
connect(sock, &address, sizeof(address));
write(sock, "Hello server", 12);
read(sock, buffer, 1024);
close(sock);
```

### 4.9.6 远程过程调用（RPC）

**基本原理：** 使程序可以调用运行在另一台机器上的函数，就像调用本地函数一样。

**RPC 的工作流程：**

```
客户端                   服务端
┌──────────┐           ┌──────────┐
│ 应用程序   │           │ 应用程序   │
│   ↓       │           │   ↑       │
│ 客户端桩  │  ═══网络═══> │ 服务端桩  │
│ (stub)   │   <═══    │ (stub)   │
│   ↓       │           │   ↑       │
│ 运行时库   │           │ 运行时库   │
└──────────┘           └──────────┘
```

**关键概念：**

| 概念 | 说明 |
|------|------|
| **客户端桩（Client Stub）** | 在客户端伪装成远程函数，负责打包参数（编组）并发送网络请求 |
| **服务端桩（Server Stub）** | 在服务端接收请求，解包参数（解组），调用实际函数，打包返回值 |
| **编组（Marshalling）** | 将函数参数转换为可在网络传输的格式 |
| **解组（Unmarshalling）** | 将网络数据恢复为函数参数 |
| **绑定（Binding）** | 客户端找到并提供 RPC 服务的过程 |

**RPC 的三种语义：**

| 语义 | 说明 | 特点 |
|------|------|------|
| **至多一次（At-Most-Once）** | RPC 执行 0 次或 1 次 | 不重试，适合非幂等操作 |
| **恰好一次（Exactly-Once）** | RPC 确保执行 1 次 | 需要去重，实现复杂 |
| **至少一次（At-Least-Once）** | RPC 至少执行 1 次 | 不断重试直到收到回复，适合幂等操作 |

## 4.10 死锁（Deadlock）

### 4.10.1 死锁的定义

**死锁（Deadlock）** 是指多个进程因竞争资源而处于一种永久阻塞的状态——每个进程都在等待其他进程释放资源，但没有任何一个进程能够继续执行。

**银行家算法故事中的死锁类比：**
> 两个人在银行保险库门口相遇，各有半个钥匙。A 拿着自己那半把钥匙，等待 B 的那半把；B 拿着自己那半把钥匙，等待 A 的那半把。两人都在等待对方把钥匙给自己——谁也不肯先放弃自己手里已有的半把。这就是死锁。

### 4.10.2 死锁的两种常见原因

#### 原因一：竞争不可抢占资源

多个进程竞争有限的非抢占式资源（如打印机、磁带机、文件锁）：

```
进程 A 持有打印机，请求扫描仪
进程 B 持有扫描仪，请求打印机
```

**非抢占式资源**是指不能被操作系统强制剥夺的资源，必须由持有者主动释放。

#### 原因二：进程推进顺序不当

即使资源充足，如果进程请求资源的顺序不当，也可能导致死锁。

```
进程 A: P(S1) → P(S2) → ... → V(S1) → V(S2)
进程 B: P(S2) → P(S1) → ... → V(S2) → V(S1)

如果 A 拿到 S1 的瞬间 B 拿到 S2，则死锁。
但如果 B 也先请求 S1 再请求 S2，不会死锁。
```

### 4.10.3 死锁的四个必要条件

**所有四个条件必须同时成立，才会发生死锁。**

| 条件 | 定义 | 通俗理解 |
|------|------|---------|
| **互斥（Mutual Exclusion）** | 资源一次只能分配给一个进程 | 椅子一次只能坐一个人 |
| **请求并保持（Hold and Wait）** | 进程持有资源的同时请求其他资源 | 手里拿着一张票，还要抢另一张 |
| **不可抢占（No Preemption）** | 已分配给进程的资源不能被强制剥夺 | 别人手里的东西不能硬抢 |
| **循环等待（Circular Wait）** | 存在一个进程-资源的循环等待链 | A 等 B 的资源，B 等 C 的，C 等 A 的 |

> **考试提示：** 四个条件是**必要不充分**条件。循环等待的存在不一定意味着死锁（见资源分配图分析）。但破坏任意一个条件，就可以防止死锁。

### 4.10.4 资源分配图（Resource Allocation Graph, RAG）

#### 形式化定义

资源分配图是一个有向图 G = (V, E)，其中：

**节点 V 包含两类：**
- **进程节点 P**（用圆圈表示）：P = {P₁, P₂, ..., Pₙ}
- **资源节点 R**（用矩形表示）：R = {R₁, R₂, ..., Rₘ}
  - 每个资源可能有多个实例（矩形内的小圆点表示实例数量）

**边 E 包含两类：**
- **请求边（Request Edge）：** Pᵢ → Rⱼ（进程 Pᵢ 请求资源 Rⱼ）
- **分配边（Assignment Edge）：** Rⱼ → Pᵢ（资源 Rⱼ 已分配给进程 Pᵢ）

**图示示例：**

```
     P₁ ──→ R₁ (请求)      R₁ ──→ P₁ (分配)
     (圆圈)     (矩形)      (矩形)     (圆圈)
                ● ●                     ● ●
              (实例)                   (实例)
```

#### 资源分配图分析

**规则一：没有环 → 没有死锁**

如果资源分配图中不包含任何环，系统中一定没有死锁。

**规则二：有环 + 每种资源只有一个实例 → 死锁**

如果图中包含环，且环中涉及的每种资源都只有一个实例，则一定有死锁。

**规则三：有环 + 部分资源有多个实例 → 可能死锁**

如果图中有环，但某些资源有多个实例，则不能确定是否死锁，需要进一步分析。

**死锁示例（每种资源一个实例）：**

```
        P₁
       /  \
      ↓    ↑
     R₁    R₂
      ↑    ↓
       \  /
        P₂
```

P₁ 持有 R₁，请求 R₂；P₂ 持有 R₂，请求 R₁。这是**典型的死锁**。

**有环但非死锁的示例（多个实例）：**

```
        P₁
       /  \
      ↓    ↑
     R₁    R₂ (有 2 个实例)
      ↑    ↓
       \  /
        P₂
```

假设 R₂ 有 2 个实例。P₁ 持有 1 个 R₂，P₂ 持有 1 个 R₂。如果 P₁ 请求 R₁，P₂ 请求 R₂——P₂ 会等待 R₂（被 P₁ 持有 1 个，但还有 1 个空闲），不会死锁。

### 4.10.5 等待图（Wait-For Graph）

**定义：** 等待图是从资源分配图简化而来的，用于分析死锁。

**简化方法：** 删除所有资源节点，如果进程 Pᵢ 等待进程 Pⱼ 释放资源，则画一条从 Pᵢ 到 Pⱼ 的边。

**转换规则：**
- 如果资源分配图中有 Pᵢ → R → Pⱼ（Pᵢ 请求 R，R 分配给 Pⱼ）
- 则等待图中有 Pᵢ → Pⱼ（Pᵢ 等待 Pⱼ）

**等待图示例：**

```
资源分配图:          等待图:
P₁ → R₁ → P₂        P₁ → P₂
P₂ → R₂ → P₃        P₂ → P₃
P₃ → R₃ → P₁        P₃ → P₁
                    (有环 → 死锁)
```

**定理：** 在每种资源只有一个实例的系统中，**死锁存在当且仅当等待图中有环**。

**等待图检测死锁的时间复杂度：** O(n²)，其中 n 是进程数（使用深度优先搜索检测环）。

### 4.10.6 死锁的处理方法总览

| 处理方法 | 策略 | 资源利用率 | 实现难度 | 适用场景 |
|---------|------|-----------|---------|---------|
| **预防（Prevention）** | 破坏四个必要条件之一 | 低 | 简单 | 对性能要求不高的系统 |
| **避免（Avoidance）** | 动态判断是否进入不安全状态 | 中 | 中等 | 资源需求可预知的系统 |
| **检测（Detection）** | 允许死锁发生，检测并恢复 | 高 | 中等 | 死锁不频繁的系统 |
| **忽略（Ostrich）** | 假装死锁不会发生 | 最高 | 最小 | 通用操作系统 |

### 4.10.7 死锁预防（Deadlock Prevention）

预防的核心是**破坏死锁的四个必要条件之一**。

#### 方法一：破坏互斥条件（Mutual Exclusion）

**策略：** 让资源可以共享，使其不再是互斥资源。

**具体做法：SPOOLing（假脱机技术）**

```c
// 以打印机为例：
// 不使用互斥访问打印机，而是使用 SPOOLing 守护进程
// 所有进程将打印任务写入磁盘（共享），守护进程逐个打印

// 进程不直接访问打印机，而是写入 SPOOL 目录
write("/var/spool/printer/myjob", data);

// 打印机守护进程从 SPOOL 目录读取任务打印
```

**局限性：** 很多资源本质上就是互斥的（如写入同一个文件），无法破坏互斥条件。

#### 方法二：破坏请求并保持条件（Hold and Wait）

**策略：** 要求进程在开始执行前一次性请求所有需要的资源。

**两种实现方式：**

```c
// 方式一：进程启动时请求所有资源
void process() {
    // 一次性请求所有需要的资源
    P(&resource1);
    P(&resource2);
    P(&resource3);
    
    // 执行任务（不再请求新资源）
    do_work();
    
    // 释放所有资源
    V(&resource1);
    V(&resource2);
    V(&resource3);
}

// 方式二：进程在请求新资源前释放已有资源
void process() {
    P(&resource1);     // 拿 R1
    
    // 需要用 R2，但在拿 R2 前先释放 R1
    V(&resource1);     // 释放 R1
    P(&resource2);     // 拿 R2
    P(&resource1);     // 再拿 R1（如果还需要）
}
```

**优点：** 简单，容易实现。

**严重缺点：**
1. **资源利用率极低**——进程可能很长时间只用一小部分资源，但占用着所有资源
2. **可能导致饥饿**——需要大量资源的进程可能永远无法获得全部资源
3. **需要预知未来**——进程需要知道它将要使用的所有资源，这通常不可能
4. **效率低**——大量资源被长期占用

#### 方法三：破坏不可抢占条件（No Preemption）

**策略：** 当一个进程请求资源被拒绝时，操作系统可以**抢占**该进程已持有的资源。

**具体做法：**

```c
void process_request_resource(Resource r) {
    if (try_allocate(r) == FAIL) {
        // 请求失败，释放所有已占有的资源
        release_all_held_resources();
        // 然后重新请求所有资源（包括刚才失败的那个）
        request_all_resources_again();
    }
}
```

**优点：** 资源利用率比"一次性请求"方案高。

**严重缺点：**
1. **实现复杂**——需要保存和恢复资源状态
2. **可能丢失工作**——如果资源是中间状态的（比如文件写入了一半），抢占后恢复困难
3. **代价高昂**——抢占和恢复的代价可能非常大
4. **不适用于所有资源**——有些资源不能抢占（如打印机——打印了一半，不能把纸张扯出来给别人用）

#### 方法四：破坏循环等待条件（Circular Wait）

**策略：** 对资源类型进行排序，要求所有进程按照相同的顺序请求资源。

**具体做法：**

```c
// 资源排序（编号）
#define RES_PRINTER   1
#define RES_SCANNER   2
#define RES_TAPE      3
#define RES_DISK      4

// 所有进程必须按编号递增顺序请求资源
// 如果进程需要打印机和扫描仪，必须先请求打印机(1)再请求扫描仪(2)
// 不能先请求扫描仪再请求打印机

// ✅ 正确顺序
P(&printer);    // 1
P(&scanner);    // 2（必须大于 1 的资源编号）

// ❌ 错误顺序（可能导致循环等待）
P(&scanner);    // 2
P(&printer);    // 1（小于 2，不允许在请求 2 之后请求 1）
```

**为什么有效？**

假设所有进程都按编号递增顺序请求资源。如果有进程 A 持有资源 Rᵢ 请求 Rⱼ（i < j），则另一个持有 Rⱼ 的进程 B 不可能请求 Rᵢ（因为不能从大编号请求小编号资源）。因此不会形成循环等待链。

**优点：**
- 实现简单
- 资源利用率相对较高
- 不需要预知未来的资源需求

**缺点：**
- 资源编号难以确定（多个子系统需要协调）
- 可能导致低效使用（一个进程可能必须提前请求实际不需要的资源）
- 限制编程灵活性

#### 四种预防方法的对比

| 方法 | 破坏的条件 | 资源利用率 | 实现难度 | 实用性 |
|------|-----------|-----------|---------|--------|
| SPOOLing/共享 | 互斥 | 高 | 中等 | 仅适用于部分资源 |
| 一次性请求 | 请求并保持 | 很低 | 简单 | 不实用 |
| 抢占 | 不可抢占 | 中等 | 困难 | 适用于部分资源 |
| 有序请求 | 循环等待 | 较高 | 简单 | 实际使用较多 |

### 4.10.8 死锁避免（Deadlock Avoidance）

**核心思想：** 与预防不同，避免不破坏必要条件，而是**每次资源分配时判断是否会导致死锁**，如果会则拒绝分配。

**关键概念：安全状态（Safe State）**

#### 安全状态（Safe State）

**定义：** 系统处于安全状态，是指存在一个**安全序列（Safe Sequence）**，按照这个序列分配资源，所有进程都能完成。

**安全序列的定义：** 存在一个进程序列 <P₁, P₂, ..., Pₙ>，对于每个 Pᵢ，它需要的额外资源可以由当前可用资源加上 P₁ 到 Pᵢ₋₁ 释放的资源满足。

**不安全状态：** 不存在这样的安全序列。在不安全状态下，系统**可能**发生死锁（但不一定发生）。

```
安全状态 → 一定不死锁
不安全状态 → 可能死锁（也可能不死锁）
```

#### 资源分配图算法（单一实例资源类型）

当资源类型只有一个实例时，可以使用资源分配图的一个变体——**带有声明边的资源分配图**。

**声明边（Claim Edge, 用虚线表示）：** Pᵢ ---→ Rⱼ 表示进程 Pᵢ 在未来**可能**请求资源 Rⱼ。

**算法：**
1. 进程在开始前声明所有可能需要的资源（声明边）
2. 当进程请求资源时，将声明边转换为请求边
3. 如果转换后图中出现环，则拒绝分配（因为会导致死锁）
4. 如果转换后图中没有环，则分配

```
初始状态（虚线 = 声明边）：
P₁ - - → R₁     P₁ - - → R₂
P₂ - - → R₁     P₂ - - → R₂

P₁ 请求 R₁：检查发现无环，分配
P₂ 请求 R₂：检查发现无环，分配
P₂ 请求 R₁：检查发现 P₁ → R₁ → P₂ → R₂ → P₁ 形成环，拒绝！
```

#### 银行家算法（Banker's Algorithm, 多实例资源类型）

**为什么叫"银行家算法"？**
> 类比银行家管理贷款：银行家有一定资金（资源），多个客户（进程）需要贷款（请求资源）。银行家必须谨慎地决定是否批准贷款，以避免陷入无法满足任何客户需求的困境。

**银行家算法的数据结构：**

```c
int n;                     // 进程数量
int m;                     // 资源类型数量

int Available[m];          // 每种资源的可用实例数量
                           // Available[j] = 资源类型 Rj 的可用实例数

int Max[n][m];             // 每个进程对每种资源的最大需求
                           // Max[i][j] = 进程 Pi 对 Rj 的最大需求

int Allocation[n][m];      // 当前已分配给每个进程的资源数
                           // Allocation[i][j] = 进程 Pi 当前持有的 Rj 实例数

int Need[n][m];            // 每个进程还需要的资源数
                           // Need[i][j] = Max[i][j] - Allocation[i][j]
```

**关系式：** `Need[i][j] = Max[i][j] - Allocation[i][j]`

#### 安全性算法（Safety Algorithm）

**步骤：**

```
1. 初始化：
   Work[m] = Available[m]   // Work 表示可用资源的工作副本
   Finish[n] = {false}      // Finish[i] 表示进程 Pi 能否完成

2. 找一个 i 满足：
   Finish[i] == false AND Need[i] <= Work
   （逐元素比较：对所有 j, Need[i][j] <= Work[j]）
   
   如果找到，转步骤 3
   如果没找到，转步骤 4

3. 假设进程 Pi 完成，释放其占用的资源：
   Work[j] = Work[j] + Allocation[i][j]  对所有 j
   Finish[i] = true
   转步骤 2

4. 如果对所有 i, Finish[i] == true，则系统处于安全状态
   否则，系统处于不安全状态
```

#### 资源请求算法（Resource-Request Algorithm）

当进程 Pi 请求资源 Request[i] 时：

```
1. 检查请求是否超过声明：
   如果 Request[i] > Need[i]，错误（进程超过了最大声明）

2. 检查资源是否可用：
   如果 Request[i] > Available，进程必须等待

3. 尝试分配（预分配）：
   Available = Available - Request[i]
   Allocation[i] = Allocation[i] + Request[i]
   Need[i] = Need[i] - Request[i]

4. 运行安全性算法：
   如果新状态是安全的 → 正式分配
   如果新状态是不安全的 → 回滚（恢复原状），进程等待
```

#### 银行家算法完整示例

**初始状态：**

```
系统有 5 个进程 P0~P4，3 种资源类型 A(10), B(5), C(7)

       Allocation        Max         Need        Available
       A  B  C        A  B  C      A  B  C       A  B  C
P0     0  1  0        7  5  3      7  4  3       3  3  2
P1     2  0  0        3  2  2      1  2  2
P2     3  0  2        9  0  2      6  0  0
P3     2  1  1        2  2  2      0  1  1
P4     0  0  2        4  3  3      4  3  1
```

**安全性检查过程：**

```
Work = [3, 3, 2]

检查 P0: Need[0]=[7,4,3] > Work=[3,3,2] → 不满足
检查 P1: Need[1]=[1,2,2] ≤ Work=[3,3,2] → 满足
  Work = [3,3,2] + [2,0,0] = [5,3,2]
  Finish[1] = true

检查 P0: Need[0]=[7,4,3] > Work=[5,3,2] → 不满足
检查 P2: Need[2]=[6,0,0] > Work=[5,3,2] → 不满足
检查 P3: Need[3]=[0,1,1] ≤ Work=[5,3,2] → 满足
  Work = [5,3,2] + [2,1,1] = [7,4,3]
  Finish[3] = true

检查 P0: Need[0]=[7,4,3] ≤ Work=[7,4,3] → 满足
  Work = [7,4,3] + [0,1,0] = [7,5,3]
  Finish[0] = true

检查 P2: Need[2]=[6,0,0] ≤ Work=[7,5,3] → 满足
  Work = [7,5,3] + [3,0,2] = [10,5,5]
  Finish[2] = true

检查 P4: Need[4]=[4,3,1] ≤ Work=[10,5,5] → 满足
  Work = [10,5,5] + [0,0,2] = [10,5,7]
  Finish[4] = true

安全序列：<P1, P3, P0, P2, P4>
```

**收到请求：P1 请求 1 个 A 资源和 0 个 B 资源和 2 个 C 资源：**

```
Request[1] = [1, 0, 2]

1. Request[1] ≤ Need[1] = [1,2,2] ✓
2. Request[1] ≤ Available = [3,3,2] ✓
3. 预分配：
   Available = [3,3,2] - [1,0,2] = [2,3,0]
   Allocation[1] = [2,0,0] + [1,0,2] = [3,0,2]
   Need[1] = [1,2,2] - [1,0,2] = [0,2,0]

新状态：
       Allocation        Need        Available
       A  B  C        A  B  C       A  B  C
P0     0  1  0        7  4  3       2  3  0
P1     3  0  2        0  2  0
P2     3  0  2        6  0  0
P3     2  1  1        0  1  1
P4     0  0  2        4  3  1

运行安全检查：
Work = [2,3,0]

检查 P0: Need[0]=[7,4,3] > Work → 不满足
检查 P1: Need[1]=[0,2,0] ≤ Work → 满足
  Work = [2,3,0] + [3,0,2] = [5,3,2]
  Finish[1] = true

检查 P0: Need[0]=[7,4,3] > Work → 不满足
检查 P2: Need[2]=[6,0,0] > Work → 不满足
检查 P3: Need[3]=[0,1,1] ≤ Work → 满足
  Work = [5,3,2] + [2,1,1] = [7,4,3]
  Finish[3] = true

检查 P0: Need[0]=[7,4,3] ≤ Work → 满足
  Work = [7,4,3] + [0,1,0] = [7,5,3]
  Finish[0] = true

检查 P2: Need[2]=[6,0,0] ≤ Work → 满足
  Work = [7,5,3] + [3,0,2] = [10,5,5]
  Finish[2] = true

检查 P4: Need[4]=[4,3,1] ≤ Work → 满足
  Work = [10,5,5] + [0,0,2] = [10,5,7]
  Finish[4] = true

新状态是安全的！允许分配。
```

### 4.10.9 死锁检测（Deadlock Detection）

#### 单一实例资源类型

**方法：** 使用等待图（Wait-For Graph），检测图中是否有环。

**等待图维护：**
- 当进程 Pi 请求被 Pj 持有的资源时，在等待图中添加 Pi → Pj
- 当资源释放时，删除相应的等待边

**检测算法：** 定期使用深度优先搜索（DFS）检测环，时间复杂度 O(n²)。

**检测时机：**
- 每经过一个固定时间间隔（如每 5 分钟）
- 当 CPU 利用率下降到某个阈值时（死锁导致所有进程阻塞，CPU 空闲）

#### 多实例资源类型

**检测算法（类似于银行家算法的安全检查）：**

```c
// 数据结构和银行家算法类似，但没有 Max
// 因为检测算法基于"当前已经分配"的情况

int Available[m];          // 当前可用资源
int Allocation[n][m];      // 当前已分配
int Request[n][m];         // 请求矩阵：Request[i][j] = Pi 当前请求的 Rj 数量
```

**检测步骤：**

```
1. 初始化：
   Work[m] = Available[m]
   对所有 Allocation[i] != 0 的进程，Finish[i] = false
   对一个 Allocation[i] == 0 的进程，Finish[i] = true（没有资源不需要等待）

2. 找一个 i 满足：
   Finish[i] == false AND Request[i] <= Work
   如果找到，转步骤 3
   如果没找到，转步骤 4

3. 假设进程 Pi 可以完成：
   Work[j] = Work[j] + Allocation[i][j]  对所有 j
   Finish[i] = true
   转步骤 2

4. 如果对所有 i, Finish[i] == true → 无死锁
   否则 → 死锁（所有 Finish[i] == false 的进程构成死锁进程集）
```

**检测算法与银行家算法安全检测的区别：**
- 银行家用 Need（最大需求），检测用 Request（当前请求）
- 银行家是"避免"（事前），检测是"发现"（事后）
- 银行家要求死锁不发生，检测允许死锁发生

### 4.10.10 死锁恢复（Deadlock Recovery）

一旦检测到死锁，必须采取措施恢复。

#### 方法一：进程终止

**方案 A：终止所有死锁进程**
- 优点：简单彻底
- 缺点：可能丢失大量计算成果
- 代价：进程已执行的全部计算被浪费

**方案 B：逐个终止死锁进程**
每次终止一个进程，然后重新运行检测算法，直到死锁消除。

**进程终止的选择标准（优先级）：**
1. **进程优先级**——终止优先级最低的进程
2. **已计算时间**——终止已计算时间最短的进程（损失最小）
3. **已使用资源**——终止已使用资源最少的进程
4. **还需资源**——终止还需资源最多的进程
5. **进程类型**——优先终止批处理进程（交互式进程掉失更严重）
6. **进程重要性**——优先终止非关键进程

#### 方法二：资源抢占

**从死锁进程中抢占资源，分配给其他进程。**

**三个问题及其解决方案：**

| 问题 | 解决方案 |
|------|---------|
| **选择受害者（Victim Selection）** | 抢占代价最小的进程的资源。代价因素包括：已占用资源数、已执行时间、进程优先级 |
| **回滚（Rollback）** | 将进程回滚到安全状态（检查点/Checkpoint）。实现检查点：定期保存进程状态。回滚到最接近的检查点 |
| **饥饿（Starvation）** | 如果总是选择同一个进程作为受害者，该进程永远无法完成。解决：确保进程被选为受害者的次数有限（如使用回滚次数作为代价因子） |

### 4.10.11 鸵鸟算法（Ostrich Algorithm）

**定义：** 假装死锁不会发生，如果发生则重启系统。

**为什么许多操作系统采用鸵鸟算法？**

1. **死锁很少发生**——在典型的桌面系统中，死锁的概率非常低
2. **死锁预防和避免的代价过高**——资源利用率下降、系统性能降低
3. **检测和恢复的代价也高**——需要额外的计算和复杂机制
4. **许多资源本身已经用其他方式解决了**——如 SPOOLing 打印机、网络协议超时重传

**实际案例：**
- 早期 Windows 和 UNIX 对死锁基本采用鸵鸟算法
- 现代操作系统在关键资源上使用死锁预防，对非关键资源采用鸵鸟算法
- 许多数据库系统使用死锁检测（因为事务中的资源争用更频繁）

**鸵鸟算法的适用场景：**

| 场景 | 是否适合鸵鸟算法 |
|------|----------------|
| 桌面操作系统 | ✅ 适合（死锁极少发生） |
| 嵌入式系统 | ❌ 不适合（死锁可能导致设备失效） |
| 数据库系统 | ❌ 不适合（事务死锁常见） |
| 航空航天系统 | ❌ 绝对不适合（安全关键） |

## 4.11 本章易错点总结

总结本章学生在学习和考试中常见的易错点，共计 12 条：

### 易错点 1：临界区不是实现互斥的代码

**最常见概念混淆。** 临界区是**访问临界资源的那段代码**，不是实现互斥的代码。信号量的 P/V 操作、锁等是实现互斥的**同步机制**，不是临界区本身。

- ❌ 误解：临界区是实现互斥的代码
- ✅ 正解：临界区是访问临界资源（共享变量等）的代码

### 易错点 2：P 操作顺序在生产者-消费者问题中的重要性

很多学生记住 P(empty) 和 P(mutex) 的顺序，但不知道为什么。本质原因：
- 先 P(empty) 再 P(mutex)：如果缓冲区满，生产者阻塞时没有持有互斥锁，消费者可以继续消费
- 先 P(mutex) 再 P(empty)：如果缓冲区满，生产者持有互斥锁阻塞，消费者无法进入缓冲区——死锁

### 易错点 3：V 操作不影响互斥正确性

V 操作的顺序无所谓，不会导致死锁或同步错误。但 P 操作的顺序必须严格遵守规范。

```c
// 以下两种写法都正确：
V(&mutex); V(&full);   // 顺序 1
V(&full); V(&mutex);   // 顺序 2（同样正确）
```

### 易错点 4：读者优先方案中写者可能饿死

读者优先方案中，如果读者持续不断到来，readcount 永远不会降到 0，写者永远无法获得 wmutex。这是读者优先方案**固有的设计缺陷**，不是实现错误。

### 易错点 5：Mesa 语义下必须用 while 而非 if 检查条件

在管程中，使用 Mesa 语义时，wait() 返回后条件可能已经改变（因为 signal 后不能保证条件仍然成立），必须用 while 重新检查：

```c
// ✅ 正确：Mesa 语义用 while
while (buffer.is_empty()) {
    cond.wait();
}

// ❌ 错误：if 只检查一次
if (buffer.is_empty()) {
    cond.wait();
}
```

### 易错点 6：条件变量不同于信号量

条件变量**没有值**，只有队列。signal 操作如果队列为空则信号丢失（不累积）。而信号量的 V 操作会增加计数器的值。

### 易错点 7：死锁的四个条件是"必要不充分"

四个条件**同时成立**不一定导致死锁（如多实例资源的资源分配图有环时不一定死锁），但**缺少任意一个**一定不会死锁。

### 易错点 8：不安全状态 ≠ 死锁

系统处于不安全状态，**可能**发生死锁，但不是**一定**发生死锁。不安全状态是"可能出问题"，安全状态是"一定不出问题"。

### 易错点 9：银行家算法的 Need 矩阵计算

```c
// Need = Max - Allocation（容易混淆哪个减哪个）
// 常见错误是写成 Allocation - Max
Need[i][j] = Max[i][j] - Allocation[i][j];
```

### 易错点 10：安全序列不唯一

一个安全状态可能有**多个**安全序列。安全检查只需要找到一个安全序列即可，不一定是唯一的那个。

### 易错点 11：预防和避免的区别

- **死锁预防（Prevention）**：破坏四个必要条件之一，**静态**策略，资源请求前决定
- **死锁避免（Avoidance）**：动态判断分配是否安全，**动态**策略，每次请求时决定
- 两者经常混淆。简单说：预防是"定规则"，避免是"动态判断"

### 易错点 12：进程通信中的"同步"与进程同步中的"同步"不同

- **进程同步（Process Synchronization）**：保证多个进程有序访问共享资源，防止竞态条件
- **进程通信的同步模式（Synchronous Communication）**：发送方和接收方在消息传递时是否需要等待对方

两者是不同的概念，前者是并发控制，后者是通信模式。

## 4.12 本章常考题型与解题思路

### 题型一：信号量实现互斥与同步

**典型题目：** 给出多个并发执行的进程（或语句），要求用信号量实现指定的执行顺序。

**解题步骤：**
1. 确定前驱关系（哪个操作必须在哪个操作之前）
2. 每个前驱关系对应一个信号量，初始化为 0
3. 前驱操作后 V，后继操作前 P
4. 检查是否有互斥需求（多个进程访问同一资源）

**示例：**
```
有 S1、S2、S3、S4 四个操作，要求：
S1 → S2（S1 在 S2 之前）
S1 → S3
S2 → S4
S3 → S4
```

**解答：** 使用 4 个信号量 a=0, b=0, c=0, d=0
- S1 后：V(a), V(b)
- S2 前：P(a)；S2 后：V(c)
- S3 前：P(b)；S3 后：V(d)
- S4 前：P(c), P(d)

### 题型二：生产者-消费者问题变体

**典型题目：** 有一个缓冲区，多个生产者和多个消费者，要求用信号量实现同步。

**常见变体：**
- 有多个缓冲区（而不是一个大的环形缓冲区）
- 生产者和消费者操作不同的缓冲区
- 缓冲区中存的是另一种数据结构的指针

**解题思路：** 始终牢记三个信号量：
- mutex：保护缓冲区互斥访问
- empty：记录空槽位数量
- full：记录满槽位数量
- 根据题目要求调整信号量的初始值

### 题型三：读者-写者问题变体

**典型题目：** 实现读者优先或写者优先的解决方案。

**解题思路：**
- **读者优先**：第一个读者获取写锁，最后一个读者释放写锁
- **写者优先**：第一个写者获取"读者准入"信号量，阻止新读者进入
- 需要仔细跟踪 readcount 和 writecount 的更新

### 题型四：哲学家就餐问题

**典型题目：** 用信号量或管程实现哲学家就餐，要求不死锁、不饥饿。

**解题方法：**
- 方案一：限制就餐人数（最多 4 人）
- 方案二：不对称拿筷子（奇偶不同顺序）
- 方案三：管程集中分配

### 题型五：银行家算法

**典型题目：** 给出 Allocation、Max、Available，问是否安全，或者能否分配指定资源。

**解题步骤：**
1. 计算 Need = Max - Allocation
2. 运行安全性算法，尝试找安全序列
3. 如果题目有资源请求，先预分配再检查安全性

**注意事项：**
- 计算 Need 时注意是 Max - Allocation，不要搞反
- 安全检查中 Work 是累积的（加 Allocation）
- 多个安全序列存在时，只需要找到一个即可

### 题型六：资源分配图与死锁判断

**典型题目：** 给出资源分配图，判断是否有死锁。

**解题步骤：**
1. 标注资源实例数量
2. 找出所有环
3. 对于每种资源只有一个实例的类型，有环即死锁
4. 对于多实例资源，需要看能否化简（找到不等待的进程释放资源）

### 题型七：死锁必要条件的判断

**典型题目：** 判断一个给定的场景是否会发生死锁。

**解题方法：** 检查四个必要条件是否同时成立：
1. 互斥？资源是否能被多个进程共享
2. 请求并保持？进程是否持有资源同时还请求别的
3. 不可抢占？资源能否被强制剥夺
4. 循环等待？是否存在循环链

### 题型八：信号量实现经典同步问题代码填空题

**典型题目：** 在给出的代码中填入正确的 P/V 操作。

**解题方法：**
1. 先确定互斥需求（哪些操作不能同时执行）
2. 再确定同步需求（哪些操作必须在某些操作之后执行）
3. 互斥用 mutex（初始 1），同步用 sync（初始 0）
4. 注意 P 操作的顺序（先同步后互斥）
